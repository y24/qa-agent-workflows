# シナリオ観点抽出ポリシー

## 観点カテゴリ

| カテゴリ | 内容 |
|---|---|
| 中核業務 | 主要な業務目的を達成する流れ |
| 分岐・例外 | 代替経路、異常系、差戻し、取消、再実行 |
| データ状態 | ステータス、締め状態、承認状態、履歴、マスタ状態 |
| ロール・権限 | 利用者、承認者、管理者、参照・更新権限 |
| 出力・連携 | 帳票、ファイル、API、外部IF、バッチ、後続処理 |
| リスク・過去不具合 | 既知リスク、過去障害、変更影響 |
| 運用・タイミング | 月末、年度末、締め日、ピーク、運用手順 |

## 観点の書き方

観点は「何を確認したいか」と「なぜ重要か」が分かる粒度にする。

粗すぎる:

```text
承認を確認する
```

適切:

```text
申請、承認、差戻し、再申請の状態ごとに、操作可否と後続処理への反映を確認する
```

細かすぎる:

```text
承認ボタンを押したときに確認ダイアログが表示される
```

## 重要度

| 重要度 | 目安 |
|---|---|
| High | 中核業務、重大な業務影響、金額・会計・監査、P0/P1相当リスク、過去障害 |
| Medium | 主要分岐、代表的な例外、重要データ状態、利用頻度が高い |
| Low | 補助的な確認、限定的な条件、他観点で同時確認できる |

## 未カバー・保留

観点化できない不明点、対象外、保留は削除しない。理由と後続Stepでの扱いを記録する。

## 禁止事項

- テスト手順や期待結果を書き始める
- 画面操作単位まで細分化する
- 根拠のないリスクを観点として断定する
- 関連Flow IDや根拠を落とす
