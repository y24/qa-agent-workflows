# Automation Recommendation Notes

## 目的

品質クライテリアや確認方法に対して、自動化推奨度を判断する基準を定義する。

## 自動化区分

| 区分 | 意味 |
|---|---|
| 自動化推奨 | 継続的または反復的に自動確認する価値が高い |
| 一部自動化 | データ準備、実行、ログ収集、比較など一部だけ自動化するとよい |
| 手動推奨 | 人間の判断、探索、業務レビューが中心 |
| 要確認 | 判断材料が不足している |

## 自動化推奨にしやすい条件

- 回帰確認が必要で、今後も繰り返し実行する。
- 判定が機械的で、期待値や閾値が明確である。
- 複数環境、複数データ、複数権限で同じ確認を行う。
- 実行ログ、メトリクス、差分、スクリーンショットなどの証跡を残しやすい。
- 手動実行では漏れやばらつきが出やすい。

## 一部自動化にしやすい条件

- 人間の最終判断は必要だが、準備や測定は自動化できる。
- 大量データ作成、環境疎通、ログ収集、レポート生成だけ自動化すると効果がある。
- 初回は手動探索し、安定した観点だけ後から自動化する。

## 手動推奨にしやすい条件

- 初回探索や仮説検証が中心である。
- 利用者の理解しやすさ、迷いにくさ、業務妥当性など、人間の認知評価が中心である。
- 業務有識者のレビューや合意が必要である。
- 環境や条件が頻繁に変わり、自動化の保守コストが高い。

## 注意事項

自動化推奨は「必ず自動化する」という意味ではない。テスト計画では、実装コスト、保守コスト、実行頻度、証跡要求、リリース頻度を合わせて判断する。
