# Step 5: 整合性レビュー・最終カタログ化ガイド

## 目的

品質クライテリア全体を見直し、テスト計画・テスト設計に渡せる最終カタログとして整理する。

## 入力

- `04_confirmation_policy.md`
- 必要に応じて Step 1からStep 3の成果物
- 根拠確認に必要な入力文書の該当箇所

## 出力

- `05_quality_criteria_catalog.md`
- 形式: `templates/quality_criteria_catalog_template.md`

## 手順

1. 品質特性間の重複を確認し、主特性と副特性を整理する。
2. 根拠のないクライテリアを除外、保留、または確認事項に移す。
3. 粒度が細かすぎるものは統合し、粗すぎるものは分割する。
4. 優先度、確認方法、自動化推奨、証跡、テスト計画反映方針の妥当性を見直す。
5. `Criterion ID -> Concern ID -> Confirmation ID -> 根拠` の追跡性を確認する。
6. 未確認事項、対象外、保留、残課題を最終成果物に明示する。
7. 最終カタログを作り、次工程への引き継ぎをまとめる。

## 判断基準

- 品質特性の分類: `references/quality_characteristic_scope_notes.md`
- 確認方法の妥当性: `references/confirmation_method_catalog.md`
- 共通のトレーサビリティ方針: `../../shared/traceability_policy.md`

## このStepでやらないこと

- 根拠不足のクライテリアを黙って最終採用すること
- 詳細テストケースへ展開すること
- 未確認事項を削除して完了扱いにすること

## 完了条件

- 品質クライテリア一覧が、テスト計画やテスト設計に渡せる表になっている。
- 各行に品質特性、対象、品質クライテリア、確認方法、自動化、優先度、根拠、未確認事項がある。
- 除外、保留、統合、分割の理由が説明されている。
- 次工程への引き継ぎが明確である。

## 停止

`05_quality_criteria_catalog.md` を出力したら停止する。必要に応じて、ユーザーに次工程への展開可否を確認する。
