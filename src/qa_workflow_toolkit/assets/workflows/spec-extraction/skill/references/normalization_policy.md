# 正規化判断ポリシー

## 同一仕様として統合してよい条件

- 同じ対象、同じ条件、同じ振る舞いを指している
- 表記だけが異なる
- 一方が詳細、一方が概要で矛盾しない
- 同じ文書内または優先度が明確な文書で補完関係にある

## 別仕様として分ける条件

- 対象機能、画面、API、データが異なる
- 発生条件や状態が異なる
- 期待される振る舞いが異なる
- 業務ルールと入力チェックなど、カテゴリが実質的に異なる
- 同じ言葉でも文脈上の意味が違う

## 表記揺れ・用語統一

表記揺れは、根拠がある場合のみ統一する。根拠が弱い場合は候補として記録し、確認事項にする。

## 矛盾の扱い

矛盾は解消しない。以下を記録する。

- 矛盾している Raw ID / Doc ID
- 矛盾内容
- 影響するカテゴリ、機能、データ、状態
- 後続Stepで断定してはいけない事項

## 粒度不足

テスト設計に必要な条件、範囲、値、状態、期待結果が不足している場合は `missing_detail` とする。

## 未確定・要確認

仕様未確定、TODO、要検討、別途確認などの記述は `tentative` または `open_issue` として扱う。

## 禁止事項

- 根拠なしに同義語として統合する
- 矛盾を多数決や常識で解消する
- 詳細条件を一般論で補う
- Raw IDへの追跡を切る
