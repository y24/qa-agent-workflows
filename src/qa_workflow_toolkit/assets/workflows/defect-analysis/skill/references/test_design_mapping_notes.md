# 不具合傾向からテスト設計への変換メモ

## 使い方

Step 5で、品質リスクや弱点パターンをテスト設計ガイダンスへ変換する際に参照する。
この表は出発点であり、入力チケットの根拠に合わせて調整する。

## 変換対応表

| 不具合傾向 | 後続テスト設計での反映 |
|---|---|
| 境界値・特殊値漏れが多い | 境界値分析、特殊値、空、最大、最小、異常値の観点を重点化する |
| 複数条件の組み合わせ漏れが多い | デシジョンテーブル、組み合わせテスト、状態と条件の組み合わせを検討する |
| 業務シナリオでのみ発生する | シナリオテスト、E2Eテスト、例外運用シナリオを強化する |
| 回帰不具合が多い | 変更影響範囲を回帰テストセットへ固定化する |
| 環境依存が多い | OS、ブラウザ、設定、外部IF、負荷条件など環境差分テストを明示する |
| データ移行・マスタ関連が多い | マスタ組み合わせ、履歴、世代、移行前後のテストデータを強化する |
| 権限不備が多い | ロール別、所属別、状態別の参照・操作権限マトリクスを作る |
| エラーハンドリング不備が多い | 異常系、リトライ、外部IF失敗、タイムアウト、復旧確認を追加する |
| 発見が遅い重大不具合がある | 早期工程のレビュー観点、結合前の観点、リスクベースの優先確認を追加する |
| 既存テストで検出可能だった不具合が多い | 既存テストの実施漏れ、期待結果の曖昧さ、レビュー観点の不足を確認する |

## 自動化候補の判断

- 同じ条件を繰り返し確認する回帰観点は自動化候補にする。
- データ組み合わせが多く、期待結果が機械判定しやすい観点は自動化候補にする。
- 画面の見た目、業務判断、探索的確認が中心の観点は手動確認候補にする。
- 外部環境やデータ準備が不安定な観点は、自動化前に前提条件を整理する。

## レビュー観点への変換

- 仕様理解漏れは、仕様レビューの確認観点へ変換する。
- 条件分岐漏れは、分岐表、状態遷移表、例外条件レビューへ変換する。
- データ不整合は、CRUD、トランザクション、整合性制約、移行レビューへ変換する。
- 権限制御不備は、権限マトリクスと否認系レビューへ変換する。
