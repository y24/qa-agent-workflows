# Test Design Granularity Policy

## Purpose

テスト観点、シナリオ、テストケース骨子、詳細テストケースの粒度が混ざらないようにするための共通方針を定義する。

## General Rules

- 1つの項目には主要な確認目的を明示する。
- 事前条件、入力データ、操作、確認対象、期待結果を混同しない。
- 期待結果を `正常に動作する` のような曖昧表現で終えない。
- 具体値、画面文言、手順は、入力に根拠がある場合だけ確定的に書く。
- 高リスクまたは中核業務は厚く、低リスクや周辺確認は過剰に詳細化しない。

## Level Guidance

| Level | Description |
|---|---|
| Test viewpoint | 何を確認すべきかを表す抽象的な切り口 |
| Scenario candidate | 業務上意味のある流れ、条件、期待する確認の組み合わせ |
| Testcase outline | 詳細手順に入る前の実行可能な骨子 |
| Detailed testcase | 事前条件、手順、入力、期待結果を実行できる粒度で示したもの |

## Boundary Rules

- 観点抽出の段階で詳細テストケースを作らない。
- リスク評価の段階でテスト手順や具体値に踏み込みすぎない。
- 詳細化できない箇所は、推測で埋めず `要確認` または `情報不足` とする。
