# テストケース詳細化ポリシー

## テストケースの粒度

テストケースは実行可能な概要粒度にする。ただし、詳細テスト手順書ではない。

適切:

```text
承認済み契約を締め対象に含めた場合、締め後の帳票と契約明細の金額が一致することを確認する
```

細かすぎる:

```text
契約ID C001 を検索し、詳細ボタンを押し、金額欄に 1000 と入力し...
```

## 必須項目

- TC ID
- テストケース名
- 対応Scenario ID
- 対応Viewpoint ID
- 対応Flow ID
- 目的
- 前提条件
- テストデータ条件
- 手順概要
- 期待結果概要
- 優先度と理由
- 根拠

## 分割する条件

- 前提条件が大きく異なる
- 期待結果が異なる
- 正常系と異常系を分けたほうがよい
- ロール、状態、データ条件が複雑
- 失敗時の原因切り分けが難しい

## 統合してよい条件

- 同じ業務フローで確認できる
- 同じ前提データで確認できる
- 低優先度観点を高優先度ケースで同時確認できる
- 回帰確認としてまとめられる

## 期待結果

期待結果は、仕様に根拠がある範囲で記載する。文言、画面項目、具体値が未確認の場合は、確認事項として残す。

## 優先度

シナリオ候補の優先度、業務影響、リスク、利用頻度、テスト制約を踏まえて High / Medium / Low を付ける。

## 未テストの扱い

未テスト、対象外、保留、代替確認は、理由、影響、残存リスク、後続対応を記録する。
