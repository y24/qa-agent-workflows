# シナリオ候補設計ポリシー

## シナリオ候補の種類

| 種類 | 内容 |
|---|---|
| 中核正常系 | 主要業務が標準的に完了する流れ |
| 分岐 | 条件や業務判断により流れが変わるケース |
| 例外 | エラー、差戻し、取消、再実行、リカバリ |
| データ状態 | 承認状態、締め状態、履歴、マスタ差異 |
| ロール・権限 | 利用者、承認者、管理者などの操作差 |
| 出力・連携 | 帳票、ファイル、API、バッチ、後続処理 |
| 過去不具合・リスク再現 | 既知リスクや過去障害を再発確認する流れ |

## 粒度

シナリオは、業務目的と確認ポイントが分かる単位にする。

粗すぎる:

```text
契約業務を確認する
```

適切:

```text
新規契約を登録し、承認後に帳票へ反映されることを確認する
```

細かすぎる:

```text
契約名欄に50文字を入力して保存ボタンを押す
```

## 優先度

| 優先度 | 目安 |
|---|---|
| P0 | 中核業務、重大影響、法令・会計・監査、過去重大障害、必須確認 |
| P1 | 主要分岐、代表例外、重要データ状態、主要ロール差 |
| P2 | 標準的な補完確認、代表条件、他シナリオで一部同時確認可能 |
| P3 | 簡易確認、探索的確認、対象外候補、詳細化しない候補 |

## 採用判断

| 判断 | 扱い |
|---|---|
| 採用 | Step 5でテストケース化する |
| 統合 | 他シナリオに含めて確認する |
| 保留 | 仕様未確定、根拠不足、条件不明として残す |
| 対象外 | 理由と残存懸念を明示する |

## よくある失敗

- 観点名をそのままシナリオ名にする
- 1シナリオに複数目的を詰め込みすぎる
- 細かい画面操作をシナリオ候補にする
- 根拠のないシナリオを確定事項にする
- 未カバー観点を消す
