# テスト方針設計ポリシー

## テスト深度

| 優先度 | 基本方針 |
|---|---|
| P0 | 正常系、異常系、境界、状態遷移、組合せ、回帰影響、帳票・集計・連携、復旧を厚く確認する |
| P1 | 主要パターン、代表的な異常系、重要な境界条件、主要な権限差、変更影響を確認する |
| P2 | 代表条件、基本的な正常系・異常系、他リスクのテストで同時確認できる観点を扱う |
| P3 | 簡易確認、同時確認、探索的確認、レビュー確認、対象外、残存リスクから扱いを選ぶ |

## リスクから観点への変換

- リスクは「起きると困ること」。
- テスト観点は「そのリスクを検出・軽減するために確認すべきこと」。
- 確認すべき状態、入力条件、権限、業務タイミング、データパターン、境界値、異常系、回帰影響、出力・帳票・連携結果を整理する。

例:

| 種別 | 内容 |
|---|---|
| リスク | 締め後にデータ修正できることで、確定済み帳票と明細が不整合になる |
| テスト観点 | 締め状態ごとの編集可否、取消可否、再集計可否を確認する |
| 推奨技法 | 状態遷移テスト、デシジョンテーブル、業務シナリオテスト |

## テストデータ粒度

Step 4では、具体値ではなくデータの種類と条件を整理する。

適切:

```text
締め前データ、締め後データ、取消済みデータ、再締め済みデータ
```

細かすぎる:

```text
部門コード D001、勘定科目 売上高、金額 999,999,999、対象年月 2026/03
```

## 対象外・簡易確認

対象外、簡易確認、レビュー確認、運用確認にする場合は、理由、残存リスク、代替対応を記載する。

理由が「優先度が低い」だけでは不足。影響が限定的か、他テストで同時確認できるか、環境制約があるか、レビューのほうが適切かを明示する。

## よくある失敗

- リスク文をそのまま「確認する」と書き、確認条件に変換できていない。
- P0からP3まで同じ深さのテストを割り当てている。
- 技法名を羅列して、なぜ有効かを書いていない。
- Step 4で大量の具体値データを作り始める。
- 対象外理由や残存リスクが曖昧。
