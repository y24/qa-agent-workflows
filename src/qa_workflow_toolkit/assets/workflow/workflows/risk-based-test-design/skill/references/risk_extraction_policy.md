# リスク候補抽出ポリシー

## リスクの書き方

リスクは、単なる機能名や確認項目ではなく「起きると困ること」として書く。

悪い例:

```text
締め処理のリスク
```

良い例:

```text
締め処理後のデータ修正が可能な場合、確定済み帳票と明細データが不整合になるリスク
```

## 必須要素

各リスク候補には以下を含める。

- Risk ID
- 分類
- 発生条件。不明な場合は `発生条件不明`
- 影響対象
- 想定される悪影響
- 根拠
- 確度

根拠には、文書名、章・節・項目名、Step 1の該当セクション、過去障害、仕様変更点、制約、不足・矛盾・曖昧な点を含める。

## 仮説リスク

不足、矛盾、曖昧な点から抽出したリスクは、通常リスクと区別する。

仮説リスクには以下を記載する。

- 仮説であること
- 根拠となる不足、矛盾、曖昧さ
- ユーザー確認が必要な内容
- 後続Stepでの扱い

## 確度

| 確度 | 定義 |
|---|---|
| 高 | 入力文書に明確な根拠があり、リスク候補として妥当性が高い |
| 中 | 入力文書に一部根拠があるが、条件や影響範囲に不明点がある |
| 低 | 根拠が弱く、仮説に近い。確認事項として扱う必要がある |

確度は優先度ではない。優先度は Step 3 で評価する。

## 粒度

適切:

```text
承認前データが帳票出力対象に含まれることで、未確定情報が利用者に提示されるリスク
```

粗すぎる:

```text
帳票出力のリスク
```

細かすぎる:

```text
画面Aの項目Bに13桁の値を入力したときだけ保存エラーになるリスク
```

細かすぎる内容は、Step 4以降でテスト観点またはテストケースとして扱う。

## よくある失敗

- 一般論だけで `性能が悪くなるリスク` のような候補を増やす。
- `金額計算を確認する` のように確認項目になっている。
- 入力、承認、帳票出力など複数リスクを1件にまとめている。
- 根拠が書かれていない。
