# 入力スコープ整理ポリシー

## 重要領域の判断

以下に該当する領域は、品質上重要な領域として扱う。

- 業務影響が大きい
- 金額、契約、会計、請求、支払に関係する
- データ確定、承認、締め、取消に関係する
- 外部システム連携がある
- 運用で手戻りしにくい
- 障害時の影響範囲が広い
- 利用頻度が高い
- 仕様が複雑、条件分岐が多い
- 過去障害がある
- 今回の変更量が大きい
- テスト環境やデータ準備が難しい

この時点ではリスクIDを付与しない。後続Stepで重点確認すべき領域として整理する。

## 対象外・保留の判断

以下に該当する場合は、対象外または保留候補として扱う。

- 今回の変更対象ではない
- テスト対象外と明記されている
- 別システム、別チームの責務である
- 現行仕様から変更がない
- テスト環境で確認できない
- 仕様が未確定
- 入力文書だけでは判断できない

対象外にする場合も、理由と残存懸念を簡潔に記載する。

## 不足・矛盾・曖昧さ

以下に該当する場合は、確認事項として記録する。

- 入力条件、出力条件、エラー時仕様が分からない
- 権限ごとの操作可否が分からない
- 状態遷移、締め、承認、取消などの業務ルールが分からない
- データの単位、桁数、丸め、期間、範囲が分からない
- 外部IFの正常系・異常系が分からない
- 文書間で用語や条件が一致しない
- 仕様変更前後の差分が分からない

## 出力粒度

- 文書一覧は重要な文書を中心に記載する。
- 対象範囲は主要な業務・機能単位で整理する。
- 業務フローは3-10件程度を目安にする。
- 重要データ、状態、条件、変更点はリスク抽出に関係するものを優先する。
- 入力文書の内容を長く引用しすぎず、文書名、章、節、項目名で根拠を示す。
