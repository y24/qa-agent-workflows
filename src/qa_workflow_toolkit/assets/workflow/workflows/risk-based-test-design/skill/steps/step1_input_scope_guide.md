# step1_input_scope_guide.md

# Step 1: 入力文書整理・分析スコープ確定ガイド

## 目的

入力された開発ドキュメントを読み取り、リスク抽出に必要な分析スコープを整理する。

このStepではリスク候補を抽出しない。後続Stepで参照する「圧縮済みの分析コンテキスト」を作る。

## 入力

- ユーザーが指定した要件、設計、画面、帳票、外部IF、バッチ、権限、データ定義、業務フロー、過去障害、変更要求、テスト制約
- 必要に応じて、ユーザーが指定した文書優先度

## 出力

- `01_input_scope_summary.md`
- 形式: `templates/input_scope_summary_template.md`

## 手順

1. 入力文書一覧を作り、各文書の種別、主な内容、リスク抽出での利用方針を整理する。
2. 対象システム、対象業務、対象機能、主な利用者、対象外・保留範囲を整理する。
3. リスク抽出に関係しそうな業務フロー、利用シナリオ、重要データ、状態、条件を整理する。
4. 新規点、変更点、廃止点、既存機能への影響がありそうな箇所を整理する。
5. 後続Stepで重点確認すべき品質上重要な領域を整理する。
6. テスト環境、期間、データ、外部接続、仕様未確定などの制約を整理する。
7. 入力文書の不足、矛盾、曖昧な点を確認事項として記録する。
8. Step 2への引き継ぎ事項をまとめる。

## 判断基準

- 重要領域、対象外、不足・矛盾の判断に迷う場合: `references/input_scope_policy.md`
- リスク分類の観点を先に確認したい場合: `references/risk_taxonomy.md`
- 共通の推測・根拠ルール: `rules.md`

## このStepでやらないこと

- リスク候補の詳細抽出
- リスクIDの付与
- 影響度、発生可能性、検出困難度の評価
- テスト観点やテストケースの作成
- 入力文書にない仕様の補完

## 完了条件

- 入力文書の種類と利用方針が整理されている
- 対象範囲、対象外、保留範囲が理由付きで整理されている
- 主要な業務フロー、重要データ、状態、条件、変更点が整理されている
- 品質上重要そうな領域とテスト制約が整理されている
- 不足、矛盾、曖昧な点が確認事項として残っている
- Step 2でリスク抽出に使える粒度に圧縮されている

## 停止

`01_input_scope_summary.md` を出力したら停止する。ユーザーから明示的な指示があるまで Step 2 へ進まない。
