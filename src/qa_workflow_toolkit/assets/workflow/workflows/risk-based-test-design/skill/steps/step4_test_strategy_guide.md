# step4_test_strategy_guide.md

# Step 4: リスクベースドテスト方針設計ガイド

## 目的

`03_risk_register.md` で評価したリスクに対して、テスト方針、テスト観点、確認条件、推奨技法、テスト深度を設計する。

このStepでは詳細なテストケースまでは作成しない。Step 5でテストケース骨子に変換できる粒度にする。

## 入力

- `03_risk_register.md`
- `rules.md`
- `references/test_technique_mapping.md`
- 必要に応じて `02_risk_candidate_list.md`

## 出力

- `04_risk_based_test_strategy.md`
- 形式: `templates/risk_based_test_strategy_template.md`

## 手順

1. 優先度別にテスト深度を決める。P0/P1を厚く、P2/P3は代表条件、同時確認、簡易確認、対象外候補に重点化する。
2. 各リスクを「何を、どの条件で確認すれば検出・軽減できるか」というテスト観点に変換する。
3. 確認すべき状態、入力条件、権限、業務タイミング、データパターン、境界値、異常系、回帰影響を整理する。
4. リスクの性質に応じて推奨技法を選び、選定理由を書く。
5. 必要なテストデータや環境を、具体値ではなく種類・条件として整理する。
6. 簡易確認、対象外、レビュー確認、運用確認、残存リスクを理由付きで整理する。
7. Step 5で骨子化すべき主要観点、優先リスク、未解決事項を引き継ぐ。

## 判断基準

- 技法選定: `references/test_technique_mapping.md`
- 技法カタログ詳細: `references/test_technique_catalog.md`
- テスト深度、粒度、対象外判断: `references/test_strategy_policy.md`

## このStepでやらないこと

- 詳細なテストケース作成
- 詳細な操作手順や具体的な期待結果の全文作成
- テストデータの具体値を大量に作成すること
- テスト実行スケジュール、工数見積り、自動化スクリプト作成
- 不明仕様の補完

## 完了条件

- 重点テスト領域が整理されている
- リスクごとのテスト観点に Viewpoint ID があり、Risk ID と紐づいている
- 確認条件、必要データ・環境、推奨技法、選定理由が整理されている
- 優先度別のテスト方針が整理されている
- 簡易確認、対象外、代替対応、残存リスクが理由付きで整理されている
- Step 5でテストケース骨子に変換できる粒度になっている

## 停止

`04_risk_based_test_strategy.md` を出力したら停止する。ユーザーから明示的な指示があるまで Step 5 へ進まない。
