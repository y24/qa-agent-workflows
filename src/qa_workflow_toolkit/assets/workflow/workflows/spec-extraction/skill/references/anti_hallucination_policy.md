# anti_hallucination_policy.md

# 推測混入防止ポリシー

## 成果物別の注意

| 成果物 | 注意 |
|---|---|
| `document_inventory.md` | 文書の内容を読まずに種別や優先度を断定しない |
| `raw_extraction.md` | 原文にない条件や期待結果を追加しない |
| `normalized_spec_inventory.md` | 矛盾を独自判断で解消しない |
| `test_design_input_catalog.md` | テストケースや具体的な期待結果に踏み込まない |
| `gap_and_review_report.md` | ギャップを独自判断で解消済みにしない |

## 期待結果を書くときの制限

期待結果の「種類」は書いてよい。

例:

```text
保存可否、エラーメッセージ、帳票出力内容、連携結果の確認が必要
```

具体的な期待結果は、入力文書に根拠がある場合だけ書く。

## 入力データ観点の制限

データの種類や条件は書いてよい。

例:

```text
有効なコード、存在しないコード、使用停止中のコード
```

具体的な値は、入力文書に根拠がある場合だけ書く。

## 用語の扱い

- 用語を勝手に同義扱いしない。
- 用語定義がない場合は確認事項にする。
- 表記揺れは候補として記録し、確定しない。

## 自己チェック

- 推測が事実として混ざっていないか
- 参照元が残っているか
- 矛盾を解消していないか
- 不明点が確認事項として残っているか
- 禁止表現や断定表現を使っていないか
