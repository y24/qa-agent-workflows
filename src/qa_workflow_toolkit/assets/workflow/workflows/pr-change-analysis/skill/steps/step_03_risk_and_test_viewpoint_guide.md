# Step 3: PRごとのリスク・テスト観点分析ガイド

## 目的

各PRの差分レポートをもとに、テストで注意すべきリスクと、そこから導かれるテスト観点（特に非機能）を抽出し、1つの分析レポートにまとめる。

このStepではリスク抽出とテスト観点提案を統合して行う。

## 入力

- 対象PRの `<pr-slug>/diff_report.md`
- 必要に応じて元差分・コメント（根拠確認のため）
- `references/nonfunctional_risk_catalog.md`、必要時 `references/qa_translation_guide.md`

## 出力

- PRごと: `<pr-slug>/analysis_report.md`
- 形式: `templates/analysis_report_template.md`
- 必要に応じて `issue_log.md`

## 手順（対象PRごとに繰り返す）

1. 差分レポートの変更単位 `CH-xxx` を確認し、各変更が直接触れる機能・データ・連携・設定を特定する。
2. 直接変更だけでなく、波及しうる機能、共有モジュール、呼び出し元/呼び出し先、データ整合性、後方互換性への影響を検討する。
3. リスクに内部ID `RK-001`... を付け、機能リスクと非機能リスクに分けて記録する。各リスクに根拠（`CH-xxx`、ファイル、コメントID、作業項目ID）と確度を添える。
4. 非機能リスクは `references/nonfunctional_risk_catalog.md` の品質特性を必ず一巡し、該当する観点を抽出する。該当なしと判断した特性は、その根拠を残す。
5. 各リスクに対応するテスト観点に `VP-001`... を付け、確認内容、想定条件・データ、確認手段の例、優先度を記す。
6. リスクとテスト観点のトレーサビリティ（`CH -> RK -> VP`）を保つ。
7. 根拠が弱いリスク・観点は `推測` または `仮説` として分け、確認事項に `Q-xxx` で残す。

## 判断基準

- リスクの粒度は、QAが確認対象を判断できる程度にする（粗すぎる一般論、細かすぎる手順は避ける）。
- 認証・権限、課金・計算、データ移行、外部連携、共通基盤、設定・フラグ変更は重点的に扱う。
- 「影響が大きい」と書く場合は、影響先と根拠を添える。
- テスト観点は実行可能な確認に落とし込み、詳細な手順レベルのテストケースには踏み込みすぎない。

## このStepでやらないこと

- 差分レポートの作り直し
- 横断サマリの作成（Step 4で行う）
- 根拠のないリスク・観点の通常リスク化

## 完了条件

- 対象PRすべてに `analysis_report.md` が作成されている。
- 機能リスクと非機能リスクが分けて整理され、非機能特性を一巡した記録がある。
- 各リスク・観点に根拠と確度があり、`CH -> RK -> VP` を追跡できる。
- 推測・仮説・確認事項が分離されている。

## 停止

対象PRすべての `analysis_report.md` を出力したら停止する。ユーザーから明示的な指示があるまでStep 4へ進まない。
