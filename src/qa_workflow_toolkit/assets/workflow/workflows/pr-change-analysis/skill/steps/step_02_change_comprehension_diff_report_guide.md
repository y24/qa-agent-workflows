# Step 2: 全PRの変更内容理解・差分レポート作成ガイド

## 目的

対象PRそれぞれについてAzure DevOpsから情報を取得し、変更内容をQAが読める形に翻訳した差分レポートを作る。

このStepではリスク評価やテスト観点の作り込みをしない。「何がどう変わったか」の理解と整理に集中する。

## 入力

- `00_collection_plan.md`
- 各PRのMCP取得結果（メタ情報、変更ファイル/差分、コメントスレッド、リンク作業項目）
- `references/azure_devops_mcp_usage.md`、`references/qa_translation_guide.md`、`references/change_type_taxonomy.md`

## 出力

- PRごと: `<pr-slug>/diff_report.md`
- 形式: `templates/diff_report_template.md`
- 必要に応じて `issue_log.md`

## 手順（対象PRごとに繰り返す）

1. `references/azure_devops_mcp_usage.md` の手順で、対象PRのメタ情報・変更ファイル/差分・コメントスレッド・リンク作業項目を取得する。
2. 取得できた範囲とできなかった範囲（差分が大きい、バイナリ、権限不足等）を「カバー範囲」として記録する。
3. PR概要を整理する: タイトル、Description要約、source→targetブランチ、状態、作成者、関連作業項目。
4. 変更ファイルを一覧化し、`references/change_type_taxonomy.md` に従って変更種別タグを付け、規模感（ファイル数、追加/削除の概算）を記す。
5. 論理的な変更単位ごとに内部ID `CH-001`... を付け、`references/qa_translation_guide.md` に従って「この変更で何が起きる/変わるか」を挙動・影響の言葉で要約する。
6. Descriptionとレビューコメントから、変更意図、判断の経緯、懸念、残課題、未解決の議論を抽出する（コメントIDを残す）。
7. コードから読み取れない意図や挙動は、推測欄または確認事項に分ける。
8. 後続Step（リスク・テスト観点分析）に渡すべき着目点を引き継ぎ事項として控える。

## 判断基準

- 差分の行単位の解説ではなく、QAが影響範囲を判断できる粒度に要約する。
- 自動生成・整形・リネームのみの変更は、その旨を明示して影響を限定的に扱う。
- レビューコメントの「未解決」「要対応」は重要な手掛かりとして残す。
- 取得できた事実と、そこからの推測を必ず分ける。

## このStepでやらないこと

- リスクの抽出・評価
- テスト観点の提案・優先度付け
- コードに書かれていない原因・挙動の断定

## 完了条件

- 対象PRすべてに `diff_report.md` が作成されている。
- 各レポートにPR概要、変更ファイルと種別、変更単位ごとのQA向け要約、Description/コメントの要点、カバー範囲が含まれる。
- 元PR ID・作業項目ID・コメントIDと内部IDが追跡できる。
- 推測と確認事項が事実から分離されている。

## 停止

対象PRすべての `diff_report.md` を出力したら停止する。ユーザーから明示的な指示があるまでStep 3へ進まない。
