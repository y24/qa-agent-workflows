# Step 1: 対象PR特定・情報収集スコープ整理ガイド

## 目的

ユーザーから渡されたPull RequestのURL群を解釈し、Azure DevOps MCPで何をどの順で取得するかの計画を立てる。

このStepでは差分の詳細解釈やリスク抽出をしない。

## 入力

- Pull Request の URL（1件以上）
- 任意: 着目してほしい機能・品質特性、対象環境、テスト設計の目的やスコープ

## 出力

- `00_collection_plan.md`
- 形式: `templates/collection_plan_template.md`
- 必要に応じて `issue_log.md`

## 手順

1. 各URLを `references/azure_devops_mcp_usage.md` の規則で `organization` / `project` / `repository` / `pullRequestId` に分解する。
2. 分解できないURL、重複URL、別組織のURLを切り分け、対象PRに内部ID `PR-001`... を付与する。元のPR IDとURLを保持する。
3. 各PRについて、取得する項目（メタ情報、変更ファイル/差分、コメントスレッド、リンク作業項目）と、取得に使うMCPツールを計画として記す。
4. ユーザー指定の着目点・スコープ・目的を記録し、後続Stepで重点化する観点を控える。
5. 取得上の前提・制約（権限、差分サイズ上限、巨大PRの分割取得方針）を明示する。
6. この時点で判明している取得不能・要確認事項を `issue_log.md` に記録する。

## 判断基準

- URLから一意にPRを特定できない場合は、推測で補わず確認事項にする。
- 同一PRが重複指定された場合は1件に正規化し、その旨を残す。
- ユーザーが着目点を示していない場合は、全体を均等に見る前提とし、想定スコープを明示する。

## このStepでやらないこと

- 差分の内容理解・要約
- リスク抽出、テスト観点の提案
- レビューコメントの解釈

## 完了条件

- 対象PRが内部IDと元ID・URLで一覧化されている。
- PRごとの取得項目と使用MCPツールが計画されている。
- 着目点・スコープ・前提・制約が記録されている。

## 進行

`00_collection_plan.md` を作成したら、停止せずにStep 2（全PRの差分レポート作成）へ進んでよい。全PRの差分レポートを出力した時点で停止する。
