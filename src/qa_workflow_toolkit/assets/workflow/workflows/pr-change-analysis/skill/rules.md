# pr-change-analysis 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティ、入力文書、テスト設計粒度は `../../shared/` 配下の方針に従う。
このファイルでは `$pr-change-analysis` 固有の責務、ID、禁止事項を定義する。

## 目的

Azure DevOpsのPull Requestの差分・Description・レビューコメント・作業項目を、QAが読める差分レポートと、リスク・テスト観点（特に非機能）の分析レポートへ変換するための共通ルールを定義する。

## 基本方針

- 差分、Description、レビューコメント、作業項目、前Step成果物、ユーザー指定を根拠にする。
- 読み手はコード詳細を普段読まないQAエンジニアである。コードを「何が起きるか・何が変わるか」という挙動・影響の言葉に翻訳する。
- 事実、推測、仮説、確認事項、未確定事項を分けて扱う。
- 分析の最終目的はテスト設計に使える材料の提供であり、リスクと観点は実行可能な確認に落とし込む。
- 非機能（性能、可用性、セキュリティ、互換性、運用性、移行・ロールバック等）を必ず一度走査する。

## 厳守ルール

### コードにないことを事実化しない

挙動、意図、原因、影響範囲、設定値の意味は、差分・Description・コメント・作業項目・参照資料に明記または読み取れる場合だけ事実として扱う。読み取れない箇所は推測欄または確認事項に分ける。

### 取得範囲を偽らない

差分が大きすぎる、バイナリ、生成物、権限不足などで読めなかった部分は、読んだことにしない。各レポートに「カバー範囲」を明示し、未取得・未読部分を `issue_log.md` に残す。

### 規模だけでリスクを決めない

小さな差分でも、認証・権限、課金・計算、データ移行、外部連携、共通基盤、設定変更は重点的に扱う。逆に大規模でも自動生成・整形のみの変更は影響を限定的に扱ってよい（根拠を添える）。

### 分析と提案を分ける

「この変更は認証処理に触れている」は事実・分析であり、「権限昇格時の境界値で認可が通るかを確認する」はテスト観点である。差分レポート（Step 2）と分析レポート（Step 3）で混同しない。

## ID体系

| 対象 | 形式 | 備考 |
|---|---|---|
| Pull Request | `PR-001` | 内部連番。元のAzure DevOps PR IDは別列で保持する |
| Change | `CH-001` | PR内の論理的な変更単位 |
| Risk | `RK-001` | 機能・非機能リスク |
| Test Viewpoint | `VP-001` | テスト観点 |
| Question | `Q-001` | 確認事項 |

元のPR ID、作業項目ID、コメントIDは変更しない。内部IDを付ける場合も元IDを必ず残す。

## Step進行ルール

- Step 1では差分の詳細解釈やリスク抽出をしない。取得計画と前提の整理に留める。
- Step 2では変更内容の理解と差分レポート化を行い、リスク評価やテスト観点の作り込みをしない。
- Step 3ではリスク抽出とテスト観点提案を統合して行うが、詳細な手順レベルのテストケースには踏み込みすぎない。
- Step 4では横断的な集約と重点領域の提示を行い、個別PRの分析を作り直さない。
- Step 1からStep 2（全PRの差分レポート）までは停止せず進めてよい。全差分レポート出力後、Step 3前後で停止する。

## 出力ルール

- 出力はMarkdownを基本にする。
- 表は列名を安定させ、CSV変換しやすい形にする。
- 各成果物には入力・参照元、カバー範囲、主成果物、確認事項、次工程への引き継ぎを含める。
- 根拠は変更ファイル名・差分箇所・コメントID・作業項目ID・前StepのIDで追跡できるようにする。
- 不明点は補完せず、`要確認`、`情報不足`、`未確定`、`対象外`、`未取得` などで状態を示す。

## 禁止事項

- 差分やコメントから読み取れない挙動・原因・意図を断定する。
- 取得できなかった差分を読んだ前提でリスクや観点を作る。
- 根拠となる変更箇所のないリスク・観点を、根拠ありのものと混ぜる。
- 規模や件数だけでテスト優先度を決める。
- 非機能観点の走査を省略する（該当なしと判断した場合も根拠を残す）。
- 最終成果物から未取得範囲・確認事項・推測の区別を消す。
