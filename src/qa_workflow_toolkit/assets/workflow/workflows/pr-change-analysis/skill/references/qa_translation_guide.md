# コード→QA言語 翻訳ガイド

読み手は普段コードの詳細を読まないQAエンジニアである。差分レポートと分析レポートは、コードの逐語的説明ではなく、「何が起きるようになるか・何が変わるか・どこに影響するか」を伝える。

## 基本方針

- 主語を「ユーザー」「システム」「画面」「API」「データ」など、QAが観測できる対象にする。
- 実装の手段（クラス名、関数名）ではなく、結果・挙動・条件を書く。識別子は根拠として括弧で添える程度にする。
- 変更前と変更後の差を対比で示す（Before/After）。
- 観測可能な入口（画面操作、API、バッチ、イベント）から書き起こす。

## 翻訳の例

| コード上の事実 | QA向けの翻訳 |
|---|---|
| `if user.role == 'admin'` の分岐を追加 | 管理者ロールのときだけ表示/操作できる項目が増えた。一般ユーザーとの差を確認する |
| バリデーションの正規表現を変更 | 入力チェックの許容範囲が変わった。これまで通る/弾かれていた値の境界が動く可能性 |
| クエリに `JOIN` を追加 | 一覧の取得条件・件数・表示内容が変わりうる。性能（件数増時）も観点になる |
| 例外時に既定値を返すよう変更 | エラー時にエラー表示ではなく既定値で続行する。異常系の見え方が変わる |
| ライブラリをメジャー更新 | 依存の挙動が変わる可能性。既存機能の回帰と既知の非互換を確認 |
| 設定キー `feature.x.enabled` を追加 | フラグのON/OFFで挙動が分岐する。両系統の確認が要る |

## 粒度の目安

| 粒度 | 例 | 扱い |
|---|---|---|
| 細かすぎる | 「変数名を `a` から `count` に変更」 | 挙動に影響しなければ要約に含めない（種別 `リファクタ`/`自動生成・整形`） |
| 適切 | 「在庫が0のときも注文できてしまう経路が塞がれた」 | 差分レポートの変更単位として記す |
| 粗すぎる | 「いろいろ改善された」 | 何が変わったかが分からないので不可 |

## やってはいけないこと

- コードに書かれていない意図・原因・効果を断定する（推測は推測欄に分ける）。
- 実装詳細をそのまま貼り付けて説明としない。
- 「リファクタなので影響なし」と根拠なく言い切る（挙動不変の根拠を示す）。

## 不明なときの書き方

- 差分から挙動を特定できない場合: 「この変更の利用者影響は差分からは特定できない（要確認: ○○）」と確認事項に回す。
- Descriptionとコードが食い違う場合: 両方を併記し、確認事項にする。
