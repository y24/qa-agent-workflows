# 品質リスク・弱点パターン解釈方針

## 目的

不具合傾向を、根拠付きで後続テスト設計に使える品質リスクへ変換する。

## 解釈の粒度

| 粒度 | 例 | 扱い |
|---|---|---|
| 粗すぎる | 入力チェック不備が多い | Step 3の分析結果として扱う |
| 適切 | 複数マスタ条件の組み合わせ時に入力チェック仕様が漏れやすい | 品質リスク候補にする |
| 細かすぎる | 画面Aの項目Bに値Cを入れる | Step 5以降のテスト条件候補に回す |

## 解釈手順

1. 多発している分類、重大度が高い分類、発見が遅れた分類を確認する。
2. 共通する機能領域、処理パターン、データ、状態、条件を探す。
3. 不具合が発生しやすい状況を1文で表現する。
4. 影響と見逃されやすい理由を、根拠がある範囲で整理する。
5. 後続テストで注意すべき観点を短く書く。
6. 根拠が弱いものは `推測` または `仮説` に分ける。

## 根拠レベル

| レベル | 条件 | 表記 |
|---|---|---|
| 高 | 複数チケットまたは集計結果で同じ傾向が確認できる | `根拠あり` |
| 中 | 少数だが重大度、発見遅延、回帰性が高い | `限定的根拠` |
| 低 | チケット記述から可能性はあるが明示根拠が弱い | `推測` |
| 不足 | 根拠がなく一般論に近い | `採用しない` または `要確認` |

## 注意事項

- 工程、体制、レビュー不足などの原因は、記録に根拠がある場合だけ扱う。
- 「多い」「集中している」と書く場合は、母数、件数、割合、比較対象を添える。
- 1件だけの不具合でも、業務停止、データ破壊、権限逸脱、リリース後発見は重点候補にできる。
- 品質リスクは、後続テスト設計で確認可能な形にする。
