# Step 01: レビュー対象と入力情報の整理

## 目的

レビュー対象の成果物、参照すべき根拠資料、レビュー範囲、対象外範囲、不足資料を整理し、後続レビューの土台を作る。

## 読む入力

- レビュー依頼内容
- テスト設計書、観点表、テスト条件、テストケース一覧
- 要件、設計、画面、業務、API、テスト計画、過去障害などの根拠候補
- 前段成果物がある場合は、仕様インベントリ、リスク一覧、観点カタログ

## 実施手順

1. 入力ファイルを一覧化し、`RT-xxx` または `EV-xxx` を付与する。
2. 各入力を「レビュー対象」「根拠資料」「参考資料」「未確認資料」に分類する。
3. レビュー範囲、対象外範囲、ユーザー指定の重点観点を整理する。
4. 成果物の粒度、構成、ID体系、トレーサビリティの有無を概観する。
5. 不足資料、矛盾、前提不明点を抽出する。
6. 後続Stepで重点確認すべき観点を仮設定する。

## 判断基準

- レビュー対象は、直接レビューコメントを付ける成果物である。
- 根拠資料は、仕様、業務ルール、期待動作、テスト範囲を判断するために使える資料である。
- 参考資料は、背景理解には使えるが、単独では指摘根拠にしない資料である。
- 不明な範囲は推測で埋めず、確認事項にする。

## 出力

`01_input_summary.md` を `templates/input_summary_template.md` に沿って作成する。

## やってはいけないこと

- Step 01で詳細なレビュー指摘を確定しない。
- 根拠資料の位置づけが不明なまま、仕様断定に使わない。
- 不足資料を理由にレビュー全体を止めず、実施可能範囲を明示する。
