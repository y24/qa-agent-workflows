# Step 03: テスト観点・テスト条件・カバレッジレビュー

## 目的

テスト設計書に記載された観点、条件、カバレッジが、仕様やリスクに対して十分かを確認する。

## 読む入力

- `01_input_summary.md`
- `02_design_structure_review.md`
- テスト観点表、テスト条件、カバレッジ表
- 要件、設計、業務フロー、リスク、過去障害、変更点
- 必要時: `references/review_viewpoint_catalog.md`

## 実施手順

1. 正常系、異常系、境界値、業務フロー上の重要パターンを確認する。
2. 条件分岐、状態遷移、権限差異、データ整合性など、対象に応じた観点を確認する。
3. 入力、出力、保存、検索、帳票、外部連携、エラー処理などの不足可能性を確認する。
4. 過去障害、変更影響、回帰確認の反映状況を確認する。
5. 非機能、運用、保守、セキュリティ観点が必要な対象かを判断する。
6. 不足、過剰、重複、対象外妥当性、追加確認事項を整理する。

## 判断基準

- 観点カタログはチェックリストとして参照するが、全項目を機械的に要求しない。
- システム特性、テストレベル、レビュー範囲に照らして必要性を判断する。
- 不足している可能性があるが根拠が弱い場合は、未確認事項にする。
- 追加案は、既存成果物との差分が分かる表現にする。

## 出力

`03_viewpoint_coverage_review.md` を `templates/viewpoint_coverage_review_template.md` に沿って作成する。

## やってはいけないこと

- 一般論だけで網羅不足を断定しない。
- カタログの全観点を必須要求にしない。
- テストケースを大量生成してカバレッジレビューの代わりにしない。
