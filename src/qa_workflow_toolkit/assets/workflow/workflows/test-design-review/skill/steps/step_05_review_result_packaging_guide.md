# Step 05: レビュー結果の統合・優先度付け・成果物化

## 目的

Step 02からStep 04のレビュー結果を統合し、修正判断に使えるレビュー報告とissue logとしてまとめる。

## 読む入力

- `02_design_structure_review.md`
- `03_viewpoint_coverage_review.md`
- `04_testcase_quality_review.md`
- 各Stepで作成したissue、確認事項、改善提案
- 必要時: `references/severity_priority_criteria.md`

## 実施手順

1. 指摘候補を収集し、重複を統合する。
2. 指摘を分類し、対象成果物、対象箇所、根拠、理由を補正する。
3. 重要度と修正優先度を付ける。
4. 即修正すべき問題、改善提案、未確認事項を分ける。
5. 重要指摘サマリ、カバレッジ懸念、テストケース品質懸念を整理する。
6. 次アクションと、必要な追加確認を明確にする。

## 判断基準

- 重要度はテスト結果の信頼性や欠陥見逃しへの影響で判断する。
- 優先度は修正順、後続作業への影響、修正容易性を加味して判断する。
- 同じ原因の指摘は統合し、代表例と影響範囲を示す。
- 未確認事項は、確認できれば指摘になる可能性があるものとして分離する。

## 出力

- `05_review_report.md` を `templates/review_report_template.md` に沿って作成する。
- `05_issue_log.md` を `templates/issue_log_template.md` に沿って作成する。

## やってはいけないこと

- 根拠が弱い指摘を、最終報告で断定表現に変えない。
- 重要度と優先度を同じ意味で扱わない。
- 低重要度の表記指摘で、重大なカバレッジ懸念を埋もれさせない。
