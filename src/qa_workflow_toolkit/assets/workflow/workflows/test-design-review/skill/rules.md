# Test Design Review 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティ、テスト設計粒度は `../../shared/` 配下の方針に従う。
このファイルでは `$test-design-review` 固有の責務、ID、禁止事項を定義する。

## 目的

既存のテスト設計成果物をレビューし、根拠付きの指摘、未確認事項、改善提案として整理するための共通ルールを定義する。

## 基本方針

- レビュー対象と根拠資料を分けて扱う。
- 指摘には、対象成果物、対象箇所、根拠、理由、推奨対応を付ける。
- 仕様に基づく問題、推測、前提、未確認事項、改善提案を混同しない。
- 断定できない不足は「不足の可能性」または「要確認」として扱う。
- 良い点は必要に応じて記録するが、レビュー結果の中心は修正判断に使える指摘とする。

## Step責務

| Step | 責務 | やらないこと |
|---|---|---|
| 01 | レビュー対象、根拠資料、範囲、不足資料の整理 | 詳細なレビュー指摘の確定 |
| 02 | テスト設計書の構造、粒度、トレーサビリティ確認 | 個別テストケースの細部レビュー |
| 03 | 観点、条件、カバレッジの妥当性確認 | カタログ項目の機械的な全要求 |
| 04 | テストケースの実行可能性、期待結果、データ妥当性確認 | 仕様にない期待結果の創作 |
| 05 | 指摘の統合、重複排除、重要度・優先度付け | 新規レビュー観点の追加調査 |

## ID体系

| 対象 | 形式 |
|---|---|
| Review Target | `RT-001` |
| Evidence | `EV-001` |
| Review Viewpoint | `RV-001` |
| Issue | `ISS-001` |
| Question | `Q-001` |

既存IDは可能な限り維持する。追加、削除、統合、分割した場合は理由を記録する。

## 指摘分類

- 要件・仕様との不整合
- テスト観点不足
- カバレッジ不足
- 粒度不適切
- 期待結果不明確
- 手順不明確
- 前提条件不足
- テストデータ不足
- トレーサビリティ不足
- 重複・冗長
- 自動化候補
- 未確認事項
- 改善提案

## 禁止事項

- 根拠資料にない仕様や業務ルールを創作する。
- 「一般的には必要」という理由だけで必須指摘にする。
- 不足を断定できない内容を欠陥として扱う。
- テストケースを大量生成してレビュー成果物の代わりにする。
- 重要度を付けずに修正判断できない指摘を並べる。
- issueや確認事項を削除して見かけを整える。

## 品質基準

- 根拠性: 指摘がレビュー対象または根拠資料に紐づく。
- 判定可能性: 何が問題で、なぜ問題で、どう直すべきかが分かる。
- 分離性: 事実、推測、未確認事項、改善提案が区別される。
- 優先判断性: 重要度と修正優先度が分かる。
- 追跡性: 指摘から対象成果物、根拠、関連IDへ追える。
