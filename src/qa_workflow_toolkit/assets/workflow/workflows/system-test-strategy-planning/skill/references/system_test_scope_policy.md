# System Test Scope Policy

## 目的

システムテスト戦略で扱う範囲、対象外、入力根拠の扱いを定義する。

## システムテスト戦略で扱うもの

| 領域 | 扱い |
|---|---|
| 対象機能・業務 | 入力文書で確認できる機能、業務フロー、画面、API、外部IF、バッチ、帳票を対象候補にする |
| 非機能 | 入力文書に品質要求、制約、利用条件、運用条件、リスクがある場合に対象候補にする |
| テスト種別 | 対象候補と品質特性に基づき、根拠付きで計画対象または計画候補に分類する |
| テスト観点 | 詳細テスト設計へ渡せる確認目的と対象を定義する |
| 付属資料 | 品質特性マトリクスと観点トレーサビリティを作る |

## 扱わないもの

- スケジュール、体制、要員、工数、役割分担
- 詳細テストケース、具体的な操作手順、入力値、期待結果
- テスト環境の構築作業手順
- 根拠のない品質目標値や性能閾値

## スコープ判定

| 状態 | 意味 | 扱い |
|---|---|---|
| 対象 | 入力文書からシステムテストで扱う必要性を説明できる | 戦略に含める |
| 候補・要確認 | 必要性は読み取れるが条件不足 | 確認事項付きで候補にする |
| 対象外 | ユーザー指定または入力文書上で対象外が明確 | 対象外理由を記録する |
| 根拠不足 | 一般論では重要だが入力から必要性を説明できない | 最終計画対象に混ぜない |
| 情報不足 | 文書不足で判断できない | issueまたは確認事項に残す |

## 根拠の扱い

- 根拠は `文書名、章、節、ページ、項目名、前段成果物ID` のいずれかで示す。
- 原文を長く引用せず、参照位置と要約を記録する。
- 「通常必要」「一般に重要」は根拠にしない。
- 入力文書が複数ある場合、矛盾や差分は統合せず issue として記録する。
