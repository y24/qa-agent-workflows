# Test Type Catalog

## 目的

システムテスト戦略で選択し得るテスト種別の候補、主な狙い、必要な入力根拠を定義する。

この catalog は選択肢であり、全種別を必ず採用するためのチェックリストではない。

## テスト種別候補

| テスト種別 | 主な狙い | 主な品質特性 | 必要な入力根拠の例 |
|---|---|---|---|
| 機能確認テスト | 要件・仕様どおりに機能が成立することを確認する | 機能適合性 | 機能要件、画面仕様、API仕様、帳票仕様、バッチ仕様 |
| シナリオテスト | 業務上意味のある流れでシステム全体の成立性を確認する | 機能適合性、使用性、信頼性 | 業務フロー、利用者、状態遷移、例外フロー |
| 外部IF・連携テスト | 他システムやファイル、APIとの相互運用を確認する | 互換性、信頼性、セキュリティ | 外部IF仕様、データ形式、連携順序、認証方式 |
| データ・移行確認 | データ変換、移行、初期データ、整合性を確認する | 機能適合性、信頼性、移植性 | 移行方針、データ定義、変換仕様、整合性条件 |
| バッチ・ジョブ確認 | 定時処理、非同期処理、再実行、異常時動作を確認する | 信頼性、性能効率性、保守性 | バッチ仕様、処理量、運用時間帯、リカバリ条件 |
| 帳票・出力確認 | 帳票、通知、ファイル出力の内容と形式を確認する | 機能適合性、互換性 | 帳票仕様、出力形式、配信条件、利用者 |
| 性能テスト | 応答時間、処理時間、スループット、資源使用を確認する | 性能効率性 | 性能目標、ピーク条件、データ量、同時利用、SLA |
| 負荷・容量テスト | 増加負荷、容量上限、劣化傾向を確認する | 性能効率性、信頼性 | 最大件数、同時接続、容量見込み、増加率 |
| 信頼性・リカバリテスト | 障害時動作、復旧、再実行、データ保護を確認する | 信頼性、保守性 | 障害条件、復旧目標、再実行仕様、バックアップ方針 |
| セキュリティテスト | 認証、認可、監査、入力攻撃、機密性を確認する | セキュリティ | 権限仕様、認証方式、監査要件、保護対象情報 |
| 権限・ロールテスト | ロール別に許可・禁止される操作や参照範囲を確認する | セキュリティ、機能適合性 | ロール定義、権限表、組織・部門ルール |
| 互換性テスト | 対応ブラウザ、OS、端末、バージョン差異を確認する | 互換性、移植性、使用性 | 対応環境、利用端末、ブラウザ、OS、ミドルウェア |
| 使用性・アクセシビリティ確認 | 利用者が迷わず、誤り少なく、効率よく使えることを確認する | 使用性 | 利用者種別、利用シーン、UI方針、アクセシビリティ要件 |
| 運用・監視確認 | ログ、監視、通知、運用保守で必要な情報を確認する | 保守性、信頼性、セキュリティ | 運用設計、監視項目、ログ要件、通知条件 |
| インストール・デプロイ確認 | 導入、設定、リリース、環境差分への適応を確認する | 移植性、保守性 | デプロイ方式、構成管理、環境差分、設定方針 |
| 回帰テスト方針 | 変更影響範囲の既存品質維持を確認する | 機能適合性、保守性 | 変更範囲、既存機能、影響分析、過去不具合 |

## 選定結果の分類

| 分類 | 意味 |
|---|---|
| 計画対象 | 入力根拠があり、システムテストで実施方針を示す |
| 計画候補・要確認 | 必要性はあるが、条件や基準が不足している |
| 対象外 | 対象外の根拠がある、または別工程で扱うことが明確 |
| 根拠不足 | 一般論以外の根拠がないため採用しない |

## 記述粒度

- テスト種別の方針は、目的、対象、主な観点、根拠、未確認事項までに留める。
- 詳細テストケースの事前条件、操作手順、入力値、期待結果は作らない。
