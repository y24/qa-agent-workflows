# システムテスト戦略策定 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティ、入力文書、テスト設計粒度は `../../shared/` 配下の方針に従う。
このファイルでは `$system-test-strategy-planning` 固有の責務、ID、禁止事項を定義する。

## 基本方針

- 成果物の中心は、システムテストフェーズにおける戦略、カバレッジ、テスト種別、テスト観点の方針化とする。
- 入力文書、前Step成果物、ユーザー指定を根拠にする。
- 事実、推測、前提、確認事項、未確定事項、提案を分けて扱う。
- テスト観点は、詳細テストケースより上位で、テスト設計担当者が詳細化できる粒度にする。
- 非機能は、入力文書から必要性が読み取れるものだけを対象候補にする。
- ISO/IEC 25010の品質特性は分類軸であり、根拠のない観点追加の口実にしない。

## 対象外

- 詳細テストケース、操作手順、入力値、期待結果の作成
- スケジュール、マイルストーン、体制、役割分担、工数、担当者の策定
- テスト環境の構築手順、ツール導入手順、運用手順書の作成
- 入力文書にない品質基準値、性能閾値、同時接続数、データ量、対応ブラウザ、障害条件、復旧目標の補完

## ID体系

| 対象 | 形式 |
|---|---|
| Input | `IN-001` |
| Source Fact | `SF-001` |
| Coverage | `CV-001` |
| Test Type | `TT-001` |
| Viewpoint | `VP-001` |
| Matrix Row | `QM-001` |
| Question | `Q-001` |
| Issue | `I-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## テスト種別の選定ルール

- テスト種別は、入力文書上の要件、品質要求、制約、業務重要度、利用環境、外部IF、運用条件などに根拠がある場合だけ「計画対象」にする。
- 根拠はあるが条件不足で具体化できない場合は「計画候補・要確認」にする。
- 一般論として重要でも、入力から必要性を説明できない場合は「根拠不足」に分類し、最終計画の計画対象に混ぜない。
- 1つのテスト種別が複数品質特性を扱う場合は、主品質特性と副品質特性を分ける。

## テスト観点の作成ルール

- 各観点には、確認目的、対象、品質特性、関連テスト種別、根拠、詳細設計への引き継ぎを含める。
- 観点は「何を確認すべきか」を示し、詳細手順や期待値を作らない。
- 入力にない判定条件は `要確認` とし、値を仮置きしない。
- 未確認事項がある観点は、最終付属資料のトレーサビリティにも残す。

## 禁止事項

- 入力文書にない内容を推測で補完する
- すべての非機能品質特性を均等に埋める
- 品質特性名だけを根拠にテスト種別を必須化する
- テスト種別名だけを列挙し、観点や根拠を示さない
- 具体的なテストケースを作成する
- スケジュール、体制、工数を計画書に含める
- 未確認事項を最終成果物から消す
