# 非機能品質クライテリア策定 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティ、入力文書、テスト設計粒度は `../../shared/` 配下の方針に従う。
このファイルでは `$nonfunctional-quality-criteria-planning` 固有の責務、ID、禁止事項を定義する。

## 基本方針

- 成果物の中心は、非機能リスクの大量列挙ではなく、テスト計画や設計に使える品質クライテリアの作成とする。
- 品質クライテリアは `品質特性 -> 対象領域・利用状況 -> 品質上の懸念 -> 品質クライテリア -> 確認方法・測定方法 -> テスト計画への反映方針` の流れで整理する。
- 入力文書、前Step成果物、ユーザー指定を根拠にする。
- 事実、推測、仮説、確認事項、未確定事項を分けて扱う。
- 品質クライテリアは、詳細テストケースより上位で、判定や確認に落とせる粒度にする。

## 対象品質特性

| 特性 | ID接頭辞 | 扱い |
|---|---|---|
| 性能効率性 | `PE` | 応答時間、処理時間、スループット、リソース使用量など |
| 互換性 | `CO` | ブラウザ、OS、端末、外部IF、共存性、相互運用性など |
| 使用性 | `US` | 入力支援、エラー理解、操作効率、アクセシビリティ、誤操作防止など |
| 信頼性 | `RE` | 可用性、復旧、障害時動作、データ保護、再実行性など |
| セキュリティ | `SEC` | 認証、認可、監査、機密性、完全性、脆弱性対策など |
| 保守性 | `MA` | 変更容易性、解析容易性、テスト容易性、運用保守性など |
| 移植性 | `PO` | 環境移行、設定変更、インストール、デプロイ、クラウド・基盤変更など |

## 厳守ルール

### 根拠のない品質要求を作らない

入力にない目標値、閾値、最大データ量、同時接続数、利用端末、対応ブラウザ、復旧時間、権限ルールを事実として創作しない。

### 確認手段を先に決めない

品質特性だけを理由に、特定のテスト種別を必須扱いしない。小さな画面改修では、通常の機能テスト、画面レビュー、静的レビューで十分な場合がある。

### 定量化できないものを無理に数値化しない

数値目標が必要だが入力にない場合は、`要確認` として残す。仮値を置く場合は `仮説` と明記し、採用条件を添える。

### 品質特性の重複を整理する

1つの懸念が複数特性に関係する場合は、主特性と副特性を分ける。例: 大量データ時の応答遅延は性能効率性が主、使用性が副。

## ID体系

| 対象 | 形式 |
|---|---|
| Scope | `S-001` |
| Input | `IN-001` |
| Concern | `QC-001` |
| Criterion | `<特性接頭辞>-001` |
| Confirmation | `CF-001` |
| Question | `Q-001` |
| Issue | `I-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## 出力ルール

- 出力はMarkdownを基本にする。
- 表は列名を安定させ、後続Stepで参照しやすくする。
- 各成果物には入力・参照元、主成果物、確認事項、次Stepへの引き継ぎを含める。
- 根拠は文書名、章、節、項目名、前StepのIDで追跡できるようにする。
- 不明点は補完せず、`要確認`、`情報不足`、`未確定`、`対象外` などで状態を示す。

## Step進行ルール

- Step 1では品質クライテリアを作り込まない。
- Step 2では基準値や確認方法を無理に決めない。
- Step 3では完全なテストケースを作らない。
- Step 4で初めて確認方法、測定方法、テスト計画への反映方針を整理する。
- Step 5では重複、粒度、根拠、未確認事項を見直し、使える表へ整える。
- 各Stepの出力後は停止し、ユーザーの続行指示を待つ。

## 禁止事項

- 入力にない非機能要求や基準値を断定する
- 一般論だけで品質クライテリアを作る
- すべての品質特性を均等に埋めようとして薄い項目を増やす
- 確認方法をテスト種別名だけで終わらせる
- 未確認事項を成果物から消す
- 機能適合性の詳細テストケースをこのskill内で作り込む
