# step_05_confirmation_policy_planning_guide.md

# Step 5: クライテリア確認方針・テスト計画組み込み案の策定ガイド

## 目的

各品質クライテリアについて、どの確認活動で妥当性を検証するかを整理し、テスト計画やレビュー計画に組み込める形へ落とす。

## 入力

- `04_quality_criteria_catalog.md`
- Step 3の優先度、Step 1の環境・運用制約
- テスト計画、レビュー計画、自動化方針があれば参照する

## 出力

- `05_confirmation_policy_and_test_planning.md`
- 形式: `templates/confirmation_policy_and_test_planning_template.md`

## 手順

1. 各品質クライテリアの確認対象と優先度を確認する。
2. 確認方法を、テスト、機能テスト組み込み、レビュー、静的解析・ツール、運用確認・リハーサルから選ぶ。
3. 推奨テスト種別または確認活動を決め、理由を記録する。
4. 必要な環境、データ、ツール、証跡、前提条件を整理する。
5. 自動化推奨度を判定し、繰り返し性、判定容易性、証跡取得性、環境準備コストを理由として書く。
6. テスト計画への組み込み先を、システムテスト、受入テスト、性能テスト、環境別テスト、設計レビュー、運用テストなどで整理する。
7. 詳細テストケース化が必要なもの、レビューで足りるもの、保留するものを分ける。
8. 最終トレーサビリティをまとめる。

## 判断基準

- 確認方法の選び方: `references/confirmation_method_catalog.md`
- 自動化推奨度の判定: `references/automation_recommendation_guide.md`
- 詳細化しすぎない粒度: `rules.md`

## このStepでやらないこと

- 詳細な画面操作手順や入力値の完全定義
- テスト自動化コードの作成
- 根拠のない本番相当データ量やSLAの断定
- レビューや合意が必要な事項を、テストで確認済み扱いにすること

## 完了条件

- 各クライテリアに推奨確認方法、推奨テスト種別、組み込み先がある。
- 自動化推奨度と理由がある。
- 必要な環境、データ、未確認事項が明示されている。
- `NFC ID -> NFR-C ID -> NFR-P ID` の対応が追跡できる。

## 停止

`05_confirmation_policy_and_test_planning.md` を出力したら停止する。追加で詳細テストケース設計が必要な場合は、別タスクとして提案する。
