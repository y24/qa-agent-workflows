# 非機能品質クライテリア策定・確認計画化 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティ、入力文書、テスト設計粒度は `../../../shared/` 配下の方針に従う。
このファイルでは `$nonfunctional-quality-criteria-planning` 固有の責務、ID、禁止事項を定義する。

## 目的

このskillで作成する非機能関連情報、品質懸念、優先度評価、品質クライテリア、確認方針に共通するルールを定義する。

## 基本方針

- 入力文書、前Step成果物、ユーザー指定を根拠にする。
- 事実、推測、仮説、確認事項、未確定事項を分けて扱う。
- 品質懸念は「非機能品質上、利用者、業務、運用、保守、セキュリティなどに困りごとを起こしうること」として書く。
- 品質クライテリアは「どの条件で、何が、どの程度なら許容できるか」として書く。
- 確認方針は「何を、どの条件で、どの手段で確認するか」として書く。
- 高重要度の懸念を厚く扱い、低重要度の懸念を過剰にクライテリア化しない。

## 対象と対象外

- 対象品質特性は、性能効率性、互換性、使用性、信頼性、セキュリティ、保守性、移植性に限定する。
- 機能適合性は対象外とし、機能仕様そのものの正しさは他skillに委ねる。
- 安全性は原則対象外とする。ただし、業務、法令、人身、環境への重大影響がある場合は補足事項として扱う。
- 期間不足、体制不足、環境不足などは、品質懸念ではなくテスト制約またはプロジェクトリスクとして分ける。

## ID体系

| 対象 | 形式 |
|---|---|
| Document | `DOC-001` |
| Scope Item | `SCOPE-001` |
| Constraint | `CON-001` |
| Question | `Q-001` |
| Quality Concern | `NFC-001` |
| Hypothesis Concern | `NFCH-001` |
| Quality Criteria | `NFR-C-001` |
| Confirmation Policy | `NFR-P-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## Step進行ルール

- Step 1では品質懸念を作り込まない。
- Step 2では優先度評価、品質クライテリア、確認方針を作らない。
- Step 3では品質クライテリアや詳細テストケースを作らない。
- Step 4では確認方針を軽く示してもよいが、テスト計画化や自動化判断は作り込まない。
- Step 5では詳細な画面操作、具体データ、期待結果の完全一致に踏み込みすぎない。
- 各Stepの出力後は停止し、ユーザーの続行指示を待つ。

## 出力ルール

- 出力はMarkdownを基本にする。
- 表は列名を安定させ、後続Stepで参照しやすくする。
- 根拠は文書名、章、節、項目名、前StepのIDで追跡できるようにする。
- 判断できない項目は補完せず、`要確認`、`情報不足`、`未確定`、`対象外` などで状態を示す。
- 数値基準は、入力に明記されている場合は事実として扱い、推定する場合は `推測:` と明記する。

## 禁止事項

- 入力にない非機能要求や数値目標を断定する
- 一般論だけで重大な品質懸念を作る
- 品質懸念、品質クライテリア、確認方針を混同する
- すべての品質懸念を高優先度にする
- 非機能確認をすべて専用テストに割り当てる
- 自動化できないものを無理に自動化推奨にする
- 未確認事項や対象外を成果物から消す

## 品質基準

- 追跡性: 非機能関連情報、品質懸念、優先度、クライテリア、確認方針がIDで追跡できる。
- 妥当性: 品質懸念の根拠、重要度、クライテリア化判断、確認方法の対応が説明できる。
- 現実性: 確認方法がテスト計画、レビュー、静的解析、運用確認に組み込める粒度である。
- 透明性: 未確認事項、対象外、保留、残存懸念が隠れていない。
