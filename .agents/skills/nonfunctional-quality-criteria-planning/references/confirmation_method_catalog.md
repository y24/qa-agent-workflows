# Confirmation Method Catalog

## 目的

品質クライテリアを確認方法へ落とし込む際の候補を定義する。

確認方法は、品質特性だけで機械的に決めない。対象範囲、変更規模、根拠、重要度、利用環境、確認コストを踏まえて選ぶ。

## 確認方法カタログ

| 品質特性 | 確認方法候補 | 補足 |
|---|---|---|
| 性能効率性 | 性能テスト、負荷テスト、大量データテスト、リソース使用量測定、通常機能テスト内の応答確認 | 目標値、測定条件、データ量、環境差分を明示する |
| 互換性 | ブラウザ・OS確認、端末確認、外部IF疎通、ファイル形式確認、共存確認 | 対象組合せを根拠付きで絞る |
| 使用性 | 画面レビュー、ユーザビリティ評価、入力補助確認、エラーメッセージ確認、アクセシビリティ確認 | 人間の判断が必要な項目は評価観点を明示する |
| 信頼性 | 障害時動作確認、復旧テスト、再実行確認、フェイルオーバー確認、データ整合性確認 | 障害条件、復旧条件、許容停止時間を明示する |
| セキュリティ | 権限テスト、認証確認、監査ログ確認、脆弱性診断、静的解析、設定レビュー | 診断が必要か、機能テスト内で足りるかを分ける |
| 保守性 | コードレビュー、設計レビュー、ログ・監視レビュー、設定変更確認、テスト容易性レビュー | 実行テストではなくレビューが主になる場合がある |
| 移植性 | インストール確認、デプロイ確認、環境差分確認、設定移行確認、データ移行リハーサル | 移行先環境と移行条件を明示する |

## 選択方針

- 高影響かつ判定が実行時にしかできない項目は、実行テストを優先する。
- 設計、設定、ログ、構成で確認できる項目は、レビューや静的確認も候補にする。
- 小規模改修では、通常の機能テスト内確認で十分な非機能項目がある。
- 外部専門診断が必要な項目は、テスト計画上の別タスクとして扱う。

## 記載に必要な要素

確認方法を書くときは、少なくとも以下を含める。

- 確認対象
- 確認条件
- 判定基準
- 証跡
- 必要環境または必要データ
- 未確定事項
