# 自動化推奨度ガイド

## 判定区分

| 判定 | 意味 |
|---|---|
| 自動化を推奨 | 繰り返し確認が必要で、期待結果が機械的に判定しやすい |
| 一部自動化を推奨 | 測定、準備、証跡取得は自動化できるが、最終判断に人の確認が必要 |
| 条件付きで自動化を推奨 | 環境構築、データ準備、頻度、保守コスト次第で判断する |
| 自動化は非推奨 | 判断が主観的、頻度が低い、または人間の観察が重要 |
| 自動化対象外 | レビュー、合意、方針確認など、実行テストではない |

## 推奨しやすい条件

- 回帰確認が繰り返し発生する。
- 判定条件が明確で、機械的に合否判定できる。
- 環境やデータを安定して準備できる。
- 証跡取得や計測に自動化の効果がある。
- 手作業だと漏れやばらつきが大きい。

## 推奨しにくい条件

- 利用者にとって分かりやすいかなど主観評価が中心である。
- 実施頻度が低く、自動化保守コストが効果を上回る。
- 本番相当環境や外部連携の準備が重い。
- レビュー、合意、設計判断が中心である。

## 例

| 対象 | 推奨度 | 理由 |
|---|---|---|
| API応答時間の定点測定 | 自動化を推奨 | 繰り返し測定でき、閾値判定しやすい |
| 権限別アクセス可否 | 自動化を推奨 | 期待結果が明確で回帰確認に向く |
| ユーザビリティ確認 | 一部自動化を推奨 | 操作ログ取得は自動化できるが、迷いやすさは人の判断が残る |
| 障害復旧訓練 | 条件付きで自動化を推奨 | 手順や証跡取得は自動化できるが、環境準備コストが大きい |
| 文言の自然さ | 自動化は非推奨 | 業務文脈に基づく人の判断が重要 |
| ログ設計レビュー | 自動化対象外 | 設計合意・レビューが中心 |
