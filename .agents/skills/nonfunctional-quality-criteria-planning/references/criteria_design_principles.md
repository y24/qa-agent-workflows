# 品質クライテリア設計原則

## 必須要素

品質クライテリアには、可能な限り次の要素を含める。

| 要素 | 内容 |
|---|---|
| 対象 | 機能、画面、処理、データ、運用、構成 |
| 条件 | 利用者、環境、データ量、時間帯、操作状態 |
| 期待品質 | 満たすべき品質状態 |
| 判定基準 | 合格、不合格、許容範囲、保留条件 |
| 測定・確認方法候補 | 測定、レビュー、静的解析、運用確認など |
| 根拠 | 入力文書、前Step ID、ユーザー指定 |
| 未確定事項 | 合意、調査、追加情報が必要な事項 |

## 書き方

- 「速い」「安全」「使いやすい」だけで終わらせない。
- 「どの条件で、何が、どの程度なら許容できるか」を書く。
- 数値基準は、入力に根拠がある場合だけ断定する。
- 数値が未確定でも、測定対象、条件、合意すべき項目は書く。
- 1つのクライテリアに複数の品質特性や対象を詰め込みすぎない。

## 良い例

| 悪い表現 | 改善例 |
|---|---|
| 月次集計画面は速いこと | 月次集計画面は、合意済みの本番相当データ量で検索結果表示が業務を阻害しない時間内に完了すること |
| セキュリティを担保すること | 権限を持たない利用者が対象データを閲覧・更新できず、拒否結果が監査可能な証跡として残ること |
| 保守しやすいこと | 障害調査時に、対象処理、利用者、時刻、エラー種別を特定できるログが参照できること |

## 未確定事項の扱い

- 目標値が未定なら、`判定基準: 要確認` とし、決めるべき項目を明示する。
- 本番相当データ量が不明なら、仮値を断定せず、データ量確認を確認事項にする。
- 環境条件が未定なら、対象ブラウザ、OS、外部接続、バージョンを確認事項にする。
