# risk_and_concern_scale_definition.md

# 品質懸念の評価基準

## 評価軸

| 評価軸 | 1 | 2 | 3 |
|---|---|---|---|
| 業務影響 | 軽微、回避容易 | 一部業務に影響 | 業務停止、重大な顧客・監査・法令影響 |
| 利用頻度 | 低頻度 | 定期利用 | 日常的またはピーク時に集中利用 |
| 発生可能性 | 根拠上低い | 条件次第で発生 | 変更量、複雑性、過去障害などから高い |
| 検知しやすさ | 通常確認で見つけやすい | 条件設定が必要 | 本番相当環境、長時間、専門確認が必要 |
| 復旧・回避しやすさ | 即時回避可能 | 手順があれば回避可能 | 復旧困難、データ・信頼への影響が残る |

必要に応じて、顧客・監査・法令影響を補正理由として別記する。

## 優先度

| 優先度 | 意味 | 扱い |
|---|---|---|
| P0 | 最重点 | 品質クライテリア化し、確認方針を明確にする |
| P1 | 重点 | 原則クライテリア化する |
| P2 | 標準 | テスト観点として保持し、必要に応じて一部クライテリア化する |
| P3 | 低 | 簡易確認、保留、対象外候補として扱う |

## クライテリア化判断

| 判断 | 条件 |
|---|---|
| クライテリア化 | P0/P1、または合否判断を明確にしないと受入・計画に支障が出る |
| テスト観点保持 | 懸念はあるが、既存テストやレビュー内で確認できる |
| 保留・確認待ち | 判定に必要な情報が不足している |
| 対象外 | 機能適合性、プロジェクトリスク、今回範囲外 |

## 補正ルール

- 情報漏えい、監査不備、業務停止などはスコアが中程度でも優先度を上げてよい。
- 根拠信頼度が低い懸念は、高優先度にする場合でも `仮説` または `要確認` を残す。
- 低頻度でも復旧困難なものは、保留にせず確認対象として残す。
