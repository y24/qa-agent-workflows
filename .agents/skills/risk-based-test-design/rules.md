# リスクベースドテスト設計 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、トレーサビリティ、入力文書、テスト設計粒度は `../../../shared/` 配下の方針に従う。
このファイルでは `$risk-based-test-design` 固有の責務、ID、禁止事項を定義する。

## 目的

このskillで作成する入力整理、リスク候補、リスク評価、テスト方針、テストケース骨子に共通するルールを定義する。

## 基本方針

- 入力文書、前Step成果物、ユーザー指定を根拠にする。
- 事実、推測、仮説、確認事項、未確定事項を分けて扱う。
- リスクは「起きると困ること」として書く。
- テスト観点は「リスクを検出・軽減するために確認すべきこと」として書く。
- テストケース骨子は詳細テストケースの一歩手前の粒度にする。
- 高リスク領域にテストを集中させ、低リスク領域を過剰に詳細化しない。

## 厳守ルール

### 根拠のない断定をしない

入力にない仕様、業務ルール、画面操作、期待結果、テストデータを事実として創作しない。

### 一般論だけでリスクを作らない

性能、セキュリティ、権限、データ不整合などの一般論は、対象文書、変更点、制約、過去障害などの根拠がある場合に扱う。

### リスクとテスト観点を混同しない

| 種別 | 書き方 |
|---|---|
| リスク | 金額計算の丸め誤りにより、請求金額が不正になるリスク |
| テスト観点 | 税率、端数処理、対象期間の組合せごとに金額計算結果を確認する |

### リスクは分割して扱う

入力、保存、承認、帳票出力、外部連携など異なる問題を1件に混ぜない。評価やテスト方針に変換できる単位にする。

### プロダクトリスクを中心に扱う

このskillの中心は、システム品質や業務結果に影響するプロダクトリスクである。期間不足、体制不足、環境不足などはテスト制約またはプロジェクトリスクとして分ける。

## ID体系

| 対象 | 形式 |
|---|---|
| Flow | `F-001` |
| Change | `C-001` |
| Constraint | `CON-001` |
| Question | `Q-001` |
| Risk | `R-001` |
| Hypothesis Risk | `RH-001` |
| Viewpoint | `V-001` |
| Test Case | `TC-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## 出力ルール

- 出力はMarkdownを基本にする。
- 表は列名を安定させ、後続Stepで参照しやすくする。
- 各成果物には入力・参照元、主成果物、確認事項、次Stepへの引き継ぎを含める。
- 根拠は文書名、章、節、項目名、前StepのIDで追跡できるようにする。
- 不明点は補完せず、`要確認`、`情報不足`、`未確定`、`対象外` などで状態を示す。

## Step進行ルール

- Step 1ではリスク候補を作らない。
- Step 2ではリスク評価やテストケースを作らない。
- Step 3ではテスト観点やテストケースを作らない。
- Step 4では詳細テストケースや具体値を作らない。
- Step 5では詳細な画面操作、具体値、文言完全一致に踏み込みすぎない。
- 各Stepの出力後は停止し、ユーザーの続行指示を待つ。

## 品質基準

- 追跡性: 対象範囲、リスク、評価、テスト観点、テストケース骨子がIDで追跡できる。
- 妥当性: リスクの悪影響、評価理由、テスト方針の対応が説明できる。
- 重点化: P0/P1が厚く扱われ、P2/P3が過剰に詳細化されていない。
- 透明性: 未確認事項、対象外、未カバー、残存リスクが隠れていない。

## 禁止事項

- 入力にない仕様や期待結果を断定する
- すべてのリスクを高優先度にする
- 根拠が弱い重大リスクを黙って削除する
- スコアだけで機械的に優先度を決める
- リスクに対応しないテストケースを大量に作る
- 未カバーリスクを成果物から消す
