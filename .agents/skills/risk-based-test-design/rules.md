# リスクベースドテスト設計 共通ルール

## 1. 目的

このプロンプト群は、プロジェクト計画書、要件定義書、設計書、画面仕様、外部IF仕様、ドメイン知識、過去障害などの開発ドキュメントを入力として、リスクベースドテスト設計を段階的に行うためのものである。

最終的な目的は、以下を明確にすることである。

- どのようなプロダクトリスクが存在するか
- どのリスクを重点的にテストすべきか
- 各リスクに対して、どのようなテスト観点・テストケースが必要か
- リスク、テスト観点、テストケースの対応関係が追跡可能であるか

---

## 2. 基本方針

リスクベースドテスト設計では、すべてを均等にテストしようとしない。

限られた時間・人員・環境・データ条件の中で、影響が大きく、発生しやすく、検出しにくいリスクに対して、より厚いテストを割り当てる。

ただし、リスクの評価やテスト方針は、入力文書・過去障害・仕様変更内容・業務上の重要性など、確認可能な根拠に基づいて行う。

---

## 3. 厳守ルール

### 3.1 根拠のない断定をしない

入力文書に記載されていない内容を、事実として扱ってはならない。

根拠が弱い内容は、以下のいずれかとして明示する。

- 仮説
- 推測
- 確認事項
- 前提条件
- 未確定事項

悪い例:

```text
この機能では承認処理が行われるため、権限リスクが高い。
````

良い例:

```text
承認処理に関する明確な記載はないが、業務フロー上、承認または確認行為が存在する可能性がある。
そのため、権限・統制リスクの仮説として扱う。
```

---

### 3.2 一般論だけでリスクを作らない

どのシステムにも当てはまる一般的なリスクを、根拠なしに混入させてはならない。

たとえば、以下のようなリスクは、根拠がなければ採用しない。

* 性能が劣化するリスク
* 権限が不正になるリスク
* データが壊れるリスク
* 外部連携に失敗するリスク
* 画面表示が崩れるリスク

ただし、入力文書に以下のような情報がある場合は、リスク候補として扱ってよい。

* 大量データ処理がある
* 権限やロールの変更がある
* データ移行や集計処理がある
* 外部システム連携がある
* UI変更やブラウザ・OS差異がある
* 過去に類似障害が発生している

---

### 3.3 リスクとテスト観点を混同しない

リスクは「起きると困ること」である。
テスト観点は「そのリスクを検出・軽減するために確認すべきこと」である。

例:

| 種別     | 内容                                  |
| ------ | ----------------------------------- |
| リスク    | 締め処理後にデータ修正できてしまい、確定済み帳票とデータが不整合になる |
| テスト観点  | 締め状態ごとの編集可否、取消可否、再集計可否を確認する         |
| テストケース | 締め済み期間の明細を修正しようとした場合、エラーとなり保存されないこと |

---

### 3.4 リスクは分割して扱う

1つのリスクに複数の論点を詰め込まない。

悪い例:

```text
入力、保存、承認、帳票出力が正しく行われないリスク
```

良い例:

```text
R-001 入力値の桁数超過により保存時にエラーとなるリスク
R-002 承認前データが帳票出力対象に含まれるリスク
R-003 承認後の修正により帳票と明細が不整合になるリスク
```

---

### 3.5 プロダクトリスクを中心に扱う

本プロンプト群では、主にプロダクトリスクを扱う。

プロダクトリスクとは、システムや成果物に欠陥が存在した場合に、業務・利用者・データ・運用・品質へ悪影響を与えるリスクである。

例:

* 計算結果が誤る
* 権限外の操作ができる
* 必要なデータが帳票に出力されない
* 外部システムとの連携結果が不整合になる
* 過去障害が再発する

プロジェクトリスクは、テスト設計の制約として扱う。

例:

* テスト期間が短い
* テスト環境が不足している
* 仕様確定が遅れている
* 有識者レビューを受けにくい
* テストデータ準備が難しい

---

## 4. リスク分類

リスク分類は、原則として以下を使用する。

| 分類       | 内容                                 |
| -------- | ---------------------------------- |
| 業務リスク    | 業務処理、判断、承認、締め、会計処理などに悪影響が出るリスク     |
| 機能リスク    | 入力、保存、検索、更新、削除、計算、表示などが正しく動作しないリスク |
| データリスク   | データ不整合、欠落、重複、桁あふれ、丸め誤差、履歴不整合などのリスク |
| 連携リスク    | 外部IF、バッチ、API、ファイル連携、他システム連携に関するリスク |
| 権限・統制リスク | 権限逸脱、承認漏れ、監査証跡不足、不正操作などのリスク        |
| 非機能リスク   | 性能、可用性、操作性、互換性、保守性、セキュリティなどのリスク    |
| 回帰リスク    | 既存機能への影響、過去障害の再発、仕様変更による副作用のリスク    |
| 運用リスク    | 運用手順、リカバリ、例外対応、締め日運用、手作業介入に関するリスク  |

分類に迷う場合は、最も主要な分類を1つ選び、必要に応じて補足に副分類を記載する。

---

## 5. リスク評価の基本方針

リスク評価では、原則として以下の観点を使用する。

| 評価軸   | 内容                                    |
| ----- | ------------------------------------- |
| 影響度   | リスクが顕在化した場合の業務・利用者・金額・信頼性・法令対応などへの影響  |
| 発生可能性 | 仕様の複雑さ、変更量、実装難度、過去障害、利用頻度などから見た起こりやすさ |
| 検出困難度 | 通常の確認で見逃しやすいか、複合条件や特殊データが必要か          |
| 根拠信頼度 | 入力文書・過去障害・仕様記述など、判断根拠がどの程度明確か         |

評価点は、各Stepのガイドまたは scoring_model.md に従う。

---

## 6. 出力ルール

### 6.1 Markdownで出力する

出力はMarkdown形式を基本とする。
表形式で整理できるものは、原則としてMarkdown表を使う。

### 6.2 IDを付与する

リスク、テスト観点、テストケースにはIDを付与する。

推奨ID体系:

| 成果物    | ID例    |
| ------ | ------ |
| リスク    | R-001  |
| テスト観点  | V-001  |
| テストケース | TC-001 |
| 確認事項   | Q-001  |

### 6.3 トレーサビリティを維持する

以下の対応関係を維持する。

```text
入力文書
  → リスク
    → テスト観点
      → テストケース
```

テストケースは、少なくとも1つ以上のリスクまたはテスト観点に紐づいている必要がある。

---

## 7. 推測・確認事項の扱い

不明点がある場合、作業を止めずに以下のように扱う。

| 状態           | 扱い                   |
| ------------ | -------------------- |
| 根拠が明確        | 通常のリスク・観点として扱う       |
| 根拠が一部ある      | 根拠信頼度を中または低として扱う     |
| 根拠がないが重要そう   | 仮説リスクとして扱う           |
| 判断不能         | 確認事項として記録する          |
| テスト設計に影響が大きい | 未解決リスクまたは保留事項として記録する |

---

## 8. ステップ進行ルール

このプロンプト群は、段階的に実行する。

各ステップの成果物を出力したら、そこで停止する。
ユーザーから明示的に次の指示があるまで、次ステップへ進んではならない。

許可される次ステップ指示の例:

```text
次へ
次をお願いします
Step 2に進んでください
この内容で続けてください
```

---

## 9. 各ステップの概要

| Step   | 内容                  | 主な成果物                             |
| ------ | ------------------- | --------------------------------- |
| Step 1 | 入力文書整理・分析スコープ確定     | input_scope_summary               |
| Step 2 | リスク候補抽出             | risk_candidate_list               |
| Step 3 | リスク評価・優先順位付け        | risk_register                     |
| Step 4 | リスクベースドテスト方針設計      | risk_based_test_strategy          |
| Step 5 | テストケース骨子・トレーサビリティ作成 | testcase_outline_and_traceability |

---

## 10. 品質基準

各ステップの出力は、以下を満たす必要がある。

* 入力文書に基づいている
* 推測と事実が分離されている
* リスク、観点、テストケースの粒度が揃っている
* 重要度の判断理由が説明されている
* 確認事項が明示されている
* 次ステップで利用しやすい形式になっている
* 不要に長大化していない

---

## 11. 禁止事項

以下を禁止する。

* 入力にない仕様を断定する
* 一般論だけでリスクを水増しする
* リスクとテストケースを直接混同する
* すべてのリスクを同じ重要度にする
* 重要度の理由を書かずにスコアだけを出す
* 未確認事項を勝手に補完する
* 1つのリスクに複数の異なる問題を詰め込む
* ユーザーのレビューなしに次ステップへ進む
* 詳細テスト手順を早すぎる段階で作成する

---

## 12. 最終成果物の考え方

最終的には、以下の成果物が揃うことを目指す。

```text
01_input_scope_summary.md
02_risk_candidate_list.md
03_risk_register.md
04_risk_based_test_strategy.md
05_testcase_outline_and_traceability.md
```

これらにより、リスク抽出からテストケース骨子までを、根拠付きで追跡可能な形にする。
