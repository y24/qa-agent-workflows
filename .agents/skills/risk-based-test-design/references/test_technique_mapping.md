# テスト技法マッピングリファレンス

## 1. 目的

このファイルは、リスクの性質に応じて、適切なテスト観点・テスト技法を選定するためのリファレンスである。

主に Step 4: リスクベースドテスト方針設計 で使用する。

このファイルの目的は、リスクをテストケースへ直接変換することではない。  
まず、リスクに対して有効なテスト観点・テスト技法・確認条件を整理するために使用する。

---

## 2. 基本方針

テスト技法は、技法名を付けること自体が目的ではない。

重要なのは、以下である。

- そのリスクを検出しやすい確認条件を作ること
- そのリスクに対して過不足のないテスト深度を決めること
- 高リスク領域にテストを集中させること
- 低リスク領域を必要以上に詳細化しないこと

技法は、リスクの性質に応じて選ぶ。  
1つのリスクに複数の技法を組み合わせてもよい。

---

## 3. リスクの性質と推奨技法

| リスクの性質 | 主な確認対象 | 推奨技法 |
|---|---|---|
| 入力値、金額、数量、日付、期間 | 入力範囲、桁数、境界、単位 | 同値分割、境界値分析 |
| 条件分岐、判定、権限、承認可否 | 条件の組合せ、可否判断 | デシジョンテーブル |
| 状態、ステータス、締め、取消、再実行 | 状態遷移、遷移可否、状態別動作 | 状態遷移テスト |
| 業務フロー、複数機能の連鎖 | 一連の業務処理、後続影響 | シナリオテスト |
| 複数条件の組合せ | 条件組合せ、パターン圧縮 | 組合せテスト、ペアワイズ |
| 過去障害、経験則上の弱点 | 再発確認、抜け漏れ確認 | エラー推測、回帰テスト |
| 帳票、集計、計算 | 期待値、集計結果、差分 | 期待値検証、突合、差分確認 |
| 外部IF、バッチ、API | 連携成功、連携失敗、再実行 | インターフェーステスト、異常系テスト、再実行テスト |
| 性能、処理時間、大量データ | 応答時間、処理時間、スループット | 性能テスト、負荷テスト、ボリュームテスト |
| 操作性、メッセージ、運用 | 利用者判断、復旧容易性 | ユーザビリティ確認、運用シナリオ確認 |
| 互換性、環境差異 | OS、ブラウザ、Excel、PDF、端末差異 | 互換性テスト、構成テスト |
| セキュリティ、権限、情報保護 | 不正アクセス、情報露出、監査証跡 | 権限テスト、セキュリティ観点確認 |

## 4. 同値分割

## 4.1 適用するリスク

同値分割は、入力値や条件値を代表的なグループに分けて確認する技法である。

以下のリスクに向いている。

- 入力値の範囲誤り
- 必須・任意の扱い誤り
- 金額、数量、日付、期間の扱い誤り
- コード値、区分値の扱い誤り
- 有効値・無効値の判定誤り
- マスタ存在有無の扱い誤り

## 4.2 確認観点

- 有効な値が受け付けられるか
- 無効な値が拒否されるか
- 未入力、NULL、空白の扱いが正しいか
- 存在するマスタ値と存在しないマスタ値で動作が正しいか
- 有効な区分と無効な区分で動作が正しいか

## 4.3 テスト観点例

```text
有効な勘定科目コード、存在しない勘定科目コード、使用停止中の勘定科目コードを入力した場合の登録可否を確認する。
````

```text
有効な対象期間、対象外期間、未入力期間を指定した場合の検索・集計結果を確認する。
```

## 5. 境界値分析

## 5.1 適用するリスク

境界値分析は、範囲や桁数の境界付近で不具合が発生しやすい場合に使用する。

以下のリスクに向いている。

* 最大桁数・最小桁数の誤り
* 金額、数量、比率、日付の境界誤り
* 期間開始日・終了日の扱い誤り
* 上限値・下限値の扱い誤り
* 0、負数、小数、最大値の扱い誤り
* 年度末、月末、うるう年などの日付境界誤り

## 5.2 確認観点

* 最小値
* 最小値未満
* 最大値
* 最大値超過
* 0
* 1
* -1
* 小数
* 月初
* 月末
* 年度末
* 期間開始日
* 期間終了日
* 期間外の日付

## 5.3 テスト観点例

```text
金額項目について、0、1、最大桁数、最大桁数超過、小数、負数の扱いを確認する。
```

```text
対象期間の開始日、終了日、期間外日付を指定した場合に、集計対象が正しく判定されることを確認する。
```

## 6. デシジョンテーブル

## 6.1 適用するリスク

デシジョンテーブルは、複数条件の組合せによって結果が変わる場合に使用する。

以下のリスクに向いている。

* 権限による操作可否の誤り
* 承認状態による操作可否の誤り
* 締め状態による編集可否の誤り
* 条件分岐の漏れ
* 複数フラグの組合せ誤り
* 業務区分ごとの処理差異
* 入力条件による計算ロジックの切替誤り

## 6.2 確認観点

* 条件の組合せが網羅されているか
* 結果が一意に決まるか
* 矛盾する条件がないか
* 未定義の条件組合せがないか
* 禁止すべき操作が許可されていないか
* 許可すべき操作が拒否されていないか

## 6.3 テスト観点例

```text
ユーザー権限、申請状態、承認状態の組合せごとに、登録・更新・承認・取消の可否を確認する。
```

```text
取引区分、対象期間、税区分の組合せごとに、金額計算ロジックが正しく切り替わることを確認する。
```

## 7. 状態遷移テスト

## 7.1 適用するリスク

状態遷移テストは、データや業務対象が状態を持ち、操作によって状態が変化する場合に使用する。

以下のリスクに向いている。

* 申請、承認、差戻し、取消の状態遷移誤り
* 締め前、締め中、締め後の操作制御誤り
* 再実行、再承認、再締めの扱い誤り
* 状態不整合
* 禁止遷移が可能になる
* 必要な遷移ができない
* 状態変更後の後続処理が誤る

## 7.2 確認観点

* 正常な状態遷移ができるか
* 禁止された状態遷移が拒否されるか
* 状態ごとの操作可否が正しいか
* 状態変更後の表示・出力・連携結果が正しいか
* 取消や差戻し後の再操作が正しいか
* 異常終了時に中間状態が残らないか

## 7.3 テスト観点例

```text
未申請、申請中、承認済み、差戻し、取消済みの各状態で、更新・承認・差戻し・取消の可否を確認する。
```

```text
締め前、締め後、締め取消後、再締め後の各状態で、明細修正と帳票出力結果が正しく制御されることを確認する。
```

## 8. シナリオテスト

## 8.1 適用するリスク

シナリオテストは、複数機能をまたぐ業務フロー全体の整合性を確認する場合に使用する。

以下のリスクに向いている。

* 業務フロー全体での不整合
* 前工程のデータが後工程で誤って扱われる
* 利用者の実業務に即した操作で不具合が発生する
* 複数機能の連携で結果が破綻する
* 例外処理や再処理を含む業務で不整合が起きる
* 単機能テストでは検出しにくい業務上の問題

## 8.2 確認観点

* 実業務の流れに沿って処理できるか
* 前工程の結果が後工程に正しく反映されるか
* 状態やデータが一貫しているか
* 例外処理後に業務を再開できるか
* 帳票や連携結果まで一貫して正しいか
* 利用者が自然な手順で操作できるか

## 8.3 テスト観点例

```text
データ登録から承認、締め処理、帳票出力までの一連の業務フローで、状態・金額・出力結果が一貫していることを確認する。
```

```text
外部データ取込後、エラー修正、再取込、集計、帳票出力までを通して、重複や欠落が発生しないことを確認する。
```

## 9. 組合せテスト・ペアワイズ

## 9.1 適用するリスク

組合せテストやペアワイズは、複数の条件が絡むが、すべての組合せをテストするのが現実的でない場合に使用する。

以下のリスクに向いている。

* 複数条件の組合せで発生する不具合
* 権限、状態、区分、期間、データ種別の組合せ
* ブラウザ、OS、権限、設定値の組合せ
* 帳票条件や検索条件の組合せ
* すべての組合せを網羅すると件数が爆発する場合

## 9.2 確認観点

* 重要な条件が組合せに含まれているか
* 高リスクな組合せを優先しているか
* ペアワイズで十分か、特定の3因子以上の組合せが必要か
* 禁止条件や業務上重要な条件を落としていないか
* P0/P1リスクに関わる組合せが薄くなりすぎていないか

## 9.3 テスト観点例

```text
権限、データ状態、処理区分、対象期間の組合せから、代表的かつ高リスクな条件を選定して操作可否を確認する。
```

```text
ブラウザ、Excelバージョン、帳票形式の組合せについて、主要環境を優先して出力結果を確認する。
```

## 10. エラー推測

## 10.1 適用するリスク

エラー推測は、過去障害、経験則、仕様の複雑さ、設計上の弱点から、起こりやすい不具合を狙って確認する技法である。

以下のリスクに向いている。

* 過去障害に類似する不具合
* 仕様書に明記されにくい例外条件
* 利用者がしがちな誤操作
* 開発者が見落としやすい境界条件
* 業務有識者が経験的に危ないと感じる箇所
* 複雑な運用手順の抜け漏れ

## 10.2 確認観点

* 過去に似た障害がないか
* よくある入力ミスや操作ミスはないか
* 仕様の抜けや曖昧な箇所はないか
* 例外時の動作は定義されているか
* 開発者が正常系中心に実装しそうな箇所はないか
* 業務上「実際には起こるが仕様に書かれにくい」条件はないか

## 10.3 テスト観点例

```text
過去に締め後修正で不整合が発生しているため、締め後・取消後・再締め後の修正操作を重点的に確認する。
```

```text
利用者が誤って同じ連携ファイルを再取込する可能性を想定し、重複登録が防止されることを確認する。
```

## 11. 回帰テスト

## 11.1 適用するリスク

回帰テストは、変更によって既存機能が壊れていないことを確認する場合に使用する。

以下のリスクに向いている。

* 共通処理変更による既存機能影響
* 既存帳票の出力差異
* 既存IFの変更影響
* 既存データとの互換性破綻
* 過去障害の再発
* 変更対象外機能への副作用

## 11.2 確認観点

* 変更対象外の主要機能が従来どおり動作するか
* 共通処理を利用する機能に副作用がないか
* 既存帳票や既存出力の差分が許容範囲か
* 過去障害の再発条件を確認しているか
* 既存データでエラーが起きないか
* 既存運用手順が破綻しないか

## 11.3 テスト観点例

```text
共通計算ロジック変更後も、変更対象外の帳票で既存期待値と一致することを確認する。
```

```text
過去障害で発生した承認後修正の条件を再実行し、同種不具合が再発しないことを確認する。
```

## 12. 期待値検証・突合・差分確認

## 12.1 適用するリスク

期待値検証、突合、差分確認は、計算結果、集計結果、帳票、連携ファイルなどの出力結果が正しいかを確認する場合に使用する。

以下のリスクに向いている。

* 金額計算誤り
* 集計対象の過不足
* 帳票出力誤り
* 丸め誤差
* 連携ファイル内容の誤り
* 明細と合計の不一致
* 変更前後の意図しない差分

## 12.2 確認観点

* 明細と合計が一致するか
* 集計条件どおりに対象が選ばれているか
* 丸め、桁数、符号、単位が正しいか
* 帳票と画面表示が一致するか
* 出力ファイルと登録データが一致するか
* 変更前後の差分が意図したものだけか
* 期待値を独立した方法で算出できるか

## 12.3 テスト観点例

```text
明細データから独立計算した期待値と、帳票の合計金額が一致することを確認する。
```

```text
変更前後の帳票出力を比較し、仕様変更対象外の項目に差分が発生していないことを確認する。
```

## 13. インターフェーステスト

## 13.1 適用するリスク

インターフェーステストは、外部システム、API、ファイル、バッチなどの連携仕様を確認する場合に使用する。

以下のリスクに向いている。

* 連携項目の欠落
* 項目型、桁数、形式の不一致
* 必須項目の扱い誤り
* 連携失敗時の処理誤り
* 再送・再実行時の重複
* 連携順序による不整合
* 文字コードや改行コードの差異
* 外部システム変更の影響

## 13.2 確認観点

* 正常な連携データを処理できるか
* 不正な連携データを検知できるか
* 必須項目欠落時の扱いが正しいか
* 形式不正、桁数超過、型不一致を検知できるか
* 再送・再実行で重複登録されないか
* 連携失敗時にエラー内容を追跡できるか
* 一部成功・一部失敗時の扱いが明確か

## 13.3 テスト観点例

```text
外部IFファイルに必須項目欠落、桁数超過、形式不正が含まれる場合に、エラー検知され、不完全なデータが登録されないことを確認する。
```

```text
同一連携ファイルを再実行した場合に、重複登録されず、処理結果が追跡可能であることを確認する。
```

## 14. 性能テスト・負荷テスト・ボリュームテスト

## 14.1 適用するリスク

性能系テストは、大量データ、同時利用、処理時間制約などがある場合に使用する。

以下のリスクに向いている。

* 応答時間が許容値を超える
* バッチ処理が業務開始までに終わらない
* 大量データでタイムアウトする
* 同時利用時に処理が不安定になる
* 帳票出力や集計処理が遅延する
* 月末・年度末の利用ピークに耐えられない

## 14.2 確認観点

* 想定データ量で処理できるか
* ピーク時の同時利用に耐えられるか
* バッチ処理が許容時間内に完了するか
* 大量データでタイムアウトしないか
* 処理遅延時に利用者が状況を判断できるか
* 性能劣化時の運用回避策があるか

## 14.3 テスト観点例

```text
月末相当の大量データを用いて集計処理を実行し、許容時間内に完了することを確認する。
```

```text
複数ユーザーが同時に検索・登録を行った場合に、応答時間とデータ整合性が維持されることを確認する。
```

## 15. ユーザビリティ確認・運用シナリオ確認

## 15.1 適用するリスク

ユーザビリティ確認や運用シナリオ確認は、利用者が迷わず操作・判断・復旧できるかを確認する場合に使用する。

以下のリスクに向いている。

* エラーメッセージが分かりにくい
* 操作手順が複雑で誤操作しやすい
* 運用手順と画面動作が合わない
* 異常時に利用者が次の対応を判断できない
* 手作業補正が必要だが判断材料が不足している
* 業務タイミングに合わない操作フローになっている

## 15.2 確認観点

* 利用者が次に何をすべきか分かるか
* エラー原因と対処が分かるか
* 操作順序が業務実態に合っているか
* 誤操作を防止できるか
* 重要操作の確認や警告が適切か
* 運用手順書と画面動作が整合しているか

## 15.3 テスト観点例

```text
連携エラー発生時に、利用者がエラー原因と再実行可否を判断できるメッセージ・ログが表示されることを確認する。
```

```text
月末締め処理の運用手順に沿って、担当者が迷わず処理状態を確認し、必要な後続操作を実行できることを確認する。
```

## 16. 互換性テスト・構成テスト

## 16.1 適用するリスク

互換性テストや構成テストは、利用環境や構成差異によって不具合が発生する可能性がある場合に使用する。

以下のリスクに向いている。

* OS差異
* ブラウザ差異
* Excelバージョン差異
* PDF出力環境差異
* 画面解像度差異
* 文字コード・フォント差異
* クライアント設定差異
* 外部ツールやアドイン差異

## 16.2 確認観点

* サポート対象環境で動作するか
* 主要環境を優先して確認しているか
* 環境差異で表示・出力・操作に問題がないか
* ExcelやPDFなど外部アプリ連携に問題がないか
* 非サポート環境の扱いが明確か
* すべての組合せを確認できない場合、優先順位が妥当か

## 16.3 テスト観点例

```text
サポート対象の主要ブラウザで、帳票出力操作とダウンロード結果が同じになることを確認する。
```

```text
Excelアドインを使用する環境で、対象バージョンごとの起動・入力・保存・出力結果を確認する。
```

## 17. 権限テスト・セキュリティ観点確認

## 17.1 適用するリスク

権限テストやセキュリティ観点確認は、権限外操作、不正参照、情報露出、監査証跡不足などを確認する場合に使用する。

以下のリスクに向いている。

* 権限外参照
* 権限外更新
* 承認者以外による承認
* 職務分離違反
* 操作履歴不足
* 機密情報の表示
* URL直打ちやAPI直接呼び出しによる制御回避
* 特権ユーザー操作の追跡不能

## 17.2 確認観点

* ロールごとの参照範囲が正しいか
* 操作権限が正しく制御されるか
* 画面上の非表示だけでなく、直接アクセスも拒否されるか
* 承認者と申請者の分離が保たれるか
* 操作履歴や監査ログが残るか
* エラー時に機密情報が露出しないか

## 17.3 テスト観点例

```text
一般ユーザー、承認者、管理者それぞれの権限で、参照・登録・更新・削除・承認の可否が仕様どおり制御されることを確認する。
```

```text
権限のないユーザーがURL直接指定またはAPI直接呼び出しを行った場合に、対象データへアクセスできないことを確認する。
```

## 19. 優先度別の技法選定方針

## 19.1 P0リスク

P0リスクでは、単一技法だけに頼らず、複数の観点で厚く確認する。

例:

* シナリオテスト
* 状態遷移テスト
* デシジョンテーブル
* 境界値分析
* 回帰テスト
* 異常系テスト
* 期待値検証

P0では、リスクを検出できる条件を明示することを優先する。

---

## 19.2 P1リスク

P1リスクでは、主要な正常系、代表的な異常系、重要な境界条件を確認する。

例:

* デシジョンテーブル
* 境界値分析
* シナリオテスト
* 回帰テスト
* 期待値検証

---

## 19.3 P2リスク

P2リスクでは、代表条件を中心に確認する。
必要に応じて、P0/P1のテストケースに統合してよい。

例:

* 同値分割
* 代表的な境界値
* 基本的な正常系・異常系
* 探索的確認

---

## 19.4 P3リスク

P3リスクでは、簡易確認、他テストへの統合、探索的確認、または対象外を検討する。

例:

* 簡易確認
* 目視確認
* レビュー確認
* 他ケースでの同時確認
* 残存リスクとして記録

---

## 20. 技法選定時の注意

## 20.1 技法を並べるだけにしない

悪い例:

```text
同値分割、境界値分析、デシジョンテーブルで確認する。
```

良い例:

```text
金額項目は桁数・0・負数・小数で誤りが起きやすいため、同値分割と境界値分析で代表値と境界値を確認する。
```

---

## 20.2 リスクの性質と技法を対応させる

技法は、リスクの性質に基づいて選ぶ。

例:

| リスク                | 適した技法              |
| ------------------ | ------------------ |
| 締め状態ごとの操作可否が誤る     | 状態遷移テスト            |
| 権限と承認状態の組合せで可否が変わる | デシジョンテーブル          |
| 金額の最大桁数で保存に失敗する    | 境界値分析              |
| 外部IF再実行で重複登録される    | インターフェーステスト、再実行テスト |
| 過去障害と同種の問題が再発する    | 回帰テスト、エラー推測        |

---

詳細な操作手順、具体的な入力値、期待結果の詳細は Step 5 または後続工程で扱う。
