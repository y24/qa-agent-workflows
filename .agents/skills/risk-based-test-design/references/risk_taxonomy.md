# リスク分類リファレンス

## 1. 目的

このファイルは、リスクベースドテスト設計で使用するリスク分類を定義する。

Step 2 のリスク候補抽出、および Step 3 のリスク評価で参照する。

リスク分類の目的は、抽出したリスクを整理し、見落としや偏りを防ぐことである。  
分類そのものが目的ではない。

---

## 2. 基本方針

リスク分類は、原則として以下の8分類を使用する。

| 分類 | 概要 |
|---|---|
| 業務リスク | 業務処理・業務判断・業務結果に悪影響が出るリスク |
| 機能リスク | システム機能が仕様どおりに動作しないリスク |
| データリスク | データの不整合、欠落、重複、誤りに関するリスク |
| 連携リスク | 外部IF、API、バッチ、ファイル連携に関するリスク |
| 権限・統制リスク | 権限、承認、監査証跡、不正操作に関するリスク |
| 非機能リスク | 性能、可用性、操作性、互換性、セキュリティなどのリスク |
| 回帰リスク | 既存機能への副作用、過去障害の再発に関するリスク |
| 運用リスク | 運用手順、例外対応、リカバリ、締め日運用に関するリスク |

1つのリスクに複数分類が当てはまる場合は、主分類を1つ選ぶ。  
必要に応じて、副分類を補足欄に記載する。

---

## 3. 業務リスク

## 3.1 定義

業務リスクとは、システムの不具合や仕様不備により、業務処理、業務判断、業務結果に悪影響が出るリスクである。

## 3.2 典型例

- 業務上必要な処理が完了できない
- 誤った業務判断につながる情報が表示される
- 承認前の情報が確定情報として扱われる
- 締め後の業務結果が変わってしまう
- 業務フローの順序を無視できてしまう
- 取消・差戻し・再実行の扱いが誤る
- 後続業務に誤ったデータが引き継がれる

## 3.3 リスク記述例

```text
承認前の申請データが確定済みデータとして集計され、業務判断に誤りが生じるリスク
````

```text
締め処理後に明細データを修正できることで、確定済み帳票と実データが不整合になるリスク
```

## 3.4 抽出観点

* 業務結果が誤ると何に影響するか
* 業務フロー上、順序制御が必要な処理はあるか
* 確定、承認、締め、取消、再実行があるか
* 誤った状態のデータが後続業務に流れないか
* 利用者が誤判断する表示や出力がないか

---

## 4. 機能リスク

## 4.1 定義

機能リスクとは、入力、保存、更新、削除、検索、表示、計算、出力などのシステム機能が仕様どおりに動作しないリスクである。

## 4.2 典型例

* 入力チェックが不足している
* 保存条件が誤っている
* 更新対象が誤っている
* 削除できてはいけないデータを削除できる
* 検索条件が正しく反映されない
* 表示内容が実データと一致しない
* 計算結果が誤る
* エラー時の動作が仕様と異なる

## 4.3 リスク記述例

```text
必須項目が未入力でも保存できてしまい、後続処理で必要なデータが欠落するリスク
```

```text
検索条件に指定した対象期間が正しく反映されず、対象外データが一覧に表示されるリスク
```

## 4.4 抽出観点

* 入力値の妥当性確認は十分か
* 保存・更新・削除の条件は明確か
* 検索・表示条件に漏れはないか
* 計算式や判定条件は明確か
* エラー時の動作は定義されているか
* 状態によって操作可否が変わるか

---

## 5. データリスク

## 5.1 定義

データリスクとは、データの整合性、完全性、一意性、正確性、履歴、桁数、単位、期間などに関するリスクである。

## 5.2 典型例

* データが欠落する
* データが重複する
* マスタとトランザクションが不整合になる
* 集計元と集計結果が一致しない
* 丸めや桁数の扱いを誤る
* 日付・期間の境界が誤る
* 履歴や世代管理が破綻する
* 既存データとの互換性が失われる
* データ移行後に値が変わる

## 5.3 リスク記述例

```text
金額の丸め単位が誤ることで、明細合計と帳票合計が一致しなくなるリスク
```

```text
マスタ変更後に過去データへ新マスタ値が誤適用され、過年度帳票の表示内容が変わるリスク
```

## 5.4 抽出観点

* 重要データの欠落・重複が起きないか
* マスタ変更が既存データに影響しないか
* 集計条件は明確か
* 桁数、単位、丸め、符号の扱いは明確か
* 日付・期間の境界条件はあるか
* 履歴、世代、改訂情報を持つか
* データ移行や既存データ互換が必要か

---

## 6. 連携リスク

## 6.1 定義

連携リスクとは、外部システム、API、ファイル、バッチ、メッセージ、帳票出力など、システム間または処理間のデータ受け渡しに関するリスクである。

## 6.2 典型例

* 外部IFの項目定義が一致しない
* 必須項目が連携されない
* 連携ファイルの形式が異なる
* 連携失敗時の扱いが不明
* 再送・再実行で重複データが発生する
* 連携順序に依存して不整合が起きる
* バッチ処理の一部失敗が検知されない
* 外部システム側の変更影響を受ける

## 6.3 リスク記述例

```text
外部システムから受信する取引データの必須項目が欠落した場合に、エラー検知できず不完全なデータが登録されるリスク
```

```text
連携バッチの再実行時に同一データが重複登録され、集計結果が過大になるリスク
```

## 6.4 抽出観点

* 連携項目、型、桁数、必須条件は明確か
* 正常系だけでなく異常系が定義されているか
* 再送、再実行、取消、リカバリがあるか
* 連携順序やタイミングに依存するか
* 連携失敗を検知できるか
* 外部システム側の仕様変更が影響するか

---

## 7. 権限・統制リスク

## 7.1 定義

権限・統制リスクとは、利用者権限、承認、職務分離、監査証跡、不正操作防止などに関するリスクである。

## 7.2 典型例

* 権限外のデータを参照できる
* 権限外の登録・更新・削除ができる
* 申請者が自分で承認できる
* 承認前データを確定できる
* 操作履歴が残らない
* 誰が処理したか追跡できない
* 特権ユーザー操作の制御が不十分
* 権限変更後の既存データアクセスが不適切

## 7.3 リスク記述例

```text
一般ユーザーが管理者権限のデータ更新を実行できてしまい、業務統制が破綻するリスク
```

```text
承認操作の履歴が残らず、監査時に誰が確定判断を行ったか追跡できないリスク
```

## 7.4 抽出観点

* ロールや権限ごとの操作可否は明確か
* 参照、登録、更新、削除、承認で権限制御が異なるか
* 申請者と承認者の分離が必要か
* 操作履歴や監査証跡が必要か
* 特権ユーザーの扱いは明確か
* 権限変更時の影響はあるか

---

## 8. 非機能リスク

## 8.1 定義

非機能リスクとは、機能の正しさ以外の品質特性に関するリスクである。

主に以下を含む。

* 性能
* 可用性
* 信頼性
* 使用性
* 互換性
* 保守性
* セキュリティ
* 運用性

## 8.2 典型例

* 大量データで処理時間が許容値を超える
* バッチが業務開始時刻までに完了しない
* 長時間操作でセッションや状態が破綻する
* 特定ブラウザやOSで表示・操作が崩れる
* エラーメッセージが分かりにくく、利用者が復旧できない
* 障害時に復旧できない
* ログが不足し、原因調査ができない
* セキュリティ上不適切な情報が露出する

## 8.3 リスク記述例

```text
月末の大量データ集計時に処理時間が許容範囲を超え、締め業務が遅延するリスク
```

```text
エラー発生時のメッセージが不十分で、利用者が原因や次の対応を判断できないリスク
```

## 8.4 抽出観点

* 大量データ、ピーク利用、月末・年度末処理があるか
* 応答時間や処理時間の要求があるか
* 利用環境、OS、ブラウザ、Excel、PDFなどの差異があるか
* 障害時の復旧やログ確認が必要か
* 利用者が自己解決すべきエラーがあるか
* セキュリティや情報漏洩の懸念があるか

---

## 9. 回帰リスク

## 9.1 定義

回帰リスクとは、変更によって既存機能や既存データに副作用が発生するリスク、または過去障害が再発するリスクである。

## 9.2 典型例

* 既存機能が動作しなくなる
* 共通部品変更により複数機能に影響が出る
* 既存帳票の出力内容が変わる
* 既存IFの項目や形式が変わる
* 既存データでエラーが発生する
* 過去障害と同種の不具合が再発する
* 旧仕様を前提とした運用が破綻する

## 9.3 リスク記述例

```text
共通計算ロジックの変更により、変更対象外の帳票でも金額計算結果が変わるリスク
```

```text
過去に発生した締め後修正の不具合が、今回の状態管理変更により再発するリスク
```

## 9.4 抽出観点

* 共通処理、共通部品、共通マスタの変更があるか
* 既存機能への影響範囲は明確か
* 既存データでの互換性確認が必要か
* 過去障害と類似した変更があるか
* 既存帳票や既存IFに影響があるか
* 変更対象外とされた範囲に副作用がないか

---

## 10. 運用リスク

## 10.1 定義

運用リスクとは、システム利用時の運用手順、例外対応、リカバリ、手作業介入、締め日運用などに関するリスクである。

## 10.2 典型例

* 運用手順が複雑で誤操作しやすい
* 異常終了時の復旧手順が不明
* 再実行時に重複や欠落が起きる
* 手作業で補正すべき箇所が明確でない
* 締め日や月末だけ発生する特殊処理が漏れる
* 運用でカバーする前提が過剰である
* 障害時に誰が何を判断するか不明
* 本番移行時の切替手順に不整合がある

## 10.3 リスク記述例

```text
バッチ異常終了後の再実行手順が不明確なため、同一データの重複登録または未処理データの残存が発生するリスク
```

```text
月末締め処理時の手作業補正手順が明確でなく、担当者によって処理結果がばらつくリスク
```

## 10.4 抽出観点

* 運用手順が仕様として明記されているか
* 異常終了、取消、再実行、リカバリがあるか
* 手作業介入が必要か
* 締め日、月末、年度末など特殊タイミングがあるか
* 本番移行やデータ切替があるか
* 運用でカバーする前提が妥当か

---

## 11. 分類に迷う場合の判断

分類に迷う場合は、以下の優先順で主分類を選ぶ。

| 判断観点                    | 主分類候補    |
| ----------------------- | -------- |
| 業務結果や業務判断への影響が中心        | 業務リスク    |
| 入力・保存・検索・表示など単一機能の動作が中心 | 機能リスク    |
| データ整合性、集計、履歴、桁数、丸めが中心   | データリスク   |
| 外部システムやバッチ、APIとの受け渡しが中心 | 連携リスク    |
| 権限、承認、監査、職務分離が中心        | 権限・統制リスク |
| 性能、可用性、操作性、互換性などが中心     | 非機能リスク   |
| 変更による既存機能影響が中心          | 回帰リスク    |
| 運用手順、例外対応、再実行、リカバリが中心   | 運用リスク    |

複数にまたがる場合は、最終的に「どの悪影響を最も避けたいか」で判断する。

---

## 12. 分類の粒度に関する注意

分類は、リスクを整理するための補助である。
分類を細かくしすぎると、以下の問題が起きやすい。

* 分類作業に時間を使いすぎる
* リスクの本質が曖昧になる
* 同じリスクが複数分類に重複する
* 後続の評価やテスト設計で扱いにくくなる

そのため、本プロンプト群では8分類を標準とする。

必要に応じて補足欄に副分類を記載してよいが、分類体系そのものを増やしすぎない。

---

## 13. リスク分類チェックリスト

リスク候補抽出後、以下を確認する。

* 業務リスクだけに偏っていないか
* 機能リスクだけに偏っていないか
* データ整合性のリスクを見落としていないか
* 連携・バッチ・外部IFのリスクを見落としていないか
* 権限・承認・監査証跡のリスクを見落としていないか
* 性能・操作性・互換性などの非機能リスクを見落としていないか
* 変更影響・過去障害再発の回帰リスクを見落としていないか
* 運用手順・リカバリ・例外対応のリスクを見落としていないか
* 一般論だけで作られたリスクが混ざっていないか
* 根拠のあるリスクと仮説リスクが分離されているか
