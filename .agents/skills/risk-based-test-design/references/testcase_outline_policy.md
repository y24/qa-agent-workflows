# テストケース骨子ポリシー

## 骨子の粒度

テストケース骨子は、詳細テストケースの一歩手前の粒度にする。

粗すぎる:

```text
締め処理を確認する
```

適切:

```text
締め済み期間の明細を修正しようとした場合、保存が拒否され、確定済み帳票に影響しないことを確認する
```

細かすぎる:

```text
2026/03/31の締め済みデータを開き、明細一覧から部門D001の売上高1000円を1200円に変更し、保存ボタンをクリックした場合、エラーダイアログに特定文言が表示されること
```

具体値、画面操作、文言完全一致は詳細テストケース作成工程で扱う。

## 分割する条件

- 前提条件が大きく異なる
- 期待結果が異なる
- リスクの種類が異なる
- 正常系と異常系を分けたほうが分かりやすい
- 権限や状態の組合せが複雑
- 1ケースに複数目的が混在している
- 実行時に失敗原因を切り分けにくい

## 統合してよい条件

- 同じ業務フローで確認できる
- 同じデータで確認できる
- 同じ期待結果を確認する
- 低優先度リスクを高優先度ケースの中で同時確認できる
- 回帰確認としてまとめられる

統合する場合は、対応Risk ID、Viewpoint ID、統合理由、注意点を記載する。

## カバレッジ

| カバレッジ | 定義 |
|---|---|
| 十分 | 主要な発生条件・影響を検出できるテストケースがある |
| 一部 | 代表条件のみ確認しており、未確認条件が残る |
| 未カバー | 対応するテストケースがない |
| 対象外 | 理由を明示してテスト対象外としている |
| 代替確認 | レビュー、静的確認、運用確認などで扱う |

P0/P1リスクは、原則として `十分` または `一部` 以上にする。未カバー、対象外、代替確認にする場合は理由を明確にする。

## よくある失敗

- 対応Risk IDのない一般的な正常系ケースが増える。
- P0/P1リスクに対応するテストケースがない。
- 1ケースに入力、承認、締め、帳票、外部連携まで詰め込みすぎる。
- 画面名、ボタン名、具体値、メッセージ文言まで詳細に書き始める。
- テストケース化できなかったリスクが成果物から消える。
