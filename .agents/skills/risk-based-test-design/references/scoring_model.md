# リスク評価スコアリングモデル

## 1. 目的

このファイルは、リスクベースドテスト設計におけるリスク評価基準を定義する。

主に Step 3: リスク評価・優先順位付け で使用する。

このスコアリングモデルの目的は、抽出されたリスク候補に対して、テスト設計上の優先度を一貫した基準で判断することである。

---

## 2. 基本方針

リスク評価では、以下の4つの観点を使用する。

| 評価項目 | 用途 | スコア対象 |
|---|---|---|
| 影響度 | リスクが顕在化した場合の悪影響の大きさを評価する | 対象 |
| 発生可能性 | リスクが実際に発生しやすいかを評価する | 対象 |
| 検出困難度 | 通常のテストで見逃しやすいかを評価する | 対象 |
| 根拠信頼度 | 評価判断の根拠がどの程度明確かを表す | スコア対象外 |

リスクスコアは、以下で算出する。

```text
Risk Score = 影響度 × 発生可能性 × 検出困難度
````

根拠信頼度はスコアに掛け算しない。
ただし、評価理由や確認事項、補正判断に必ず反映する。

---

## 3. 評価軸

## 3.1 影響度

影響度は、リスクが顕在化した場合に、業務・利用者・データ・金額・法令・監査・信頼性などへ与える悪影響の大きさを評価する。

| 点数 | 評価    | 判断基準                                               |
| -: | ----- | -------------------------------------------------- |
|  5 | 極めて高い | 業務停止、重大な金額誤り、会計・法令・監査上の重大問題、広範囲の利用者影響、重大な信用失墜につながる |
|  4 | 高い    | 主要業務への大きな影響、重要データの不整合、後続業務への波及、顧客・利用部門への明確な影響がある   |
|  3 | 中     | 一部業務に影響するが、回避策や修正余地がある                             |
|  2 | 低     | 限定的な条件や一部利用者にのみ影響し、業務影響は小さい                        |
|  1 | 極めて低い | 表示上の軽微な問題、業務結果や後続処理にほぼ影響しない                        |

---

## 3.2 発生可能性

発生可能性は、リスクが実際に発生しやすいかを評価する。

| 点数 | 評価    | 判断基準                                         |
| -: | ----- | -------------------------------------------- |
|  5 | 極めて高い | 仕様が非常に複雑、変更量が大きい、過去障害が多い、実装難度が高い、同種不具合が起きやすい |
|  4 | 高い    | 条件分岐が多い、変更影響が広い、類似機能で障害実績がある、設計上の注意点が多い      |
|  3 | 中     | 一定の複雑さや変更があるが、仕様はおおむね明確                      |
|  2 | 低     | 変更範囲が限定的で、処理も比較的単純                           |
|  1 | 極めて低い | 既存安定機能で変更がなく、発生条件も限定的                        |

---

## 3.3 検出困難度

検出困難度は、通常のテストや単純な確認でリスクを見逃しやすいかを評価する。

| 点数 | 評価    | 判断基準                                            |
| -: | ----- | ----------------------------------------------- |
|  5 | 極めて高い | 複数条件、特殊データ、長い業務フロー、タイミング依存、外部連携などが絡み、通常確認では検出困難 |
|  4 | 高い    | 境界値、組合せ条件、権限差、状態遷移、過去データなどを意識しないと見逃しやすい         |
|  3 | 中     | 正常系だけでは見逃すが、基本的な異常系・境界条件を確認すれば検出可能              |
|  2 | 低     | 標準的な確認で検出しやすい                                   |
|  1 | 極めて低い | 画面上、ログ上、処理結果上ですぐに分かる                            |

---

## 3.4 根拠信頼度

根拠信頼度は、リスク評価の判断根拠がどの程度明確かを表す。

| 信頼度 | 定義                        | 扱い                |
| --- | ------------------------- | ----------------- |
| 高   | 入力文書、仕様、過去障害、変更点に明確な根拠がある | 通常評価として扱う         |
| 中   | 一部根拠はあるが、条件や影響範囲に不明点がある   | 評価理由に不確実性を記載する    |
| 低   | 根拠が弱く、仮説または確認事項に依存している    | 仮説リスクまたは確認事項として扱う |

根拠信頼度が低いことを理由に、重大リスクを削除してはならない。
影響が大きい場合は、仮説リスクとして残し、確認事項にする。

---

## 4. リスクスコア

リスクスコアは、以下で算出する。

```text
Risk Score = 影響度 × 発生可能性 × 検出困難度
```

スコア範囲は 1〜125 とする。

|  Score | 優先度 | 基本的な扱い       |
| -----: | --- | ------------ |
| 80〜125 | P0  | 最重点テスト対象     |
|  40〜79 | P1  | 重点テスト対象      |
|  15〜39 | P2  | 標準テスト対象      |
|   1〜14 | P3  | 簡易確認または対象外候補 |

---

## 5. 優先度定義

## 5.1 P0: 最重点テスト対象

P0は、顕在化した場合の影響が大きく、かつ発生可能性または検出困難度が高いリスクである。

### 扱い

* 必ずテスト方針に含める
* 正常系、異常系、境界、状態遷移、組合せ、回帰影響を厚く確認する
* 必要に応じて、有識者レビューや仕様確認を行う
* テストケースとの対応関係を必ず維持する
* 未テストの場合は残存リスクとして明示する

### 例

```text
締め処理後のデータ修正により、確定済み帳票と会計データが不整合になるリスク
```

---

## 5.2 P1: 重点テスト対象

P1は、主要な業務・機能に影響する重要リスクである。

### 扱い

* 原則としてテスト対象に含める
* 主要パターン、代表的な異常系、境界条件を確認する
* テストしない場合は理由を記載する
* 必要に応じて、P0テストケースに統合して確認する

### 例

```text
承認前データが帳票出力対象に含まれ、未確定情報が利用者に提示されるリスク
```

---

## 5.3 P2: 標準テスト対象

P2は、影響や発生可能性が中程度のリスクである。

### 扱い

* 標準的な確認対象とする
* 代表値、主要条件、基本的な異常系を確認する
* 高優先リスクのテストで同時確認できる場合は統合してよい
* 制約が強い場合は簡易確認へ落とすこともできる

### 例

```text
一部検索条件の組合せで、一覧表示結果に対象外データが含まれるリスク
```

---

## 5.4 P3: 簡易確認または対象外候補

P3は、影響が限定的、または発生可能性・検出困難度が低いリスクである。

### 扱い

* 簡易確認、探索的確認、または対象外候補とする
* 対象外にする場合は理由を記載する
* 高優先リスクと同じテストで確認できる場合は、あわせて確認する
* 完全に削除せず、必要に応じて残存リスクとして扱う

### 例

```text
軽微な表示崩れにより、補足情報の一部が見づらくなるリスク
```

---

## 6. 補正ルール

スコアは判断補助であり、機械的に優先度を決めるためのものではない。
以下に該当する場合は、優先度を補正してよい。

---

## 6.1 影響度が5の場合

影響度が5のリスクは、スコアが低くても原則として P1 以上を検討する。

### 理由

発生可能性が低くても、顕在化した場合の被害が重大であるため。

### 記載例

```text
Scoreは30でP2相当だが、影響度が5であり、会計処理に重大な影響を与えるため、P1へ補正する。
```

---

## 6.2 会計・金額・法令・監査に関係する場合

以下に該当するリスクは、優先度を1段階上げることを検討する。

* 会計処理
* 金額計算
* 請求・支払
* 契約
* 税務
* 法令対応
* 監査対応
* 内部統制
* 個人情報
* 権限・承認

### 記載例

```text
金額計算の誤りが請求額に直接影響するため、Score上はP2だがP1として扱う。
```

---

## 6.3 過去障害に類似する場合

過去障害と類似するリスクは、発生可能性または検出困難度を高めに評価する。

### 理由

同種の設計・実装ミスが再発する可能性があるため。
また、過去に見逃された障害であれば、通常のテストでは検出しにくい可能性がある。

### 記載例

```text
過去に同種の締め後修正不具合が発生しているため、発生可能性を4として評価する。
```

---

## 6.4 変更影響が広い場合

共通部品、共通ロジック、共通マスタ、共通帳票、共通IFなどに影響するリスクは、影響度または発生可能性を高めに評価する。

### 記載例

```text
共通計算ロジックの変更であり、複数帳票に影響する可能性があるため、影響度を4として評価する。
```

---

## 6.5 テストで検出しにくい場合

以下のようなリスクは、検出困難度を高めに評価する。

* 複数条件の組合せが必要
* 特殊な状態やタイミングが必要
* 大量データが必要
* 外部システム連携が必要
* 過去データや移行データが必要
* 権限やロールの組合せが必要
* 長い業務フローを通さないと検出できない

### 記載例

```text
承認状態、締め状態、権限の組合せが必要で、通常の正常系テストでは見逃しやすいため、検出困難度を4とする。
```

---

## 6.6 根拠信頼度が低いが影響が大きい場合

根拠信頼度が低い場合でも、影響度が大きいリスクは削除しない。

### 扱い

* 仮説リスクとして残す
* 評価理由に不確実性を記載する
* 確認事項を作成する
* 必要に応じて暫定優先度を付ける

### 記載例

```text
権限仕様が未提示のため根拠信頼度は低い。
ただし、承認権限の誤りは業務統制に大きく影響するため、暫定的にP1として扱い、仕様確認事項にする。
```

---

## 7. 評価時の判断補助

## 7.1 影響度を高くしやすい条件

以下に該当する場合、影響度を高めに評価する。

* 金額・会計・請求・支払に影響する
* 法令・監査・内部統制に影響する
* 主要業務が停止する
* 多数の利用者に影響する
* 後続処理へ誤ったデータが流れる
* 本番運用後の修正が困難
* 顧客・利用部門の信頼を損なう
* データ復旧が難しい

---

## 7.2 発生可能性を高くしやすい条件

以下に該当する場合、発生可能性を高めに評価する。

* 仕様が複雑
* 条件分岐が多い
* 変更量が大きい
* 新規機能である
* 共通処理を変更している
* 過去障害がある
* 類似機能で不具合が多い
* 開発期間が短い
* 仕様が未確定または曖昧
* 複数システム・複数チームにまたがる

---

## 7.3 検出困難度を高くしやすい条件

以下に該当する場合、検出困難度を高めに評価する。

* 境界値でのみ発生する
* 特定の権限でのみ発生する
* 特定の状態遷移でのみ発生する
* 複数条件の組合せでのみ発生する
* 長い業務フローを経ないと発生しない
* 外部システム連携が必要
* 大量データが必要
* 月末・年度末など特定タイミングが必要
* 異常系やリカバリ操作でのみ発生する
* 画面上は正常に見えるが、データ内部が不整合になる

---

## 8. 評価理由の記載ルール

各リスクには、必ず評価理由を記載する。

評価理由には、以下を含める。

* なぜその影響度にしたか
* なぜその発生可能性にしたか
* なぜその検出困難度にしたか
* 補正した場合は、なぜ補正したか
* 根拠信頼度が低い場合は、どこが不確実か

---

## 8.1 悪い評価理由

```text
重要そうなのでP1。
```

```text
スコアが高いためP0。
```

```text
仕様が複雑なので影響度5。
```

---

## 8.2 良い評価理由

```text
締め後データの不整合は、確定済み帳票と後続会計処理に影響するため影響度4。
締め状態、修正操作、帳票出力条件の組合せで発生しうるため発生可能性3。
通常の正常系確認では見逃しやすいため検出困難度4。
Scoreは48であり、P1とする。
```

```text
権限仕様が未提示のため根拠信頼度は低い。
ただし、承認者以外が確定操作できる場合、内部統制に影響するため影響度5。
暫定的にP1とし、仕様確認事項として管理する。
```

---

## 9. 評価結果の出力項目

Step 3 のリスク評価一覧では、以下の項目を出力する。

| 項目      | 内容                  |
| ------- | ------------------- |
| Risk ID | Step 2で付与したリスクID    |
| リスク     | リスク内容               |
| 分類      | リスク分類               |
| 影響度     | 1〜5                 |
| 発生可能性   | 1〜5                 |
| 検出困難度   | 1〜5                 |
| Score   | 影響度 × 発生可能性 × 検出困難度 |
| 優先度     | P0 / P1 / P2 / P3   |
| 根拠信頼度   | 高 / 中 / 低           |
| 評価理由    | 点数と優先度の理由           |

## 10. スコアリング例

## 10.1 高優先度の例

| 項目    | 内容                              |
| ----- | ------------------------------- |
| リスク   | 締め後にデータ修正でき、確定済み帳票と明細が不整合になる    |
| 影響度   | 4                               |
| 発生可能性 | 3                               |
| 検出困難度 | 4                               |
| Score | 48                              |
| 優先度   | P1                              |
| 理由    | 会計・帳票結果に影響し、状態条件を意識しないと見逃しやすいため |

---

## 10.2 最重点の例

| 項目    | 内容                               |
| ----- | -------------------------------- |
| リスク   | 金額計算ロジックの変更により、請求金額が誤る           |
| 影響度   | 5                                |
| 発生可能性 | 4                                |
| 検出困難度 | 4                                |
| Score | 80                               |
| 優先度   | P0                               |
| 理由    | 金額誤りの影響が重大で、条件分岐や丸め条件により検出しにくいため |

---

## 10.3 標準確認の例

| 項目    | 内容                            |
| ----- | ----------------------------- |
| リスク   | 一部検索条件の組合せで対象外データが表示される       |
| 影響度   | 3                             |
| 発生可能性 | 3                             |
| 検出困難度 | 3                             |
| Score | 27                            |
| 優先度   | P2                            |
| 理由    | 一部業務に影響するが、代表的な検索条件確認で検出可能なため |

---

## 10.4 簡易確認の例

| 項目    | 内容                          |
| ----- | --------------------------- |
| リスク   | 補足表示の軽微な文言崩れにより、説明がやや分かりにくい |
| 影響度   | 1                           |
| 発生可能性 | 2                           |
| 検出困難度 | 1                           |
| Score | 2                           |
| 優先度   | P3                          |
| 理由    | 業務結果に影響せず、画面確認で容易に検出できるため   |

---

## 11. 評価保留の扱い

評価に必要な情報が不足している場合は、無理に断定しない。

以下のいずれかで扱う。

| 状態      | 扱い                   |
| ------- | -------------------- |
| 影響範囲が不明 | 影響度を暫定評価し、確認事項を作る    |
| 発生条件が不明 | 発生可能性を暫定評価し、確認事項を作る  |
| 検出方法が不明 | 検出困難度を高めに置くか、評価保留にする |
| 根拠が弱い   | 根拠信頼度を低にする           |
| 仕様未確定   | 暫定優先度を付け、保留事項にする     |

---

## 12. 評価の偏りを防ぐチェックリスト

評価後、以下を確認する。

* すべてのリスクがP0/P1に偏っていないか
* すべてのリスクがP2/P3に落ちすぎていないか
* 影響度と発生可能性を混同していないか
* 複雑さだけで影響度を高くしていないか
* 業務影響が大きいリスクを低く評価していないか
* 検出困難度を無視していないか
* 根拠信頼度が低いリスクを消していないか
* 補正したリスクの理由が書かれているか
* P0/P1の数が、テスト制約上扱える量になっているか
* P3に落としたリスクの残存リスクを説明できるか

---

## 13. 禁止事項

以下を禁止する。

* 評価理由を書かずに点数だけ付ける
* すべてのリスクを高優先度にする
* すべてのリスクを低優先度にする
* 根拠が弱いリスクを無条件で削除する
* スコアだけで機械的に優先度を決める
* 影響度、発生可能性、検出困難度を混同する
* 補正理由を書かずに優先度を変更する
* テストケース作成に踏み込む
* 不明点を勝手に補完して評価する

---
