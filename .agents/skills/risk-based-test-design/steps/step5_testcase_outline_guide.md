# Step 5: テストケース骨子・トレーサビリティ作成ガイド

## 1. このStepの目的

このStepでは、`04_risk_based_test_strategy.md` をもとに、リスクベースドテストのテストケース骨子を作成する。

このStepの目的は、以下を明確にすることである。

- どのリスクに対して、どのテストケースを用意するか
- 各テストケースがどのテスト観点に対応しているか
- テストケースの目的、前提条件、入力条件、期待結果は何か
- 実行優先度はどうするか
- 未カバーのリスクは残っていないか
- リスク、テスト観点、テストケースの対応関係が追跡可能か

このStepでは、詳細な操作手順までは作成しない。  
詳細な手順、画面操作、具体的な入力値、期待結果の細部は、必要に応じて別工程で作成する。

- `04_risk_based_test_strategy.md`

## 4. このStepでやること

以下の観点で、テストケース骨子とトレーサビリティを作成する。

---

### 4.1 テスト観点をテストケース骨子に変換する

Step 4 で定義したテスト観点をもとに、テストケース骨子を作成する。

テスト観点は「確認すべき方向性」である。  
テストケース骨子は「どの条件で、何を確認するか」を整理したものである。

例:

| 種別 | 内容 |
|---|---|
| リスク | 締め後にデータ修正できることで、確定済み帳票と明細が不整合になる |
| テスト観点 | 締め状態ごとの編集可否、取消可否、再集計可否を確認する |
| テストケース骨子 | 締め済み期間の明細を修正しようとした場合、保存が拒否され、帳票出力結果が変わらないことを確認する |

---

### 4.2 優先度に応じてテストケース数と粒度を調整する

すべてのリスクに同じ数・同じ粒度のテストケースを作らない。

| 優先度 | テストケース化の方針 |
|---|---|
| P0 | 複数のテストケースに分け、正常系・異常系・境界・状態・組合せ・回帰影響を厚く扱う |
| P1 | 主要パターンと代表的な異常系をテストケース化する |
| P2 | 代表条件を中心に、必要最小限のテストケースにする |
| P3 | 簡易確認、他ケースへの統合、探索的確認、または対象外とする |

P0/P1は、テストケース骨子として明示的に残す。  
P2/P3は、必要に応じて他のケースと統合してよい。

---

### 4.3 テストケース骨子に必要な項目を整理する

テストケース骨子には、最低限以下を記載する。

| 項目 | 内容 |
|---|---|
| TC ID | テストケースID |
| テストケース名 | 何を確認するテストか |
| 対応Risk ID | 対応するリスク |
| 対応Viewpoint ID | 対応するテスト観点 |
| 目的 | このテストで検出・軽減したいリスク |
| 前提条件 | 必要な状態、権限、データ、環境 |
| 入力条件・操作条件 | 確認する条件。詳細操作手順ではない |
| 期待結果 | 確認すべき結果の概要 |
| 優先度 | 実行優先度 |
| 備考 | 制約、未確定事項、統合確認など |

---

### 4.4 テストケース名の付け方

テストケース名は、確認内容が分かるように具体化する。

悪い例:

```text
締め処理確認
```

良い例:

```text
締め済み期間の明細修正が拒否されることを確認する
```

悪い例:

```text
帳票出力確認
```

良い例:

```text
承認前データが確定帳票に出力されないことを確認する
```

---

### 4.5 期待結果の粒度

このStepでは、期待結果は概要レベルでよい。

適切な例:

```text
保存が拒否され、対象データは更新されない。確定済み帳票の出力結果も変化しない。
```

細かすぎる例:

```text
エラーダイアログに「締め済み期間のため修正できません。」と表示され、OKボタンを押すと元画面に戻り、明細一覧の金額列が1,000円のままである。
```

詳細な文言、画面操作、ボタン名、具体値は、詳細テストケース作成工程で扱う。

---

### 4.6 テストケースの統合を検討する

複数のリスクや観点を、1つのテストケースで効率よく確認できる場合は統合してよい。

統合してよい例:

- 同じ業務フロー上で確認できる
- 同じ前提データで確認できる
- 期待結果が同時に確認できる
- P2/P3リスクをP0/P1ケースの中で補足確認できる
- 回帰確認を主要シナリオに含められる

ただし、統合しすぎてテスト目的が曖昧になる場合は分割する。

統合時は、対応Risk IDとViewpoint IDを複数記載してよい。

---

### 4.7 未カバーリスクを整理する

すべてのリスクがテストケース化されているとは限らない。  
テストケース化しないリスクは、未カバーリスクまたは対象外・代替確認として整理する。

未カバー理由の例:

- 優先度が低いため簡易確認にする
- 他のテストケースで間接的に確認できる
- 仕様未確定のためテストケース化できない
- テスト環境で確認できない
- レビュー確認のほうが適している
- 運用確認で扱う
- 今回のスコープ外である
- 残存リスクとして受容する

未カバーリスクを放置してはならない。  
必ず理由と扱いを記載する。

---

## 5. 実行優先度の考え方

テストケースの実行優先度は、対応するリスクの優先度を基本に決める。

| 対応リスク | テストケース実行優先度 | 扱い |
|---|---|---|
| P0 | High | 最優先で実行する |
| P1 | High / Medium | 重点的に実行する |
| P2 | Medium | 標準的に実行する |
| P3 | Low | 時間があれば実行、または他ケースに統合 |

複数リスクに対応するテストケースは、最も高いリスク優先度に合わせる。

例:

```text
R-001 が P0、R-006 が P2 の場合、そのテストケースの実行優先度は High とする。
```

## 6. トレーサビリティの作成

このStepでは、以下の対応関係を明確にする。

```text
Risk ID
  → Viewpoint ID
    → TC ID
```

最低限、以下を確認する。

- P0/P1リスクに対応するテストケースが存在するか
- 各テストケースが対応Risk IDを持っているか
- 各テストケースが対応Viewpoint IDを持っているか
- 未カバーリスクが明示されているか
- 対象外・簡易確認にした理由が記載されているか

## 8. テストケース骨子の粒度

テストケース骨子は、詳細テストケースの一歩手前の粒度にする。

### 粗すぎる例

```text
締め処理を確認する
```

### 適切な例

```text
締め済み期間の明細を修正しようとした場合、保存が拒否され、確定済み帳票に影響しないことを確認する
```

### 細かすぎる例

```text
2026/03/31の締め済みデータを開き、明細一覧から部門D001の売上高1000円を1200円に変更し、保存ボタンをクリックした場合、エラーダイアログに「締め済み期間のため保存できません」と表示されること
```

細かすぎる内容は、詳細テストケース作成工程で扱う。

---

## 9. テストケース分割の判断基準

以下に該当する場合は、テストケースを分割する。

- 前提条件が大きく異なる
- 期待結果が異なる
- リスクの種類が異なる
- 正常系と異常系を分けたほうが分かりやすい
- 権限や状態の組合せが複雑
- 1ケースに複数目的が混在している
- 実行時に失敗原因を切り分けにくい

以下に該当する場合は、統合してよい。

- 同じ業務フローで確認できる
- 同じデータで確認できる
- 同じ期待結果を確認する
- 低優先度リスクを高優先度ケースの中で同時確認できる
- 回帰確認としてまとめられる

---

## 10. カバレッジの定義

リスクに対するカバレッジは、以下で表す。

| カバレッジ | 定義 |
|---|---|
| 十分 | 主要な発生条件・影響を検出できるテストケースがある |
| 一部 | 代表条件のみ確認しており、未確認条件が残る |
| 未カバー | 対応するテストケースがない |
| 対象外 | 理由を明示してテスト対象外としている |
| 代替確認 | レビュー、静的確認、運用確認などで扱う |

P0/P1リスクは、原則として `十分` または `一部` 以上にする。  
`未カバー` または `対象外` にする場合は、理由を明確にする。

## 11. このStepでやらないこと

以下は、このStepでは行わない。

- 詳細な画面操作手順の作成
- 具体的なテストデータ値の大量作成
- エラーメッセージ文言の完全一致定義
- 自動化スクリプト作成
- テスト実行スケジュール作成
- 工数見積り
- 担当者割り当て
- テスト管理ツールへの登録形式変換

このStepでは、リスクとテストケース骨子の対応関係を明確にすることに集中する。

---

## 12. よくある失敗と回避策

### 12.1 リスクに対応しないテストケースが増える

悪い状態:

```text
一般的な正常系テストケースが大量に増え、どのリスクに対応しているか分からない。
```

回避策:

- すべてのテストケースに対応Risk IDを付ける
- Risk IDが付かないテストケースは、必要性を見直す
- 汎用的な確認は別枠に分離する

---

### 12.2 P0/P1リスクがテストケース化されていない

悪い状態:

```text
高優先度リスクは書かれているが、対応するテストケースがない。
```

回避策:

- リスク-観点-テストケース対応表で確認する
- P0/P1に未カバーがあれば、理由を記載する
- 必要に応じてテストケースを追加する

---

### 12.3 1つのテストケースに詰め込みすぎる

悪い状態:

```text
1ケースで入力、承認、締め、帳票、外部連携まで確認しようとしている。
```

回避策:

- テスト目的が複数ある場合は分割する
- 失敗時に原因を切り分けられる単位にする
- 業務シナリオとして統合する場合も、確認ポイントを明示する

---

### 12.4 詳細手順を書き始める

悪い状態:

```text
画面名、ボタン名、具体値、メッセージ文言まで詳細に書き始める。
```

回避策:

- このStepでは骨子までに留める
- 詳細化が必要な事項として、後続工程への引き継ぎに記録する

---

### 12.5 未カバーリスクを隠す

悪い状態:

```text
テストケース化できなかったリスクが成果物から消える。
```

回避策:

- 未カバーリスク表を必ず作成する
- 対象外、簡易確認、代替確認、保留、受容のいずれかで扱う
- 残存リスクを記載する

---

## 13. 完了条件

このStepは、以下を満たしたら完了とする。

- テストケース骨子が作成されている
- 各テストケースに TC ID が付与されている
- 各テストケースに対応Risk IDが付与されている
- 各テストケースに対応Viewpoint IDが付与されている
- テストケースの目的、前提条件、入力条件、期待結果の概要が記載されている
- 実行優先度が設定されている
- リスク-観点-テストケース対応表が作成されている
- P0/P1リスクのカバレッジが確認されている
- 未カバーリスクが明示されている
- 統合・簡略化した確認が明示されている
- 詳細テストケース化時の注意事項が整理されている

---

## 14. 実行後の停止

`05_testcase_outline_and_traceability.md` を出力したら、このStepを終了する。

この時点で、リスクベースドテスト設計の主要成果物は一通り作成済みとなる。

必要に応じて、次工程として以下を実施する。

- 詳細テストケース作成
- テストデータ設計
- テスト実行順序設計
- レビュー観点チェック
- テスト管理ツール投入用フォーマットへの変換
