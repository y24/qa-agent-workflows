# rules.md

## 1. このプロンプト群の目的

このプロンプト群は、要件定義書、基本設計書、詳細設計書、画面仕様書、API仕様書、DB仕様書、業務説明資料、補足メモなどの開発ドキュメントから、テスト設計に必要な情報を抽出し、整理・一覧化するためのものである。

目的は、テストケースを作成することではない。

目的は、散逸した仕様情報を、後続のテスト設計工程で利用できる形に整理することである。

最終的な主成果物は、以下とする。

- `document_inventory.md`
- `raw_extraction.md`
- `normalized_spec_inventory.md`
- `test_design_input_catalog.md`
- `gap_and_review_report.md`

---

## 2. 基本原則

すべてのステップでは、以下の原則を厳守する。

### 2.1 推測しない

ドキュメントに明記されていない内容を、一般論や経験則で補完してはならない。

禁止例:

- 一般的な業務システムならこうだろうと判断する
- 通常はこの画面遷移になるはずだと補完する
- 未記載のエラーメッセージや入力制約を作る
- テーブル名や項目名を推測で補う
- 仕様の空白を自然な動作で埋める

不明な内容は、不明として扱う。

---

### 2.2 原文に基づく

抽出・整理・統合・分類は、必ず入力ドキュメントの記載に基づいて行う。

各抽出結果には、可能な限り以下を記録する。

- 参照元ドキュメント
- 章番号、見出し、ページ、行番号、表名などの位置情報
- 原文または要約
- 抽出理由
- 確度

参照元を示せない情報は、原則として仕様情報として確定扱いしない。

---

### 2.3 事実と解釈を分ける

ドキュメントに書かれている事実と、テスト設計上の解釈を混同してはならない。

例:

```text
事実:
金額は0以上の整数のみ入力可能である。

解釈:
金額入力では、境界値、不正値、型制約の観点が必要である。
````

事実情報は主に `raw_extraction.md` と `normalized_spec_inventory.md` に記録する。

テスト設計上の観点は `test_design_input_catalog.md` に記録する。

---

### 2.4 矛盾を勝手に解消しない

複数のドキュメント間で記載が異なる場合、AIがどちらか一方を正しいと判断してはならない。

矛盾がある場合は、以下のように扱う。

* `normalized_spec_inventory.md` では状態を `contradiction` とする
* `gap_and_review_report.md` に確認事項として記録する
* 影響範囲と確認すべき相手・資料を可能な範囲で記載する

---

### 2.5 不足を補完しない

テスト設計に必要な情報が不足している場合、AIが補ってはならない。

不足している情報は、以下のいずれかとして扱う。

* `missing`
* `insufficient_detail`
* `unknown`
* `to_be_confirmed`

不足情報は `gap_and_review_report.md` に記録する。

---

### 2.6 テストケースを作らない

このプロンプト群の範囲では、具体的なテストケースを作成しない。

作成してよいもの:

* テスト対象
* テスト観点
* テスト条件の候補
* 必要な入力データ観点
* 期待結果の種類
* 関連仕様
* 不明点
* 矛盾
* リスク

作成してはならないもの:

* 詳細な操作手順付きテストケース
* 期待結果を断定したテストケース
* テストデータの具体値を伴うケース一覧
* 実行順序を持つテストシナリオ

ただし、原文に具体例が明記されている場合は、参照元付きで記録してよい。

---

## 3. ステップ構成

このプロンプト群は、以下の5ステップで進行する。

| Step   | 名称             | 主な成果物                          |
| ------ | -------------- | ------------------------------ |
| Step 1 | ドキュメント棚卸し      | `document_inventory.md`        |
| Step 2 | 事実情報の抽出        | `raw_extraction.md`            |
| Step 3 | 正規化・統合         | `normalized_spec_inventory.md` |
| Step 4 | テスト設計インプット化    | `test_design_input_catalog.md` |
| Step 5 | 不明点・矛盾・不足のレビュー | `gap_and_review_report.md`     |

各ステップは独立して実行する。

1つのステップが完了したら、必ず停止し、ユーザーの確認を待つ。

ユーザーから次のステップ実行を指示されるまで、勝手に次ステップへ進んではならない。

---

## 4. 入力ドキュメントの扱い

### 4.1 入力ドキュメントの例

入力対象となるドキュメントには、以下を含めてよい。

* 要件定義書
* 業務要件資料
* 基本設計書
* 詳細設計書
* 画面仕様書
* 帳票仕様書
* API仕様書
* 外部連携仕様書
* バッチ設計書
* DB設計書
* テーブル定義書
* 権限定義書
* 運用設計書
* 移行設計書
* 補足メモ
* QAメモ
* チーム別設計資料
* 過去障害資料
* 仕様確認メモ

---

### 4.2 正本と補足資料を区別する

資料には優先度を付ける。

推奨する優先度は以下。

| 優先度     | 意味                 |
| ------- | ------------------ |
| High    | 正本、または仕様判断の基準となる資料 |
| Medium  | 補足資料、チーム別資料、詳細資料   |
| Low     | 参考資料、古い資料、未確定資料    |
| Unknown | 位置づけ不明の資料          |

優先度が低い資料の記載を、高い資料の記載より優先してはならない。

ただし、優先度が低い資料にしか存在しない情報は、確度を下げたうえで記録してよい。

---

### 4.3 古い資料の扱い

古い資料、更新日不明の資料、過去版の資料を使用する場合は、以下を明記する。

* 古い可能性があること
* 最新性が不明であること
* 現行仕様として確定できないこと
* 確認が必要であること

---

## 5. 抽出対象カテゴリ

仕様情報は、原則として以下のカテゴリに分類する。

| カテゴリ               | 内容                      |
| ------------------ | ----------------------- |
| `feature`          | 機能、処理単位、ユースケース          |
| `screen`           | 画面、画面遷移、画面部品            |
| `ui_item`          | 入力項目、表示項目、ボタン、リンク       |
| `report`           | 帳票、ファイル出力、印刷、Excel/PDF等 |
| `api`              | API、インターフェース、外部連携       |
| `batch`            | バッチ、ジョブ、非同期処理           |
| `data`             | DB、テーブル、項目、マスタ、トランザクション |
| `business_rule`    | 業務ルール、業務上の判定条件          |
| `validation`       | 入力チェック、制約、必須、桁数、型       |
| `state_transition` | ステータス、状態遷移、ライフサイクル      |
| `condition_branch` | 条件分岐、場合分け               |
| `calculation`      | 計算、集計、変換、丸め             |
| `permission`       | 権限、ロール、操作可否             |
| `error_handling`   | エラー、警告、例外、リトライ          |
| `operation`        | 運用、保守、移行、監視、手動作業        |
| `non_functional`   | 性能、セキュリティ、可用性、互換性等      |
| `open_issue`       | 未確定、要確認、不足、矛盾           |

分類に迷う場合は、最もテスト設計で使いやすいカテゴリを選び、メモに理由を記載する。

---

## 6. 確度の定義

抽出した情報には、原則として確度を付与する。

| 確度      | 意味                     |
| ------- | ---------------------- |
| High    | 原文に明確に記載されている          |
| Medium  | 原文から直接読み取れるが、表現に曖昧さがある |
| Low     | 関連記述はあるが、仕様として確定しにくい   |
| Unknown | 判断できない                 |

確度は、情報の重要度ではない。

重要度が高くても、原文が曖昧であれば確度は低くする。

---

## 7. 状態の定義

正規化・統合後の仕様情報には、状態を付与する。

| 状態                    | 意味                      |
| --------------------- | ----------------------- |
| `confirmed`           | 参照元が明確で、仕様として扱える        |
| `partial`             | 一部は分かるが、詳細が不足している       |
| `contradiction`       | 資料間または同一資料内に矛盾がある       |
| `missing`             | 必要な情報が存在しない             |
| `insufficient_detail` | 記載はあるが、テスト設計には粒度が不足している |
| `to_be_confirmed`     | 仕様責任者や追加資料への確認が必要       |
| `deprecated_or_stale` | 古い資料の可能性がある             |
| `out_of_scope`        | 今回の抽出対象外                |

---

## 8. 出力ルール

### 8.1 Markdownで出力する

すべての成果物は、原則としてMarkdown形式で出力する。

表形式を基本とし、必要に応じて補足説明を付ける。

---

### 8.2 IDを付与する

主要な成果物にはIDを付与する。

推奨ID体系:

| 成果物        | ID例        |
| ---------- | ---------- |
| ドキュメント     | `DOC-001`  |
| 抽出情報       | `EXT-001`  |
| 正規化仕様      | `SPEC-001` |
| テスト設計インプット | `TDI-001`  |
| ギャップ・確認事項  | `GAP-001`  |

IDは後続ステップで参照できるように、一貫して使用する。

---

### 8.3 参照元を残す

抽出・正規化・テスト設計インプット化した情報には、可能な限り参照元を記録する。

参照元の記載例:

```text
DOC-001 3.2.1
DOC-002 画面項目一覧「金額」
DOC-003 p.12 外部連携エラー時の処理
EXT-014
SPEC-008
```

---

### 8.4 不明点を明示する

不明点は空欄にしない。

以下のように明示する。

* `未記載`
* `不明`
* `要確認`
* `資料なし`
* `判断不可`
* `記載粒度不足`

---

### 8.5 長文を避ける

成果物は、後続工程で参照しやすいように簡潔に書く。

1セルに長文を詰め込みすぎない。

必要な場合は、概要と詳細メモを分ける。

---

## 9. コンテキスト長対策

### 9.1 一度に大量の文書を処理しない

1回の実行で読み込むドキュメント量は制限する。

推奨:

* 1回につき1〜3ファイル程度
* または、明確に区切られた章・節単位
* または、数十〜数百行程度

大量の入力がある場合は、分割して処理する。

---

### 9.2 中間成果物を必ず残す

会話履歴だけに依存してはならない。

各ステップの出力は、Markdownファイルとして保存・参照できる形にする。

後続ステップでは、前ステップの成果物を入力として扱う。

---

### 9.3 生情報と統合情報を分ける

以下を混在させてはならない。

* ドキュメントから抜き出した生情報
* 複数情報を正規化・統合した仕様情報
* テスト設計向けに変換した観点情報

それぞれ以下に分ける。

| 情報種別      | 出力先                            |
| --------- | ------------------------------ |
| 生情報       | `raw_extraction.md`            |
| 正規化・統合情報  | `normalized_spec_inventory.md` |
| テスト設計向け情報 | `test_design_input_catalog.md` |

---

### 9.4 確信がない情報は gap に逃がす

判断に迷う情報、不明確な情報、矛盾する情報は、無理に仕様化しない。

必要に応じて `gap_and_review_report.md` に記録する。

---

## 10. 禁止事項

以下を禁止する。

* ドキュメントにない仕様を作る
* 曖昧な記載を断定表現に変える
* 参照元のない情報を確定仕様として扱う
* 複数資料の矛盾を勝手に解消する
* 古い資料の記載を現行仕様として断定する
* テストケースを作成する
* 具体的な期待結果を推測で作成する
* 未記載の入力値、境界値、エラーメッセージを作成する
* 情報源の確度を無視して統合する
* ユーザー確認なしに次ステップへ進む

---

## 11. 例外的に許可されること

以下は許可する。

### 11.1 明示的なラベル付け

原文に書かれている内容を、カテゴリ分類することは許可する。

例:

```text
原文:
承認済みの申請のみ支払処理に進める。

分類:
business_rule
state_transition
```

---

### 11.2 テスト観点への抽象化

Step 4では、仕様情報をテスト設計向けの観点に変換してよい。

ただし、具体的なテストケースにはしない。

例:

```text
仕様:
金額は0以上の整数のみ入力可能。

観点:
金額入力の境界値、不正値、型制約。
```

---

### 11.3 不足情報の指摘

テスト設計に必要だが資料にない情報を、不足情報として指摘してよい。

例:

```text
税込金額の丸め単位が未記載。
権限別の操作可否が未記載。
APIタイムアウト時の再試行条件が未記載。
```

---

## 12. 完了判定

各ステップの完了時には、以下を確認する。

* ステップの目的に合った成果物が作成されている
* 参照元が記録されている
* 不明点や矛盾が隠されていない
* 推測による補完が混入していない
* 後続ステップで利用できる形式になっている
* 次ステップへ勝手に進んでいない

---

## 13. レビュー待ちルール

各ステップの成果物を出力したら、そこで停止する。

次のステップに進むには、ユーザーから明示的な指示を受ける必要がある。

ユーザー指示の例:

```text
次へ進んでください。
Step 2を実行してください。
この内容で続けてください。
修正したので次をお願いします。
```

ユーザーの確認前に、後続ステップの成果物を作成してはならない。
