# Spec Extraction 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、出力先、トレーサビリティは `../../../shared/` 配下の方針に従う。
このファイルでは `$spec-extraction` 固有の責務、ID、禁止事項を定義する。

## 目的

開発文書から事実を抽出し、根拠付きで正規化し、後続のQA設計に渡せるインプットへ変換するための共通ルールを定義する。

## 基本原則

- 推測しない。
- 原文に基づく。
- 事実、解釈、推測、確認事項を分ける。
- 矛盾を勝手に解消しない。
- 不足を補完しない。
- このskillでは詳細テストケースを作らない。

## Step責務

| Step | 責務 | やらないこと |
|---|---|---|
| 1 | ドキュメント棚卸し | 仕様抽出、正規化 |
| 2 | 原文ベースの事実抽出 | 統合、矛盾解消 |
| 3 | 正規化・統合 | 期待結果断定、テストケース作成 |
| 4 | テスト設計インプット化 | 詳細テストケース作成 |
| 5 | ギャップレビュー | ギャップの独断解消 |

## ID体系

| 対象 | 形式 |
|---|---|
| Document | `DOC-001` |
| Scope | `SCOPE-001` |
| Raw Extraction | `RAW-001` |
| Normalized Spec | `SPEC-001` |
| Test Design Input | `TDI-001` |
| Gap | `GAP-001` |
| Question | `Q-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## カテゴリ・確度・状態

- 抽出カテゴリは `references/extraction_category_definitions.md` の固定値を使う。
- 確度は `高` / `中` / `低` を使う。
- 状態は `confirmed` / `ambiguous` / `conflicting` / `missing_detail` / `tentative` / `out_of_scope` を使う。
- 根拠は Doc ID、章、節、項目、Raw ID、Spec IDで追跡できるようにする。

## 出力ルール

- Markdownで出力する。
- 表は列名を安定させる。
- 未処理範囲、対象外、確認事項を明示する。
- 長文引用を避け、原文要約と参照元で表す。
- 後続Stepへの引き継ぎを各成果物に含める。

## 許可されること

- 原文に基づく分類
- 仕様情報の要約
- テスト観点への抽象化
- 不足、矛盾、曖昧さの指摘
- 期待結果の「種類」の整理
- 入力データ観点の種類・条件の整理

## 禁止事項

- 入力にない仕様値、画面操作、文言、期待結果を断定する
- 一般論で不足仕様を補う
- 矛盾を独自判断で解消する
- 旧版や参考資料を現行仕様として断定する
- 詳細テストケースを作成する
- 不明点を隠す

## レビュー待ち

各Stepの成果物を出力したら停止する。ユーザーから明示的な続行指示があるまで次Stepへ進まない。
