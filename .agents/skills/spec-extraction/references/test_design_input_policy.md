# テスト設計インプット化ポリシー

## 目的

正規化仕様を、後続のシナリオ設計やテストケース設計で使える確認対象へ変換する。

## インプット化してよい情報

- 仕様として根拠がある機能、画面、API、データ、状態、条件、計算、権限、例外
- テストで確認すべき振る舞い、条件、前提、入力データ観点、期待結果の種類
- 未確定、矛盾、不足があるが、注意付きで確認対象にすべき情報

## 書いてよい粒度

- テスト観点の抽象化
- 入力データの種類・条件
- 期待結果の種類
- 関連コンポーネント
- リスクや注意事項

## 書いてはいけない粒度

- 詳細テストケース
- 具体的な操作手順
- 根拠のない具体値
- 根拠のない期待結果
- エラーメッセージ文言の断定

## 優先度

| 優先度 | 目安 |
|---|---|
| High | 中核機能、業務影響大、金額・会計・権限・連携・状態遷移、重大リスク |
| Medium | 主要機能、代表的な分岐、重要データ、通常確認が必要 |
| Low | 補助的仕様、影響限定、他観点で同時確認可能 |
| Unknown | 根拠不足、未確定、矛盾により判断不能 |

## 断定禁止

未確定・矛盾・根拠弱の仕様は、注意付きで扱い、後続テスト設計で断定してはいけない事項として明示する。
