# extraction_category_definitions.md

## 1. このファイルの目的

このファイルは、開発ドキュメントから抽出した情報を分類するためのカテゴリ定義を示す。

主に以下の成果物で使用する。

- `raw_extraction.md`
- `normalized_spec_inventory.md`
- `test_design_input_catalog.md`
- `gap_and_review_report.md`

目的は、AIエージェントが抽出情報を分類するときの判断基準を統一し、カテゴリの揺れを抑えることである。

---

## 2. 基本方針

### 2.1 主カテゴリを1つ選ぶ

1つの抽出情報または仕様情報には、原則として主カテゴリを1つ設定する。

複数カテゴリに関係する場合でも、後続のテスト設計で最も扱いやすいカテゴリを主カテゴリとする。

関連カテゴリがある場合は、備考に記録する。

例:

```text
主カテゴリ: business_rule
備考: validationにも関連
````

---

### 2.2 分類に迷った場合

分類に迷った場合は、以下の順で判断する。

1. テスト設計で最も使いやすい分類を優先する
2. 原文の見出し、表名、章名を参考にする
3. 仕様の対象が何かを確認する
4. 仕様が何を制御しているかを確認する
5. それでも判断できない場合は、`open_issue` または備考付きで仮分類する

---

### 2.3 推測で分類しない

カテゴリ分類のために、原文にない仕様を補完してはならない。

例えば、入力項目名から「たぶん必須だろう」と判断して `validation` に分類してはならない。

必須・桁数・型・範囲などの制約が明記されている場合にのみ、`validation` として扱う。

---

## 3. カテゴリ一覧

使用するカテゴリは以下とする。

| カテゴリ               | 概要                      |
| ------------------ | ----------------------- |
| `feature`          | 機能、処理単位、ユースケース          |
| `screen`           | 画面、画面遷移、画面部品            |
| `ui_item`          | 入力項目、表示項目、ボタン、リンク       |
| `report`           | 帳票、ファイル出力、印刷、Excel/PDF等 |
| `api`              | API、インターフェース、外部連携       |
| `batch`            | バッチ、ジョブ、非同期処理           |
| `data`             | DB、テーブル、項目、マスタ、トランザクション |
| `business_rule`    | 業務ルール、業務上の判定条件          |
| `validation`       | 入力チェック、制約、必須、桁数、型       |
| `state_transition` | ステータス、状態遷移、ライフサイクル      |
| `condition_branch` | 条件分岐、場合分け               |
| `calculation`      | 計算、集計、変換、丸め             |
| `permission`       | 権限、ロール、操作可否             |
| `error_handling`   | エラー、警告、例外、リトライ          |
| `operation`        | 運用、保守、移行、監視、手動作業        |
| `non_functional`   | 性能、セキュリティ、可用性、互換性等      |
| `open_issue`       | 未確定、要確認、不足、矛盾           |

---

# 4. 各カテゴリの定義

## 4.1 `feature`

### 定義

機能、処理単位、ユースケース、業務操作のまとまりを表す情報。

画面、API、バッチなどの具体的な実装単位よりも、利用者や業務から見た機能のまとまりを扱う。

### 該当する情報

* 機能一覧
* 業務機能
* ユースケース
* 処理概要
* 機能の目的
* 機能の対象範囲
* 利用者が実施する操作のまとまり
* システムが提供する主要な処理

### 例

```text
請求データを登録する。
承認済みの申請を締処理する。
月次集計結果を確認する。
外部システムへ取引データを連携する。
```

### 分類時の注意

画面名やAPI名が中心の場合は、`screen` または `api` を優先する。

業務ルールが中心の場合は、`business_rule` を優先する。

---

## 4.2 `screen`

### 定義

画面、画面構成、画面遷移、画面部品の配置や表示制御に関する情報。

### 該当する情報

* 画面一覧
* 画面名
* 画面概要
* 画面遷移
* 初期表示
* 表示条件
* ボタン押下時の遷移
* ダイアログ表示
* タブ、一覧、詳細領域
* 画面レイアウト上の仕様

### 例

```text
検索画面から詳細画面へ遷移する。
初期表示時は当月のデータを表示する。
登録ボタン押下後、確認ダイアログを表示する。
一覧行をダブルクリックすると詳細画面を開く。
```

### 分類時の注意

入力項目そのものの仕様は `ui_item` とする。

入力制約が中心の場合は `validation` とする。

画面遷移によってステータスが変わる場合、主眼が状態変化なら `state_transition` を優先する。

---

## 4.3 `ui_item`

### 定義

画面上の入力項目、表示項目、ボタン、リンク、チェックボックス、ラジオボタン、プルダウンなどのUI項目に関する情報。

### 該当する情報

* 入力項目
* 表示項目
* 項目名
* ボタン
* リンク
* チェックボックス
* ラジオボタン
* プルダウン
* 一覧列
* 初期値
* 表示順
* 活性・非活性
* 表示・非表示

### 例

```text
金額項目を表示する。
取引日は入力項目である。
承認ボタンは承認権限を持つユーザーにのみ表示する。
状態が確定済みの場合、削除ボタンは非活性とする。
```

### 分類時の注意

入力制約が中心の場合は `validation` とする。

権限による表示・操作可否が中心の場合は `permission` とする。

画面全体の遷移や構成が中心の場合は `screen` とする。

---

## 4.4 `report`

### 定義

帳票、ファイル出力、印刷、Excel、PDF、CSVなどの出力物に関する情報。

### 該当する情報

* 帳票名
* 帳票項目
* 出力条件
* 出力形式
* ファイル名
* ファイル形式
* 印刷条件
* Excel出力
* PDF出力
* CSV出力
* 帳票編集ルール
* 並び順
* 集計行
* 改ページ
* ヘッダー・フッター

### 例

```text
月次集計結果をExcel形式で出力する。
帳票には取引先コード、取引先名、金額を出力する。
出力対象は承認済みデータのみとする。
金額は税抜金額と税込金額を別列で表示する。
```

### 分類時の注意

出力値の計算が中心の場合は `calculation` とする。

帳票の出力対象条件が業務ルールである場合は `business_rule` としても扱える。

ファイル連携仕様の場合は `api` または `external interface` 相当として `api` に分類する。

---

## 4.5 `api`

### 定義

API、外部インターフェース、システム間連携、通信、リクエスト・レスポンスに関する情報。

このカテゴリには、REST APIだけでなく、外部システムとのファイル連携やメッセージ連携も含めてよい。

### 該当する情報

* API一覧
* エンドポイント
* HTTPメソッド
* リクエスト項目
* レスポンス項目
* ステータスコード
* エラーコード
* 認証方式
* 外部システム連携
* ファイル連携
* 送受信タイミング
* タイムアウト
* リトライ
* 連携結果

### 例

```text
POST /orders で注文データを登録する。
連携先から受信した取引データを取り込む。
タイムアウト時は連携エラーとして扱う。
レスポンスには処理結果コードを含める。
```

### 分類時の注意

リトライや例外処理が中心の場合は `error_handling` を優先する。

バッチで実行される連携処理の場合、主眼がジョブ実行なら `batch` を優先する。

---

## 4.6 `batch`

### 定義

バッチ、ジョブ、スケジュール実行、非同期処理、定期処理に関する情報。

### 該当する情報

* バッチ一覧
* ジョブ名
* 起動条件
* 実行スケジュール
* 処理順序
* 対象データ抽出
* 正常終了
* 異常終了
* 再実行
* リカバリ
* 排他制御
* 非同期処理
* キュー処理

### 例

```text
日次バッチは毎日午前2時に起動する。
対象データは未連携状態の取引データとする。
異常終了した場合、次回実行時に再処理する。
月次締処理後に集計バッチを実行する。
```

### 分類時の注意

対象データの業務条件が中心の場合は `business_rule` とする。

異常終了時の扱いが中心の場合は `error_handling` とする。

スケジュールや運用手順が中心の場合は `operation` としても扱える。

---

## 4.7 `data`

### 定義

DB、テーブル、カラム、データ構造、マスタ、トランザクション、データ更新に関する情報。

### 該当する情報

* テーブル
* カラム
* 主キー
* 外部キー
* データ型
* 桁数
* NULL可否
* マスタ
* トランザクション
* 登録
* 更新
* 削除
* 履歴
* データ整合性
* データ保持
* データ移行

### 例

```text
取引データは取引テーブルに登録する。
承認時にステータスを承認済みに更新する。
削除時は物理削除せず、削除フラグを更新する。
履歴テーブルに変更前後の値を保存する。
```

### 分類時の注意

入力項目の制約は `validation` とする。

業務上のデータ抽出条件は `business_rule` とする。

状態変更が中心の場合は `state_transition` とする。

---

## 4.8 `business_rule`

### 定義

業務上の判断、制約、処理対象、処理可否、業務例外に関する情報。

システム仕様というより、業務として守るべきルールを表す。

### 該当する情報

* 処理対象条件
* 業務上の制約
* 申請・承認ルール
* 締処理ルール
* 確定・取消ルール
* 業務例外
* 対象データ抽出条件
* 業務的な可否判定
* 法令・会計基準・社内規程に基づく条件

### 例

```text
承認済みデータのみ締処理対象とする。
締済み期間の取引データは変更できない。
取消済み申請は承認対象外とする。
親会社の取引のみ連結調整対象とする。
```

### 分類時の注意

入力項目のチェック条件は `validation` とする。

状態変化そのものが中心の場合は `state_transition` とする。

権限による操作可否が中心の場合は `permission` とする。

---

## 4.9 `validation`

### 定義

入力値、項目値、リクエスト値などに対するチェック・制約に関する情報。

### 該当する情報

* 必須
* 任意
* 桁数
* 文字種
* 型
* 範囲
* 上限
* 下限
* 日付妥当性
* 相関チェック
* 重複チェック
* 存在チェック
* 入力不可条件
* フォーマット
* 禁止文字
* 入力エラー

### 例

```text
金額は0以上の整数のみ入力可能とする。
取引日は必須入力とする。
コードは10桁以内の半角英数字とする。
開始日は終了日以前でなければならない。
```

### 分類時の注意

業務上の処理対象条件は `business_rule` とする。

DBカラムの型や桁数そのものは `data` とする。

ただし、画面入力やAPI入力の制約として使われる場合は `validation` を優先する。

---

## 4.10 `state_transition`

### 定義

ステータス、状態、ライフサイクル、状態遷移条件に関する情報。

### 該当する情報

* ステータス一覧
* 遷移前状態
* 遷移後状態
* 遷移条件
* 不正遷移
* 申請状態
* 承認状態
* 締状態
* 連携状態
* 処理状態
* ライフサイクル

### 例

```text
申請するとステータスは下書きから承認待ちに変わる。
承認するとステータスは承認済みになる。
締処理後、対象データは締済みとなる。
連携失敗時、連携状態はエラーとなる。
```

### 分類時の注意

状態による操作可否が中心の場合は `permission` または `business_rule` とする。

状態による画面表示制御が中心の場合は `ui_item` または `screen` とする。

---

## 4.11 `condition_branch`

### 定義

条件によって処理、表示、出力、計算、連携などが分岐する情報。

### 該当する情報

* if条件
* 区分別処理
* パターン別処理
* 条件別表示
* 条件別出力
* 条件別計算
* 条件別連携
* 組み合わせ条件

### 例

```text
請求区分が月額の場合は月額料金を使用する。
請求区分が従量の場合は利用量を集計する。
海外取引の場合は為替換算を行う。
承認状態が未承認の場合は編集可能とする。
```

### 分類時の注意

分岐条件が業務ルールとして重要な場合は `business_rule` とする。

状態の変化が中心の場合は `state_transition` とする。

計算式の違いが中心の場合は `calculation` とする。

---

## 4.12 `calculation`

### 定義

計算、集計、変換、丸め、換算、按分に関する情報。

### 該当する情報

* 計算式
* 集計条件
* 合計
* 差額
* 税計算
* 為替換算
* 丸め
* 端数処理
* 按分
* 変換
* 単位変換
* 金額計算
* 比率計算

### 例

```text
税込金額は税抜金額に消費税額を加算して算出する。
小数点以下は四捨五入する。
外貨金額は為替レートを乗じて円換算する。
部門別金額は配賦率に基づき按分する。
```

### 分類時の注意

計算対象データの抽出条件は `business_rule` とする。

計算結果の帳票表示は `report` とする。

入力値の数値制約は `validation` とする。

---

## 4.13 `permission`

### 定義

権限、ロール、ユーザー種別、操作可否、参照可否、更新可否に関する情報。

### 該当する情報

* ロール
* 権限
* 操作可否
* 参照可否
* 登録可否
* 更新可否
* 削除可否
* 承認可否
* 管理者権限
* 一般ユーザー権限
* 所属部門による制御
* データ範囲制御

### 例

```text
管理者のみユーザーを登録できる。
承認権限を持つユーザーのみ承認ボタンを押下できる。
一般ユーザーは自部門のデータのみ参照できる。
締済みデータは管理者でも変更できない。
```

### 分類時の注意

画面上の表示・非表示が中心の場合は `ui_item` としても扱える。

業務上の可否条件が中心の場合は `business_rule` とする。

---

## 4.14 `error_handling`

### 定義

エラー、警告、例外、異常系、リトライ、ロールバック、リカバリに関する情報。

### 該当する情報

* エラー条件
* エラーメッセージ
* 警告
* 例外処理
* 異常終了
* リトライ
* ロールバック
* スキップ
* 再実行
* ログ出力
* 通知
* タイムアウト
* 復旧手順

### 例

```text
連携先がタイムアウトした場合はエラーとしてログ出力する。
登録処理に失敗した場合はトランザクションをロールバックする。
必須項目が未入力の場合はエラーを表示する。
バッチ異常終了時は運用担当者へ通知する。
```

### 分類時の注意

入力チェックそのものは `validation` とする。

バッチ異常終了の運用手順が中心の場合は `operation` とする。

外部連携の通信仕様が中心の場合は `api` とする。

---

## 4.15 `operation`

### 定義

運用、保守、移行、監視、手動作業、リリース後の業務手順に関する情報。

### 該当する情報

* 運用手順
* 保守手順
* 監視
* 通知
* リカバリ
* 手動作業
* データ補正
* 移行作業
* 初期設定
* バックアップ
* リストア
* ジョブ監視
* 障害対応
* 問い合わせ対応

### 例

```text
異常終了時は運用担当者が再実行する。
毎月末に管理者が締処理を実行する。
移行前に既存データのバックアップを取得する。
障害発生時はログを確認し、必要に応じて手動補正する。
```

### 分類時の注意

運用手順内の具体的なシステム動作は、該当カテゴリに分類してよい。

例えば、再実行時のエラー処理が中心なら `error_handling` とする。

---

## 4.16 `non_functional`

### 定義

性能、可用性、セキュリティ、互換性、保守性、運用性など、機能そのものではない品質特性に関する情報。

### 該当する情報

* 性能
* 応答時間
* 処理時間
* 同時接続数
* 可用性
* 障害復旧
* セキュリティ
* 認証
* 認可
* 監査ログ
* 互換性
* ブラウザ対応
* OS対応
* 保守性
* 運用性
* 拡張性
* 移行性

### 例

```text
検索結果は3秒以内に表示する。
同時に100ユーザーが利用できること。
操作ログを90日間保持する。
Windows 11上で動作すること。
```

### 分類時の注意

認可による操作可否は `permission` とする。

監査ログの具体的な出力仕様は `data` または `operation` としても扱える。

性能値が明記されている場合は `non_functional` とする。

---

## 4.17 `open_issue`

### 定義

未確定、要確認、不足、矛盾、TODO、検討中の情報。

仕様として確定できない情報を表す。

### 該当する情報

* 未定
* TBD
* TODO
* 検討中
* 別途確認
* 後日決定
* 暫定
* 仮
* 要調整
* 仕様確認中
* 矛盾
* 不足
* 判断不可

### 例

```text
リトライ回数は別途検討。
権限別の操作範囲は後日確定。
帳票レイアウトは仮。
APIエラーコードは未定。
```

### 分類時の注意

`open_issue` は、確定仕様ではない。

Step 3以降では、状態を `to_be_confirmed`、`missing`、`insufficient_detail`、`contradiction` などとして扱う。

---

# 5. 判断に迷いやすいカテゴリの切り分け

## 5.1 `business_rule` と `validation`

### `business_rule` にする場合

業務上の処理可否、対象条件、業務制約を表す場合。

```text
承認済みデータのみ締処理対象とする。
締済み期間のデータは変更できない。
```

### `validation` にする場合

入力値や項目値に対するチェックを表す場合。

```text
金額は0以上の整数のみ入力可能とする。
取引日は必須入力とする。
```

---

## 5.2 `ui_item` と `validation`

### `ui_item` にする場合

項目の存在、表示、活性・非活性などが中心の場合。

```text
金額項目を表示する。
締済みの場合、削除ボタンは非活性とする。
```

### `validation` にする場合

入力値の制約が中心の場合。

```text
金額は0以上の整数のみ入力可能とする。
コードは10桁以内とする。
```

---

## 5.3 `condition_branch` と `business_rule`

### `condition_branch` にする場合

処理が条件によって分岐することが中心の場合。

```text
請求区分が月額の場合は月額料金を使用する。
請求区分が従量の場合は利用量を集計する。
```

### `business_rule` にする場合

業務上の可否判断や対象条件が中心の場合。

```text
承認済みデータのみ支払処理に進める。
締済み期間のデータは変更不可とする。
```

---

## 5.4 `condition_branch` と `calculation`

### `condition_branch` にする場合

分岐の存在が中心の場合。

```text
請求区分により計算方法を切り替える。
```

### `calculation` にする場合

具体的な計算式や丸めが中心の場合。

```text
従量料金は利用量に単価を乗じて算出する。
小数点以下は四捨五入する。
```

---

## 5.5 `state_transition` と `business_rule`

### `state_transition` にする場合

状態がどう変わるかが中心の場合。

```text
承認するとステータスは承認済みになる。
```

### `business_rule` にする場合

状態を条件にした業務上の可否が中心の場合。

```text
承認済みデータのみ締処理対象とする。
```

---

## 5.6 `permission` と `business_rule`

### `permission` にする場合

ユーザー、ロール、権限による操作可否が中心の場合。

```text
管理者のみ再実行できる。
```

### `business_rule` にする場合

業務状態や業務条件による処理可否が中心の場合。

```text
締済み期間のデータは変更できない。
```

---

## 5.7 `error_handling` と `validation`

### `validation` にする場合

入力値チェックの条件が中心の場合。

```text
必須項目が未入力の場合はエラーとする。
```

### `error_handling` にする場合

エラー発生後の動作が中心の場合。

```text
エラー発生時は処理を中断し、ログを出力する。
```

---

## 5.8 `api` と `error_handling`

### `api` にする場合

APIの入出力、通信、連携仕様が中心の場合。

```text
POST /orders は注文データを登録する。
レスポンスには処理結果コードを返す。
```

### `error_handling` にする場合

エラー時の振る舞いが中心の場合。

```text
連携先タイムアウト時は3回までリトライする。
```

---

## 5.9 `batch` と `operation`

### `batch` にする場合

バッチ処理そのものの仕様が中心の場合。

```text
日次集計バッチは未集計データを対象に実行する。
```

### `operation` にする場合

人手の運用、監視、復旧手順が中心の場合。

```text
異常終了時は運用担当者がログを確認し、再実行する。
```

---

## 5.10 `data` と `calculation`

### `data` にする場合

保存先、項目、データ更新が中心の場合。

```text
税込金額を取引明細テーブルに保存する。
```

### `calculation` にする場合

税込金額の算出方法が中心の場合。

```text
税込金額は税抜金額に消費税額を加算して算出する。
```

---

# 6. カテゴリ選択の簡易フロー

分類に迷った場合は、以下の順で考える。

```text
1. 未確定・TODO・要確認か？
   → open_issue

2. 入力値や項目値の制約か？
   → validation

3. 状態が変わる話か？
   → state_transition

4. 権限・ロールによる操作可否か？
   → permission

5. 計算・集計・丸め・換算か？
   → calculation

6. エラー時・例外時・リトライ・ロールバックか？
   → error_handling

7. API・外部連携・通信・ファイル連携か？
   → api

8. バッチ・ジョブ・非同期処理か？
   → batch

9. DB・テーブル・データ更新か？
   → data

10. 帳票・ファイル出力・印刷か？
    → report

11. 画面・画面遷移・画面構成か？
    → screen

12. UI項目・ボタン・表示項目か？
    → ui_item

13. 業務上の判断・制約・処理対象条件か？
    → business_rule

14. 機能や処理のまとまりか？
    → feature

15. どれにも分類できないが確認が必要か？
    → open_issue
```

---

# 7. カテゴリごとのテスト観点例

この表は、Step 4でテスト設計インプット化するときの参考にする。

| カテゴリ               | テスト観点例                   |
| ------------------ | ------------------------ |
| `feature`          | 機能の成立条件、処理対象、正常系、主要業務フロー |
| `screen`           | 初期表示、画面遷移、表示条件、操作後遷移     |
| `ui_item`          | 表示・非表示、活性・非活性、初期値、操作可否   |
| `report`           | 出力条件、出力項目、編集ルール、形式、並び順   |
| `api`              | リクエスト、レスポンス、認証、エラー、連携条件  |
| `batch`            | 起動条件、対象データ、処理順、異常終了、再実行  |
| `data`             | 登録、更新、削除、整合性、履歴、排他       |
| `business_rule`    | 業務条件、処理可否、対象データ、業務例外     |
| `validation`       | 必須、型、桁数、範囲、相関、境界値、不正値    |
| `state_transition` | 遷移前状態、遷移後状態、遷移条件、不正遷移    |
| `condition_branch` | 条件別動作、分岐網羅、組み合わせ         |
| `calculation`      | 計算式、丸め、端数、集計範囲、ゼロ・マイナス   |
| `permission`       | ロール別操作可否、参照可否、更新可否       |
| `error_handling`   | エラー条件、警告、リトライ、ロールバック、ログ  |
| `operation`        | 運用手順、監視、再実行、復旧、手動作業      |
| `non_functional`   | 性能、セキュリティ、互換性、可用性、運用性    |
| `open_issue`       | 確認事項、保留事項、テスト設計前の解消要否    |

# 8. 出力時の記載ルール

## 8.1 カテゴリ名は固定値を使う

カテゴリ名は、このファイルで定義された英字の固定値を使用する。

良い例:

```text
business_rule
validation
state_transition
```

悪い例:

```text
業務ルール
入力チェック
状態
```

ただし、説明文や見出しでは日本語を使用してよい。

---

## 8.2 独自カテゴリを増やさない

原則として、AIエージェントは独自カテゴリを追加しない。

分類できない場合は、以下のいずれかで対応する。

* 既存カテゴリのうち最も近いものを選ぶ
* 備考に分類理由を書く
* `open_issue` として扱う

---

## 8.3 関連カテゴリは備考に書く

複数カテゴリに関係する場合、主カテゴリを1つ選び、関連カテゴリを備考に記録する。

例:

```text
カテゴリ: business_rule
備考: state_transitionにも関連。締済み状態を条件とする処理可否。
```

---

# 9. 禁止事項

以下を禁止する。

* 原文にない仕様を補ってカテゴリ分類する
* 独自カテゴリを増やす
* 同じ種類の情報に別カテゴリを使い続ける
* 未確定事項を確定カテゴリに分類する
* 入力制約が明記されていない項目を `validation` と断定する
* 業務ルールと入力チェックを混同する
* 状態遷移と条件分岐を混同する
* エラー条件とエラー後動作を混同する
* カテゴリ分類のために期待動作を推測する

---

# 10. 完了確認

カテゴリ分類後は、以下を確認する。

* 定義済みカテゴリのいずれかを使用している
* 主カテゴリが1つに絞られている
* 関連カテゴリは備考に記載されている
* 分類理由が不明なものは備考に残している
* 未確定事項を確定仕様として扱っていない
* 推測による分類が混入していない
