# step4_test_design_input_guide.md

## 1. このステップの目的

Step 4では、Step 3で作成した `normalized_spec_inventory.md` を入力として、テスト設計に利用できるインプット情報へ変換する。

このステップの目的は、正規化された仕様情報を、後続のテスト設計工程で扱いやすい形に整理することである。

このステップでは、以下を行う。

- テスト対象を整理する
- 確認すべき仕様を整理する
- テスト観点を抽出する
- 必要な事前条件・状態を整理する
- 入力データ観点を整理する
- 期待結果の種類を整理する
- 関連する画面・API・DB・帳票・バッチ等を整理する
- 優先度やリスクを付与する
- 参照元 `SPEC-ID` を記録する

このステップでは、以下を行わない。

- 詳細なテストケースを作成する
- 操作手順を作成する
- 具体的なテストデータ値を網羅展開する
- 期待結果を推測で断定する
- 未確定仕様を確定扱いする

---

## 2. このステップで作成する成果物

このステップでは、以下の成果物を作成する。

```text
test_design_input_catalog.md
````

`test_design_input_catalog.md` は、後続のテスト設計工程に引き渡すための入力カタログである。

この成果物は、テストケース一覧ではない。

## 5. 実施手順

### 5.1 入力成果物を確認する

まず、以下を確認する。

* `normalized_spec_inventory.md` が存在するか
* 各仕様に `SPEC-ID` が付与されているか
* 各仕様にカテゴリが付与されているか
* 各仕様に状態が付与されているか
* 各仕様に参照 `EXT-ID` があるか
* 矛盾・不足・未確定事項が明示されているか

不足がある場合は、可能な範囲で処理し、不足内容を明示する。

---

### 5.2 テスト設計インプット化の対象を選別する

`normalized_spec_inventory.md` の仕様情報から、テスト設計インプット化する対象を選別する。

原則として、以下は対象とする。

* 状態が `confirmed` の仕様
* 状態が `partial` だが、確認可能な範囲がある仕様
* 状態が `insufficient_detail` だが、確認観点として扱える仕様
* 状態が `to_be_confirmed` で、後続工程への注意喚起が必要な仕様
* 状態が `contradiction` で、テスト設計前に確認が必要な仕様

ただし、扱いは状態によって変える。

| 状態                    | Step 4での扱い         |
| --------------------- | ------------------ |
| `confirmed`           | 通常のテスト設計インプットとして扱う |
| `partial`             | 分かっている範囲のみインプット化する |
| `insufficient_detail` | テスト観点候補と不足情報を記録する  |
| `to_be_confirmed`     | 確認前提のインプットとして扱う    |
| `contradiction`       | テスト設計を保留または注意付きで扱う |
| `missing`             | テスト設計に必要な不足情報として扱う |
| `deprecated_or_stale` | 原則として参考扱いにする       |
| `out_of_scope`        | 原則として対象外にする        |

---

### 5.3 テスト対象を整理する

各仕様について、テスト対象を整理する。

テスト対象の例:

* 機能
* 業務処理
* 画面
* 画面項目
* API
* 帳票
* ファイル出力
* バッチ
* 外部連携
* DB更新
* 権限制御
* 状態遷移
* 計算処理
* 入力チェック
* エラー処理
* 運用手順

テスト対象は、後続のテスト設計者が「何を確認する話か」をすぐ理解できる粒度で記載する。

---

### 5.4 確認すべき仕様を整理する

各テスト対象について、確認すべき仕様を簡潔に記録する。

記録例:

```text
承認済みデータのみ締処理対象となること。
金額は0以上の整数のみ入力可能であること。
APIタイムアウト時の扱いは未確定であること。
```

注意点:

* 原文にない期待結果を追加しない
* 未確定仕様を断定しない
* 矛盾がある場合は、矛盾があることを明記する
* 詳細テスト手順に展開しない

---

### 5.5 テスト観点を抽出する

確認すべき仕様から、テスト観点を抽出する。

テスト観点は、具体的なテストケースではなく、確認の切り口として記録する。

観点の例:

| 仕様カテゴリ             | テスト観点例                     |
| ------------------ | -------------------------- |
| `validation`       | 入力値制約、必須、桁数、型、境界値、不正値      |
| `business_rule`    | 業務条件、対象データ抽出、処理可否、業務例外     |
| `state_transition` | 遷移前状態、遷移後状態、遷移条件、不正遷移      |
| `condition_branch` | 条件別動作、分岐網羅、組み合わせ           |
| `calculation`      | 計算式、丸め、集計範囲、ゼロ・マイナス値       |
| `permission`       | ロール別操作可否、参照可否、更新可否         |
| `error_handling`   | エラー条件、警告、リトライ、ロールバック       |
| `api`              | リクエスト、レスポンス、エラーコード、認証、連携条件 |
| `batch`            | 起動条件、対象データ、再実行、異常終了、順序     |
| `report`           | 出力条件、出力項目、編集ルール、ファイル形式     |
| `data`             | 登録、更新、削除、整合性、排他、履歴         |
| `non_functional`   | 性能、セキュリティ、互換性、可用性、運用性      |

---

### 5.6 必要な条件・前提を整理する

テスト設計に必要な条件や前提があれば記録する。

例:

* 対象データの状態
* ユーザー権限
* 処理日時
* 締状態
* マスタ設定
* 外部連携先の状態
* バッチ実行前後の状態
* 入力済みデータの有無
* システム設定
* 運用モード
* 移行前後の状態

条件が原文から分からない場合は、推測せず `不明` または `要確認` とする。

---

### 5.7 入力データ観点を整理する

仕様から、必要な入力データ観点を整理する。

これは具体的なテストデータ値を作成することではない。

許可される例:

```text
0、正の整数、負数、小数、空欄、最大桁数超過などの観点が必要。
```

禁止される例:

```text
テストデータ1: 金額 = 0
テストデータ2: 金額 = 1
テストデータ3: 金額 = -1
```

このステップでは、データの種類・切り口までに留める。

---

### 5.8 期待結果の種類を整理する

仕様に基づき、どのような期待結果を確認する必要があるかを整理する。

期待結果の種類の例:

* 画面表示
* 入力可否
* エラーメッセージ
* 警告表示
* 登録結果
* 更新結果
* 削除結果
* ステータス変更
* DB更新
* APIレスポンス
* 帳票出力
* ファイル出力
* ログ出力
* 通知
* 外部連携結果
* バッチ終了状態
* 権限による操作可否
* データ整合性

注意点:

* 具体的な期待結果文は、仕様に明記されている範囲でのみ書く
* 未記載の場合は、期待結果の種類だけを書く
* エラーメッセージ文言を推測しない

---

### 5.9 関連コンポーネントを整理する

テスト設計時に関連しそうなコンポーネントを記録する。

例:

* 画面
* API
* DB
* テーブル
* 帳票
* バッチ
* 外部システム
* ファイル
* マスタ
* 権限定義
* 設定値
* ログ
* 通知

関連コンポーネントが不明な場合は、`不明` とする。

---

### 5.10 優先度を設定する

各テスト設計インプットには、優先度を付与する。

優先度は以下から選択する。

| 優先度     | 意味                      |
| ------- | ----------------------- |
| High    | 重要機能、業務影響大、障害影響大、仕様確認必須 |
| Medium  | 通常の確認対象                 |
| Low     | 影響が限定的、参考扱い、後回し可能       |
| Unknown | 判断材料が不足している             |

優先度は、仕様の重要度、リスク、影響範囲、確認の必要性を踏まえて設定する。

判断できない場合は `Unknown` とする。

---

### 5.11 リスクを整理する

必要に応じて、テスト設計上のリスクを記録する。

リスクの例:

* 業務影響が大きい
* 金額・会計値に影響する
* データ不整合につながる
* 権限漏れにつながる
* 外部連携失敗につながる
* 障害時の復旧が困難
* 仕様が曖昧
* 資料間に矛盾がある
* テストデータ準備が難しい
* 期待結果を確定できない

リスクは、後続のテスト設計で優先度や深さを決める材料になる。

---

### 5.12 未確定・矛盾ありの仕様を扱う

状態が `to_be_confirmed`、`contradiction`、`missing`、`insufficient_detail` の仕様は、断定的なテスト設計インプットにしない。

扱い例:

```text
仕様未確定のため、タイムアウト時の期待動作確認が必要。
```

```text
資料間で金額桁数が異なるため、境界値設計前に仕様確認が必要。
```

この場合、テスト観点は書いてよいが、期待結果を断定してはならない。

後続工程に渡す注意事項を記録する。

* 外部連携エラーはリトライ仕様確認後に異常系を設計する
* 帳票項目は編集条件確認後に出力検証観点へ展開する

## 8. TDI-ID の採番ルール

テスト設計インプットには、以下の形式でIDを付与する。

```text
TDI-001
TDI-002
TDI-003
```

カテゴリ別に分けたい場合は、以下の形式を使用してもよい。

```text
TDI-VAL-001
TDI-BR-001
TDI-API-001
```

ただし、プロジェクト内で採番形式は統一する。

## 9. テスト観点化の判断基準

### 9.1 テスト観点に変換してよい例

仕様:

```text
金額は0以上の整数のみ入力可能である。
```

テスト観点:

```text
金額入力の境界値、不正値、型制約。
```

入力データ観点:

```text
0、正の整数、負数、小数、空欄、最大桁数超過。
```

---

仕様:

```text
承認済みデータのみ締処理対象とする。
```

テスト観点:

```text
対象データ抽出条件、承認状態別の処理可否。
```

必要条件・前提:

```text
承認済み、未承認、差戻しなどの状態を持つデータ。
```

---

仕様:

```text
管理者のみ再実行できる。
```

テスト観点:

```text
ロール別操作可否、権限境界。
```

必要条件・前提:

```text
管理者ユーザー、一般ユーザー。
```

---

### 9.2 具体的なテストケースにしてしまう悪い例

以下のような出力は禁止する。

```text
TC-001: 金額に0を入力し、登録できることを確認する。
TC-002: 金額に-1を入力し、エラーになることを確認する。
TC-003: 一般ユーザーで再実行ボタンを押し、エラーになることを確認する。
```

このステップでは、そこまで展開しない。

---

### 9.3 期待結果を断定してよい場合

期待結果を断定してよいのは、元仕様に明記されている場合のみである。

元仕様:

```text
未承認データを締処理対象に含めた場合、エラーとする。
```

許可される記載:

```text
期待結果の種類: エラー表示、処理中断
備考: 未承認データを含む場合はエラーと明記あり
```

---

### 9.4 期待結果を断定してはならない場合

元仕様:

```text
未承認データは締処理対象外とする。
```

禁止される記載:

```text
未承認データを含む場合、エラーメッセージ「未承認データがあります」を表示する。
```

理由:

```text
エラー表示やメッセージ文言は原文にないため。
```

## 10. 優先度判断の補足

### 10.1 Highにしやすいもの

以下は、優先度 `High` にしやすい。

* 金額・会計値に影響する仕様
* データ更新に関係する仕様
* 業務フローの主要分岐
* 承認・締処理・確定処理
* 権限制御
* 外部連携
* エラー時の復旧
* 法令・会計基準・監査に関係する仕様
* 仕様が曖昧だが影響が大きいもの
* 資料間で矛盾しているもの

---

### 10.2 Mediumにしやすいもの

以下は、優先度 `Medium` とする。

* 通常の画面項目確認
* 一般的な入力チェック
* 通常のAPI入出力
* 通常の帳票出力
* 業務影響が限定的な条件分岐

---

### 10.3 Lowにしやすいもの

以下は、優先度 `Low` とする。

* 表示文言など軽微な仕様
* 補足的な画面表示
* 影響範囲が狭い仕様
* 対象外に近い参考情報
* 旧仕様由来で現行対象外の可能性が高いもの

---

### 10.4 Unknownにするもの

以下は、優先度 `Unknown` とする。

* 影響範囲が不明
* 対象機能が不明
* 仕様状態が不明
* 資料の位置づけが不明
* 判断材料が不足している

---

## 11. 禁止事項

このステップでは、以下を禁止する。

* 詳細なテストケースを作成する
* 操作手順を作成する
* テストケースIDを作成する
* 具体的なテストデータ値を網羅展開する
* 期待結果を推測で作成する
* エラーメッセージ文言を推測する
* 未記載の画面遷移を作成する
* 未記載のDB更新結果を断定する
* 未確定仕様を確定扱いする
* 矛盾する仕様のどちらかを勝手に採用する
* 参照 `SPEC-ID` を失わせる
* Step 5のレビュー結果を先に確定する

---

## 12. 例外的に許可されること

以下は許可する。

### 12.1 テスト観点の抽象化

仕様情報を、テスト設計で使いやすい観点に抽象化してよい。

---

### 12.2 入力データ観点の列挙

具体的なテストケースではなく、データの種類や境界の観点を列挙してよい。

---

### 12.3 期待結果の種類の整理

具体的な期待結果ではなく、確認すべき結果の種類を整理してよい。

---

### 12.4 リスクの記録

後続のテスト設計に影響するリスクを記録してよい。

---

### 12.5 未確定事項の注意喚起

未確定・矛盾・不足がある仕様について、後続工程への注意事項として記録してよい。

---

## 13. 完了条件

このステップは、以下を満たしたら完了とする。

* `test_design_input_catalog.md` が作成されている
* 各項目に `TDI-ID` が付与されている
* 各項目に参照 `SPEC-ID` が記録されている
* テスト対象が整理されている
* 確認すべき仕様が整理されている
* テスト観点が整理されている
* 必要条件・前提が整理されている
* 入力データ観点が整理されている
* 期待結果の種類が整理されている
* 優先度とリスクが記録されている
* 未確定・矛盾ありの仕様が断定されていない
* 具体的なテストケースを作成していない
* Step 5には進まず停止している
