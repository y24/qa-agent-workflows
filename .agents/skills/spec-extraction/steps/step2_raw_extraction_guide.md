# step2_raw_extraction_guide.md

## 1. このステップの目的

Step 2では、Step 1で整理した入力ドキュメントから、テスト設計に必要な仕様情報を抽出する。

このステップの目的は、ドキュメントに明記されている事実情報を、後続工程で扱いやすい形に記録することである。

このステップでは、以下を行う。

- ドキュメントに書かれている仕様情報を抽出する
- 抽出情報に `EXT-ID` を付与する
- 参照元を記録する
- 抽出カテゴリを付与する
- 確度を記録する
- 曖昧な記述や未確定情報を明示する

このステップでは、以下を行わない。

- 複数資料の内容を統合する
- 仕様の正誤を判断する
- 不足情報を補完する
- テスト観点へ変換する
- テストケースを作成する

---

## 2. このステップで作成する成果物

このステップでは、以下の成果物を作成する。

```text
raw_extraction.md
````

`raw_extraction.md` は、ドキュメントから抽出した生の仕様情報を記録する成果物である。

後続の Step 3 では、この `raw_extraction.md` を入力として、仕様情報の正規化・統合を行う。

---

## 3. 入力

このステップの入力は、以下である。

* `document_inventory.md`
* 抽出対象ドキュメント本文
* 抽出対象の章・節・ページ・範囲
* 必要に応じて、ユーザーからの補足説明

`document_inventory.md` に記載された優先度、対象領域、読む範囲に従って抽出する。

## 5. 実施手順

### 5.1 抽出対象範囲を確認する

まず、今回処理する対象範囲を確認する。

確認する項目:

* 対象 `DOC-ID`
* ドキュメント名
* 対象章・節・ページ・行
* 対象機能・対象画面・対象APIなど
* 今回処理しない範囲

対象範囲が不明な場合は、入力された範囲だけを対象とする。

対象外の範囲を勝手に補完してはならない。

---

### 5.2 テスト設計に関係する情報を抽出する

ドキュメント本文から、テスト設計に必要となる可能性がある情報を抽出する。

抽出対象の例:

* 機能
* 処理
* 業務ルール
* 画面
* 画面項目
* 入力項目
* 表示項目
* ボタン
* 画面遷移
* 帳票
* ファイル出力
* API
* 外部連携
* バッチ
* データ更新
* DB項目
* マスタ
* 入力チェック
* 桁数
* 型
* 必須条件
* 条件分岐
* 状態遷移
* 計算
* 集計
* 変換
* 丸め
* 権限
* ロール
* エラー
* 警告
* 例外処理
* リトライ
* 運用制約
* 移行条件
* 非機能要件
* 未確定事項
* 注意事項

---

### 5.3 抽出単位を小さく保つ

1つの抽出情報には、原則として1つの仕様事実だけを記録する。

悪い例:

```text
承認済みデータのみ締処理対象となり、未承認データはエラーとなり、管理者のみ再実行できる。
```

この場合は、以下のように分割する。

```text
承認済みデータのみ締処理対象となる。
未承認データは締処理対象外となる。
未承認データが含まれる場合の扱いはエラーである。
再実行できるのは管理者のみである。
```

ただし、原文の1文が不可分の業務ルールである場合は、無理に分割しすぎない。

---

### 5.4 原文位置を記録する

各抽出情報には、可能な限り原文位置を記録する。

記録例:

```text
DOC-001 3.2.1
DOC-002 p.12
DOC-003 画面項目一覧「金額」
DOC-004 API一覧 / POST /orders
DOC-005 第4章 エラー処理
```

行番号が分かる場合は、行番号も記録する。

原文位置が不明な場合は、`位置不明` と記録する。

---

### 5.5 抽出内容を要約する

抽出内容は、原文の意味を変えずに簡潔に記録する。

原文をそのまま引用してもよい。

ただし、長すぎる場合は要約する。

要約する場合の注意:

* 意味を変えない
* 条件を落とさない
* 例外を落とさない
* 主語を勝手に変えない
* 曖昧な表現を断定に変えない
* 未記載の語を補わない

---

### 5.6 カテゴリを付与する

抽出情報には、カテゴリを付与する。

カテゴリは、原則として以下から選択する。

| カテゴリ               | 内容                      |
| ------------------ | ----------------------- |
| `feature`          | 機能、処理単位、ユースケース          |
| `screen`           | 画面、画面遷移、画面部品            |
| `ui_item`          | 入力項目、表示項目、ボタン、リンク       |
| `report`           | 帳票、ファイル出力、印刷、Excel/PDF等 |
| `api`              | API、インターフェース、外部連携       |
| `batch`            | バッチ、ジョブ、非同期処理           |
| `data`             | DB、テーブル、項目、マスタ、トランザクション |
| `business_rule`    | 業務ルール、業務上の判定条件          |
| `validation`       | 入力チェック、制約、必須、桁数、型       |
| `state_transition` | ステータス、状態遷移、ライフサイクル      |
| `condition_branch` | 条件分岐、場合分け               |
| `calculation`      | 計算、集計、変換、丸め             |
| `permission`       | 権限、ロール、操作可否             |
| `error_handling`   | エラー、警告、例外、リトライ          |
| `operation`        | 運用、保守、移行、監視、手動作業        |
| `non_functional`   | 性能、セキュリティ、可用性、互換性等      |
| `open_issue`       | 未確定、要確認、不足、矛盾           |

分類に迷う場合は、主カテゴリを1つ選び、備考に関連カテゴリを記録する。

---

### 5.7 確度を付与する

抽出情報には、確度を付与する。

| 確度      | 意味                     |
| ------- | ---------------------- |
| High    | 原文に明確に記載されている          |
| Medium  | 原文から直接読み取れるが、表現に曖昧さがある |
| Low     | 関連記述はあるが、仕様として確定しにくい   |
| Unknown | 判断できない                 |

確度は、情報の重要度ではない。

重要な仕様でも、原文が曖昧であれば `Medium` または `Low` とする。

---

### 5.8 曖昧さを記録する

抽出した情報に曖昧さがある場合は、備考に記録する。

曖昧さの例:

* 「必要に応じて」と書かれているが、条件が不明
* 「通常は」と書かれているが、例外条件が不明
* 「管理者」とあるが、具体的なロール名が不明
* 「エラーとする」とあるが、エラーメッセージが不明
* 「一定時間」とあるが、具体的な時間が不明
* 「対象データ」とあるが、抽出条件が不明
* 「最新データ」とあるが、判定基準が不明

---

### 5.9 未確定事項を抽出する

ドキュメント内に未確定事項、TODO、検討中、要確認などの記載がある場合は、必ず抽出する。

カテゴリは原則として `open_issue` とする。

例:

```text
エラー時のリトライ回数は別途検討。
権限別の操作範囲は後日確定。
帳票レイアウトは仮。
```

未確定事項を仕様として確定扱いしてはならない。

---

### 5.10 抽出しない情報を判断する

以下は、原則として抽出対象外とする。

* 仕様と関係しない一般的な説明
* 単なる背景説明
* 議事録上の雑談
* 作業担当者の個人的なコメント
* テスト設計に影響しない文章装飾
* 既に廃止された仕様
* 今回の対象外と明記された機能
* 単なる画像やデザイン上の装飾で、仕様判断に使えないもの

ただし、テスト設計に影響する可能性がある場合は、備考付きで抽出してよい。

## 6. 出力形式

成果物は `raw_extraction.md` として、以下の構成で出力する。

```md
# raw_extraction.md

## 1. 概要

## 2. 抽出対象範囲

## 3. 抽出結果

## 5. 曖昧・未確定な記述

## 6. 未処理範囲

## 7. 完了判定
```

## 7. `raw_extraction.md` の記載項目

### 7.1 抽出結果

以下の表を使用する。

| EXT-ID | DOC-ID | 原文位置 | カテゴリ | 抽出内容 | 確度 | 備考 |
| ------ | ------ | ---- | ---- | ---- | -- | -- |

各項目の説明:

| 項目     | 説明                            |
| ------ | ----------------------------- |
| EXT-ID | 抽出情報に付与する一意のID                |
| DOC-ID | 参照元ドキュメントID                   |
| 原文位置   | 章、節、ページ、行、表名など                |
| カテゴリ   | 抽出カテゴリ                        |
| 抽出内容   | 原文に基づく仕様情報                    |
| 確度     | High / Medium / Low / Unknown |
| 備考     | 曖昧さ、関連カテゴリ、注意点など              |

---

### 7.2 抽出しなかった情報

抽出対象外とした情報がある場合は、以下の表を使用する。

| DOC-ID | 原文位置 | 抽出しなかった内容 | 理由 |
| ------ | ---- | --------- | -- |

理由の例:

```text
テスト設計への影響が薄い
背景説明のみ
旧仕様として明記
対象外機能
仕様情報ではない
```

---

### 7.3 曖昧・未確定な記述

曖昧または未確定な記述は、抽出結果とは別に一覧化する。

| EXT-ID | DOC-ID | 原文位置 | 曖昧・未確定な内容 | 後続影響 |
| ------ | ------ | ---- | --------- | ---- |

この一覧は、Step 5の `gap_and_review_report.md` の入力になる。

---

### 7.4 未処理範囲

処理していない範囲がある場合は、以下の表を使用する。

| DOC-ID | 未処理範囲 | 理由 | 次回処理要否 |
| ------ | ----- | -- | ------ |

次回処理要否は、以下のいずれかとする。

```text
必要
不要
不明
```

---

## 8. EXT-ID の採番ルール

抽出情報には、以下の形式でIDを付与する。

```text
EXT-001
EXT-002
EXT-003
```

分割処理で追記する場合は、既存IDと重複しないように採番する。

ドキュメント単位で分けたい場合は、以下の形式を使用してもよい。

```text
EXT-DOC001-001
EXT-DOC001-002
EXT-DOC002-001
```

ただし、プロジェクト内で採番形式は統一する。

---

## 9. 抽出粒度の判断基準

### 9.1 分割すべき場合

以下の場合は、抽出情報を分割する。

* 1文に複数の条件が含まれる
* 1文に複数の期待動作が含まれる
* 正常系と異常系が同時に書かれている
* 権限と処理条件が混在している
* 入力制約とエラー処理が混在している
* 複数の画面・API・帳票にまたがる
* 複数の状態遷移が含まれる

---

### 9.2 分割しすぎない場合

以下の場合は、無理に分割しすぎない。

* 1つの業務ルールとして不可分である
* 条件と動作が一体で意味を持つ
* 原文上、分割すると意味が曖昧になる
* テスト設計上、まとめて扱った方が自然である

---

## 10. カテゴリ分類の補足

### 10.1 `business_rule` と `validation` の違い

`business_rule` は、業務上の判断や制約を表す。

例:

```text
承認済みの申請のみ支払処理に進める。
締済み期間のデータは変更できない。
```

`validation` は、入力値や項目に対するチェックを表す。

例:

```text
金額は0以上の整数のみ入力可能。
取引日は必須入力。
コードは10桁以内。
```

---

### 10.2 `condition_branch` と `state_transition` の違い

`condition_branch` は、条件によって処理が分かれることを表す。

例:

```text
請求区分が月額の場合は月次処理を行う。
請求区分が従量の場合は利用量を集計する。
```

`state_transition` は、ステータスや状態が変化することを表す。

例:

```text
申請後、ステータスは未承認から承認待ちに変わる。
承認するとステータスは承認済みになる。
```

---

### 10.3 `error_handling` と `open_issue` の違い

`error_handling` は、エラー時の仕様が記載されている場合に使う。

例:

```text
連携先がタイムアウトした場合は、エラーとしてログ出力する。
```

`open_issue` は、エラー時の仕様が未確定または要確認の場合に使う。

例:

```text
タイムアウト時のリトライ回数は別途検討。
```

---

## 11. 確度判断の例

### 11.1 High

原文:

```text
金額は0以上の整数のみ入力可能とする。
```

抽出:

```text
金額は0以上の整数のみ入力可能である。
```

確度:

```text
High
```

---

### 11.2 Medium

原文:

```text
金額には不正な値を入力できないようにする。
```

抽出:

```text
金額には不正な値を入力できない。
```

確度:

```text
Medium
```

理由:

```text
不正な値の具体条件が不明。
```

---

### 11.3 Low

原文:

```text
必要に応じて管理者が再実行する。
```

抽出:

```text
管理者が再実行する場合がある。
```

確度:

```text
Low
```

理由:

```text
再実行条件と管理者ロールの定義が不明。
```

---

### 11.4 Unknown

原文:

```text
この処理は既存仕様に準ずる。
```

抽出:

```text
既存仕様に従う処理が存在する。
```

確度:

```text
Unknown
```

理由:

```text
既存仕様の参照先が不明。
```

---

## 12. 禁止事項

このステップでは、以下を禁止する。

* 原文にない仕様を追加する
* 曖昧な表現を断定に変える
* 一般的な業務知識で補完する
* 実装上こうなるはず、という推測を書く
* 複数ドキュメントの内容を統合する
* 矛盾を解消する
* テスト観点に変換する
* テストケースを作成する
* 期待結果を推測で作成する
* 未記載の入力値や境界値を作る
* エラーメッセージを推測する
* 権限名やロール名を推測する
* DBテーブルやカラムを推測する

---

## 13. 例外的に許可されること

以下は許可する。

### 13.1 原文の簡潔な要約

原文が長い場合、意味を変えない範囲で要約してよい。

---

### 13.2 カテゴリ分類

原文に明記された内容を、カテゴリに分類してよい。

---

### 13.3 曖昧さの指摘

原文が曖昧な場合、その曖昧さを備考に記録してよい。

---

### 13.4 未確定事項の抽出

TODO、検討中、別途確認、未定などの記載は、未確定事項として抽出してよい。

---

### 13.5 関連カテゴリのメモ

主カテゴリとは別に関連カテゴリがある場合、備考に記載してよい。

例:

```text
主カテゴリは business_rule。validation にも関連。
```

---

## 14. 完了条件

このステップは、以下を満たしたら完了とする。

* 対象ドキュメント範囲が明示されている
* 抽出情報に `EXT-ID` が付与されている
* 各抽出情報に `DOC-ID` が記録されている
* 原文位置が記録されている
* カテゴリが付与されている
* 確度が付与されている
* 曖昧・未確定な記述が明示されている
* 未処理範囲が明示されている
* 推測による補完が混入していない
* Step 3には進まず停止している

---

## 15. 完了時の応答

Step 2完了時には、以下を出力する。

```md
# Step 2: 事実情報の抽出

## 入力

- 使用した `document_inventory.md`
- 対象ドキュメント
- 対象範囲
- 未提供の情報

## 実行結果

<raw_extraction.md の内容>

## 未処理・要確認

- 未処理の範囲
- 曖昧な記述
- 追加確認が必要な事項

## 完了判定

- [x] raw_extraction.md が作成されている
- [x] EXT-ID が付与されている
- [x] DOC-ID と原文位置が記録されている
- [x] カテゴリが付与されている
- [x] 確度が付与されている
- [x] 推測による補完がない
- [x] 次ステップへ進まず停止している
```

最後に、次のようにユーザー確認を待つ。

```text
Step 2の成果物を作成しました。
内容を確認し、問題なければ「次へ」と指示してください。
```
