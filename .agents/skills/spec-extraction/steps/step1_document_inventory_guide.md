# step1_document_inventory_guide.md

## 1. このステップの目的

Step 1では、入力された開発ドキュメントを棚卸しし、後続ステップで抽出対象とする資料・範囲・優先度を整理する。

このステップの目的は、仕様を詳細に抽出することではない。

目的は、以下を明確にすることである。

- どのドキュメントが存在するか
- 各ドキュメントが何の資料か
- どの機能・領域・チームに関係するか
- 仕様判断上の優先度はどれくらいか
- Step 2で読むべき範囲はどこか
- 抽出対象外または後回しにする範囲はどこか
- 資料間の位置づけに不明点があるか

---

## 2. このステップで作成する成果物

このステップでは、以下の成果物を作成する。

```text
document_inventory.md
````

`document_inventory.md` は、後続の Step 2 で抽出対象を決めるための入力となる。

---

## 3. 入力

このステップの入力は、以下のいずれか、またはその組み合わせである。

* 開発ドキュメント一覧
* ファイル名一覧
* フォルダ構成
* ドキュメント本文の冒頭部分
* ドキュメントの目次
* ドキュメントの更新日・版数
* 対象チーム・対象機能に関する説明
* ユーザーからの補足説明

入力情報が不足していても、可能な範囲で棚卸しを行う。

不足している項目は、推測で補完せず、`不明` または `要確認` と記録する。

## 5. 実施手順

### 5.1 入力ドキュメントを一覧化する

まず、入力されたドキュメントを1件ずつ一覧化する。

各ドキュメントには、連番で `DOC-ID` を付与する。

IDの形式は以下とする。

```text
DOC-001
DOC-002
DOC-003
```

同じドキュメントの版違いがある場合も、原則として別の `DOC-ID` を付与する。

例:

```text
DOC-003: API仕様書_v1.0
DOC-004: API仕様書_v1.2
```

ただし、明らかに同一ファイルの重複である場合は、重複候補として記録する。

---

### 5.2 ドキュメント種別を分類する

各ドキュメントの種別を分類する。

分類例:

| 種別     | 内容                  |
| ------ | ------------------- |
| 要件定義   | 業務要件、機能要件、制約、スコープ   |
| 業務資料   | 業務フロー、業務ルール、業務用語    |
| 基本設計   | 機能構成、画面構成、処理概要      |
| 詳細設計   | 処理詳細、条件分岐、項目仕様      |
| 画面仕様   | 画面項目、入力制御、画面遷移      |
| 帳票仕様   | 帳票項目、出力条件、編集ルール     |
| API仕様  | API、リクエスト、レスポンス、エラー |
| DB仕様   | テーブル、カラム、制約、データ更新   |
| バッチ仕様  | ジョブ、起動条件、処理順、異常時処理  |
| 外部連携仕様 | 連携先、ファイル連携、通信、再送    |
| 権限仕様   | ロール、操作可否、参照範囲       |
| 運用設計   | 運用手順、監視、リカバリ、保守     |
| 移行設計   | データ移行、移行条件、移行チェック   |
| 補足資料   | 補足メモ、QAメモ、議事録、検討資料  |
| 不明     | 種別を判断できない資料         |

分類できない場合は、無理に分類せず `不明` とする。

---

### 5.3 対象領域を整理する

各ドキュメントが扱っている対象領域を整理する。

対象領域の例:

* 全体
* 共通基盤
* 認証・権限
* マスタ管理
* 取引入力
* 承認
* 締処理
* 集計
* 帳票出力
* 外部連携
* バッチ
* データ移行
* 運用
* チームA担当領域
* チームB担当領域

対象領域が複数ある場合は、複数記載してよい。

ただし、判断できない場合は `不明` とする。

---

### 5.4 関係する開発チームを整理する

プロジェクトが複数チームで構成されている場合、各ドキュメントがどのチームに関係するかを整理する。

記録例:

```text
チームA
チームB
共通チーム
外部連携チーム
不明
```

チーム名が資料から判断できない場合は、推測せず `不明` とする。

---

### 5.5 仕様判断上の優先度を設定する

各ドキュメントに、仕様判断上の優先度を設定する。

優先度は、以下のいずれかとする。

| 優先度     | 意味                 |
| ------- | ------------------ |
| High    | 正本、または仕様判断の基準となる資料 |
| Medium  | 詳細資料、補足資料、チーム別資料   |
| Low     | 参考資料、古い資料、検討中資料    |
| Unknown | 位置づけが不明な資料         |

優先度は、資料の重要度ではなく、**仕様判断上どの程度信頼できるか**を表す。

判断例:

| 状況          | 優先度             |
| ----------- | --------------- |
| 最新版の要件定義書   | High            |
| 承認済みの基本設計書  | High            |
| 最新版の画面仕様書   | High または Medium |
| チーム内メモ      | Medium または Low  |
| 古い設計資料      | Low             |
| 更新日・版数不明の資料 | Unknown         |
| ドラフト資料      | Low または Unknown |

判断できない場合は `Unknown` とする。

---

### 5.6 主に抽出する情報を整理する

各ドキュメントから、後続ステップで主に抽出すべき情報を整理する。

例:

* 業務要件
* 機能要件
* 業務ルール
* 画面項目
* 入力制約
* 画面遷移
* API仕様
* データ更新
* DB項目
* 外部連携条件
* エラー処理
* 権限
* 状態遷移
* 計算・集計ルール
* 帳票出力条件
* 非機能要件
* 運用制約
* 未確定事項

この時点では、具体的な仕様内容までは抽出しない。

---

### 5.7 Step 2で読む範囲を決める

各ドキュメントについて、Step 2で読むべき範囲を整理する。

範囲指定の例:

```text
全体
目次のみ確認済み。詳細抽出は第3章以降
画面項目一覧のみ
APIエンドポイント一覧とエラー仕様
第2章 業務フロー
第5章 例外処理
未指定
```

ドキュメント量が多い場合は、優先的に読む範囲を限定する。

推奨する優先順:

1. 要件・業務ルールに関する範囲
2. 機能単位・処理単位が分かる範囲
3. 画面/API/帳票/バッチなどの入出力仕様
4. 条件分岐・状態遷移・例外処理
5. DB・データ更新仕様
6. 運用・非機能・移行に関する範囲
7. 補足メモ・参考資料

---

### 5.8 抽出対象外・後回し範囲を整理する

現時点で抽出対象外にする範囲や、後回しにする範囲があれば記録する。

例:

```text
旧仕様のため今回は対象外
未承認ドラフトのため参考扱い
画面デザインのみで仕様情報が少ないため後回し
テスト設計との関係が薄いため対象外
本文未提供のため未処理
```

対象外と判断した理由を必ず記録する。

---

### 5.9 不明点・確認事項を記録する

棚卸し段階で不明な点を記録する。

例:

* 最新版かどうか不明
* 正本か補足資料か不明
* 対象チームが不明
* どの機能に関係する資料か不明
* 版違いのどちらを優先すべきか不明
* ファイル名からは資料種別を判断できない
* 本文がないため内容を確認できない

これらは、Step 5だけでなく、Step 2以降の前提にも関係するため、Step 1の時点で記録しておく。

## 7. `document_inventory.md` の記載項目

### 7.1 入力ドキュメント一覧

以下の表を使用する。

| DOC-ID | ドキュメント名 | 種別 | 対象領域 | 関連チーム | 優先度 | 主に抽出する情報 | Step 2で読む範囲 | 備考 |
| ------ | ------- | -- | ---- | ----- | --- | -------- | ----------- | -- |

各項目の説明:

| 項目          | 説明                            |
| ----------- | ----------------------------- |
| DOC-ID      | ドキュメントに付与する一意のID              |
| ドキュメント名     | ファイル名または資料名                   |
| 種別          | 要件定義、画面仕様、API仕様など             |
| 対象領域        | 対象機能、業務領域、システム領域              |
| 関連チーム       | 関係する開発チーム                     |
| 優先度         | High / Medium / Low / Unknown |
| 主に抽出する情報    | Step 2で抽出すべき情報の種類             |
| Step 2で読む範囲 | 抽出対象範囲                        |
| 備考          | 版数、更新日、不明点など                  |

---

### 7.2 抽出優先度

以下の表を使用する。

| 優先順位 | DOC-ID | 理由 |
| ---- | ------ | -- |

優先順位は、Step 2で読む順番の判断材料とする。

---

### 7.3 Step 2での処理順

以下の表を使用する。

| 処理順 | DOC-ID | 処理単位 | 想定する抽出内容 | 備考 |
| --- | ------ | ---- | -------- | -- |

処理単位の例:

* 全体
* 章単位
* 画面単位
* API単位
* 機能単位
* チーム単位

---

### 7.4 抽出対象外・後回し範囲

以下の表を使用する。

| DOC-ID | 対象外・後回し範囲 | 理由 | 再確認要否 |
| ------ | --------- | -- | ----- |

再確認要否は、以下のいずれかとする。

```text
必要
不要
不明
```

---

### 7.5 不明点・確認事項

以下の表を使用する。

| 確認ID | 対象DOC-ID | 確認事項 | 影響 | 優先度 |
| ---- | -------- | ---- | -- | --- |

確認IDは以下の形式とする。

```text
INV-Q-001
INV-Q-002
```

優先度は以下のいずれかとする。

```text
High
Medium
Low
```

## 8. 判断基準

### 8.1 Highにしてよい資料

以下のような資料は、優先度 `High` としてよい。

* プロジェクトで正本として扱われている資料
* 最新版であることが明確な資料
* 承認済みの要件定義書
* 承認済みの基本設計書
* テスト対象機能の仕様判断に直接使う資料
* ユーザーが正本と明示した資料

---

### 8.2 Mediumにしてよい資料

以下のような資料は、優先度 `Medium` としてよい。

* チーム別の詳細設計資料
* 画面/API/DBなどの詳細資料
* 正本を補足する資料
* 最新性は高いが、仕様判断の最上位資料ではない資料
* 実装寄りの資料

---

### 8.3 Lowにする資料

以下のような資料は、優先度 `Low` とする。

* 古い資料
* 検討中メモ
* 未承認ドラフト
* 参考資料
* 現行仕様か不明な過去資料
* 仕様というより背景説明に近い資料

---

### 8.4 Unknownにする資料

以下の場合は、優先度 `Unknown` とする。

* 資料の位置づけが分からない
* 最新版かどうか不明
* ファイル名だけでは種別を判断できない
* 本文や目次がなく、内容を確認できない

---

## 9. 禁止事項

このステップでは、以下を禁止する。

* ドキュメント本文から詳細仕様を大量に抽出する
* 仕様の正しさを判断する
* 資料間の矛盾を解消する
* ファイル名だけで内容を断定する
* 正本かどうか不明な資料を正本扱いする
* 古い資料を現行仕様として断定する
* Step 2以降の成果物を作成する
* テスト観点やテストケースを作成する
* 不明項目を推測で埋める

---

## 10. 例外的に許可されること

以下は許可する。

* ファイル名や見出しから、仮の種別を設定する
* 仮判断であることを備考に明記する
* ユーザーが明示した情報に基づき、優先度を設定する
* 明らかに重複している資料を重複候補として記録する
* 古い可能性がある資料を参考資料として記録する
* 本文未提供の資料を未処理として一覧に含める

---

## 11. 完了条件

このステップは、以下を満たしたら完了とする。

* 入力ドキュメントに `DOC-ID` が付与されている
* 各ドキュメントの種別が整理されている
* 対象領域が記録されている
* 関連チームが分かる範囲で記録されている
* 仕様判断上の優先度が設定されている
* 主に抽出する情報が整理されている
* Step 2で読む範囲が整理されている
* 抽出対象外・後回し範囲が記録されている
* 不明点・確認事項が明示されている
* Step 2には進まず停止している

---

## 12. 完了時の応答

Step 1完了時には、以下を出力する。

```md
# Step 1: ドキュメント棚卸し

## 入力

- 使用した入力情報
- 対象範囲
- 未提供の情報

## 実行結果

<document_inventory.md の内容>

## 未処理・要確認

- 未処理の資料
- 不明な資料
- ユーザー確認が必要な事項

## 完了判定

- [x] document_inventory.md が作成されている
- [x] DOC-ID が付与されている
- [x] 優先度が設定されている
- [x] Step 2で読む範囲が整理されている
- [x] 推測による仕様補完がない
- [x] 次ステップへ進まず停止している
```

最後に、次のようにユーザー確認を待つ。

```text
Step 1の成果物を作成しました。
内容を確認し、問題なければ「次へ」と指示してください。
```
