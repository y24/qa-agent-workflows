# step3_normalization_guide.md

## 1. このステップの目的

Step 3では、Step 2で作成した `raw_extraction.md` を入力として、抽出情報を仕様単位に正規化・統合する。

このステップの目的は、ドキュメント単位でばらばらに抽出された情報を、後続のテスト設計工程で利用しやすい形に整理することである。

このステップでは、以下を行う。

- 同じ対象に関する抽出情報を集約する
- 表記揺れや同義表現を整理する
- 粒度の違う情報を仕様単位に整える
- 関連する `EXT-ID` を紐づける
- 仕様の状態を付与する
- 矛盾、不足、未確定事項を明示する

このステップでは、以下を行わない。

- 不足している仕様を推測で補完する
- 矛盾する仕様を勝手に解消する
- 正しい仕様を独断で決める
- テスト観点へ詳細変換する
- テストケースを作成する

---

## 2. このステップで作成する成果物

このステップでは、以下の成果物を作成する。

```text
normalized_spec_inventory.md
````

`normalized_spec_inventory.md` は、抽出された仕様情報を、仕様単位に整理した一覧である。

後続の Step 4 では、この `normalized_spec_inventory.md` を入力として、テスト設計インプットへ変換する。

---

## 3. 入力

このステップの入力は、以下である。

* `document_inventory.md`
* `raw_extraction.md`
* 必要に応じて、既存の `normalized_spec_inventory.md`
* 必要に応じて、ユーザーからの補足説明

既存の `normalized_spec_inventory.md` がある場合は、追記・更新として扱う。

ただし、既存の `SPEC-ID` と重複しないようにする。

## 5. 実施手順

### 5.1 入力成果物を確認する

まず、以下を確認する。

* `raw_extraction.md` が存在するか
* 各抽出情報に `EXT-ID` があるか
* 各抽出情報に `DOC-ID` があるか
* 原文位置が記録されているか
* カテゴリと確度が付与されているか
* 曖昧・未確定な記述が記録されているか

不足がある場合は、可能な範囲で処理し、不足内容を明示する。

参照元が不明な情報は、確定仕様として扱わない。

---

### 5.2 正規化対象をグルーピングする

`raw_extraction.md` の抽出情報を、以下の観点でグルーピングする。

* 対象機能
* 対象画面
* 対象API
* 対象帳票
* 対象バッチ
* 対象DB・テーブル
* 対象業務ルール
* 対象入力項目
* 対象ステータス
* 対象権限
* 対象エラー
* 対象外部連携
* 対象運用

グルーピングの目的は、関連する情報を見つけやすくすることである。

この時点では、仕様を確定しない。

---

### 5.3 同義・類似表現を整理する

同じ意味を持つ表現が複数ある場合は、代表表現に整理する。

例:

```text
承認済みデータ
承認完了データ
ステータスが承認済のデータ
```

これらが同じ意味であることが明確な場合は、代表表現として以下のように整理してよい。

```text
承認済みデータ
```

ただし、同じ意味か判断できない場合は、無理に統合しない。

その場合は、状態を `to_be_confirmed` とする。

---

### 5.4 重複情報を統合する

同一内容が複数の抽出情報に存在する場合は、1つの `SPEC-ID` に統合する。

統合する場合は、元の `EXT-ID` をすべて記録する。

例:

| SPEC-ID  | 正規化した内容           | 参照EXT-ID         |
| -------- | ----------------- | ---------------- |
| SPEC-001 | 承認済みデータのみ締処理対象とする | EXT-001, EXT-014 |

統合しても、参照元を失ってはならない。

---

### 5.5 粒度の違いを整理する

同じ対象について、粒度の違う情報がある場合は、関係性が分かるように整理する。

例:

```text
EXT-001: 締処理を行う。
EXT-002: 承認済みデータのみ締処理対象とする。
EXT-003: 締処理後、対象データは締済みとなる。
```

正規化例:

```text
SPEC-001: 締処理は、承認済みデータを対象に実行する。
SPEC-002: 締処理後、対象データの状態は締済みとなる。
```

ただし、原文にない処理順や因果関係を追加してはならない。

---

### 5.6 仕様単位に整理する

1つの `SPEC-ID` には、原則として1つの仕様を記録する。

仕様単位の例:

* 1つの業務ルール
* 1つの入力制約
* 1つの状態遷移
* 1つの条件分岐
* 1つの計算ルール
* 1つの権限制御
* 1つのエラー処理
* 1つの外部連携仕様
* 1つの帳票出力条件
* 1つのデータ更新仕様

複数の仕様が混在する場合は、分割する。

---

### 5.7 仕様カテゴリを設定する

正規化した仕様には、カテゴリを付与する。

カテゴリは、原則として以下から選択する。

| カテゴリ               | 内容                      |
| ------------------ | ----------------------- |
| `feature`          | 機能、処理単位、ユースケース          |
| `screen`           | 画面、画面遷移、画面部品            |
| `ui_item`          | 入力項目、表示項目、ボタン、リンク       |
| `report`           | 帳票、ファイル出力、印刷、Excel/PDF等 |
| `api`              | API、インターフェース、外部連携       |
| `batch`            | バッチ、ジョブ、非同期処理           |
| `data`             | DB、テーブル、項目、マスタ、トランザクション |
| `business_rule`    | 業務ルール、業務上の判定条件          |
| `validation`       | 入力チェック、制約、必須、桁数、型       |
| `state_transition` | ステータス、状態遷移、ライフサイクル      |
| `condition_branch` | 条件分岐、場合分け               |
| `calculation`      | 計算、集計、変換、丸め             |
| `permission`       | 権限、ロール、操作可否             |
| `error_handling`   | エラー、警告、例外、リトライ          |
| `operation`        | 運用、保守、移行、監視、手動作業        |
| `non_functional`   | 性能、セキュリティ、可用性、互換性等      |
| `open_issue`       | 未確定、要確認、不足、矛盾           |

分類に迷う場合は、主カテゴリを1つ選び、備考に関連カテゴリを記録する。

---

### 5.8 仕様状態を設定する

正規化した仕様には、状態を付与する。

状態は以下から選択する。

| 状態                    | 意味                      |
| --------------------- | ----------------------- |
| `confirmed`           | 参照元が明確で、仕様として扱える        |
| `partial`             | 一部は分かるが、詳細が不足している       |
| `contradiction`       | 資料間または同一資料内に矛盾がある       |
| `missing`             | 必要な情報が存在しない             |
| `insufficient_detail` | 記載はあるが、テスト設計には粒度が不足している |
| `to_be_confirmed`     | 仕様責任者や追加資料への確認が必要       |
| `deprecated_or_stale` | 古い資料の可能性がある             |
| `out_of_scope`        | 今回の抽出対象外                |

状態は、仕様をどう扱うべきかを示すためのものである。

重要度とは異なる。

---

### 5.9 矛盾を記録する

複数の抽出情報の間で矛盾がある場合は、勝手に解消しない。

矛盾の例:

* 同じ項目の桁数が資料によって異なる
* 同じ処理の対象条件が資料によって異なる
* 同じエラー時動作が資料によって異なる
* 画面仕様とAPI仕様で必須項目が異なる
* 要件定義と詳細設計で権限制御が異なる
* 新旧資料のどちらが正しいか不明

矛盾がある場合は、状態を `contradiction` とする。

備考には、矛盾している内容と参照元を記録する。

例:

```text
DOC-002では金額は最大10桁、DOC-004では最大12桁と記載されている。
```

---

### 5.10 不足・粒度不足を記録する

テスト設計に必要な情報が不足している場合は、補完せずに状態として記録する。

不足の例:

* 入力制約が記載されていない
* エラー時の期待動作が記載されていない
* 状態遷移後のステータスが不明
* 権限別の操作可否が不明
* 帳票項目の編集条件が不明
* APIエラーコードが不明
* バッチ異常終了時のリカバリ方法が不明
* 計算式はあるが丸めルールが不明

状態は、内容に応じて以下を使用する。

```text
partial
missing
insufficient_detail
to_be_confirmed
```

---

### 5.11 古い資料・参考資料由来の情報を扱う

参照元ドキュメントの優先度が `Low` または `Unknown` の場合、その情報を現行仕様として断定しない。

必要に応じて状態を以下にする。

```text
to_be_confirmed
deprecated_or_stale
```

備考に、参照元の位置づけを記録する。

例:

```text
DOC-007は旧版の可能性があるため、現行仕様として扱うには確認が必要。
```

---

### 5.12 open_issue を仕様化しない

Step 2で `open_issue` として抽出された情報は、仕様として確定しない。

例:

```text
エラー時のリトライ回数は別途検討。
```

この場合、以下のように扱う。

```text
状態: to_be_confirmed
内容: エラー時のリトライ回数は未確定である。
```

未確定事項を、確定仕様に変換してはならない。

## 7. `normalized_spec_inventory.md` の記載項目

### 7.1 正規化仕様一覧

以下の表を使用する。

| SPEC-ID | カテゴリ | 対象 | 正規化した仕様内容 | 状態 | 確度 | 参照EXT-ID | 備考 |
| ------- | ---- | -- | --------- | -- | -- | -------- | -- |

各項目の説明:

| 項目        | 説明                                     |
| --------- | -------------------------------------- |
| SPEC-ID   | 正規化仕様に付与する一意のID                        |
| カテゴリ      | 仕様カテゴリ                                 |
| 対象        | 機能、画面、項目、API、帳票など                      |
| 正規化した仕様内容 | 抽出情報を仕様単位に整理した内容                       |
| 状態        | confirmed / partial / contradiction など |
| 確度        | High / Medium / Low / Unknown          |
| 参照EXT-ID  | 元となる抽出情報のID                            |
| 備考        | 矛盾、不足、統合理由、注意事項など                      |

---

### 7.2 矛盾がある仕様

矛盾がある場合は、以下の表にまとめる。

| SPEC-ID | 対象 | 矛盾内容 | 参照EXT-ID | 影響 | 備考 |
| ------- | -- | ---- | -------- | -- | -- |

影響には、テスト設計上どこに影響するかを記載する。

例:

```text
期待結果を確定できない
境界値を決められない
権限テストの前提が決まらない
異常系テストの期待動作が決まらない
```

---

### 7.3 不足・粒度不足がある仕様

不足や粒度不足がある場合は、以下の表にまとめる。

| SPEC-ID | 対象 | 不足内容 | 状態 | 影響 | 備考 |
| ------- | -- | ---- | -- | -- | -- |

---

### 7.4 未確定・要確認事項

未確定事項や確認が必要な事項は、以下の表にまとめる。

| 確認ID | 関連SPEC-ID | 対象 | 確認事項 | 確認理由 | 優先度 |
| ---- | --------- | -- | ---- | ---- | --- |

確認IDは以下の形式とする。

```text
NORM-Q-001
NORM-Q-002
```

---

### 7.5 未処理範囲

未処理の抽出情報がある場合は、以下の表にまとめる。

| EXT-IDまたはDOC-ID | 未処理範囲 | 理由 | 次回処理要否 |
| --------------- | ----- | -- | ------ |

## 8. SPEC-ID の採番ルール

正規化仕様には、以下の形式でIDを付与する。

```text
SPEC-001
SPEC-002
SPEC-003
```

分割処理で追記する場合は、既存IDと重複しないように採番する。

カテゴリ別に分けたい場合は、以下の形式を使用してもよい。

```text
SPEC-BR-001
SPEC-VAL-001
SPEC-API-001
```

ただし、プロジェクト内で採番形式は統一する。

---

## 9. 正規化判断の基準

### 9.1 統合してよい場合

以下の場合は、複数の抽出情報を統合してよい。

* 同一仕様を別資料で説明している
* 表記は違うが、対象・条件・動作が一致している
* 上位資料と詳細資料が補完関係にある
* 同じ項目の説明が複数箇所に分かれている
* 同じエラー処理が複数資料に記載されている

統合する場合は、必ずすべての `EXT-ID` を記録する。

---

### 9.2 統合してはならない場合

以下の場合は、無理に統合してはならない。

* 条件が異なる可能性がある
* 対象範囲が異なる可能性がある
* 資料間で値や期待動作が異なる
* 同じ用語だが意味が違う可能性がある
* 新旧資料のどちらが正しいか不明
* 権限やロールの粒度が異なる
* 状態名は似ているが遷移条件が異なる

この場合は、別の `SPEC-ID` として残すか、状態を `to_be_confirmed` または `contradiction` とする。

---

### 9.3 confirmed にしてよい場合

以下を満たす場合は、状態を `confirmed` としてよい。

* 参照元が明確である
* 原文に仕様が明記されている
* 矛盾する情報が見つかっていない
* テスト設計に使える粒度である
* 資料の位置づけが `High` または信頼できる

---

### 9.4 partial にする場合

以下の場合は、状態を `partial` とする。

* 仕様の一部だけが分かる
* 条件は分かるが期待動作が不足している
* 処理概要は分かるが詳細条件が不足している
* 入力項目は分かるが制約が不足している
* 状態遷移は分かるが遷移条件が不足している

---

### 9.5 contradiction にする場合

以下の場合は、状態を `contradiction` とする。

* 資料間で記載が異なる
* 同一資料内で記載が矛盾している
* 上位資料と下位資料で条件が異なる
* 画面仕様とAPI仕様で項目仕様が異なる
* 仕様書と補足資料で期待動作が異なる

---

### 9.6 insufficient_detail にする場合

以下の場合は、状態を `insufficient_detail` とする。

* 記載はあるが、テスト設計には粒度が足りない
* 「適切に」「必要に応じて」などの表現で具体条件がない
* エラーとだけ書かれており、エラー内容が不明
* チェックするとだけ書かれており、チェック条件が不明
* 連携するとだけ書かれており、連携条件が不明

---

### 9.7 to_be_confirmed にする場合

以下の場合は、状態を `to_be_confirmed` とする。

* 仕様責任者の確認が必要
* 資料の位置づけが不明
* 最新版か不明
* 未確定と明記されている
* 複数解釈が可能
* 用語の意味が不明
* 対象範囲が不明

---

## 10. 確度の決め方

正規化仕様の確度は、参照元の抽出情報を踏まえて設定する。

基本方針:

| 参照EXTの確度    | 正規化後の確度       |
| ----------- | ------------- |
| Highのみ      | Highにしてよい     |
| HighとMedium | HighまたはMedium |
| Medium中心    | Medium        |
| Lowを含む      | MediumまたはLow  |
| Unknownを含む  | LowまたはUnknown |

ただし、矛盾がある場合は、情報が明記されていても確度を下げる。

例:

```text
EXT-001はHigh、EXT-002もHighだが、内容が矛盾している。
→ 状態は contradiction、確度は Medium または Low とする。
```

---

## 11. 禁止事項

このステップでは、以下を禁止する。

* 不足情報を推測で補う
* 矛盾を勝手に解消する
* 正しい仕様を独断で選ぶ
* 資料優先度だけで一方を破棄する
* 参照EXT-IDを失わせる
* 原文にない条件を追加する
* 原文にない処理順を追加する
* 原文にない因果関係を追加する
* テスト観点へ過度に変換する
* テストケースを作成する
* 具体的なテストデータを作成する
* エラーメッセージや境界値を推測する

---

## 12. 例外的に許可されること

以下は許可する。

### 12.1 表記揺れの統一

同じ意味であることが明確な用語は、代表表現に統一してよい。

ただし、備考に元表現を記録する。

---

### 12.2 情報の集約

同一仕様に関する複数の抽出情報を、1つの仕様として整理してよい。

ただし、参照元 `EXT-ID` を必ず残す。

---

### 12.3 矛盾の明示

矛盾を解消せず、矛盾として整理してよい。

---

### 12.4 不足情報の明示

テスト設計に必要だが不足している情報を、不足として記録してよい。

---

### 12.5 テスト設計への影響メモ

矛盾や不足が、後続のテスト設計にどう影響するかを簡潔に記録してよい。

ただし、具体的なテストケースにはしない。

---

## 13. 完了条件

このステップは、以下を満たしたら完了とする。

* `normalized_spec_inventory.md` が作成されている
* 正規化仕様に `SPEC-ID` が付与されている
* 各仕様にカテゴリが付与されている
* 各仕様に状態が付与されている
* 各仕様に確度が付与されている
* 各仕様に参照 `EXT-ID` が記録されている
* 矛盾がある仕様が明示されている
* 不足・粒度不足がある仕様が明示されている
* 未確定・要確認事項が明示されている
* 推測による補完が混入していない
* Step 4には進まず停止している

---
