# シナリオテスト設計 固有ルール

共通の行動契約、根拠、issue、レビュー停止、出力形式、トレーサビリティ、テスト設計粒度は `../../../shared/` 配下の方針に従う。
このファイルでは `$scenario-test-design` 固有の責務、ID、禁止事項を定義する。

## 目的

このskillで作成する入力サマリ、業務フロー、シナリオ観点、シナリオ候補、テストケースに共通するルールを定義する。

## 基本方針

- 入力文書、前Step成果物、ユーザー指定を根拠にする。
- 事実、推測、仮説、確認事項、未確定事項を分けて扱う。
- 各Stepの責務を守り、早いStepでテストケース化しない。
- IDによるトレーサビリティを維持する。
- 後続Stepで扱いやすい粒度に圧縮する。

## Step責務

| Step | 責務 | やらないこと |
|---|---|---|
| 1 | 入力文書の要点抽出 | 業務フロー、観点、シナリオ、テストケース作成 |
| 2 | 業務フロー・利用シーン整理 | シナリオ候補、詳細テストケース作成 |
| 3 | シナリオ観点抽出 | シナリオ候補、テスト手順作成 |
| 4 | シナリオ候補設計 | 詳細手順、具体値、期待結果全文作成 |
| 5 | テストケース化・優先度付け | 自動化、実行計画、管理ツール固有形式化 |

## ID体系

| 対象 | 形式 |
|---|---|
| Document | `DOC-001` |
| Role | `ROLE-001` |
| Feature | `FEAT-001` |
| Business Flow | `BF-001` |
| Use Scene | `US-001` |
| Viewpoint | `SV-001` |
| Scenario | `SC-001` |
| Test Case | `TC-001` |
| Question | `Q-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## 根拠と推測

- 入力にない業務ルール、画面操作、期待結果、データ値を事実として書かない。
- 推測が必要な場合は `推測:` と明記し、根拠または確認事項を添える。
- 文書間の矛盾、不足、曖昧さは確認事項として記録する。
- 根拠は文書名、章、節、項目名、前StepのIDで追跡できるようにする。

## 出力ルール

- 出力はMarkdownを基本にする。
- 表は列名を安定させる。
- 各成果物には概要、入力・参照元、主成果物、確認事項、次Stepへの引き継ぎを含める。
- `Flow ID -> Viewpoint ID -> Scenario ID -> TC ID` の対応を崩さない。
- 不明点は補完せず、`要確認`、`情報不足`、`未確定`、`対象外` などで状態を示す。

## 優先度

- High/P0: 中核業務、重大影響、法令・会計・監査、過去重大障害、必須確認。
- Medium/P1-P2: 主要分岐、代表例外、重要データ状態、主要ロール差。
- Low/P3: 簡易確認、探索的確認、他ケースで同時確認、対象外候補。

## 品質基準

- 網羅性: 中核業務、重要分岐、例外、データ状態、ロール差、出力・連携を必要範囲で扱う。
- 追跡性: 入力、フロー、観点、シナリオ、テストケースがIDでつながる。
- 実行可能性: Step 5のテストケースは実行可能な概要粒度になっている。
- 業務妥当性: 画面操作羅列ではなく、業務目的に沿っている。
- 透明性: 未確認、対象外、未カバー、保留が隠れていない。

## 禁止事項

- 入力にない仕様や期待結果を断定する
- Step 1-4で詳細テストケースを作る
- 画面遷移だけを業務フローとして扱う
- 観点名をそのままシナリオ名にする
- リスクや未カバー観点を成果物から消す
- すべてを同じ優先度・同じ深さで扱う
