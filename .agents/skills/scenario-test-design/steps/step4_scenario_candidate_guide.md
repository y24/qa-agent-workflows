# Step 4: シナリオ候補の設計ガイド

## 1. このstepの目的

Step 4では、Step 3で作成した `outputs/03_scenario_viewpoints.md` をもとに、実施候補となるシナリオテストを設計する。

このstepの目的は、シナリオ観点を業務上意味のある一連の確認単位にまとめ、Step 5で詳細なテストケースへ落とし込める状態にすることである。

このstepでは、詳細な操作手順や期待結果までは作成しない。  
あくまで「どの業務シナリオをテスト対象にするか」を設計する。

---

主な入力は以下とする。


## 5. シナリオ候補の基本方針

シナリオ候補は、単なる機能確認や画面操作の羅列ではない。

シナリオ候補は、以下を満たす一連の業務確認単位として設計する。

* 業務目的が明確である
* 利用者・ロールが明確である
* 開始条件と終了条件が分かる
* 関連する業務フロー・観点が追跡できる
* 使用するデータ状態が分かる
* 重要な分岐・例外を扱える
* 確認ポイントが明確である
* Step 5でテストケース化できる粒度である

---

## 6. シナリオ候補に含める情報

各シナリオ候補には、以下を記載する。

### 6.1 シナリオID

シナリオ候補には `SC-001` の形式でIDを付与する。

例:

```md
SC-001: 新規契約を登録し、承認後に帳票へ反映されることを確認する
```

---

### 6.2 シナリオ名

シナリオ名は、業務目的と確認対象が分かる名前にする。

悪い例:

```md
SC-001: 契約登録確認
```

良い例:

```md
SC-001: 担当者が新規契約を登録し、承認者の承認後に後続処理対象となることを確認する
```

---

### 6.3 業務目的

そのシナリオが、業務上どの目的を確認するものかを記載する。

例:

```md
- 業務目的: 新規契約情報を登録し、承認済み状態にして後続業務で利用可能にする
```

---

### 6.4 対象ユーザー・ロール

シナリオで登場する利用者・ロールを記載する。

例:

```md
- 対象ユーザー・ロール:
  - 担当者
  - 承認者
```

ロールによって操作可否や確認結果が変わる場合は、必ず記載する。

---

### 6.5 前提条件

シナリオ開始時に必要な前提を記載する。

例:

```md
- 前提条件:
  - 担当者ユーザーが存在する
  - 承認者ユーザーが存在する
  - 契約登録に必要なマスタが登録済みである
```

仕様にない前提は断定せず、`要確認` または `仮置き` とする。

---

### 6.6 使用データ概要

詳細なテストデータ値ではなく、どのようなデータが必要かを概要として記載する。

例:

```md
- 使用データ概要:
  - 新規登録可能な契約データ
  - 承認対象となる契約データ
  - 帳票出力対象となる承認済み契約データ
```

具体的な値や組み合わせは、Step 5で詳細化する。

---

### 6.7 操作・業務の大まかな流れ

シナリオ内で実施する業務の流れを、業務レベルで記載する。

例:

```md
- 操作・業務の大まかな流れ:
  1. 担当者が契約情報を登録する
  2. 担当者が承認依頼を行う
  3. 承認者が内容を確認する
  4. 承認者が承認する
  5. 承認済み契約が後続処理対象になることを確認する
```

この段階では、画面項目やボタン操作を細かく書きすぎない。

---

### 6.8 確認ポイント

シナリオで確認したいことを記載する。

例:

```md
- 確認ポイント:
  - 契約登録後、適切な状態になること
  - 承認後、ステータスが承認済みになること
  - 承認済みデータが後続処理対象になること
  - 登録内容が出力結果に反映されること
```

期待結果の詳細はStep 5で作成する。

---

### 6.9 関連フロー・関連観点

Step 2・Step 3のIDを使って追跡できるようにする。

例:

```md
- 関連フロー:
  - BF-001: 契約登録から承認完了まで
  - BF-002: 承認済み契約の帳票出力

- 関連観点:
  - SV-001: 新規契約登録から承認完了までの基本業務成立
  - SV-002: 登録内容の後続処理・帳票への反映
  - SV-003: 承認ステータスの状態遷移
```

---

### 6.10 優先度

シナリオ候補には優先度を付与する。

推奨優先度:

| 優先度 | 意味                          |
| --- | --------------------------- |
| P0  | 業務上致命的。必ず実施すべき中核シナリオ        |
| P1  | 重要度が高く、通常のシステムテストで実施すべきシナリオ |
| P2  | 条件付きで実施すべきシナリオ              |
| P3  | 余裕があれば実施する補助的シナリオ           |
| 未判断 | 判断材料が不足している                 |

優先度を付ける場合は、必ず理由を記載する。

例:

```md
- 優先度: P0
- 優先度理由: 契約登録から承認完了までの中核業務であり、後続処理や帳票出力に影響するため
```

---

### 6.11 採用判断

シナリオ候補の扱いを記載する。

| 採用判断  | 意味                     |
| ----- | ---------------------- |
| 採用候補  | Step 5でテストケース化する候補     |
| 要レビュー | 内容や優先度についてユーザー確認が必要    |
| 保留    | 仕様不足や対象範囲不明により一旦保留     |
| 対象外候補 | 今回のシナリオテスト対象外の可能性がある   |
| 統合候補  | 他のシナリオと統合したほうがよい可能性がある |

例:

```md
- 採用判断: 採用候補
- 判断理由: 主要業務フローかつ高重要度観点に対応しているため
```

## 7. シナリオ候補の種類

以下の種類を参考に、シナリオ候補を設計する。

ただし、すべての種類を無理に作成する必要はない。
入力文書・業務フロー・シナリオ観点に基づくものを優先する。

---

### 7.1 中核正常系シナリオ

主要な業務フローが、標準的な条件で最後まで成立することを確認するシナリオ。

例:

```md
SC-001: 担当者が新規契約を登録し、承認者が承認して契約を利用可能状態にする
```

対象になりやすい観点:

* 代表業務フロー
* データ整合性
* 状態遷移
* 帳票・出力
* 外部連携

---

### 7.2 分岐シナリオ

業務上重要な分岐条件で、正しい流れになることを確認するシナリオ。

例:

```md
SC-002: 承認者が契約申請を差戻し、担当者が修正後に再申請できることを確認する
```

対象になりやすい分岐:

* 承認 / 差戻し
* 正常 / エラー
* 登録 / 更新 / 取消
* 締め前 / 締め後
* 連携成功 / 連携失敗
* 対象データあり / なし

---

### 7.3 データ状態シナリオ

データ状態やステータスの違いによって、処理対象や操作可否が変わることを確認するシナリオ。

例:

```md
SC-003: 承認済み契約のみが請求予定作成の対象となることを確認する
```

対象になりやすい状態:

* 未申請
* 申請中
* 承認済み
* 差戻し
* 締め済み
* 取消済み
* 連携済み
* エラー

---

### 7.4 ロール・権限シナリオ

利用者ロールによって操作可否や表示内容が正しく制御されることを確認するシナリオ。

例:

```md
SC-004: 担当者は承認操作を実行できず、承認者のみが承認できることを確認する
```

この種のシナリオでは、同一データを複数ロールで確認する構成が有効な場合がある。

---

### 7.5 出力・帳票シナリオ

業務フローの結果が帳票、ファイル、レポートに正しく反映されることを確認するシナリオ。

例:

```md
SC-005: 承認済み契約が契約一覧帳票に正しく出力されることを確認する
```

対象になりやすい出力:

* Excel
* PDF
* CSV
* 画面一覧
* 詳細画面
* 集計レポート
* 外部連携ファイル

---

### 7.6 外部連携・バッチシナリオ

外部システム連携やバッチ処理を含む一連の流れを確認するシナリオ。

例:

```md
SC-006: 連携対象データを作成し、外部連携後に連携結果が画面へ反映されることを確認する
```

対象になりやすい確認:

* 連携対象データ
* 連携タイミング
* 連携成功
* 連携失敗
* 再送
* 重複連携防止
* バッチ実行後の反映

---

### 7.7 締め・取消・再実行シナリオ

締め、取消、再実行など、業務上影響の大きい状態変更を確認するシナリオ。

例:

```md
SC-007: 月次締め後に対象データを編集できず、締め取消後に再編集できることを確認する
```

対象になりやすい確認:

* 締め後の編集制限
* 締め取消後の再処理
* 取消後の後続処理対象外
* 再実行時の重複防止
* エラー修正後の再実行

---

### 7.8 過去不具合・リスク再現シナリオ

過去不具合や既知リスクに関連する条件を含めて、再発しないことを確認するシナリオ。

例:

```md
SC-008: 過去に不整合が発生した条件で、承認済みデータと帳票出力が一致することを確認する
```

この種のシナリオでは、過去不具合IDやリスクIDを明記する。

## 11. 観点別のシナリオ対応

Step 3で作成したシナリオ観点に対して、どのシナリオ候補で確認するかを整理する。

```md
| 観点ID | 観点名 | 対応シナリオ | カバー状況 | 補足 |
|---|---|---|---|---|
| SV-001 | 新規契約登録から承認完了までの基本業務成立 | SC-001 | カバー | 中核正常系として確認 |
| SV-004 | 承認と差戻しの分岐 | SC-002 | カバー | 差戻しシナリオで確認 |
| SV-008 | 外部連携結果の反映 | - | 未カバー | 連携仕様が不足しているため要確認 |
```

カバー状況は以下を使う。

| カバー状況 | 意味               |
| ----- | ---------------- |
| カバー   | シナリオ候補で確認できる     |
| 一部カバー | 一部のみ確認できる        |
| 未カバー  | 現時点では対応するシナリオがない |
| 保留    | 仕様不足や対象範囲不明により保留 |
| 対象外候補 | 今回対象外の可能性がある     |

```md
|---|---|---|---|
```

## 13. 統合・除外候補の整理

類似シナリオや対象外候補がある場合は整理する。

```md
| ID | 処理案 | 理由 | 統合先・扱い |
|---|---|---|---|
| SC-003 | SC-001へ統合候補 | 基本フロー内で同時確認可能 | SC-001 |
| SC-009 | 対象外候補 | 今回の改修範囲外の可能性 | 要確認 |
```

AIエージェントが勝手に削除・除外を確定しない。
除外は `対象外候補` としてユーザーレビューに回す。

## 15. 判断ルール

### 15.1 観点をそのままシナリオ名にしない

シナリオ観点は「確認したいこと」であり、シナリオは「業務上の流れ」である。

悪い例:

```md
SC-001: 状態遷移を確認する
```

良い例:

```md
SC-001: 担当者が申請した契約を承認者が承認し、契約ステータスが承認済みになることを確認する
```

---

### 15.2 1シナリオに目的を詰め込みすぎない

1つのシナリオに複数の業務目的を詰め込みすぎると、実行も失敗時の原因分析も難しくなる。

悪い例:

```md
SC-001: 契約登録、承認、請求、入金、締め、帳票出力、外部連携まで全部確認する
```

良い例:

```md
SC-001: 契約登録から承認完了までを確認する
SC-002: 承認済み契約から請求予定が作成されることを確認する
SC-003: 請求結果が帳票へ正しく出力されることを確認する
```

ただし、シナリオテストとして一連の業務連携を確認する意味がある場合は、適切な範囲で複数機能を含めてよい。

---

### 15.3 細かすぎるシナリオを避ける

単一の画面項目やボタン操作だけをシナリオ候補にしない。

悪い例:

```md
SC-001: 契約番号を入力する
SC-002: 契約名を入力する
SC-003: 保存ボタンを押す
```

良い例:

```md
SC-001: 必要な契約情報を入力して新規契約を登録する
```

---

### 15.4 複数観点を効率よくカバーする

1つのシナリオで複数の観点を自然に確認できる場合は、統合してよい。

例:

```md
SC-001: 新規契約を登録し、承認後に後続処理対象となることを確認する

関連観点:
- SV-001: 基本業務成立
- SV-002: データ整合性
- SV-003: 状態遷移
- SV-005: ロール・権限
```

ただし、分岐条件や期待結果が大きく異なる場合は別シナリオにする。

---

### 15.5 根拠のないシナリオを断定しない

文書根拠がないが重要そうなシナリオは、以下のように扱う。

```md
SC-XXX: 推測: 締め後の編集制限を確認するシナリオ

- 根拠: 締め処理機能は記載されているが、締め後の編集可否は未確認
- 採用判断: 要レビュー
- 関連確認事項: Q-XXX
```

原則として、根拠の薄いシナリオを大量に作らない。

---

### 15.6 未カバー観点を確認する

Step 3の観点一覧に対して、対応するシナリオ候補があるか確認する。

特に、重要度が高い観点が未カバーの場合は、以下のいずれかを行う。

* シナリオ候補を追加する
* 既存シナリオ候補に関連観点として追加する
* 仕様不足として確認事項にする
* 対象外候補としてユーザーレビューに回す

## 16. シナリオ候補設計の手順

このstepでは、以下の順で作業する。

### 16.1 高重要度の観点を確認する

`outputs/03_scenario_viewpoints.md` から、重要度が高い観点を確認する。

優先する観点:

* 中核業務に関する観点
* データ整合性に関する観点
* 状態遷移に関する観点
* 帳票・出力に関する観点
* 外部連携に関する観点
* 過去不具合・既知リスクに関する観点

---

### 16.2 観点を業務フロー単位に束ねる

関連する観点を、業務フローや利用シーン単位にまとめる。

例:

```md
BF-001: 契約登録から承認完了まで
- SV-001: 基本業務成立
- SV-003: 状態遷移
- SV-005: ロール・権限
```

このまとまりをもとに、シナリオ候補を設計する。

---

### 16.3 中核正常系シナリオを作る

まず、業務上もっとも重要な正常系シナリオを作る。

正常系シナリオは、後続の分岐・例外シナリオの基準になる。

---

### 16.4 分岐・例外シナリオを作る

次に、業務上重要な分岐・例外をシナリオ化する。

ただし、すべての異常系を細かく作るのではなく、業務影響が大きいものを優先する。

---

### 16.5 データ状態・出力・連携シナリオを補う

必要に応じて、以下を確認するシナリオを追加する。

* データ状態の違い
* 承認済み・締め済みなどの状態制御
* 帳票・ファイル出力
* 外部連携・バッチ
* 取消・再実行
* 過去不具合条件

---

### 16.6 優先度と採用判断を付ける

各シナリオ候補に優先度と採用判断を付ける。

判断が難しい場合は、`未判断` や `要レビュー` とする。

---

### 16.7 未カバー観点を確認する

最後に、Step 3の観点がどのシナリオでカバーされているか確認する。

重要観点が未カバーの場合は、必ず理由を記載する。

---

## 17. 確認事項の形式

確認事項は、以下の形式で記載する。

```md
| ID | 種別 | 内容 | 影響 | 関連箇所 | 状態 |
|---|---|---|---|---|---|
| Q-001 | 仕様不足 | 差戻し後に再申請できるか不明 | SC-002の採用判断・Step 5の期待結果に影響 | SV-004, BF-001 | 未確認 |
```

種別は以下を使う。

| 種別    | 意味                    |
| ----- | --------------------- |
| 不明    | 必要な情報が見つからない          |
| 矛盾    | 文書間または同一文書内で内容が矛盾している |
| 仕様不足  | シナリオ候補設計に必要な仕様が不足している |
| 判断待ち  | ユーザーまたは関係者の判断が必要      |
| 対象外候補 | 今回のシナリオテスト対象外か確認が必要   |

## 19. 禁止事項

このstepでは、以下を禁止する。

* 詳細なテストケースを作成すること
* 詳細な操作手順を作成すること
* 詳細な期待結果を作成すること
* 画面項目単位の確認リストを作ること
* 単機能確認をシナリオ候補として乱立すること
* 未レビューの観点をすべて無条件に採用すること
* 根拠のないシナリオを断定すること
* 対象外候補を勝手に除外確定すること
* Step 5の成果物を先取りして作成すること
* 確認事項を本文に紛れ込ませること
