# Step 3: シナリオ観点の抽出ガイド

## 1. このstepの目的

Step 3では、Step 2で整理した業務フロー・利用シーンをもとに、シナリオテストで確認すべき観点を抽出する。

このstepの目的は、業務上重要な流れ、分岐、データ状態、リスクを踏まえて、どのような観点でシナリオを設計すべきかを明確にすることである。

このstepでは、シナリオ候補や詳細なテストケースは作成しない。  
あくまで「シナリオテストで確認すべき観点」を整理する。

---

## 2. 入力

主な入力は以下とする。

```text
outputs/01_input_summary.md
outputs/02_business_flows.md
input_docs/ のうち必要な範囲
````

## 5. 観点抽出の基本方針

シナリオ観点は、単なる機能確認項目ではない。
業務フローを実行したときに、業務目的が正しく達成されるか、データや状態が正しくつながるか、重要な分岐やリスクが確認できるかを表す。

観点は、以下のような問いに答える形で抽出する。

* この業務フローで、最も壊れてはいけないことは何か
* 利用者が業務目的を達成できるか
* 入力したデータが後続処理や出力に正しく反映されるか
* ステータスや状態遷移が正しいか
* 権限やロールによる差異が正しく制御されるか
* 例外やエラー時に業務が破綻しないか
* 承認、締め、取消、再実行などの重要操作が正しく扱われるか
* 外部連携や帳票出力まで含めて整合しているか
* 過去不具合や既知リスクの再発を防げるか

---

## 6. 抽出対象の観点カテゴリ

以下のカテゴリを参考に、シナリオ観点を抽出する。

ただし、すべてのカテゴリを無理に埋める必要はない。
対象システム・入力文書・業務フローに根拠があるものを優先する。

---

### 6.1 代表業務フロー観点

通常業務で最も重要な基本フローが成立するかを確認する観点。

対象例:

* 新規登録から承認完了まで
* データ取込から反映確認まで
* 入力から帳票出力まで
* マスタ設定から取引入力への反映まで
* 締め処理から後続利用制限まで
* 外部連携データ作成から結果確認まで

記載例:

```md
SV-001: 新規契約登録から承認完了までの基本業務が成立すること

- 関連フロー: BF-001
- 確認したいこと: 担当者が契約を登録し、承認者が承認することで、契約が業務上利用可能な状態になること
- 重要な理由: 契約管理業務の中核フローであり、ここが破綻すると後続業務に影響する
- 根拠: 02_business_flows.md > BF-001
```

---

### 6.2 データ整合性観点

入力・更新・承認・出力・連携の間で、データが正しく引き継がれるかを確認する観点。

対象例:

* 登録内容が一覧・詳細・帳票に反映される
* 承認済みデータだけが後続処理の対象になる
* マスタ変更が取引入力や計算結果に反映される
* 外部連携データと画面表示が一致する
* 集計結果と明細が整合する

記載例:

```md
SV-002: 登録した契約情報が後続画面・帳票に一貫して反映されること

- 関連フロー: BF-001, BF-002
- 確認したいこと: 登録・承認された契約情報が、一覧、詳細、帳票、後続処理で矛盾なく扱われること
- 関連データ: DATA-001
- 重要な理由: データ不整合は業務判断や帳票出力の誤りにつながる
- 根拠: 01_input_summary.md > データ・状態、02_business_flows.md > フロー間の関係
```

---

### 6.3 状態遷移観点

データや業務のステータスが、正しい契機で正しい状態に変化するかを確認する観点。

対象例:

* 未申請 → 申請中 → 承認済み
* 取込待ち → 取込済み → エラー
* 未締め → 締め済み
* 有効 → 無効
* 登録済み → 取消済み
* 一時保存 → 確定

記載例:

```md
SV-003: 承認操作により契約ステータスが正しく遷移すること

- 関連フロー: BF-001
- 確認したいこと: 登録、申請、承認、差戻しなどの操作に応じて、ステータスが正しく変化すること
- 関連データ: DATA-001
- 重要な理由: ステータス誤りは後続処理の対象判定に影響する
- 根拠: 02_business_flows.md > データ状態の変化
```

---

### 6.4 分岐・例外観点

業務上重要な分岐や例外条件で、正しい流れになるかを確認する観点。

対象例:

* 承認 / 差戻し
* 正常データ / エラーデータ
* 権限あり / 権限なし
* 締め前 / 締め後
* 対象データあり / なし
* 連携成功 / 失敗
* 初回実行 / 再実行
* 通常取引 / 例外取引

記載例:

```md
SV-004: 承認と差戻しの分岐で、後続の業務状態が正しく分かれること

- 関連フロー: BF-001
- 関連分岐: BRN-001, BRN-002
- 確認したいこと: 承認時は後続処理に進み、差戻し時は担当者による修正・再申請が可能になること
- 重要な理由: 分岐制御の誤りは業務停滞や誤処理につながる
- 根拠: 02_business_flows.md > 分岐・例外
```

---

### 6.5 ロール・権限観点

利用者ロールによって、操作可否や表示内容が正しく制御されるかを確認する観点。

対象例:

* 担当者は登録できるが承認できない
* 承認者は承認できるが登録内容を編集できない
* 管理者のみマスタ設定できる
* 一般利用者には機密情報が表示されない
* 所属部門によって参照可能範囲が異なる

記載例:

```md
SV-005: 利用者ロールに応じて操作権限が正しく制御されること

- 関連フロー: BF-001
- 関連ロール: 担当者、承認者
- 確認したいこと: 各ロールが許可された操作のみ実行できること
- 重要な理由: 権限制御の誤りは不正操作や業務統制の破綻につながる
- 根拠: 01_input_summary.md > 利用者・ロール
```

---

### 6.6 画面遷移・操作順序観点

業務上必要な順序で操作でき、誤った順序では適切に制御されるかを確認する観点。

対象例:

* 一覧から詳細、登録、編集へ遷移できる
* 登録後に承認依頼へ進める
* 未保存状態で画面遷移する場合に警告が出る
* 前提データがない場合は後続操作できない
* 締め後は編集操作が制限される

記載例:

```md
SV-006: 業務順序に沿って画面遷移・操作が成立すること

- 関連フロー: BF-001
- 関連画面: SCR-001, SCR-002
- 確認したいこと: 登録、確認、承認依頼、承認の順で業務を進められること
- 重要な理由: 操作順序の破綻は業務完了不能につながる
- 根拠: 02_business_flows.md > 業務フロー詳細
```

---

### 6.7 帳票・出力観点

業務フローの結果が帳票・ファイル・レポートに正しく出力されるかを確認する観点。

対象例:

* 登録内容が帳票に反映される
* 集計条件が正しい
* 出力対象データが正しい
* 出力フォーマットが業務利用に適している
* 出力後の確認・提出・保管業務に使える
* Excel / PDF / CSVなどの出力内容が一致する

記載例:

```md
SV-007: 承認済みデータが帳票へ正しく出力されること

- 関連フロー: BF-002
- 関連帳票: RPT-001
- 確認したいこと: 承認済みデータのみが帳票出力対象となり、出力内容が画面上のデータと一致すること
- 重要な理由: 帳票誤りは社内外への報告誤りにつながる
- 根拠: 01_input_summary.md > 画面・帳票・出力
```

---

### 6.8 外部連携・バッチ観点

外部システム連携やバッチ処理を含む業務フローが正しく成立するかを確認する観点。

対象例:

* 連携対象データが正しい
* 連携タイミングが正しい
* 連携成功後にステータスが更新される
* 連携失敗時にエラー確認・再実行できる
* バッチ実行後に画面や帳票へ反映される
* 同一データが重複連携されない

記載例:

```md
SV-008: 外部連携後に連携結果が画面・データ状態へ正しく反映されること

- 関連フロー: BF-003
- 関連連携: IF-001
- 確認したいこと: 連携成功・失敗に応じて、連携結果とデータ状態が正しく更新されること
- 重要な理由: 連携不整合は後続システムや業務判断に影響する
- 根拠: 01_input_summary.md > 外部連携・バッチ・後続処理
```

---

### 6.9 締め・取消・再実行観点

業務上重要な締め、取消、再実行などの処理が正しく制御されるかを確認する観点。

対象例:

* 締め後はデータ編集できない
* 締め取消後に再編集できる
* 取消済みデータは後続処理対象外になる
* 再実行しても重複登録されない
* エラー修正後に再処理できる
* 取消・再実行後の帳票や連携データが整合する

記載例:

```md
SV-009: 締め後の編集制限と締め取消後の再処理が正しく制御されること

- 関連フロー: BF-004
- 確認したいこと: 締め状態に応じて編集可否や後続処理対象が正しく制御されること
- 重要な理由: 締め処理の誤りは会計・集計・報告業務に大きく影響する
- 根拠: 02_business_flows.md > 業務上重要なタイミング
```

---

### 6.10 過去不具合・既知リスク観点

過去不具合や既知リスクを踏まえ、再発防止や重点確認が必要な観点。

対象例:

* 過去に発生したデータ不整合
* 境界条件での誤計算
* 権限制御漏れ
* 帳票出力誤り
* 連携失敗時の復旧不備
* 締め後修正の不備
* ステータス更新漏れ

記載例:

```md
SV-010: 過去不具合が発生した条件でデータ不整合が再発しないこと

- 関連フロー: BF-002
- 関連リスク: R-001
- 確認したいこと: 過去不具合の発生条件を含む業務フローで、同様の不整合が発生しないこと
- 重要な理由: 再発リスクが高く、品質保証上の重点確認対象である
- 根拠: 01_input_summary.md > 過去不具合・既知リスク
```

## 8. シナリオ観点一覧の形式

シナリオ観点一覧は、以下の形式を推奨する。

```md
| ID | 観点名 | 観点カテゴリ | 関連フロー | 重要度 | 確認したいこと | 根拠 |
|---|---|---|---|---|---|---|
| SV-001 | 新規契約登録から承認完了までの基本業務成立 | 代表業務フロー | BF-001 | 高 | 登録から承認完了まで業務が成立すること | 02_business_flows.md > BF-001 |
```

重要度は以下を使う。

| 重要度 | 意味                     |
| --- | ---------------------- |
| 高   | 中核業務、重大リスク、必須確認対象      |
| 中   | 重要だが条件付き、または一部業務に限定される |
| 低   | 補助的、頻度が低い、影響範囲が限定的     |
| 未判断 | 判断材料が不足している            |

## 13. 判断ルール

### 13.1 業務フローに紐づける

シナリオ観点は、原則として業務フローまたは利用シーンに紐づける。

悪い例:

```md
SV-001: 入力チェックを確認する
```

良い例:

```md
SV-001: 契約登録フローにおいて、必須情報不足時に登録を完了できないこと
- 関連フロー: BF-001
```

---

### 13.2 一般論だけの観点を避ける

以下のような一般論だけの観点は避ける。

```md
- 入力チェックを確認する
- エラーが出ることを確認する
- 権限を確認する
- 帳票を確認する
```

業務フロー・データ状態・リスクに結びつけて具体化する。

---

### 13.3 観点を細かく分けすぎない

観点を細かく分けすぎると、後続stepでシナリオ候補が乱立する。
近い目的の観点は統合する。

細かすぎる例:

```md
SV-001: 契約番号を入力できること
SV-002: 契約名を入力できること
SV-003: 契約金額を入力できること
```

適切な例:

```md
SV-001: 契約登録に必要な主要情報を入力し、業務上有効な契約データを作成できること
```

---

### 13.4 観点を大きくしすぎない

観点が大きすぎると、何を確認するのか曖昧になる。

大きすぎる例:

```md
SV-001: 契約管理業務が正しく動くこと
```

適切な例:

```md
SV-001: 新規契約登録から承認完了までの基本業務が成立すること
SV-002: 承認済み契約のみが後続処理の対象になること
SV-003: 差戻し時に担当者が修正して再申請できること
```

---

### 13.5 根拠のない観点は区別する

文書根拠がないが、業務上重要そうな観点は、断定せず区別する。

```md
SV-XXX: 推測: 締め後の修正制限を確認する必要がある可能性

- 根拠: 締め処理機能は記載されているが、締め後の編集可否は未確認
- 扱い: 確認事項 Q-XXX として整理
```

原則として、根拠の薄い観点を大量に作らない。

---

### 13.6 優先度は仮判断とする

このstepで設定する重要度・優先度は、Step 4・Step 5で見直される可能性がある。

そのため、必要に応じて `仮` と明記する。

---

## 14. 観点抽出の手順

このstepでは、以下の順で作業する。

### 14.1 重要な業務フローを確認する

`outputs/02_business_flows.md` から、以下を確認する。

* 重要度が高い業務フロー
* 中核業務に該当するフロー
* 複数機能をまたぐフロー
* 分岐や例外が多いフロー
* データ状態の変化があるフロー
* 後続処理や帳票出力につながるフロー
* 外部連携やバッチを含むフロー

---

### 14.2 フローごとに確認すべきことを洗い出す

各業務フローについて、以下を確認する。

* フローが最後まで成立するか
* 途中の状態遷移が正しいか
* 入力データが後続処理へ正しく引き継がれるか
* 分岐条件ごとに正しい結果になるか
* ロールごとの操作可否が正しいか
* 出力・帳票・連携まで整合するか
* エラー時に復旧できるか
* 過去不具合に関連する条件が含まれるか

---

### 14.3 重複する観点を統合する

似た観点は統合する。

統合する例:

```md
- 登録内容が一覧に反映されること
- 登録内容が詳細に反映されること
- 登録内容が帳票に反映されること
```

統合後:

```md
SV-XXX: 登録内容が後続の参照・出力先に一貫して反映されること
```

ただし、リスクや確認方法が大きく異なる場合は分ける。

---

### 14.4 重要度を付与する

各観点に重要度を付与する。

判断基準:

* 中核業務か
* 障害時の業務影響が大きいか
* 金額・契約・会計・法令・顧客影響があるか
* 利用頻度が高いか
* 複数機能に影響するか
* 過去不具合や仕様変更に関連するか
* 外部連携や帳票出力に影響するか
* 復旧困難なデータ不整合につながるか

---

### 14.5 確認事項を整理する

観点化の過程で発見した不明点は、確認事項に分離する。

---

## 15. 確認事項の形式

確認事項は、以下の形式で記載する。

```md
| ID | 種別 | 内容 | 影響 | 関連箇所 | 状態 |
|---|---|---|---|---|---|
| Q-001 | 仕様不足 | 差戻し後に再申請できるか不明 | 分岐観点、シナリオ候補に影響 | BF-001, BRN-002 | 未確認 |
```

種別は以下を使う。

| 種別    | 意味                    |
| ----- | --------------------- |
| 不明    | 必要な情報が見つからない          |
| 矛盾    | 文書間または同一文書内で内容が矛盾している |
| 仕様不足  | 観点抽出に必要な仕様が不足している     |
| 判断待ち  | ユーザーまたは関係者の判断が必要      |
| 対象外候補 | 今回のシナリオテスト対象外か確認が必要   |

---

## 16. 次stepへの引き継ぎ

成果物の最後に、Step 4へ引き継ぐべき情報を記載する。

```md
## 9. 次stepへの引き継ぎ

- SV-009: ...

### データ整合性を重視する観点
- SV-002: ...
- SV-007: ...

### 確認が必要な事項
- Q-001: ...
- Q-002: ...
```

---

## 17. 禁止事項

このstepでは、以下を禁止する。

* シナリオ候補を作成すること
* 詳細なテストケースを作成すること
* 詳細な操作手順を作成すること
* 期待結果を作成すること
* 画面項目単位の確認リストを作ること
* 一般論だけで観点を大量に作ること
* 業務フローに紐づかない観点を乱立すること
* 根拠のない観点を断定すること
* Step 4以降の成果物を先取りして作成すること
* 確認事項を本文に紛れ込ませること

---

## 18. 完了条件

このstepは、以下を満たした場合に完了とする。

* シナリオ観点一覧が作成されている
* 各観点に関連フローが紐づいている
* 各観点に確認したいことが記載されている
* 各観点に重要な理由が記載されている
* 必要に応じて重要度が付与されている
* 業務フロー別の観点対応が整理されている
* リスク・確認対象が整理されている
* 不明点・矛盾・仕様不足が確認事項に分離されている
* Step 4への引き継ぎが記載されている
* ユーザーレビュー待ちになっている
