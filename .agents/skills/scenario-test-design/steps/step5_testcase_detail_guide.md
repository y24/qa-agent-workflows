# Step 5: テストケース化・優先度付けガイド

## 1. このstepの目的

Step 5では、Step 4で作成した `outputs/04_scenario_candidates.md` をもとに、採用対象のシナリオ候補を実行可能なテストケースへ落とし込む。

このstepの目的は、テスト実行者が迷わず実施でき、かつ合否判定できる粒度のシナリオテストケースを作成することである。

このstepでは、以下を明確にする。

- 何を確認するのか
- どの前提で実行するのか
- どのデータを使うのか
- どの順序で操作・確認するのか
- 何をもって合格とするのか
- どの業務フロー・観点・シナリオに対応するのか

---

## 2. 入力

主な入力は以下とする。

```text
outputs/01_input_summary.md
outputs/02_business_flows.md
outputs/03_scenario_viewpoints.md
outputs/04_scenario_candidates.md
input_docs/ のうち必要な範囲

````

原則として、Step 1〜Step 4の成果物を優先して参照する。

入力文書は、以下の場合にのみ必要範囲を参照する。

* 詳細な操作手順を確認する場合
* 画面項目名を確認する場合
* 期待結果の根拠を確認する場合
* 業務ルールや計算ルールを確認する場合
* テストデータ条件を確認する場合
* 不明点・矛盾点を再確認する場合

## 5. テストケース化の基本方針

テストケースは、シナリオ候補を単純に細かく分解したものではない。

シナリオ候補の業務目的を保ちながら、テスト実行者が実施できるように、以下を具体化する。

* 前提条件
* テストデータ
* 操作手順
* 確認手順
* 期待結果
* 合否判定条件
* 優先度
* 関連情報

ただし、入力文書にない詳細操作や仕様をAIが勝手に補完してはならない。
不明な場合は `要確認` と明記する。

## 6. テストケースに含める情報

各テストケースには、以下を記載する。

### 6.1 テストケースID

テストケースには `TC-001` の形式でIDを付与する。

例:

```md
TC-001: 新規契約を登録し、承認後に後続処理対象となることを確認する
```

---

### 6.2 対応シナリオID

Step 4のシナリオ候補IDを記載する。

例:

```md
- 対応シナリオ: SC-001
```

1つのシナリオ候補から複数のテストケースを作成する場合は、同じシナリオIDを参照してよい。

例:

```md
- TC-001: SC-001の基本正常系
- TC-002: SC-001の入力条件違い
- TC-003: SC-001のロール違い
```

---

### 6.3 テスト目的

テストで何を確認するのかを明確に記載する。

悪い例:

```md
- テスト目的: 契約登録を確認する
```

良い例:

```md
- テスト目的: 担当者が新規契約を登録し、承認者の承認後に契約ステータスが承認済みとなり、後続処理対象になることを確認する
```

---

### 6.4 前提条件

テスト開始前に必要な状態を記載する。

例:

```md
- 前提条件:
  - 担当者ユーザーでログインできる
  - 承認者ユーザーでログインできる
  - 契約登録に必要なマスタが登録済みである
  - 対象契約データは未登録である
```

前提条件が文書にない場合は、`要確認` または `仮置き` とする。

---

### 6.5 テストデータ

テストに使用するデータを記載する。

例:

```md
- テストデータ:
  - 契約番号: 自動採番または未使用の契約番号
  - 契約先: 登録済み取引先
  - 契約開始日: 当月内の日付
  - 契約金額: 正常登録可能な金額
```

具体的な値が未定の場合は、以下のように記載する。

```md
- テストデータ:
  - 契約番号: 要確認
  - 契約先: 登録済み取引先マスタから選択
  - 契約金額: 正常登録可能な金額範囲。具体値は要確認
```

---

### 6.6 操作手順

テスト実行者が実施できる粒度で、操作手順を記載する。

例:

```md
1. 担当者ユーザーでログインする
2. 契約登録画面を開く
3. 新規契約情報を入力する
4. 契約情報を保存する
5. 承認依頼を行う
6. 承認者ユーザーでログインする
7. 承認対象の契約を確認する
8. 契約を承認する
9. 契約のステータスを確認する
10. 後続処理対象として参照できることを確認する
```

画面名やボタン名が不明な場合は、無理に補完せず `要確認` とする。

---

### 6.7 期待結果

各操作または確認ポイントに対して、合否判定可能な期待結果を記載する。

悪い例:

```md
- 正しく登録されること
```

良い例:

```md
- 契約情報が保存され、契約ステータスが「申請中」になること
- 承認者の承認後、契約ステータスが「承認済み」になること
- 承認済み契約が後続処理対象として表示されること
```

期待結果に必要な仕様が不足している場合は、以下のように記載する。

```md
- 期待結果: 要確認。承認後のステータス名称および後続処理への反映タイミングが仕様上未確認。
```

---

### 6.8 確認対象

何を見て確認するのかを記載する。

対象例:

* 画面表示
* 一覧表示
* 詳細画面
* ステータス
* 帳票
* 出力ファイル
* データベース
* 外部連携結果
* ログ
* 通知
* エラーメッセージ

例:

```md
- 確認対象:
  - 契約詳細画面のステータス
  - 承認一覧画面の表示
  - 後続処理対象一覧
  - 契約帳票
```

---

### 6.9 優先度

テストケース単位で優先度を付与する。

推奨優先度:

| 優先度 | 意味                     |
| --- | ---------------------- |
| P0  | 業務上致命的。必ず実行する          |
| P1  | 重要度が高く、通常のシステムテストで実行する |
| P2  | 条件付きで実行する              |
| P3  | 余裕があれば実行する             |
| 未判断 | 判断材料が不足している            |

例:

```md
- 優先度: P0
- 優先度理由: 契約登録から承認完了までの中核業務であり、後続処理に影響するため
```

---

### 6.10 実行区分

テストケースの実行区分を記載する。

推奨区分:

| 実行区分 | 意味                  |
| ---- | ------------------- |
| 必須   | 原則として毎回実行する         |
| 標準   | 通常のシステムテストで実行する     |
| 条件付き | 対象機能・対象設定がある場合に実行する |
| 回帰   | 変更影響や過去不具合に対する回帰確認  |
| 任意   | 余裕がある場合に実行する        |
| 保留   | 仕様不足や判断待ちにより保留      |

---

### 6.11 関連情報

追跡性を確保するため、以下を記載する。

* 関連シナリオ
* 関連観点
* 関連業務フロー
* 関連機能
* 関連画面
* 関連データ
* 関連リスク
* 根拠文書

例:

```md
- 関連情報:
  - シナリオ: SC-001
  - 観点: SV-001, SV-003
  - 業務フロー: BF-001
  - 機能: FN-001, FN-002
  - 画面: SCR-001, SCR-002
  - データ: DATA-001
  - 根拠: outputs/04_scenario_candidates.md > SC-001
```

## 7. テストケースの種類

以下の種類を参考に、シナリオ候補をテストケース化する。

ただし、すべての種類を無理に作成する必要はない。
採用済みシナリオ候補とレビュー結果に基づいて作成する。

---

### 7.1 中核正常系テストケース

主要な業務フローが標準条件で成立することを確認する。

例:

```md
TC-001: 新規契約を登録し、承認後に承認済み状態になることを確認する
```

---

### 7.2 分岐テストケース

業務上重要な分岐が正しく処理されることを確認する。

例:

```md
TC-002: 承認者が契約申請を差戻し、担当者が修正後に再申請できることを確認する
```

---

### 7.3 データ状態テストケース

データ状態やステータスの違いによる操作可否・処理対象差異を確認する。

例:

```md
TC-003: 承認済み契約のみが請求予定作成の対象になることを確認する
```

---

### 7.4 ロール・権限テストケース

ロールごとの操作可否や表示制御を確認する。

例:

```md
TC-004: 担当者ユーザーでは承認操作を実行できないことを確認する
```

---

### 7.5 帳票・出力テストケース

業務フローの結果が帳票・ファイル・レポートに正しく反映されることを確認する。

例:

```md
TC-005: 承認済み契約が契約一覧帳票に正しく出力されることを確認する
```

---

### 7.6 外部連携・バッチテストケース

外部連携やバッチ処理を含む結果を確認する。

例:

```md
TC-006: 連携対象データを作成し、外部連携後に連携結果が画面へ反映されることを確認する
```

---

### 7.7 締め・取消・再実行テストケース

締め、取消、再実行などの状態変更を確認する。

例:

```md
TC-007: 月次締め後に対象データを編集できず、締め取消後に再編集できることを確認する
```

---

### 7.8 過去不具合・回帰テストケース

過去不具合や既知リスクに関連する条件で、再発しないことを確認する。

例:

```md
TC-008: 過去に不整合が発生した条件で、帳票出力結果が画面データと一致することを確認する
```

## 11. シナリオ別のテストケース対応

Step 4のシナリオ候補に対して、どのテストケースで確認するかを整理する。

```md
| シナリオID | シナリオ名 | 対応テストケース | カバー状況 | 補足 |
|---|---|---|---|---|
| SC-001 | 新規契約を登録し承認済み状態にする | TC-001 | カバー | 基本正常系として確認 |
| SC-002 | 差戻し後に再申請する | TC-002 | カバー | 分岐ケースとして確認 |
| SC-006 | 外部連携結果を確認する | - | 保留 | 連携仕様が未確定 |
```

カバー状況は以下を使う。

| カバー状況 | 意味                 |
| ----- | ------------------ |
| カバー   | テストケースで確認できる       |
| 一部カバー | 一部のみ確認できる          |
| 未カバー  | 現時点では対応するテストケースがない |
| 保留    | 仕様不足や判断待ちにより保留     |
| 対象外   | 今回はテスト対象外と判断された    |

---

## 12. 観点別のテストケース対応

Step 3のシナリオ観点に対して、どのテストケースで確認するかを整理する。

```md
| 観点ID | 観点名 | 対応テストケース | カバー状況 | 補足 |
|---|---|---|---|---|
| SV-001 | 基本業務成立 | TC-001 | カバー | 中核正常系で確認 |
| SV-003 | 状態遷移 | TC-001, TC-002 | カバー | 承認・差戻しで確認 |
| SV-008 | 外部連携結果の反映 | - | 保留 | 連携仕様が未確定 |
```

## 13. 優先度・実行区分の整理

テストケース全体を、優先度と実行区分で整理する。

```md
| TC-ID | 優先度 | 実行区分 | 判断理由 |
|---|---|---|---|
| TC-001 | P0 | 必須 | 中核業務であり、後続処理に影響するため |
| TC-002 | P1 | 標準 | 差戻し分岐は重要だが、基本正常系より優先度は下がるため |
| TC-006 | P2 | 条件付き | 外部連携設定がある環境で実行するため |
```

---

## 14. 保留・対象外・要確認ケース

仕様不足や対象範囲の問題により、保留・対象外・要確認となるケースは分離する。

```md
| ID | 種別 | 内容 | 理由 | 影響 | 関連ID |
|---|---|---|---|---|---|
| HOLD-001 | 保留 | 外部連携失敗時の再実行テスト | 連携エラー時の仕様が未確認 | SV-008が未カバー | SC-006, Q-003 |
| OUT-001 | 対象外 | 旧画面での登録シナリオ | 今回の改修対象外 | なし | SC-009 |
```

## 16. 判断ルール

### 16.1 レビュー済みシナリオを優先する

Step 4で `採用候補` またはユーザーが採用指示したシナリオを優先してテストケース化する。

`保留`、`要レビュー`、`対象外候補` のシナリオを勝手に詳細テストケース化しない。

---

### 16.2 シナリオの業務目的を維持する

テストケース化するときに、シナリオの業務目的を見失わない。

悪い例:

```md
TC-001: 保存ボタンを押せること
```

良い例:

```md
TC-001: 新規契約情報を保存し、承認依頼可能な状態になること
```

---

### 16.3 期待結果は合否判定可能にする

期待結果は、実行者が合格・不合格を判断できる表現にする。

悪い例:

```md
- 正常に処理されること
- 問題ないこと
- エラーにならないこと
```

良い例:

```md
- 登録完了メッセージが表示されること
- 契約ステータスが「申請中」になること
- 承認一覧に対象契約が表示されること
```

---

### 16.4 仕様にない詳細を補完しない

画面名、項目名、ボタン名、ステータス名、メッセージ文言、帳票項目、計算結果などが不明な場合は、想像で補完しない。

例:

```md
- 期待結果: 要確認。登録完了時のメッセージ文言が仕様上未確認。
```

---

### 16.5 テストケースを細かく分けすぎない

画面項目1つごと、ボタン1つごとのテストケースに分割しない。

細かすぎる例:

```md
TC-001: 契約番号を入力できる
TC-002: 契約名を入力できる
TC-003: 契約金額を入力できる
```

適切な例:

```md
TC-001: 必要な契約情報を入力して新規契約を登録できる
```

ただし、項目ごとに業務ルールやリスクが大きく異なる場合は、別ケース化してよい。

---

### 16.6 1ケースに詰め込みすぎない

1つのテストケースに、独立した複数の業務目的を詰め込みすぎない。

詰め込みすぎると、失敗時の原因分析が難しくなる。

悪い例:

```md
TC-001: 契約登録、承認、請求、入金、締め、帳票出力、外部連携まで全部確認する
```

適切な例:

```md
TC-001: 契約登録から承認完了までを確認する
TC-002: 承認済み契約から請求予定が作成されることを確認する
TC-003: 請求結果が帳票へ正しく出力されることを確認する
```

---

### 16.7 分岐は必要に応じて別ケース化する

正常系、差戻し、エラー、取消、再実行など、期待結果が大きく異なる分岐は別ケースにする。

例:

```md
TC-001: 承認者が申請を承認する
TC-002: 承認者が申請を差戻す
```

---

### 16.8 テストデータ条件を明確にする

具体的な値が未定でも、必要なデータ条件は明確にする。

悪い例:

```md
- テストデータ: 契約データ
```

良い例:

```md
- テストデータ:
  - 承認前の契約データ
  - 承認者が参照可能な契約データ
  - 後続処理の対象条件を満たす契約データ
```

---

### 16.9 回帰観点を明記する

過去不具合や既知リスクに基づくテストケースは、回帰確認であることを明記する。

例:

```md
- 実行区分: 回帰
- 関連リスク: R-003
- 根拠: 過去不具合情報
```

---

## 17. テストケース化の手順

このstepでは、以下の順で作業する。

### 17.1 採用対象シナリオを確認する

`outputs/04_scenario_candidates.md` から、以下を確認する。

* 採用候補
* P0/P1のシナリオ
* ユーザーレビューで採用されたシナリオ
* 保留・対象外候補
* 確認事項

---

### 17.2 シナリオごとにテストケース化方針を決める

各シナリオについて、以下を判断する。

* 1ケースで確認できるか
* 分岐ごとに分けるべきか
* ロールごとに分けるべきか
* データ状態ごとに分けるべきか
* 帳票・連携確認を同一ケースに含めるか、別ケースにするか
* 回帰確認として別扱いするか

---

### 17.3 前提条件とテストデータを具体化する

Step 4の使用データ概要をもとに、テスト実行に必要なデータ条件を整理する。

具体値が不明な場合は、無理に作らず `要確認` とする。

---

### 17.4 操作手順と期待結果を作成する

操作手順と期待結果を対応させて記載する。

推奨形式:

```md
| 手順 | 操作 | 期待結果 | 確認対象 |
|---|---|---|---|
| 1 | ... | ... | ... |
```

---

### 17.5 優先度と実行区分を設定する

Step 4のシナリオ優先度を参考に、テストケース単位で優先度と実行区分を設定する。

同じシナリオから作成されたテストケースでも、分岐や条件によって優先度が異なる場合がある。

---

### 17.6 トレーサビリティを確認する

各テストケースについて、以下との対応を確認する。

* シナリオ候補
* シナリオ観点
* 業務フロー
* 機能
* 画面
* データ
* リスク
* 根拠文書

---

### 17.7 保留・対象外・要確認を分離する

仕様不足や判断待ちによりテストケース化できないものは、本文に混ぜず分離する。

---

## 18. 確認事項の形式

確認事項は、以下の形式で記載する。

```md
| ID | 種別 | 内容 | 影響 | 関連箇所 | 状態 |
|---|---|---|---|---|---|
| Q-001 | 仕様不足 | 承認後の正確なステータス名称が不明 | TC-001の期待結果に影響 | SC-001, SV-003 | 未確認 |
```

種別は以下を使う。

| 種別    | 意味                    |
| ----- | --------------------- |
| 不明    | 必要な情報が見つからない          |
| 矛盾    | 文書間または同一文書内で内容が矛盾している |
| 仕様不足  | テストケース化に必要な仕様が不足している  |
| 判断待ち  | ユーザーまたは関係者の判断が必要      |
| 対象外候補 | 今回のテスト対象外か確認が必要       |

---

成果物の最後に、ユーザーがレビューしやすいよう、以下の観点を記載する。

```md
## 11. 最終レビュー観点

以下を確認してください。

- P0/P1のテストケースに不足がないか
- 業務上重要なシナリオが漏れていないか
- 優先度が妥当か
- 実行区分が妥当か
- テストデータ条件が現実的か
- 期待結果が合否判定可能か
- 保留・対象外の扱いが妥当か
- 確認事項が解決されているか
```

## 20. 禁止事項

このstepでは、以下を禁止する。

* 仕様にない画面項目・ボタン名・ステータス名を断定すること
* 根拠のない期待結果を作成すること
* 合否判定できない期待結果を書くこと
* 単機能確認を過剰に細分化すること
* 1ケースに複数の独立した業務目的を詰め込みすぎること
* 保留・対象外候補を勝手に採用すること
* ユーザーレビューで除外されたシナリオをテストケース化すること
* 確認事項を本文に紛れ込ませること
* 根拠のないテストデータ値を作ること
