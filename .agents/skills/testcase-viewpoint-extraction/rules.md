# Testcase Viewpoint Extraction 共通ルール

## 目的

既存テストケースから意図と再利用可能な抽象観点を抽出するための共通ルールを定義する。

## 基本方針

- 元テストケースを根拠にする。
- 推定は許可するが、推定度とレビュー要否を明示する。
- 推定を重ねない。
- 曖昧さを勝手に補完しない。
- 根拠TCのない観点を確定扱いしない。

## Step責務

| Step | 責務 | やらないこと |
|---|---|---|
| 01 | 入力整理・テストケース正規化 | 意図抽出、観点候補化 |
| 02 | 意図・観点候補抽出 | 抽象観点の確定 |
| 03 | 抽象観点カタログ化 | 根拠のない観点追加 |
| 04 | トレーサビリティ確認・最終化 | 新規観点追加、issue隠蔽 |

## ID体系

| 対象 | 形式 |
|---|---|
| Test Case | `TC-001` |
| Intent | `INT-001` |
| Candidate | `VC-001` |
| Viewpoint | `VP-001` |
| Issue | `ISS-001` |

既存IDは可能な限り維持する。削除、統合、分割、追加した場合は理由を記録する。

## 出力ルール

- Markdownで出力する。
- 根拠TCを必ず紐づける。
- 確定情報と推定情報を分ける。
- 不明点、低推定度、レビュー要を隠さない。
- 中間成果物を肥大化させず、必要な情報に絞る。

## 抽象化ルール

- 抽象化しすぎない。
- 観点名には条件と期待動作を残す。
- 個別の画面名、データ名、値を残しすぎない。
- 複数TCに再利用できる粒度を目指す。

## issue 記録

issueとして扱うもの:

- 入力欠落
- 曖昧な手順・期待結果
- 根拠不足
- TC間の矛盾
- 抽象化・統合判断の迷い
- トレーサビリティ切れ
- 対象外・保留

## 禁止事項

- 既存TCにない観点を要件から創作する
- 低推定度の内容を確定扱いする
- 根拠TCを落とす
- issueを削除して見かけを整える
- 新しいテストケースを作る

## 品質基準

- 追跡性: TC、意図、候補、抽象観点がIDでつながる。
- 根拠性: 各観点に根拠TCがある。
- 再利用性: 抽象観点が個別TCに閉じすぎていない。
- 透明性: レビュー要、保留、未対応が明示されている。
