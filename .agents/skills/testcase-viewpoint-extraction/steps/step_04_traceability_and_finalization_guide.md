# step_04_traceability_and_finalization_guide.md

## 1. このステップの目的

このステップでは、これまでの成果物をもとに、テストケース、テスト意図、観点候補、抽象観点の対応関係を確認し、最終成果物として整理する。

主な目的は以下である。

- 各テストケースに意図が紐づいていることを確認する
- 各テストケースに抽象観点が紐づいていることを確認する
- 各抽象観点に根拠テストケースが存在することを確認する
- 根拠のない観点を最終成果物から除外する
- 推定度が低い内容を確定扱いしない
- issue を統合・整理する
- 最終的なトレーサビリティ表、issue一覧、サマリを作成する

このステップは、新しい観点を追加する工程ではない。  
既存成果物を検証し、最終成果物として利用可能な状態へ整える工程である。

## 5. 基本方針

## 5.1 対応関係を最優先する

最終成果物では、以下の対応関係が追跡できる必要がある。

```text
テストケース
  ↓
テスト意図
  ↓
観点候補
  ↓
抽象観点
```

どれか1つでも対応が切れている場合は、確定成果物として扱わず、issue として記録する。

---

## 5.2 根拠のない観点を含めない

根拠TCが存在しない抽象観点は、最終トレーサビリティ表に含めてはならない。

根拠のない観点が `03_viewpoint_catalog.md` に含まれていた場合は、以下の扱いにする。

* `04_final_traceability_matrix.md` から除外する
* `04_final_issue_log.md` に除外理由を記録する

---

## 5.3 推定度が低い内容を確定扱いしない

推定度が `低` の意図、観点候補、抽象観点は、確定情報として扱わない。

扱いは以下のいずれかとする。

* レビュー対象として明示する
* 最終成果物から除外する
* issue として記録する

推定度が `中` の内容は最終成果物に含めてよいが、レビュー要否を必要に応じて明示する。

---

## 5.4 新しい観点を追加しない

このステップでは、Step 03 に存在しない抽象観点を新しく追加してはならない。

不足しているように見える観点がある場合は、以下のいずれかにする。

* `04_final_issue_log.md` に「不足している可能性」として記録する
* 必要に応じて Step 02 または Step 03 の再実行を推奨する

このステップで補完してカタログに混ぜてはならない。

---

## 5.5 issue を隠さない

最終成果物では、曖昧さ、不足、判断保留を隠してはならない。

むしろ、最終成果物の価値は以下にもある。

* 意図が明確なテストケース
* 意図が曖昧なテストケース
* 観点化できたテストケース
* 観点化しにくかったテストケース
* 既存テストケースの改善余地

きれいに見せるために issue を消してはならない。

---

## 6. 実施手順

## 6.1 テストケース一覧を確認する

`01_testcase_inventory.md` の全テストケースを確認し、対象TC一覧を作成する。

確認すること:

* 対象TC数
* TC ID の重複有無
* 仮IDの有無
* 除外対象の有無
* 入力 issue の有無

TC ID が重複している場合は、最終トレーサビリティに影響するため issue として記録する。

---

## 6.2 意図抽出結果との対応を確認する

各 TC ID について、`02_intent_and_viewpoint_candidates.md` に対応する意図が存在するか確認する。

確認結果は以下のいずれかに分類する。

| 状態    | 意味                         |
| ----- | -------------------------- |
| OK    | 主意図が存在し、根拠もある              |
| 要レビュー | 主意図はあるが、推定度が中または低、またはレビュー要 |
| 不足    | 主意図が存在しない                  |
| 対象外   | Step 01 時点で対象外とされた         |

主意図が存在しないTCは、`04_final_issue_log.md` に記録する。

---

## 6.3 観点候補との対応を確認する

各 TC ID について、観点候補が存在するか確認する。

確認結果は以下のいずれかに分類する。

| 状態    | 意味                 |
| ----- | ------------------ |
| OK    | 観点候補が存在する          |
| 要レビュー | 観点候補はあるが、根拠が弱い     |
| 不足    | 観点候補が存在しない         |
| 対象外   | Step 01 時点で対象外とされた |

観点候補が存在しないTCは、`04_final_issue_log.md` に記録する。

---

## 6.4 抽象観点との対応を確認する

各観点候補が、`03_viewpoint_catalog.md` の抽象観点に紐づいているか確認する。

確認すること:

* 観点候補が抽象観点に統合されているか
* 抽象観点に観点IDがあるか
* 抽象観点に根拠TCが含まれているか
* 根拠TCに該当TCが含まれているか
* 推定度が低い観点候補を確定扱いしていないか

観点候補が抽象観点に紐づかない場合は、以下のどちらかにする。

* `04_final_traceability_matrix.md` に「未対応」として記録する
* `04_final_issue_log.md` に記録する

---

## 6.5 抽象観点の根拠TCを確認する

`03_viewpoint_catalog.md` の各抽象観点について、根拠TCを確認する。

確認すること:

* 根拠TCが1件以上あるか
* 根拠TCが `01_testcase_inventory.md` に存在するか
* 根拠TCが `02_intent_and_viewpoint_candidates.md` に存在するか
* 根拠TCの観点候補と抽象観点が対応しているか

根拠TCが存在しない、または対応が確認できない観点は最終成果物から除外し、issue に記録する。

---

## 6.6 推定度とレビュー要否を整理する

各TC・各観点について、推定度とレビュー要否を整理する。

レビュー要否を `要` とする条件:

* 推定度が低い
* 推定度が中で、根拠がやや弱い
* 入力 issue が残っている
* 抽出 issue が残っている
* カタログ化 issue が残っている
* 観点候補と抽象観点の対応に迷いがある
* 抽象化レベルに迷いがある
* 1テストケースに複数意図が混在している
* 元テストケースの期待結果が曖昧

レビュー要否が `要` の場合は、理由を簡潔に記録する。

---

## 6.7 issue を統合する

以下の issue を統合する。

```text
01_input_issues.md
02_extraction_issues.md
03_cataloging_issues.md
```

統合時には、以下を行う。

* 重複 issue を統合する
* 同一TCに関する issue をまとめる
* 種別を整理する
* 影響を明確にする
* 推奨対応を具体化する
* 解決済みの issue がある場合は状態を明示する

issue の状態は以下を使用する。

| 状態    | 意味            |
| ----- | ------------- |
| 未解決   | 対応が必要         |
| 要レビュー | ユーザー判断が必要     |
| 情報不足  | 入力資料の追加・修正が必要 |
| 対応不要  | 記録のみでよい       |
| 解決済み  | 後続ステップで解消済み   |

## 7. 最終化時の除外基準

以下に該当する内容は、最終トレーサビリティ表または最終サマリから除外する。

* 根拠TCが存在しない抽象観点
* Step 02 の観点候補に存在しない観点
* 一般論で追加された観点
* 推定度が低く、根拠が弱い意図
* 抽象化しすぎて意味が不明な観点
* カテゴリ名だけで具体性のない観点
* 元テストケースとの対応が確認できない観点

除外した場合は、`04_final_issue_log.md` に記録する。

---

## 8. 最終化時に修正してよいもの

このステップでは、以下の軽微な修正は許可する。

* 表記ゆれの統一
* 観点IDの並び順調整
* issue の重複統合
* レビュー理由の簡潔化
* 根拠TC表記の整理
* カテゴリ順の整理
* サマリ用の件数集計
* 明らかな転記ミスの修正

ただし、以下は行ってはならない。

* 新しい抽象観点の追加
* 意図の再解釈
* 観点候補の新規作成
* 推定度の恣意的な引き上げ
* 根拠の弱い観点の確定化
* issue の削除による見かけ上の品質向上

## 10. 品質チェック

出力前に、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| すべての対象TCがトレーサビリティ表に含まれているか |  |
| 各TCに主意図が紐づいているか |  |
| 各TCに観点候補が紐づいているか |  |
| 観点候補が抽象観点に対応しているか |  |
| 未対応の観点候補が明示されているか |  |
| 各抽象観点に根拠TCがあるか |  |
| 根拠TCのない観点を最終成果物から除外しているか |  |
| 推定度が低い内容を確定扱いしていないか |  |
| issue が統合されているか |  |
| issue を削除して品質が高く見えるようにしていないか |  |
| 新しい観点をこのステップで追加していないか |  |
| 最終サマリに制約と注意事項が記録されているか |  |
```

この品質チェック表は、必要に応じて `04_final_summary.md` の末尾に含めてもよい。
ただし、成果物が過度に肥大化する場合は省略してよい。

## 11. このステップでしてはいけないこと

このステップでは、以下を行ってはならない。

* 新しい抽象観点を追加する
* Step 02 にない観点候補を作る
* Step 03 にない観点IDを勝手に追加する
* 根拠TCのない観点を最終成果物に含める
* 推定度が低い内容を確定扱いする
* issue を削除して見かけ上きれいにする
* 入力にない仕様を前提に最終サマリを書く
* 一般的な不足観点を自由に提案する
* トレーサビリティが切れている内容を隠す
* ユーザーレビューが必要な事項を不要扱いする
