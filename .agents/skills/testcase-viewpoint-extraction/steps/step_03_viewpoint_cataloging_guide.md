# step_03_viewpoint_cataloging_guide.md

## 1. このステップの目的

このステップでは、`02_intent_and_viewpoint_candidates.md` で抽出したテスト観点候補をもとに、再利用可能な抽象観点カタログを作成する。

主な目的は以下である。

- 類似する観点候補を統合する
- 個別機能に依存しすぎた観点候補を適切に抽象化する
- 抽象化しすぎて意味が薄くなった観点を避ける
- 観点カテゴリを付与する
- 観点IDを付与する
- 各抽象観点に根拠テストケースを紐づける
- 抽象化理由を記録する
- 判断が難しい観点を issue として記録する

このステップは、一般的なテスト観点一覧を新しく作る工程ではない。  
必ず Step 02 で抽出された観点候補を起点にする。

## 5. 基本方針

## 5.1 観点候補を起点にする

抽象観点は、必ず Step 02 の観点候補から作成する。

以下は禁止する。

* 元テストケースに存在しない観点を一般論で追加する
* よくあるテスト観点を補完する
* 仕様上ありそうな観点を推測して追加する
* 不足観点を自由に洗い出す

このステップで作成する観点は、すべて1件以上の根拠TCを持つ必要がある。

---

## 5.2 統合と抽象化を同時に行う

観点候補には、以下のようなばらつきがある。

```text
取引先コードの重複入力制御
ユーザーIDの重複入力制御
部門コードの重複登録チェック
伝票番号の重複エラー
```

これらは、必要に応じて以下のように統合・抽象化する。

```text
一意制約項目の重複入力制御
```

ただし、抽象化しすぎて以下のようにしてはならない。

```text
入力チェック
```

統合後も、何をどの条件で確認する観点なのか分かる表現にする。

---

## 5.3 抽象観点には条件と期待動作を残す

抽象観点名には、可能な範囲で以下の要素を残す。

```text
対象の性質 + 条件 + 期待される制御または振る舞い
```

良い例:

```text
一意制約項目の重複入力制御
必須項目未入力時の登録抑止
確定済み状態のデータに対する更新制御
権限不足ユーザーによる保護対象操作の抑止
登録・更新後の関連画面への反映確認
境界日付における期間判定
出力条件に応じた帳票内容の反映確認
```

悪い例:

```text
入力チェック
状態確認
権限確認
表示確認
帳票確認
正常系確認
異常系確認
```

悪い例はカテゴリ名としては使える場合があるが、抽象観点名としては粗すぎる。

---

## 5.4 抽象化レベルを明示する

抽象観点には、抽象化レベルを付与する。

| レベル | 意味         | 例             |
| --- | ---------- | ------------- |
| L1  | 個別機能レベル    | 取引先コードの重複入力制御 |
| L2  | 機能横断レベル    | マスタコードの重複入力制御 |
| L3  | 汎用テスト設計レベル | 一意制約項目の重複入力制御 |

最終的な観点カタログでは、L2 または L3 を中心にする。

ただし、元テストケースの文脈を失うほど無理に L3 へ引き上げてはならない。
業務ドメイン固有の意味が重要な場合は、L2 に留める。

---

## 5.5 観点IDを付与する

各抽象観点には観点IDを付与する。

形式:

```text
VP-{CATEGORY}-{連番}
```

例:

```text
VP-INPUT-001
VP-STATE-001
VP-AUTH-001
VP-DATA-001
VP-OUTPUT-001
```

カテゴリは `references/viewpoint_category_seed.md` を参考にする。

分類に迷う場合は、無理に決めず、以下のようにする。

```text
VP-UNCLEAR-001
```

分類に迷った理由は `03_cataloging_issues.md` に記録する。

---

## 5.6 根拠TCを必ず残す

各抽象観点には、必ず根拠となる TC ID を紐づける。

根拠TCがない観点は、`03_viewpoint_catalog.md` に含めてはならない。

もし作業中に根拠TCがない観点が発生した場合は、以下のいずれかにする。

* 削除する
* `03_cataloging_issues.md` に「根拠なし観点」として記録する

---

## 6. 実施手順

## 6.1 観点候補を収集する

`02_intent_and_viewpoint_candidates.md` から、すべての観点候補を収集する。

収集時には以下を保持する。

* 観点候補名
* 対応する TC ID
* 主意図
* 副次意図
* 根拠
* 推定度
* レビュー要否
* レビュー理由

この時点では、観点候補を勝手に削除しない。

---

## 6.2 観点候補をグルーピングする

類似する観点候補をグルーピングする。

グルーピングの観点:

* 同じ条件を扱っている
* 同じ期待動作を扱っている
* 同じ制約を扱っている
* 同じ確認対象を扱っている
* 表現が違うだけで意味が近い
* 個別対象は違うが、確認の切り口が同じ

例:

```text
取引先コードの重複入力制御
ユーザーIDの重複入力制御
部門コードの重複登録チェック
```

上記は、以下のグループとして扱える。

```text
識別子・コードの重複入力制御
```

ただし、意味が異なるものを無理にまとめてはならない。

例:

```text
必須項目未入力時のエラー表示
禁止文字入力時のエラー表示
重複コード入力時のエラー表示
```

これらはすべて「エラー表示」としてまとめることもできるが、確認している制約が異なる。
必要に応じて、別観点として維持する。

---

## 6.3 抽象観点名を作成する

各グループに対して、抽象観点名を作成する。

抽象観点名は、以下を満たすこと。

* 元の観点候補から説明できる
* 1件以上の根拠TCがある
* 個別機能に依存しすぎない
* 抽象化しすぎて意味が薄くならない
* 条件または期待動作が読み取れる
* 後続のテスト設計で再利用できる

### 良い抽象観点名の例

```text
一意制約項目の重複入力制御
必須項目未入力時の登録抑止
入力値の形式不正時のエラー表示
確定済み状態のデータに対する更新制御
権限不足ユーザーによる保護対象操作の抑止
登録・更新後の関連画面への反映確認
検索条件に一致するデータ抽出
出力条件に応じた帳票内容の反映確認
```

### 避けるべき抽象観点名の例

```text
入力
画面
確認
表示
登録
正常
異常
権限
状態
帳票
```

---

## 6.4 カテゴリを付与する

各抽象観点にカテゴリを付与する。

カテゴリは、`references/viewpoint_category_seed.md` を参考にする。

カテゴリ例:

```text
入力制約
業務ルール
状態制御
権限
データ整合性
表示
検索
登録・更新・削除
帳票・出力
外部連携
エラー・警告
排他・同時実行
監査ログ
性能
回帰確認
その他
```

カテゴリは観点の整理用であり、観点名そのものではない。

悪い例:

```text
カテゴリ: 入力制約
抽象観点: 入力制約
```

良い例:

```text
カテゴリ: 入力制約
抽象観点: 必須項目未入力時の登録抑止
```

---

## 6.5 抽象化レベルを判定する

各抽象観点に L1 / L2 / L3 の抽象化レベルを付与する。

### L1: 個別機能レベル

特定機能や特定画面に依存している。

例:

```text
取引先コードの重複入力制御
```

### L2: 機能横断レベル

同種の機能へ横展開できる。

例:

```text
マスタコードの重複入力制御
```

### L3: 汎用テスト設計レベル

ドメインを越えて再利用しやすい。

例:

```text
一意制約項目の重複入力制御
```

原則として、Step 03 の成果物では L2 または L3 を優先する。
ただし、L3 にすると意味が曖昧になる場合は L2 に留める。

---

## 6.6 抽象化理由を記録する

各抽象観点には、抽象化理由を記録する。

抽象化理由には、以下を含める。

* どの観点候補を統合したか
* どの個別表現を一般化したか
* なぜその抽象度が妥当か
* どのTCが根拠か

例:

```text
取引先コード、ユーザーID、部門コードの重複入力制御はいずれも、一意であるべき識別子に重複値を入力した場合の登録抑止を確認しているため、一意制約項目の重複入力制御として統合した。
```

---

## 6.7 レビュー要否を判定する

以下に該当する場合は、レビュー要否を `要` とする。

* 根拠TCが少ない
* 推定度が中または低の観点候補をもとにしている
* 抽象化レベルに迷いがある
* カテゴリ分類に迷いがある
* 複数の観点候補を統合したが、意味の差が残っている
* 業務ドメイン固有の表現を一般化しすぎた可能性がある
* 観点名が抽象的すぎる可能性がある
* 観点名が個別機能に依存しすぎている可能性がある

レビュー要否が `要` の場合は、理由を記録する。

## 8. issue として記録するもの

以下に該当する場合は、`03_cataloging_issues.md` に記録する。

* 観点候補をどのカテゴリに分類すべきか判断できない
* 観点候補を統合すべきか判断できない
* 抽象化レベルに迷いがある
* 観点名が抽象的すぎる可能性がある
* 観点名が個別機能に依存しすぎている可能性がある
* 推定度が低い観点候補をもとにしている
* 根拠TCが1件のみで、独立観点とすべきか判断が難しい
* 似た観点が複数残っている
* 元観点候補の意味差を統合で失う可能性がある
* 根拠TCがない観点が発生した
* Step 02 の意図・観点候補に戻って確認すべき内容がある

## 9. 判断例

## 9.1 重複入力制御の統合

### 入力となる観点候補

| TC ID  | 観点候補           |
| ------ | -------------- |
| TC-002 | 取引先コードの重複入力制御  |
| TC-014 | ユーザーIDの重複入力制御  |
| TC-021 | 部門コードの重複登録チェック |

### 出力例

| 観点ID         | カテゴリ | 抽象観点          | 説明                                       | 元観点候補                                      | 根拠TC                   | 抽象化レベル |
| ------------ | ---- | ------------- | ---------------------------------------- | ------------------------------------------ | ---------------------- | ------ |
| VP-INPUT-001 | 入力制約 | 一意制約項目の重複入力制御 | 一意であるべき項目に重複値を入力した場合、登録・更新が抑止されることを確認する。 | 取引先コードの重複入力制御、ユーザーIDの重複入力制御、部門コードの重複登録チェック | TC-002, TC-014, TC-021 | L3     |

## 9.2 状態による更新制御

### 入力となる観点候補

| TC ID  | 観点候補         |
| ------ | ------------ |
| TC-030 | 締め済み期間の更新制御  |
| TC-031 | 承認済みデータの削除抑止 |

### 出力例

| 観点ID         | カテゴリ | 抽象観点                  | 説明                                                   | 元観点候補                    | 根拠TC           | 抽象化レベル |
| ------------ | ---- | --------------------- | ---------------------------------------------------- | ------------------------ | -------------- | ------ |
| VP-STATE-001 | 状態制御 | 確定済み状態のデータに対する変更操作の抑止 | 締め済み・承認済みなど、確定状態にあるデータに対して更新・削除などの変更操作が制限されることを確認する。 | 締め済み期間の更新制御、承認済みデータの削除抑止 | TC-030, TC-031 | L3     |

注意:

上記は、締め済みと承認済みを「確定済み状態」としてまとめられる根拠がある場合に限る。
元テストケース上で意味差が大きい場合は、別観点として残す。

## 9.3 登録後反映の統合

### 入力となる観点候補

| TC ID  | 観点候補     |
| ------ | -------- |
| TC-001 | 登録後の一覧反映 |
| TC-008 | 更新後の詳細反映 |
| TC-015 | 登録後の帳票反映 |

### 出力例

統合しすぎると、確認対象の差が失われる。

悪い例:

```text
データ反映確認
```

| ------------- | ------ | ----------------- | ---------------------------------------------- | ----------------- | -------------- | ------ |
| VP-DATA-001   | データ整合性 | 登録・更新後の関連画面への反映確認 | 登録・更新した内容が一覧、詳細、検索結果などの関連画面に一貫して反映されることを確認する。  | 登録後の一覧反映、更新後の詳細反映 | TC-001, TC-008 | L2     |
| VP-OUTPUT-001 | 帳票・出力  | 登録・更新後の出力物への反映確認  | 登録・更新した内容が帳票、CSV、Excel、PDFなどの出力物に反映されることを確認する。 | 登録後の帳票反映          | TC-015         | L2     |

---

## 9.4 エラー表示の扱い

### 入力となる観点候補

```text
必須項目未入力時のエラー表示
重複コード入力時のエラー表示
禁止文字入力時のエラー表示
```

これらはすべて「エラー表示」として統合できるように見える。
しかし、実際には確認している制約が異なる。

推奨:

```text
必須項目未入力時の登録抑止
一意制約項目の重複入力制御
入力値の形式不正時のエラー表示
```

エラー表示自体を独立観点にする場合は、元テストケースでメッセージ内容や表示位置を明確に確認している場合に限る。

## 10. 品質チェック

出力前に、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| すべての抽象観点が Step 02 の観点候補を起点にしているか |  |
| 各抽象観点に1件以上の根拠TCがあるか |  |
| 根拠TCのない観点が混入していないか |  |
| 観点名が抽象的すぎないか |  |
| 観点名が個別機能に依存しすぎていないか |  |
| 観点名に条件または期待動作が残っているか |  |
| 類似観点が不要に重複していないか |  |
| 意味の違う観点を無理に統合していないか |  |
| カテゴリ名と観点名が同一になっていないか |  |
| 抽象化理由が説明されているか |  |
| レビューが必要な観点が明示されているか |  |
| 推定度が低い観点候補を確定扱いしていないか |  |
```

この品質チェック表は、必要に応じて成果物末尾に含めてもよい。
ただし、成果物が過度に肥大化する場合は省略してよい。

---

## 11. このステップでしてはいけないこと

このステップでは、以下を行ってはならない。

* Step 02 に存在しない観点を新規追加する
* 一般的なテスト観点一覧を作る
* 不足観点を自由に補完する
* 根拠TCのない観点をカタログに含める
* 推定度が低い観点候補を確定扱いする
* 意味の異なる観点を無理に統合する
* 観点名をカテゴリ名だけにする
* 抽象化しすぎて確認条件を失う
* 個別機能名を完全に消して意味不明にする
* レビュー要否を隠す
* 次の Step 04 に進む

---

このステップは、以下を満たしたとき完了とする。

* `03_viewpoint_catalog.md` が作成されている
* `03_cataloging_issues.md` が作成されている
