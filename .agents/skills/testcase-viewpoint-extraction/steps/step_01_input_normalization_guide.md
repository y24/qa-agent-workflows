# step_01_input_normalization_guide.md

## 1. このステップの目的

このステップでは、入力されたテストケース資料を、後続ステップで扱いやすい標準形式に整理する。

主な目的は以下である。

- 解析対象のテストケースを明確にする
- テストケースIDを確定する
- テストケースの構成要素を標準形式にそろえる
- 欠落、曖昧、矛盾、複数ケース混在を記録する
- 後続の意図抽出・観点抽出で推測が混入しにくい状態にする

このステップでは、テスト意図やテスト観点の深い抽出は行わない。  
あくまで、元テストケースを整理し、後続処理の入力として使える状態にする。

---

## 2. 入力

このステップでは、以下のような資料を入力として扱う。

- Excelテスト仕様書
- CSV化されたテストケース一覧
- Markdown形式のテストケース
- 既存テストケース管理表
- 手順書内に記載されたテストケース
- 画面名、機能名、テスト対象範囲に関する補足情報
- ユーザーからの対象範囲指定

入力資料に複数シート、複数ファイル、複数機能が含まれる場合は、対象範囲を明示する。

---

## 3. 出力

このステップでは、以下の2ファイルを出力する。

```text
01_testcase_inventory.md
01_input_issues.md
````

### 3.1 01_testcase_inventory.md

正規化したテストケース一覧。

### 3.2 01_input_issues.md

入力資料に関する問題、曖昧さ、不足、確認事項の一覧。

---

## 4. 参照ファイル

このステップでは、以下を参照する。

```text
rules.md
templates/testcase_inventory_template.md
templates/issue_log_template.md
```

必要に応じて、以下も参照してよい。

```text
references/definitions.md
references/ambiguity_policy.md
```

ただし、コンテキスト節約のため、不要な参照ファイルは読み込まない。

---

## 5. 実施手順

## 5.1 入力資料の構造を確認する

最初に、入力資料の構造を確認する。

確認する項目:

* ファイル名
* シート名またはセクション名
* 対象機能
* 対象範囲
* テストケースの記載単位
* ヘッダー行の有無
* テストケースIDの有無
* テストケース名の有無
* 前提条件の有無
* 操作手順の有無
* 入力値・テストデータの有無
* 期待結果の有無
* 備考・補足列の有無
* 既存の観点・分類列の有無

入力資料の構造が不明な場合は、無理に解釈せず `01_input_issues.md` に記録する。

---

## 5.2 解析対象を確定する

解析対象とするテストケース範囲を明確にする。

ユーザーが対象範囲を指定している場合は、その範囲を優先する。

対象範囲として記録する項目:

* 対象ファイル
* 対象シート
* 対象機能
* 対象テスト種別
* 対象行範囲
* 除外した範囲
* 除外理由

対象かどうか判断できない行は、勝手に除外せず issue として記録する。

---

## 5.3 テストケースIDを確認する

各テストケースにIDが存在するか確認する。

### IDが存在する場合

元のIDを維持する。

IDの表記ゆれがある場合も、原則として元の表記を保持する。

例:

```text
TC-001
TC_001
No.001
T001
```

表記ゆれが後続処理に影響しそうな場合は issue として記録する。

### IDが存在しない場合

仮IDを付与する。

仮ID形式:

```text
TC-TEMP-001
TC-TEMP-002
TC-TEMP-003
```

仮IDを付与した場合は、必ず `01_input_issues.md` に記録する。

---

## 5.4 テストケースの単位を確認する

入力資料の1行または1ブロックが、1つのテストケースとして扱えるか確認する。

以下の場合は注意する。

* 1行に複数の確認内容が含まれている
* 1つのテストケースが複数行に分割されている
* 手順ごとに行が分かれている
* 期待結果が複数あり、別ケースとして分けるべき可能性がある
* 見出し行や説明行がテストケースに見える
* 前提条件だけの行がある
* テストデータだけの行がある

判断が難しい場合は、無理に分割・統合せず issue として記録する。

このステップでは、テストケースの再設計は行わない。

---

## 5.5 標準項目にマッピングする

入力資料の列や記述を、以下の標準項目にマッピングする。

| 標準項目    | 内容                     |
| ------- | ---------------------- |
| TC ID   | テストケースID               |
| テストケース名 | テストケースの名称または概要         |
| 対象機能    | 対象画面、機能、業務、モジュール       |
| 前提条件    | 実行前に満たすべき状態            |
| 操作/入力   | 操作手順、入力値、テストデータ        |
| 期待結果    | 確認すべき結果                |
| 既存分類    | 元資料に記載された分類、観点、テスト種別など |
| 備考      | 補足、注意、関連情報             |
| 正規化メモ   | AIエージェントが整理時に記録すべきメモ   |

入力資料に該当する列がない場合は空欄にする。
空欄を推測で埋めてはならない。

---

## 5.6 元の表現を保つ

このステップでは、入力の意味を勝手に書き換えない。

許可される整理:

* 改行の整理
* 箇条書きの整形
* 複数列情報の結合
* 明らかな空白・記号の整理
* 表として読みやすい形への整形

禁止される整理:

* 期待結果を具体化して補う
* 仕様らしい説明を追加する
* 操作手順を推測して補う
* テスト意図を先回りして書く
* 観点名を独自に付与する
* 異常系・正常系を勝手に分類する
* 元資料にない条件を追加する

---

## 5.7 欠落・曖昧さを記録する

以下に該当するものは、`01_input_issues.md` に記録する。

* テストケースIDがない
* テストケース名がない
* 対象機能が不明
* 前提条件が不足している
* 操作/入力が不足している
* 期待結果が不足している
* 期待結果が曖昧
* テストデータの意味が不明
* 1ケースに複数の確認内容が含まれている
* 複数行を1ケースとして扱う必要がある
* 対象外かどうか判断できない
* 元資料内で記述が矛盾している
* 列の意味が不明
* 既存分類の意味が不明

issue は、後続ステップで推測を抑制するための重要な情報として扱う。

---

## 5.8 正規化結果を作成する

`01_testcase_inventory.md` には、以下の形式で出力する。

```markdown

## 1. 入力概要

| 項目 | 内容 |

| 対象ファイル |  |

| 対象テストケース数 |  |

## 2. テストケース一覧

| TC ID | テストケース名 | 対象機能 | 前提条件 | 操作/入力 | 期待結果 | 既存分類 | 備考 | 正規化メモ |
|---|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |  |  |
```

## 5.9 入力 issue を作成する

`01_input_issues.md` には、以下の形式で出力する。

```markdown
# 01_input_issues.md

## 1. Issue一覧

| Issue ID | 種別 | 対象TC | 内容 | 影響 | 推奨対応 |
|---|---|---|---|---|---|
| ISS-IN-001 |  |  |  |  |  |
```

issue がない場合も、以下のように明記する。

```markdown
# 01_input_issues.md

## 1. Issue一覧

入力整理時点で記録すべき issue はありません。
```

## 6. 判断基準

## 6.1 テストケースとして扱うもの

以下は、原則としてテストケースとして扱う。

* 前提、操作、期待結果が揃っている行
* テストケースIDが付与されている行
* テストケース名と期待結果がある行
* 手順形式で記載され、確認結果が明示されているブロック

---

## 6.2 テストケースとして扱わない可能性があるもの

以下は、テストケースではなく補足情報として扱う可能性がある。

* セクション見出し
* 機能名だけの行
* 観点名だけの行
* テストデータ定義のみの行
* 共通前提条件のみの行
* 備考のみの行
* 実行結果記録のみの行

ただし、判断に迷う場合は issue として記録する。

---

## 6.3 複数行で1ケースの場合

複数行で1つのテストケースを構成している場合は、可能な範囲で1レコードにまとめる。

例:

| 行 | 内容                    |
| - | --------------------- |
| 1 | TC-001                |
| 2 | 手順1: 画面を開く            |
| 3 | 手順2: 値を入力する           |
| 4 | 期待結果: 登録完了メッセージが表示される |

この場合、`TC-001` の `操作/入力` に手順1・手順2をまとめる。

ただし、どこまでが同一ケースか不明な場合は、無理にまとめず issue として記録する。

---

## 6.4 1行に複数ケースが混在している場合

1行に複数の確認が含まれている場合は、原則としてそのまま1レコードとして扱い、正規化メモまたは issue に記録する。

例:

```text
登録できること。登録後、一覧に表示されること。帳票にも反映されること。
```

この場合、複数意図が含まれる可能性があるため、issue として記録する。

このステップでは、勝手に複数ケースへ分割しない。
分割が必要な場合は、ユーザーレビュー後に対応する。

---

## 7. 出力時の注意

## 7.1 空欄の扱い

入力に情報がない項目は空欄にする。

以下のような補完は禁止する。

```text
不明だが、おそらくログイン済み
通常は登録画面だと思われる
仕様上はエラーになるはず
```

不明な場合は、不明として issue に記録する。

---

## 7.2 原文保持と要約のバランス

操作手順や期待結果が長い場合は、意味を損なわない範囲で整理してよい。

ただし、後続の意図抽出に必要な情報を削ってはならない。

残すべき情報:

* 条件
* 入力値
* 操作
* 期待する画面表示
* 期待するデータ状態
* 期待するエラー・警告
* 対象ユーザー・権限
* 対象期間・日付
* 対象帳票・出力物
* 対象ステータス

---

## 7.3 正規化メモの使い方

`正規化メモ` には、後続ステップで注意すべき内容を書く。

例:

```text
期待結果が複数含まれる
前提条件が別シートにある可能性
対象機能は見出し行から補った
テストケースIDがないため仮IDを付与
手順と期待結果が同一セルに混在
```

正規化メモには、テスト意図や観点候補を書かない。

---

## 8. 品質チェック

出力前に、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| 解析対象範囲が明記されているか |  |
| 各テストケースにTC IDがあるか |  |
| 仮IDを付与したケースが issue に記録されているか |  |
| 操作/入力と期待結果を取り違えていないか |  |
| 入力にない情報を補完していないか |  |
| 欠落・曖昧さが issue に記録されているか |  |
| 後続ステップで必要な情報が残っているか |  |
| テスト意図や観点をこのステップで先取りしていないか |  |
```

この品質チェック表は、必要に応じて出力末尾に含めてもよい。
ただし、成果物が過度に肥大化する場合は省略してよい。

---

## 9. このステップでしてはいけないこと

このステップでは、以下を行ってはならない。

* テスト意図を抽出する
* テスト観点候補を抽出する
* 抽象観点を作成する
* テストケースを勝手に分割する
* テストケースを勝手に統合する
* 期待結果を推測で補う
* 前提条件を一般常識で補う
* 仕様書に書いてありそうな内容を補う
* 正常系・異常系を独自判断で付与する
* 元資料にないカテゴリを追加する
* issue を隠してきれいな一覧にする

---

## 10. 完了条件

このステップは、以下を満たしたとき完了とする。

* `01_testcase_inventory.md` が作成されている
* `01_input_issues.md` が作成されている
* 解析対象のテストケースが一覧化されている
* 各テストケースにIDまたは仮IDがある
* 入力資料の曖昧さ、不足、矛盾が issue として記録されている
* 後続の Step 02 で意図・観点候補を抽出できる状態になっている

完了後は、ユーザーのレビューを待つ。
ユーザーから指示があるまで、Step 02 には進まない。
