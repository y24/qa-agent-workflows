# step_02_intent_and_candidate_extraction_guide.md

## 1. このステップの目的

このステップでは、`01_testcase_inventory.md` に整理された各テストケースから、以下を抽出する。

- 各テストケースの主意図
- 必要に応じた副次意図
- テスト観点候補
- 根拠
- 推定度
- レビューが必要な事項

このステップの目的は、テストケースを読んで「このテストは何を確認しようとしているのか」を明らかにし、その意図から読み取れるテスト観点候補を整理することである。

ただし、このステップでは観点を過度に抽象化しない。  
抽象観点カタログへの統合・一般化は Step 03 で行う。

---

## 2. 入力

このステップでは、以下を入力として使用する。

```text
01_testcase_inventory.md
01_input_issues.md
````

### 2.1 01_testcase_inventory.md

正規化されたテストケース一覧。

主に参照する項目:

* TC ID
* テストケース名
* 対象機能
* 前提条件
* 操作/入力
* 期待結果
* 既存分類
* 備考
* 正規化メモ

### 2.2 01_input_issues.md

入力整理時点で判明した曖昧さ、不足、注意点。

このファイルに記録された issue は、意図抽出時の推定度判断に利用する。


## 5. 基本方針

## 5.1 元テストケースを根拠にする

意図と観点候補は、元テストケースの記述から説明できる範囲で抽出する。

根拠として使える情報:

* テストケース名
* 対象機能
* 前提条件
* 操作/入力
* 期待結果
* 入力値・テストデータ
* 既存分類
* 備考
* 正規化メモ
* 入力 issue

入力に存在しない仕様や業務ルールを、確定事項として追加してはならない。

---

## 5.2 意図と観点候補を分ける

このステップでは、以下を明確に分ける。

### テスト意図

そのテストケースが何を確認しようとしているか。

例:

```text
取引先コードが重複した場合に、登録が抑止されることを確認する。
```

### テスト観点候補

その意図から読み取れる確認の切り口。

例:

```text
取引先コードの重複入力制御
一意性制約
エラー表示
```

意図は文章で書く。
観点候補は短い名詞句で書く。

---

## 5.3 抽象化しすぎない

このステップでは、観点候補を一般化しすぎない。

良い例:

```text
取引先コードの重複入力制御
締め済み期間の更新制御
必須項目未入力時のエラー表示
登録後の一覧反映
権限なしユーザーによる削除抑止
```

悪い例:

```text
入力チェック
業務ルール確認
画面確認
異常系確認
正常系確認
データ確認
```

後者は粗すぎる。
カテゴリとしては使える場合があるが、このステップの観点候補としては情報量が不足している。

---

## 5.4 推定度を必ず付与する

各テストケースの意図と観点候補には、推定度を付与する。

| 推定度 | 判断基準                         |
| --- | ---------------------------- |
| 高   | テストケース名、操作/入力、期待結果から明確に読み取れる |
| 中   | 妥当に推定できるが、明示されていない要素がある      |
| 低   | 可能性はあるが、根拠が弱い                |

推定度が低い場合は、`02_extraction_issues.md` に記録する。

---

## 5.5 推定を重ねない

推定を根拠に、さらに推定を重ねてはならない。

許可される例:

```text
期待結果に「重複エラーが表示される」とあるため、
重複入力制御を確認していると判断する。
```

禁止される例:

```text
重複エラーがある
→ マスタ設計上、一意制約があるはず
→ 業務上、コード体系の整合性を担保する目的である
```

最後の「コード体系の整合性を担保する目的」は、元テストケースに記述がなければ推定しすぎである。

---

## 6. 実施手順

## 6.1 テストケースを1件ずつ読む

`01_testcase_inventory.md` の各テストケースを、1件ずつ確認する。

確認する項目:

* TC ID
* テストケース名
* 対象機能
* 前提条件
* 操作/入力
* 期待結果
* 既存分類
* 備考
* 正規化メモ

複数テストケースをまとめて処理してよいが、出力では必ずTC ID単位で記録する。

---

## 6.2 主意図を抽出する

各テストケースについて、主意図を1つ抽出する。

主意図は、以下の問いに答える形で書く。

```text
このテストケースは、最も中心的に何を確認しようとしているのか？
```

主意図の表現形式:

```text
〜の場合に、〜となることを確認する。
〜操作により、〜が正しく行われることを確認する。
〜条件で、〜が制御されることを確認する。
```

例:

```text
必須項目を入力した場合に、取引先を正常に登録できることを確認する。
```

```text
既存の取引先コードを入力した場合に、重複エラーとなり登録が抑止されることを確認する。
```

```text
締め済み期間のデータに対して、更新操作ができないことを確認する。
```

---

## 6.3 副次意図を抽出する

1つのテストケースに、主意図以外の確認が含まれる場合は副次意図として記録する。

副次意図の例:

```text
登録後に一覧へ反映されることを確認する。
エラーメッセージの内容が妥当であることを確認する。
更新後に詳細画面へ反映されることを確認する。
帳票出力結果にも登録内容が反映されることを確認する。
```

ただし、副次意図を無理に作らない。
元テストケースから読み取れる場合のみ記録する。

副次意図がない場合は、`なし` と記録する。

---

## 6.4 観点候補を抽出する

主意図・副次意図から、観点候補を抽出する。

観点候補は、元テストケースに近い粒度でよい。

観点候補の例:

| 意図                         | 観点候補            |
| -------------------------- | --------------- |
| 必須項目を入力した場合に正常登録できることを確認する | 正常登録            |
| 既存コード入力時に重複エラーとなることを確認する   | コード重複入力制御、エラー表示 |
| 締め済み期間のデータを更新できないことを確認する   | 締め済み期間の更新制御     |
| 登録後に一覧に表示されることを確認する        | 登録後の一覧反映        |
| 権限なしユーザーで削除できないことを確認する     | 権限なしユーザーの削除抑止   |

観点候補は複数あってよい。
ただし、根拠が説明できるものに限る。

---

## 6.5 根拠を記録する

意図と観点候補には、必ず根拠を記録する。

根拠には以下を含める。

* 参照したテストケースID
* どの記述から判断したか
* 操作/入力、期待結果、テストケース名などの該当箇所

根拠の例:

```text
TC-002 の操作/入力に「既存の取引先コードを入力して登録」、期待結果に「重複エラーが表示される」とあるため。
```

悪い根拠の例:

```text
一般的にマスタ登録では重複チェックが必要なため。
```

これは元テストケースの根拠ではなく一般論なので不可。

---

## 6.6 推定度を判定する

意図と観点候補に対して推定度を判定する。

### 推定度: 高

以下のような場合。

* テストケース名に確認目的が明記されている
* 操作/入力と期待結果が明確に対応している
* 期待結果から確認対象が具体的に読み取れる

例:

```text
操作: 既存コードを入力して登録
期待結果: 重複エラーが表示される
→ 推定度: 高
```

### 推定度: 中

以下のような場合。

* 期待結果はあるが、確認対象がやや抽象的
* 前提条件や入力値の一部が省略されている
* テストケース名から意図は読めるが、期待結果の記述が不足している

例:

```text
期待結果: 正しく登録される
→ 何をもって正しいかは不明だが、正常登録の確認と推定できる
→ 推定度: 中
```

### 推定度: 低

以下のような場合。

* 期待結果が曖昧すぎる
* 操作/入力が不足している
* 前提条件が不明
* 複数の意図が混在し、主意図を特定しにくい
* 元資料の記述だけでは観点候補を判断できない

推定度が低い場合は、必ず issue に記録する。

---

## 6.7 レビュー要否を判定する

以下に該当する場合は、レビュー要否を `要` とする。

* 推定度が低い
* 推定度は中だが、業務上重要そうな判断を含む
* 主意図が複数考えられる
* 観点候補の粒度に迷いがある
* 期待結果が曖昧
* 前提条件が不足している
* 操作/入力と期待結果の対応が弱い
* `01_input_issues.md` で対象TCに issue が記録されている

レビュー要否が `要` の場合は、理由を記録する。

## 7.2 観点候補の書き方

観点候補が複数ある場合は、読点区切りで記録する。

例:

```text
コード重複入力制御、エラー表示
```

観点候補が多すぎる場合は、主意図に強く関係するものを優先する。

原則として、1テストケースあたりの観点候補は 1〜4 個程度に抑える。
ただし、元テストケースに複数確認が明確に含まれる場合はこの限りではない。


issue がない場合も、以下のように明記する。

```markdown

## 1. Issue一覧

```

## 8. issue として記録するもの

以下に該当する場合は、`02_extraction_issues.md` に記録する。

* 主意図が特定できない
* 主意図が複数あり、1つに絞れない
* 副次意図が多く、テストケース分割が必要に見える
* 観点候補の根拠が弱い
* 期待結果が曖昧で意図が不明確
* 操作/入力と期待結果が対応していない
* 前提条件が不足しているため意図が確定できない
* 既存分類とテスト内容が一致していない可能性がある
* 推定度が低い
* 観点候補が抽象的すぎる
* 観点候補が個別すぎて後続の抽象化に注意が必要

## 9. 判断例

## 9.1 正常登録の例

### 入力

```text
TC ID: TC-001
テストケース名: 取引先登録_正常
操作/入力: 必須項目を入力して登録する
期待結果: 登録完了メッセージが表示され、一覧に表示される
```

### 出力例

| TC ID  | 主意図                              | 副次意図                | 観点候補          | 根拠                                               | 推定度 | レビュー要否 |
| ------ | -------------------------------- | ------------------- | ------------- | ------------------------------------------------ | --- | ------ |
| TC-001 | 必須項目を入力した場合に、取引先を正常に登録できることを確認する | 登録後に一覧へ反映されることを確認する | 正常登録、登録後の一覧反映 | 操作/入力に「必須項目を入力して登録」、期待結果に「登録完了メッセージ」「一覧に表示」とあるため | 高   | 不要     |

---

## 9.2 重複エラーの例

### 入力

```text
TC ID: TC-002
テストケース名: 取引先コード重複
操作/入力: 既存の取引先コードを入力して登録する
期待結果: 重複エラーが表示され、登録されない
```

### 出力例

| TC ID  | 主意図                               | 副次意図               | 観点候補                | 根拠                                          | 推定度 | レビュー要否 |
| ------ | --------------------------------- | ------------------ | ------------------- | ------------------------------------------- | --- | ------ |
| TC-002 | 既存の取引先コードを入力した場合に、登録が抑止されることを確認する | 重複エラーが表示されることを確認する | 取引先コードの重複入力制御、エラー表示 | 操作/入力に「既存の取引先コード」、期待結果に「重複エラー」「登録されない」とあるため | 高   | 不要     |

---

## 9.3 曖昧な期待結果の例

### 入力

```text
TC ID: TC-008
テストケース名: 登録確認
操作/入力: データを登録する
期待結果: 正しく表示される
```

### 出力例

| TC ID  | 主意図                  | 副次意図 | 観点候補     | 根拠                        | 推定度 | レビュー要否 | レビュー理由                 |
| ------ | -------------------- | ---- | -------- | ------------------------- | --- | ------ | ---------------------- |
| TC-008 | 登録したデータが表示されることを確認する | なし   | 登録後の表示確認 | 操作/入力に「登録」、期待結果に「表示」とあるため | 中   | 要      | 「正しく表示される」の具体的な確認項目が不明 |

issue にも記録する。

```markdown
| Issue ID | 種別 | 対象TC | 内容 | 影響 | 推奨対応 |
|---|---|---|---|---|---|
| ISS-EX-001 | 期待結果曖昧 | TC-008 | 期待結果が「正しく表示される」のみで、表示項目・値・状態が不明 | 意図と観点候補の推定度が下がる | 期待する表示項目、値、状態を明記する |
```

---

## 9.4 複数意図が混在する例

### 入力

```text
TC ID: TC-015
テストケース名: 登録後確認
操作/入力: データを登録する
期待結果: 一覧に表示され、詳細画面にも反映され、帳票にも出力される
```

### 出力例

| TC ID  | 主意図                     | 副次意図                   | 観点候補                       | 根拠                            | 推定度 | レビュー要否 | レビュー理由               |
| ------ | ----------------------- | ---------------------- | -------------------------- | ----------------------------- | --- | ------ | -------------------- |
| TC-015 | 登録したデータが一覧に反映されることを確認する | 詳細画面への反映、帳票出力への反映を確認する | 登録後の一覧反映、登録後の詳細反映、登録後の帳票反映 | 期待結果に一覧、詳細画面、帳票への反映が記載されているため | 高   | 要      | 1テストケースに複数の確認対象が含まれる |

---

## 10. 品質チェック

出力前に、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| 各TC IDが維持されているか |  |
| 各テストケースに主意図があるか |  |
| 主意図は元テストケースから説明できるか |  |
| 副次意図を無理に作っていないか |  |
| 観点候補が抽象化されすぎていないか |  |
| 観点候補が根拠のある内容に限定されているか |  |
| 根拠にTC IDが含まれているか |  |
| 推定度が付与されているか |  |
| 推定度が低いものが issue 化されているか |  |
| 入力にない仕様や業務ルールを追加していないか |  |
| 推定を重ねていないか |  |
```

この品質チェック表は、必要に応じて成果物末尾に含めてもよい。
ただし、成果物が過度に肥大化する場合は省略してよい。

---

## 11. このステップでしてはいけないこと

このステップでは、以下を行ってはならない。

* 抽象観点カタログを作成する
* 観点IDを付与する
* 観点候補を過度に一般化する
* 入力にない仕様を確定事項として書く
* 一般的なテスト観点一覧から観点を補う
* 根拠のない観点候補を追加する
* 推定度を付けずに断定する
* 推定を重ねる
* 曖昧な期待結果を勝手に具体化する
* テストケースを勝手に分割・統合する
* 不足観点を自由に提案する
* 次の Step 03 に進む

---

## 12. 完了条件

このステップは、以下を満たしたとき完了とする。

* `02_intent_and_viewpoint_candidates.md` が作成されている
* `02_extraction_issues.md` が作成されている
* 各テストケースに主意図が記録されている
* 必要に応じて副次意図が記録されている
* 各テストケースに観点候補が記録されている
* 各意図・観点候補に根拠が記録されている
* 各意図・観点候補に推定度が付与されている
* レビューが必要な内容が明示されている
* 推定度が低い内容が issue として記録されている

完了後は、ユーザーのレビューを待つ。
ユーザーから指示があるまで、Step 03 には進まない。
