# traceability_matrix_template.md

## 1. 目的

このテンプレートは、Step 04「トレーサビリティ確認・最終化」で作成する `04_final_traceability_matrix.md` の出力形式を定義する。

目的は以下である。

- テストケース、テスト意図、観点候補、抽象観点の対応関係を明確にする
- 各抽象観点がどのテストケースに基づいているか追跡できるようにする
- 根拠のない観点や未対応の観点候補を明示する
- 推定度やレビュー要否を最終成果物上で確認できるようにする
- 後続のテスト設計、レビュー、棚卸し、改善活動に利用できる状態にする

このテンプレートは、新しい観点を作るためのものではない。  
既存成果物の対応関係を確認し、最終的に利用可能な形へ整理するために使用する。

---


複数バッチを統合した場合:


## 5. 各項目の説明

## 5.1 作成概要

### 対象テストケース数

`01_testcase_inventory.md` に含まれる対象テストケース数を記録する。

### 対応済みTC数

以下をすべて満たすTC数を記録する。

* 主意図が存在する
* 観点候補が存在する
* 観点候補が抽象観点に紐づいている
* 根拠が記録されている

### 要レビューTC数

レビュー要否が `要` のTC数を記録する。

レビュー要の理由には、以下が含まれる。

* 推定度が中または低
* 期待結果が曖昧
* 前提条件が不足
* 複数意図が混在
* 観点候補が抽象観点に未対応
* 抽象観点の分類や粒度に迷いがある

### 未対応TC数

意図、観点候補、抽象観点のいずれかが未対応のTC数を記録する。

### 抽象観点数

`03_viewpoint_catalog.md` に含まれる抽象観点数を記録する。
ただし、Step 04 で根拠TCなしなどの理由により除外した観点は含めない。

### 未対応観点候補数

Step 02 の観点候補のうち、Step 03 の抽象観点に紐づいていないものの数を記録する。

### 除外観点数

Step 04 で最終成果物から除外した観点数を記録する。

除外理由は `5. 除外観点一覧` に記録する。

---

## 5.2 トレーサビリティ表

### TC ID

`01_testcase_inventory.md` の TC ID をそのまま記録する。

仮IDの場合も維持する。

例:

```text
TC-001
TC-TEMP-001
```

### テストケース名

`01_testcase_inventory.md` のテストケース名を記録する。

### 対象機能

`01_testcase_inventory.md` の対象機能を記録する。

### 主意図

`02_intent_and_viewpoint_candidates.md` の主意図を記録する。

主意図が存在しない場合は、以下のように記録する。

```text
未対応
```

または、

```text
特定困難
```

### 副次意図

`02_intent_and_viewpoint_candidates.md` の副次意図を記録する。

副次意図がない場合は、以下のように記録する。

```text
なし
```

### 観点候補

`02_intent_and_viewpoint_candidates.md` の観点候補を記録する。

観点候補が存在しない場合は、以下のように記録する。

```text
未対応
```

または、

```text
特定困難
```

### 観点ID

観点候補に対応する `03_viewpoint_catalog.md` の観点IDを記録する。

例:

```text
VP-INPUT-001
VP-DATA-001
VP-OUTPUT-001
```

対応する抽象観点がない場合は、以下のように記録する。

```text
未対応
```

確定できない場合は、以下のように記録する。

```text
未確定
```

### 抽象観点

観点IDに対応する抽象観点名を記録する。

例:

```text
一意制約項目の重複入力制御
登録・更新後の関連画面への反映確認
```

対応する抽象観点がない場合は、以下のように記録する。

```text
未対応
```

### カテゴリ

抽象観点のカテゴリを記録する。

例:

```text
入力制約
データ整合性
帳票・出力
```

未対応の場合は空欄または `未対応` とする。

### 根拠

意図・観点候補・抽象観点の対応を説明できる根拠を記録する。

根拠には、TC ID と元テストケースの記述を含める。

例:

```text
TC-002 の操作/入力に「既存の取引先コードを入力」、期待結果に「重複エラーが表示され、登録されない」とあるため。
```

### 推定度

Step 02 の推定度を記録する。

使用する値:

```text
高
中
低
```

Step 03 で抽象観点のレビュー要否がある場合は、Step 02 の推定度と合わせて判断する。

### 対応状態

以下のいずれかを記録する。

| 対応状態    | 意味                     |
| ------- | ---------------------- |
| 対応済み    | TC、意図、観点候補、抽象観点が対応している |
| 要レビュー   | 対応はあるが、推定や曖昧さが残る       |
| 意図未対応   | 主意図が抽出されていない           |
| 観点候補未対応 | 観点候補が抽出されていない          |
| 抽象観点未対応 | 観点候補に対応する抽象観点がない       |
| 未確定     | 対応候補はあるが確定できない         |
| 対象外     | Step 01 で対象外と判断された     |

### レビュー要否

使用する値:

```text
不要
要
```

### レビュー理由

レビュー要否が `要` の場合、その理由を記録する。

レビュー不要の場合は `-` とする。

## 5.3 観点別トレーサビリティ

観点IDごとに、どのTCを根拠としているかを整理する。

### 観点ID

`03_viewpoint_catalog.md` の観点IDを記録する。

### 抽象観点

観点IDに対応する抽象観点名を記録する。

### カテゴリ

抽象観点のカテゴリを記録する。

### 根拠TC

その抽象観点の根拠TCを記録する。

例:

```text
TC-002, TC-014, TC-021
```

### 根拠TC数

根拠TCの件数を記録する。

### 元観点候補

抽象観点に統合された元観点候補を記録する。

### レビュー要否 / レビュー理由

`03_viewpoint_catalog.md` のレビュー要否・レビュー理由を記録する。
必要に応じて Step 04 の判断を加えてよい。

---

## 5.4 未対応一覧

未対応は隠さず記録する。

### 意図未対応TC

主意図が抽出されていないTCを記録する。

代表的な理由:

* 期待結果がない
* 操作/入力がない
* 操作と期待結果が矛盾している
* 主意図が複数あり特定できない

### 観点候補未対応TC

主意図はあるが、観点候補が抽出されていないTCを記録する。

代表的な理由:

* 意図が曖昧
* 確認対象が不明
* 期待結果が抽象的すぎる
* 観点候補を根拠づけられない

### 抽象観点未対応の観点候補

Step 02 の観点候補が Step 03 の抽象観点に紐づかなかった場合に記録する。

代表的な理由:

* Step 03 で統合漏れがある
* 抽象化判断が保留されている
* 推定度が低いためカタログ化しなかった
* 根拠が弱く確定できない

---

## 5.5 除外観点一覧

Step 04 で最終成果物から除外した観点を記録する。

除外対象の例:

* 根拠TCがない抽象観点
* Step 02 に存在しない観点候補から作られた観点
* 一般論で追加された観点
* 推定度が低く、根拠が弱い観点
* カテゴリ名だけで具体性がない観点
* 抽象化しすぎて意味が薄い観点

除外した場合は、必ず理由を記録する。

---

## 5.6 レビュー推奨一覧

レビューが必要なTCや観点を、優先度つきで整理する。

### 優先度

以下を使用する。

| 優先度 | 意味                    |
| --- | --------------------- |
| 高   | 最終成果物の正確性に大きく影響する     |
| 中   | 確認したほうがよいが、成果物全体は利用可能 |
| 低   | 記録・改善候補として扱う          |

### 種別

例:

```text
期待結果曖昧
前提条件不足
複数意図混在
抽象観点未対応
カテゴリ分類保留
根拠TC不足
推定度低
```

## 7. 作成時のルール

## 7.1 すべての対象TCを含める

`01_testcase_inventory.md` に含まれる対象TCは、原則としてすべて `トレーサビリティ表` に含める。

意図や観点が抽出できていない場合でも、行自体を消さない。
その場合は `未対応` として記録する。

---

## 7.2 対応関係を省略しない

以下の対応を省略しない。

```text
TC ID
主意図
観点候補
観点ID
抽象観点
```

どれかが未対応の場合は、空欄にせず `未対応` と記録する。

---

## 7.3 根拠TCのない観点を含めない

根拠TCが存在しない抽象観点は、最終トレーサビリティ表に含めない。

除外した場合は、`除外観点一覧` に記録する。

---

## 7.4 推定度が低いものは確定扱いしない

推定度が `低` のものは、以下のいずれかで扱う。

* `対応状態` を `未確定` とする
* `レビュー要否` を `要` とする
* `未対応一覧` に記録する
* issue log 側に記録する

---

## 7.5 複数観点に対応するTCの扱い

1つのTCが複数の抽象観点に対応する場合、以下のいずれかで記録する。

### 方式A: 1行に複数観点IDを記録する

```text
VP-DATA-001, VP-OUTPUT-001
```

### 方式B: TCを複数行に分ける

同じTC IDを複数行にし、観点IDごとに行を分ける。

大量の観点が紐づく場合は、方式Bのほうが読みやすい。

ただし、どちらの方式を使ったかを `作成時の補足` に記録する。

---

## 7.6 未対応を隠さない

未対応は成果物の価値を下げるものではない。
既存テストケースの改善余地を示す情報として扱う。

以下は必ず記録する。

* 意図未対応
* 観点候補未対応
* 抽象観点未対応
* 根拠TCなし
* 推定度低
* レビュー要

```markdown
| 確認項目 | 判定 |
|---|---|
| すべての対象TCがトレーサビリティ表に含まれているか |  |
| 各TCに主意図または未対応理由があるか |  |
| 各TCに観点候補または未対応理由があるか |  |
| 観点候補と抽象観点の対応が記録されているか |  |
| 未対応の観点候補が明示されているか |  |
| 各抽象観点に根拠TCがあるか |  |
| 根拠TCのない観点を除外しているか |  |
| 除外観点の理由が記録されているか |  |
| 推定度が低い内容を確定扱いしていないか |  |
| レビュー要否と理由が記録されているか |  |
| 新しい観点をStep 04で追加していないか |  |
| issue log と矛盾していないか |  |
```

---

## 9. 禁止事項

このテンプレートを使用する際、以下を行ってはならない。

* Step 04 で新しい抽象観点を追加する
* Step 02 にない観点候補を作る
* Step 03 にない観点IDを勝手に作る
* 根拠TCのない観点を含める
* 未対応TCを表から消す
* 推定度が低い内容を確定扱いする
* レビュー要の理由を省略する
* issue を隠す
* トレーサビリティが切れている内容を対応済みに見せる
