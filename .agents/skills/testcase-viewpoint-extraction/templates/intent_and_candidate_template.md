# intent_and_candidate_template.md

## 1. 目的

このテンプレートは、Step 02「意図・観点候補の抽出」で作成する `02_intent_and_viewpoint_candidates.md` の出力形式を定義する。

目的は以下である。

- 各テストケースの主意図を明確にする
- 必要に応じて副次意図を記録する
- テストケースから読み取れる観点候補を整理する
- 意図・観点候補の根拠を残す
- 推定度とレビュー要否を明示する
- 後続の抽象観点カタログ化で利用しやすい形にする

このテンプレートでは、抽象観点や観点IDは扱わない。  
それらは Step 03 で作成する。

---

## 2. 出力ファイル名

通常時:

```text

````

バッチ分割時:

```text
02_intent_and_viewpoint_candidates__{batch_name}.md
```

例:

```text
02_intent_and_viewpoint_candidates__batch_01_customer_master.md
```

## 4. 各項目の説明

## 4.1 抽出概要

### 入力ファイル

このステップで使用した入力ファイルを記録する。

通常は以下を記録する。

```text
01_testcase_inventory.md
01_input_issues.md
```

---

### 対象テストケース数

Step 02 の対象としたテストケース数を記録する。

---

### 意図抽出件数

主意図を抽出できたテストケース数を記録する。

通常は対象テストケース数と一致する。
ただし、期待結果不足や記述不備により主意図を確定できない場合は一致しないことがある。

---

### 観点候補抽出件数

抽出した観点候補の延べ件数を記録する。

1テストケースから複数の観点候補を抽出した場合は、その数を含める。

---

### レビュー要TC数

レビュー要否が `要` となったテストケース数を記録する。

---

### 推定度別件数

推定度 `高` / `中` / `低` の件数を記録する。

件数の基準は、原則としてテストケース単位とする。
1つのテストケース内で観点候補ごとに推定度が異なる場合は、最も低い推定度を代表値として扱う。

---

### バッチ名

バッチ分割している場合に記録する。

バッチ分割していない場合は `なし` とする。

---

## 4.2 意図・観点候補一覧

### TC ID

`01_testcase_inventory.md` の TC ID をそのまま記録する。

仮IDの場合も、そのまま維持する。

例:

```text
TC-001
TC-TEMP-001
```

---

### テストケース名

`01_testcase_inventory.md` のテストケース名を記録する。

空欄の場合は空欄のままでよい。
このステップで勝手に名前を作らない。

---

### 対象機能

`01_testcase_inventory.md` の対象機能を記録する。

---

### 主意図

そのテストケースが最も中心的に確認している内容を1つ記録する。

推奨表現:

```text
〜の場合に、〜となることを確認する。
〜操作により、〜が行われることを確認する。
〜条件で、〜が制御されることを確認する。
```

例:

```text
既存の取引先コードを入力した場合に、登録が抑止されることを確認する。
```

主意図が判断できない場合は、無理に補完せず以下のように記録する。

```text
特定困難
```

その場合は、レビュー要否を `要` とし、issue にも記録する。

---

### 副次意図

主意図に付随して確認している内容を記録する。

例:

```text
登録後に一覧へ反映されることを確認する。
```

副次意図がない場合は、以下のように記録する。

```text
なし
```

複数ある場合は、読点区切りまたは `<br>` で記録する。

例:

```text
詳細画面への反映を確認する。<br>帳票出力への反映を確認する。
```

---

### 観点候補

テストケースから読み取れる確認の切り口を記録する。

この段階では、元テストケースに近い粒度でよい。

例:

```text
取引先コードの重複入力制御
登録後の一覧反映
締め済み期間の更新制御
権限なしユーザーによる削除抑止
```

複数ある場合は、読点区切りで記録する。

例:

```text
取引先コードの重複入力制御、エラー表示
```

観点候補を抽出できない場合は、以下のように記録する。

```text
特定困難
```

---

### 根拠

主意図と観点候補を抽出した根拠を記録する。

根拠には、必ず TC ID と元テストケース上の記述を含める。

良い例:

```text
TC-002 の操作/入力に「既存の取引先コードを入力して登録」、期待結果に「重複エラーが表示され、登録されない」とあるため。
```

悪い例:

```text
一般的にマスタ登録では重複チェックが必要なため。
```

一般論を根拠にしてはならない。

---

### 推定度

意図・観点候補が元テストケースからどの程度確からしく読み取れるかを記録する。

使用する値:

```text
高
中
低
```

判断基準:

| 推定度 | 判断基準                 |
| --- | -------------------- |
| 高   | 操作/入力と期待結果から明確に読み取れる |
| 中   | 妥当に推定できるが、詳細が不足している  |
| 低   | 可能性はあるが、根拠が弱い        |

---

### レビュー要否

ユーザーまたは担当者による確認が必要かを記録する。

使用する値:

```text
不要
要
```

レビュー要否を `要` とする代表例:

* 推定度が低い
* 推定度が中で、確認対象が曖昧
* 期待結果が曖昧
* 前提条件が不足している
* 操作/入力が不足している
* 主意図が複数考えられる
* 観点候補の粒度に迷いがある
* 1テストケースに複数意図が混在している

---

### レビュー理由

レビュー要否が `要` の場合、その理由を記録する。

例:

```text
期待結果が「正しく表示される」のみで、具体的な表示項目が不明なため。
```

レビュー要否が `不要` の場合は、以下のように記録する。

```text
-
```

---

## 5. 抽出時の補足

`抽出時の補足` には、意図・観点候補一覧に収まりにくい説明を記録する。

例:

```markdown
| No. | 対象TC | 補足 |
|---:|---|---|
| 1 | TC-015 | 一覧、詳細、帳票への反映が1ケースに含まれているため、副次意図を複数記録した。 |
```

補足には、一般論や新しい観点を追加しない。
あくまで抽出時の判断理由を簡潔に記録する。

---

## 6. 後続ステップへの注意

`後続ステップへの注意` には、Step 03 で抽象化・統合する際に注意すべき内容を記録する。

例:

```markdown
| No. | 対象TC | 注意事項 |
|---:|---|---|
| 1 | TC-002 | 「取引先コードの重複入力制御」は、他マスタのコード重複と統合できる可能性がある。 |
| 2 | TC-015 | 一覧反映と帳票反映は確認対象が異なるため、抽象化時に無理に統合しない。 |
| 3 | TC-033 | 推定度が低いため、Step 03 で確定的な抽象観点へ含める場合はレビュー要。 |
```

## 8. 書き方のルール

## 8.1 主意図は1つにする

主意図は原則として1つにする。

複数の主意図があるように見える場合は、以下のいずれかで扱う。

* 最も中心的なものを主意図にし、他を副次意図にする
* 主意図の特定が難しい場合は `特定困難` とする
* issue に記録する

---

## 8.2 副次意図を増やしすぎない

副次意図は、元テストケースに明確に含まれる場合のみ記録する。

AIエージェントが「ついでに確認していそう」と思った内容を追加してはならない。

---

## 8.3 観点候補は具体性を残す

観点候補は、抽象化しすぎない。

悪い例:

```text
入力チェック
業務ルール
画面確認
正常系
異常系
```

良い例:

```text
取引先コードの重複入力制御
必須項目未入力時の登録抑止
登録後の一覧反映
締め済み期間の更新制御
```

---

## 8.4 根拠は必ず書く

根拠には、以下を含める。

* TC ID
* 操作/入力の該当記述
* 期待結果の該当記述
* 必要に応じてテストケース名や前提条件

---

## 8.5 推定度を省略しない

すべての行に推定度を記録する。

推定度が低い場合は、`02_extraction_issues.md` にも記録する。

---

## 9. 品質チェック

作成後、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| すべての対象TCが含まれているか |  |
| 各TCに主意図が記録されているか |  |
| 主意図が文章として書かれているか |  |
| 副次意図を無理に作っていないか |  |
| 観点候補が抽象化されすぎていないか |  |
| 観点候補が根拠のある内容に限定されているか |  |
| 根拠にTC IDが含まれているか |  |
| 推定度がすべての行に記録されているか |  |
| レビュー要否がすべての行に記録されているか |  |
| 推定度が低い内容が issue 化されているか |  |
| 入力にない仕様を追加していないか |  |
| 次ステップの抽象観点や観点IDを先取りしていないか |  |
```

---

## 10. 禁止事項

このテンプレートを使用する際、以下を行ってはならない。

* 抽象観点を作成する
* 観点IDを付与する
* 一般的な観点を追加する
* 入力にない仕様を補完する
* 曖昧な期待結果を勝手に具体化する
* 推定度を省略する
* 根拠TCを省略する
* 副次意図を想像で追加する
* 観点候補をカテゴリ名だけにする
