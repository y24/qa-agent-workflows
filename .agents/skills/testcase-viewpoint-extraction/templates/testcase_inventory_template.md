# testcase_inventory_template.md

## 1. 目的

このテンプレートは、Step 01「入力整理・テストケース正規化」で作成する `01_testcase_inventory.md` の出力形式を定義する。

目的は以下である。

- 入力テストケースを標準形式に整理する
- 後続の意図抽出・観点抽出で扱いやすくする
- テストケースID、操作、期待結果、前提条件などの情報を保持する
- 入力資料の情報を勝手に補完せず、正規化した状態で記録する

---

## 2. 出力ファイル名

通常時:

```text
01_testcase_inventory.md
````

バッチ分割時:

```text
01_testcase_inventory__{batch_name}.md
```

例:

```text
01_testcase_inventory__batch_01_customer_master.md
```

## 4. 各項目の説明

## 4.1 入力概要

### 対象ファイル

入力として扱ったファイル名を記録する。

例:

```text
test_cases_customer_master.xlsx
```

複数ファイルの場合は、読点区切りまたは箇条書きで記録する。

---

### 対象シート/範囲

Excelやスプレッドシートの場合、対象シートや行範囲を記録する。

例:

```text
取引先マスタ, 2行目〜120行目
```

MarkdownやCSVの場合は、対象セクションや対象範囲を記録する。

---

### 対象機能

対象となる機能、画面、業務、モジュールを記録する。

例:

```text
取引先マスタ
仕訳入力
帳票出力
ユーザー管理
```

入力資料から判断できない場合は空欄または `不明` とする。
推測で補ってはならない。

---

### 対象テスト種別

元資料にテスト種別がある場合のみ記録する。

例:

```text
機能テスト
回帰テスト
シナリオテスト
受入テスト
```

不明な場合は空欄でよい。

---

### 対象テストケース数

正規化後に一覧化したテストケース数を記録する。

---

### 除外範囲

解析対象から除外した範囲がある場合に記録する。

例:

```text
集計行、見出し行、実行結果のみの行
```

除外がない場合は `なし` とする。

---

### 除外理由

除外した理由を簡潔に記録する。

例:

```text
テストケースではなく、セクション見出しであるため。
```

---

### バッチ名

バッチ分割している場合に記録する。

例:

```text
batch_01_customer_master
```

バッチ分割していない場合は `なし` とする。

---

### 備考

入力資料全体に関する補足を記録する。

---

## 4.2 列・項目マッピング

元資料の列や項目が、標準項目のどれに対応するかを記録する。

### 標準項目

| 標準項目    | 内容                   |
| ------- | -------------------- |
| TC ID   | テストケースID             |
| テストケース名 | テストケースの名称または概要       |
| 対象機能    | 対象画面、機能、業務、モジュール     |
| 前提条件    | 実行前に満たすべき条件や状態       |
| 操作/入力   | 操作手順、入力値、テストデータ      |
| 期待結果    | 確認すべき結果              |
| 既存分類    | 元資料に記載された分類、観点、テスト種別 |
| 備考      | 補足情報                 |
| 正規化メモ   | 整理時に記録した注意点          |

### 記録例

```markdown
| 元資料の列/項目 | 標準項目 | マッピング理由/補足 |
|---|---|---|
| No | TC ID | テストケース番号として扱う |
| 確認項目 | テストケース名 | テスト内容の概要として扱う |
| 操作手順 | 操作/入力 | 手順と入力値が同一列に記載されている |
| 確認結果 | 期待結果 | 期待される結果として扱う |
```

---

## 4.3 テストケース一覧

正規化したテストケースを1行1ケースで記録する。

### TC ID

元資料にIDがある場合は、そのIDを維持する。

元資料にIDがない場合は、仮IDを付与する。

仮ID形式:

```text
TC-TEMP-001
TC-TEMP-002
TC-TEMP-003
```

仮IDを付与した場合は、`01_input_issues.md` に記録する。

---

### テストケース名

元資料のテストケース名、確認項目名、概要などを記録する。

ない場合は空欄にする。
勝手に名前を作らない。

---

### 対象機能

対象機能、画面、業務、モジュールを記録する。

見出し行やシート名から補う場合は、`正規化メモ` にその旨を記録する。

例:

```text
見出し「取引先マスタ」配下のため、対象機能として記録。
```

---

### 前提条件

テスト実行前に必要な条件を記録する。

例:

```text
ログイン済み
対象データが登録済み
締め済み期間のデータが存在する
管理者ユーザーでログイン
```

入力資料にない場合は空欄にする。
「通常はログイン済みのはず」などの補完は禁止。

---

### 操作/入力

操作手順、入力値、テストデータを記録する。

例:

```text
取引先コードに既存コードを入力し、登録ボタンを押下する。
```

複数手順がある場合は、同一セル内で改行または `<br>` を使ってよい。

例:

```text
1. 取引先マスタ画面を開く<br>2. 既存コードを入力する<br>3. 登録ボタンを押下する
```

---

### 期待結果

期待される結果を記録する。

例:

```text
重複エラーが表示され、登録されない。
```

期待結果が曖昧な場合でも、勝手に具体化しない。

例:

```text
正しく表示される
```

このまま記録し、issue に残す。

---

### 既存分類

元資料に分類、観点、テスト種別などがある場合に記録する。

例:

```text
正常系
異常系
入力チェック
権限
帳票
回帰
```

既存分類がない場合は空欄にする。

---

### 備考

元資料の補足、注意、関連チケット、関連仕様などを記録する。

---

### 正規化メモ

正規化時にAIエージェントが判断した内容や、後続ステップへの注意を記録する。

例:

```text
対象機能はシート名から補った
期待結果が複数含まれる
手順と期待結果が同一セルに混在
テストケースIDがないため仮IDを付与
```

正規化メモには、テスト意図や観点候補を書かない。

---

## 5. 正規化メモの例

```markdown
| No. | 内容 |
|---:|---|
| 1 | 対象機能はシート名「取引先マスタ」から補った。 |
| 2 | 一部のテストケースで、操作手順と期待結果が同一セルに混在している。 |
| 3 | TC ID が存在しない行には仮IDを付与した。 |
| 4 | 「正しく表示される」など、期待結果が曖昧な記述が複数存在する。 |
```

---

```markdown

|---:|---|---|

```

## 8. 作成時の注意

## 8.1 補完しない

入力資料にない情報を補完してはならない。

禁止例:

```text
前提条件が空欄だが、おそらくログイン済みなので「ログイン済み」と記録する。
```

許可される例:

```text
前提条件は空欄のままにし、必要に応じて issue に記録する。
```

---

## 8.2 意図を書かない

このテンプレートには、テスト意図を書かない。

禁止例:

```text
このテストは重複チェックを確認するもの。
```

これは Step 02 で扱う。

---

## 8.3 観点候補を書かない

このテンプレートには、観点候補を書かない。

禁止例:

```text
観点: 一意性制約
```

これは Step 02 以降で扱う。

---

## 8.4 曖昧さを消さない

曖昧な期待結果や不足している前提条件は、きれいに直さず、そのまま残す。

必要に応じて `01_input_issues.md` に記録する。

---

## 9. 品質チェック

作成後、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| 入力概要が記録されているか |  |
| 対象範囲が明記されているか |  |
| 各テストケースにTC IDまたは仮IDがあるか |  |
| 仮IDを付与した場合、issue に記録されているか |  |
| 操作/入力と期待結果が分離されているか |  |
| 入力にない前提条件を補完していないか |  |
| テスト意図を先取りしていないか |  |
| 観点候補を先取りしていないか |  |
| 曖昧な記述が正規化メモまたは issue に残されているか |  |
```
