# viewpoint_catalog_template.md

## 1. 目的

このテンプレートは、Step 03「抽象観点カタログ化」で作成する `03_viewpoint_catalog.md` の出力形式を定義する。

目的は以下である。

- Step 02 で抽出した観点候補を、再利用可能な抽象観点として整理する
- 類似する観点候補を統合する
- 抽象観点に観点IDとカテゴリを付与する
- 各抽象観点の根拠TCを保持する
- 抽象化レベルと抽象化理由を記録する
- レビューが必要な観点を明示する

このテンプレートは、一般的なテスト観点一覧を作るためのものではない。  
必ず `02_intent_and_viewpoint_candidates.md` の観点候補を起点にする。

---

## 2. 出力ファイル名

通常時:

```text
03_viewpoint_catalog.md
````

バッチ分割時:

```text
03_viewpoint_catalog__{batch_name}.md
```

例:

```text
03_viewpoint_catalog__batch_01_customer_master.md
```

複数バッチを統合した場合:

```text
03_viewpoint_catalog__merged.md
```

## 4. 各項目の説明

## 4.1 作成概要

### 入力ファイル

このステップで使用した入力ファイルを記録する。

通常は以下を記録する。

```text
02_intent_and_viewpoint_candidates.md
02_extraction_issues.md
```

---

### 対象観点候補数

Step 02 から収集した観点候補の延べ件数を記録する。

例:

```text
18
```

---

### 抽象観点数

統合・抽象化後に作成した抽象観点の件数を記録する。

例:

```text
9
```

---

### カテゴリ数

抽象観点に付与したカテゴリの種類数を記録する。

---

### レビュー要観点数

レビュー要否が `要` となった抽象観点数を記録する。

---

### 抽象化レベル別件数

L1 / L2 / L3 の件数を記録する。

---

### バッチ名

バッチ分割している場合に記録する。

バッチ分割していない場合は `なし` とする。

---

## 4.2 テスト観点カタログ

### 観点ID

抽象観点ごとに一意のIDを付与する。

形式:

```text
VP-{CATEGORY}-{連番}
```

例:

```text
VP-INPUT-001
VP-STATE-001
VP-AUTH-001
VP-DATA-001
VP-OUTPUT-001
```

カテゴリに迷う場合:

```text
VP-UNCLEAR-001
```

その場合は、レビュー要否を `要` とし、理由を記録する。

---

### カテゴリ

抽象観点の分類を記録する。

カテゴリ例:

```text
入力制約
業務ルール
状態制御
権限
データ整合性
表示
検索・抽出
登録・更新・削除
帳票・出力
外部連携
エラー・警告
排他・同時実行
監査ログ
性能・応答
回帰確認
分類保留
その他
```

カテゴリは `references/viewpoint_category_seed.md` を参考にする。

---

### 抽象観点

統合・抽象化した観点名を記録する。

良い例:

```text
一意制約項目の重複入力制御
必須項目未入力時の登録抑止
確定済み状態のデータに対する変更操作の抑止
権限不足ユーザーによる保護対象操作の抑止
登録・更新後の関連画面への反映確認
出力条件に応じた帳票内容の反映確認
```

悪い例:

```text
入力チェック
権限確認
状態確認
画面確認
帳票確認
正常系
異常系
```

---

### 説明

抽象観点の意味を1〜2文で説明する。

説明には、以下を含めるとよい。

* どのような条件で
* 何を操作または確認し
* どのような結果を期待する観点か

例:

```text
一意であるべき項目に重複値を入力した場合、登録・更新が抑止されることを確認する。
```

---

### 元観点候補

Step 02 で抽出された観点候補を記録する。

複数ある場合は読点区切りまたは `<br>` で記録する。

例:

```text
取引先コードの重複入力制御、ユーザーIDの重複入力制御、部門コードの重複登録チェック
```

元観点候補は、抽象観点の根拠をたどるために重要である。
省略しない。

---

### 根拠TC

抽象観点の根拠となる TC ID を記録する。

例:

```text
TC-002, TC-014, TC-021
```

根拠TCが存在しない抽象観点は、このカタログに含めてはならない。

---

### 抽象化レベル

以下のいずれかを記録する。

```text
L1
L2
L3
```

基準:

| レベル | 意味         | 例             |
| --- | ---------- | ------------- |
| L1  | 個別機能レベル    | 取引先コードの重複入力制御 |
| L2  | 機能横断レベル    | マスタコードの重複入力制御 |
| L3  | 汎用テスト設計レベル | 一意制約項目の重複入力制御 |

---

### 抽象化理由

なぜその抽象観点にしたのかを記録する。

含める内容:

* どの観点候補を統合したか
* どの個別表現を一般化したか
* なぜその抽象度にしたか
* どの根拠TCに基づくか

良い例:

```text
取引先コード、ユーザーID、部門コードはいずれも一意であるべき識別子として扱われ、重複値を入力した場合の登録抑止を確認しているため、一意制約項目の重複入力制御として統合した。
```

悪い例:

```text
似ているため統合した。
```

---

### レビュー要否

レビューが必要かどうかを記録する。

使用する値:

```text
不要
要
```

レビュー要否を `要` とする例:

* 抽象化レベルに迷いがある
* カテゴリ分類に迷いがある
* 根拠TCが1件のみ
* 元観点候補の推定度が中または低
* 複数観点を統合したが意味差が残っている
* 業務用語を一般化しすぎた可能性がある
* 抽象観点名が抽象的すぎる可能性がある

---

### レビュー理由

レビュー要否が `要` の場合、その理由を記録する。

レビュー要否が `不要` の場合は、以下のように記録する。

```text
-
```

---

## 5. カテゴリ別サマリ

カテゴリごとに、観点数と該当する観点IDを記録する。

例:

```markdown
| カテゴリ | 観点数 | 観点ID |
|---|---:|---|
| 入力制約 | 3 | VP-INPUT-001, VP-INPUT-002, VP-INPUT-003 |
| 状態制御 | 1 | VP-STATE-001 |
| 帳票・出力 | 2 | VP-OUTPUT-001, VP-OUTPUT-002 |
```

このサマリにより、観点の偏りを確認しやすくする。

---

## 6. 抽象化レベル別サマリ

抽象化レベルごとに、件数と主な観点IDを記録する。

例:

```markdown
| 抽象化レベル | 件数 | 主な観点ID |
|---|---:|---|
| L1 | 1 | VP-BIZ-003 |
| L2 | 4 | VP-DATA-001, VP-OUTPUT-001, VP-BIZ-001 |
| L3 | 5 | VP-INPUT-001, VP-INPUT-002, VP-AUTH-001 |
```

L1 が多すぎる場合は、抽象化が不足している可能性がある。
L3 が多すぎる場合は、業務文脈を消しすぎている可能性がある。

---

## 7. 統合・抽象化メモ

統合や抽象化に関する補足を記録する。

例:

```markdown
| No. | 対象観点ID | 内容 |
|---:|---|---|
| 1 | VP-INPUT-001 | 複数のコード重複チェックを、一意制約項目の重複入力制御として統合した。 |
| 2 | VP-DATA-001 | 一覧反映と詳細反映は関連画面への反映として統合したが、帳票反映は出力物への反映として分離した。 |
| 3 | VP-BIZ-001 | 「締め済み会計期間」は業務意味が強いため、L3 ではなく L2 として扱った。 |
```

---

## 8. 後続ステップへの注意

Step 04 でトレーサビリティ確認・最終化を行う際の注意事項を記録する。

例:

```markdown
| No. | 対象観点ID/TC | 注意事項 |
|---:|---|---|
| 1 | VP-UNCLEAR-001 | カテゴリ分類に迷いがあるため、Step 04 でレビュー要として扱う。 |
| 2 | VP-INPUT-003 | 根拠TCが1件のみのため、独立観点とする妥当性を確認する。 |
| 3 | TC-033 | 推定度が低い観点候補から作成されたため、確定扱いしない。 |
```

## 10. 作成時のルール

## 10.1 Step 02 の観点候補を起点にする

抽象観点は、必ず Step 02 の観点候補をもとに作成する。

禁止例:

```text
性能確認が一般的に必要なので VP-PERF-001 を追加する。
```

これは入力テストケースに根拠がなければ不可。

---

## 10.2 根拠TCを必ず記録する

すべての抽象観点に、1件以上の根拠TCを記録する。

根拠TCがない場合、その観点はカタログに含めない。

---

## 10.3 抽象化しすぎない

以下のような抽象観点名は避ける。

```text
入力チェック
表示確認
業務ルール確認
権限確認
データ確認
正常系確認
異常系確認
```

これらはカテゴリ名に近く、観点としては粗すぎる。

---

## 10.4 個別すぎる場合は一般化する

以下のような観点候補は、必要に応じて一般化する。

```text
取引先コードの重複入力制御
↓
一意制約項目の重複入力制御
```

ただし、業務ドメイン固有の意味が重要な場合は、個別表現を残してよい。

---

## 10.5 意味差のある観点を無理に統合しない

例:

```text
登録後の一覧反映
登録後の帳票反映
```

これらはどちらも「反映確認」だが、確認対象が画面と出力物で異なる。
無理に1つへ統合せず、必要に応じて別観点として扱う。

---

## 11. 品質チェック

作成後、以下を確認する。

```markdown
| 確認項目 | 判定 |
|---|---|
| すべての抽象観点が Step 02 の観点候補を起点にしているか |  |
| 各抽象観点に観点IDがあるか |  |
| 各抽象観点にカテゴリがあるか |  |
| 各抽象観点に1件以上の根拠TCがあるか |  |
| 観点名がカテゴリ名だけになっていないか |  |
| 観点名が抽象化されすぎていないか |  |
| 観点名が個別機能に依存しすぎていないか |  |
| 抽象化レベルが記録されているか |  |
| 抽象化理由が記録されているか |  |
| レビュー要否が記録されているか |  |
| 意味の異なる観点を無理に統合していないか |  |
| 根拠TCのない観点を含めていないか |  |
```

---

## 12. 禁止事項

このテンプレートを使用する際、以下を行ってはならない。

* Step 02 に存在しない観点を追加する
* 根拠TCのない観点を作る
* 一般論から観点を補完する
* 観点名をカテゴリ名だけにする
* 意味の異なる観点を無理に統合する
* 推定度が低い観点を確定扱いする
* 抽象化理由を省略する
* レビュー要否を省略する
