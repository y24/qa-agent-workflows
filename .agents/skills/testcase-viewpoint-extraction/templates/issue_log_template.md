# issue_log_template.md

## 1. 目的

このテンプレートは、各ステップで発見した issue を記録するための共通形式を定義する。

目的は以下である。

- 入力資料の曖昧さ、不足、矛盾を記録する
- 意図抽出や観点抽出の不確実性を記録する
- 抽象化やカテゴリ分類の判断保留を記録する
- トレーサビリティ上の未対応を記録する
- ユーザーレビューが必要な事項を明確にする
- 最終成果物で issue を統合しやすくする

issue は、作業失敗の記録ではない。  
既存テストケースを改善するための重要な観察結果として扱う。

---

## 2. 使用する出力ファイル

このテンプレートは、以下の issue ファイルで使用する。

```text
01_input_issues.md
02_extraction_issues.md
03_cataloging_issues.md
04_final_issue_log.md
````

バッチ分割時は、ファイル名にバッチ名を付与する。

例:

```text id="06zrnh"
01_input_issues__batch_01_customer_master.md
02_extraction_issues__batch_01_customer_master.md
03_cataloging_issues__batch_01_customer_master.md
04_final_issue_log__batch_01_customer_master.md
```

---

## 3. Issue ID ルール

issue ID は、発生ステップごとに接頭辞を分ける。

| ファイル                    | Issue ID 形式 |
| ----------------------- | ----------- |
| 01_input_issues.md      | ISS-IN-001  |
| 02_extraction_issues.md | ISS-EX-001  |
| 03_cataloging_issues.md | ISS-CAT-001 |
| 04_final_issue_log.md   | ISS-FIN-001 |

例:

```text id="dbrypc"
ISS-IN-001
ISS-EX-001
ISS-CAT-001
ISS-FIN-001
```

最終 issue log では、元 issue ID を内容欄または備考欄に残してよい。

例:

```text id="6n17wx"
元issue: ISS-IN-003, ISS-EX-002
```

## 5. 各項目の説明

## 5.1 Issue ID

issue を一意に識別するIDを記録する。

例:

```text id="xynwpu"
ISS-IN-001
ISS-EX-001
ISS-CAT-001
ISS-FIN-001
```

同じ issue を複数行に分けない。
必要な場合は、内容欄に複数の対象や詳細を記録する。

---

## 5.2 状態

issue の状態を記録する。

使用する値:

| 状態    | 意味                 |
| ----- | ------------------ |
| 未解決   | 対応が必要              |
| 要レビュー | ユーザーまたは担当者の判断が必要   |
| 情報不足  | 入力資料や補足情報が不足している   |
| 対応不要  | 記録のみでよい            |
| 解決済み  | 後続ステップまたは修正により解消済み |

### 状態の使い分け

#### 未解決

成果物の正確性や利用に影響があり、何らかの対応が必要な場合。

例:

```text id="9dvj6v"
期待結果が存在しないため、テスト意図を抽出できない。
```

#### 要レビュー

AIエージェントだけでは判断できず、ユーザー確認が必要な場合。

例:

```text id="qgefav"
締め済み会計期間を L2 観点として残すか、L3 の状態制御へ一般化するか判断が必要。
```

#### 情報不足

入力資料や補足情報が不足しており、正確な判断ができない場合。

例:

```text id="n5s1e7"
権限条件が記載されていないため、削除不可の理由が判断できない。
```

#### 対応不要

注意点として記録するが、成果物の利用を妨げない場合。

例:

```text id="q1v6ic"
TC ID の表記に軽微な揺れがあるが、対応関係は維持できている。
```

#### 解決済み

後続ステップまたはユーザー修正により解消した場合。

例:

```text id="d1wx0e"
Step 03 で観点候補が抽象観点に統合されたため解決済み。
```

---

## 5.3 種別

issue の種類を記録する。

代表的な種別は以下。

```text id="7qfrse"
TC IDなし
対象範囲不明
対象機能不明
列マッピング不明
前提条件不足
操作/入力不足
期待結果不足
期待結果曖昧
曖昧語
複数意図混在
操作結果不整合
既存分類不一致
業務用語不明
具体値の意味不明
推定度低
観点候補特定困難
観点根拠不足
抽象化判断保留
カテゴリ分類保留
根拠TC不足
根拠TCなし
観点未対応
トレーサビリティ未対応
除外観点
```

必要に応じて新しい種別を追加してよい。
ただし、種別を増やしすぎない。

---

## 5.4 対象TC/観点

issue の対象となる TC ID、観点候補、観点ID などを記録する。

例:

```text id="5k6hiy"
TC-008
TC-015
VP-INPUT-001
VP-UNCLEAR-001
TC-020 / 登録後の監査ログ出力
```

対象が複数ある場合は、読点区切りで記録する。

例:

```text id="4hl12c"
TC-008, TC-009, TC-010
```

対象が全体に関わる場合は、以下のように記録する。

```text id="0yaq6a"
全体
```

---

## 5.5 内容

issue の内容を具体的に記録する。

良い書き方:

```text id="uj9ise"
期待結果が「正しく表示される」のみで、確認すべき表示項目・値・状態が不明。
```

悪い書き方:

```text id="du2cyu"
期待結果が微妙。
```

内容欄には、何が不明なのか、何が問題なのかを明確に書く。

---

## 5.6 影響

その issue が抽出結果や最終成果物に与える影響を記録する。

例:

```text id="qwi9db"
テスト意図の推定度が下がる。
観点候補を確定できない。
抽象観点への統合判断が難しくなる。
トレーサビリティが未対応になる。
最終成果物でレビュー要として扱う必要がある。
```

影響が小さい場合も、その旨を記録する。

例:

```text id="god6z5"
成果物全体への影響は小さいが、元資料との対応確認時に注意が必要。
```

---

## 5.7 推奨対応

ユーザーまたは担当者が取るべき対応を記録する。

例:

```text id="sh34tg"
期待する表示項目、値、状態を明記する。
前提条件として対象ユーザー権限を追記する。
対象データが締め済みかどうかを明記する。
テストケースを一覧反映、詳細反映、帳票反映に分割する。
抽象観点として独立させるか、既存観点に統合するかレビューする。
```

推奨対応が不要な場合は、以下のように記録する。

```text id="k0l2y8"
記録のみ。追加対応は不要。
```

---

## 5.8 元Issue

最終 issue log で、元になった issue ID を記録する。

例:

```text id="i6j3dq"
ISS-IN-003
ISS-IN-003, ISS-EX-002
```

元 issue がない場合は `-` とする。

## 6.2 02_extraction_issues.md

Step 02 では、意図・観点候補抽出時点の issue を記録する。

### 対象となる issue

* 主意図が特定できない

* 副次意図が多い
* 観点候補が特定できない
* 観点候補の根拠が弱い

## 6.3 03_cataloging_issues.md

Step 03 では、観点カタログ化時点の issue を記録する。

* 観点候補の統合判断に迷う

## 7. issue 統合ルール

## 7.1 同じ原因の issue は統合する

同じ原因に由来する issue は、最終 issue log で統合してよい。

例:

```text id="9s68q4"
ISS-IN-002: TC-008 の期待結果が曖昧
ISS-EX-003: TC-008 の意図推定度が中
ISS-CAT-002: VP-VIEW-001 がレビュー要
```

これらは、最終的に以下のように統合できる。

```text id="5s9l8i"
TC-008 の期待結果が曖昧であり、意図抽出・観点抽象化の確実性に影響している。
```

元 issue ID は `元Issue` に残す。

---

## 7.2 解決済み issue も必要に応じて残す

後続ステップで解決した issue は、必要に応じて `解決済み` として記録する。

例:

```text id="vay8il"
Step 02 では観点候補が未対応だったが、Step 03 で抽象観点に統合された。
```

解決済み issue を残すことで、判断履歴を追跡できる。

---

## 7.3 issue を削除して見かけを整えない

最終成果物をきれいに見せるために issue を削除してはならない。

以下は避ける。

```text id="2n2ei4"
曖昧だが、たぶん問題ないので issue から消す。
```

issue は、レビューと改善のための資産として扱う。

---

## 8. 優先度の付け方

最終 issue log やレビュー推奨一覧では、必要に応じて優先度を付ける。

| 優先度 | 判断基準                           |
| --- | ------------------------------ |
| 高   | 抽出結果の正確性、トレーサビリティ、最終利用に大きく影響する |
| 中   | 成果物は利用可能だが、レビューしたほうがよい         |
| 低   | 改善候補、記録、将来的な見直し対象              |

### 高の例

```text id="jdqqga"
主意図が特定できない
観点候補が抽象観点に未対応
根拠TCがない観点が混入している
```

### 中の例

```text id="3s71p0"
期待結果がやや曖昧
根拠TCが1件のみ
抽象化レベルに迷いがある
```

### 低の例

```text id="i2r9l8"
TC ID の表記ゆれ
カテゴリ名の表記ゆれ
将来的なテストケース分割候補
```

## 9. よくある issue と推奨対応

| 種別       | 内容例                 | 推奨対応                      |
| -------- | ------------------- | ------------------------- |
| 期待結果曖昧   | 「正しく表示される」の具体内容が不明  | 表示項目、値、状態を明記する            |
| 前提条件不足   | 権限や対象データ状態が不明       | 前提条件にユーザー、データ状態、環境条件を追記する |
| 複数意図混在   | 一覧、詳細、帳票の確認が1ケースに混在 | 確認対象ごとにテストケースを分割する        |
| 推定度低     | エラー原因が不明            | 入力値、条件、期待エラー内容を明記する       |
| カテゴリ分類保留 | 業務ルールか状態制御か判断できない   | 業務上の意味をレビューする             |
| 根拠TC不足   | 根拠TCが1件のみ           | 独立観点とするか、既存観点へ統合するか確認する   |
| 根拠TCなし   | 抽象観点に根拠TCがない        | 最終成果物から除外する               |
| 観点未対応    | 観点候補が抽象観点に紐づかない     | Step 03 の観点カタログを見直す       |
| 既存分類不一致  | 既存分類と内容が合わない        | 既存分類を修正するか、分類の意味を確認する     |

---

## 10. 作成時の注意

## 10.1 issue は具体的に書く

悪い例:

```text id="uuc0if"
よく分からない。
```

良い例:

```text id="e8cvy8"
期待結果が「正常に処理される」のみで、登録完了、画面表示、DB反映のどれを確認するのか不明。
```

---

## 10.2 影響を書く

issue は、単に問題点を列挙するだけでなく、抽出結果への影響を書く。

例:

```text id="vt2lj1"
前提条件が不明なため、権限制御の観点として確定できない。
```

---

## 10.3 推奨対応を書く

レビュー時に何を確認すべきか分かるように、推奨対応を書く。

悪い例:

```text id="1sgu66"
確認する。
```

良い例:

```text id="a091ik"
対象ユーザーの権限、削除不可となる条件、期待するメッセージを明記する。
```

---

## 10.4 根拠の弱さを隠さない

推定度が低い、根拠TCが少ない、判断に迷う場合は、そのまま issue として記録する。

成果物をきれいに見せるために、根拠の弱さを隠してはならない。

---

## 11. 品質チェック

作成後、以下を確認する。

```markdown id="yezo1r"
| 確認項目 | 判定 |
|---|---|
| Issue ID が一意か |  |
| 状態が記録されているか |  |
| 種別が記録されているか |  |
| 対象TC/観点が記録されているか |  |
| 内容が具体的か |  |
| 影響が記録されているか |  |
| 推奨対応が記録されているか |  |
| 元Issueが必要に応じて記録されているか |  |
| 同一原因の issue が過剰に重複していないか |  |
| 曖昧さを隠していないか |  |
| 解決済み・対応不要の理由が明確か |  |
```

---

## 12. 禁止事項

このテンプレートを使用する際、以下を行ってはならない。

* issue を記録せずに曖昧さを補完する
* issue を削除して成果物をきれいに見せる
* 対象TCや観点IDを省略する
* 「不明」「微妙」などの曖昧な内容だけを書く
* 影響を書かない
* 推奨対応を書かない
* 元 issue との対応を失う
* 推定度が低い内容を issue 化せず確定扱いする
