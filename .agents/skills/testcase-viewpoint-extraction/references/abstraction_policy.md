# abstraction_policy.md

## 1. 目的

このファイルは、テストケースから抽出した観点候補を、再利用可能な抽象観点へ変換する際の方針を定義する。

主な目的は以下である。

- 観点候補を適切な粒度で抽象化する
- 個別機能に依存しすぎた観点を整理する
- 抽象化しすぎて意味が薄くなることを防ぐ
- 複数の観点候補を統合する際の判断基準を揃える
- 抽象観点カタログの品質を安定させる

このファイルは、主に Step 03 で参照する。

---

## 2. 基本方針

## 2.1 抽象化は「意味を薄めること」ではない

抽象化とは、個別の画面名、機能名、項目名などに依存した観点を、他の機能にも再利用しやすい表現に変換することである。

ただし、確認条件や期待動作が消えてしまうほど一般化してはならない。

悪い例:

```text
取引先コードの重複入力制御
↓
入力チェック
````

良い例:

```text
取引先コードの重複入力制御
↓
一意制約項目の重複入力制御
```

---

## 2.2 根拠TCを持たない抽象観点を作らない

抽象観点は、必ず1件以上の根拠テストケースを持つ必要がある。

以下は禁止する。

* 一般論で観点を追加する
* 仕様上ありそうな観点を推測で追加する
* 他プロジェクトでよく見る観点を補完する
* Step 02 にない観点候補を起点にする

根拠TCがないものは、抽象観点ではなく issue または検討メモとして扱う。

---

## 2.3 抽象化後も条件と期待動作を残す

良い抽象観点には、可能な範囲で以下が含まれる。

```text
対象の性質 + 条件 + 期待される制御または振る舞い
```

例:

```text
一意制約項目の重複入力制御
必須項目未入力時の登録抑止
確定済み状態のデータに対する変更操作の抑止
権限不足ユーザーによる保護対象操作の抑止
登録・更新後の関連画面への反映確認
境界日付における期間判定
```

---

## 2.4 カテゴリ名を観点名にしない

カテゴリ名は、観点を整理するための分類である。
抽象観点名としては粗すぎる場合が多い。

避けるべき観点名:

```text
入力制約
権限
状態制御
表示
帳票
外部連携
データ整合性
```

良い観点名:

```text
必須項目未入力時の登録抑止
権限不足ユーザーによる保護対象操作の抑止
確定済み状態のデータに対する変更操作の抑止
登録・更新後の関連画面への反映確認
出力条件に応じた帳票内容の反映確認
```

## 3. 抽象化レベル

抽象化レベルは、以下の3段階で扱う。

| レベル | 名称         | 意味                     |
| --- | ---------- | ---------------------- |
| L1  | 個別機能レベル    | 特定画面・特定機能・特定項目に依存する観点  |
| L2  | 機能横断レベル    | 同じ種類の機能や業務内で横展開できる観点   |
| L3  | 汎用テスト設計レベル | システムやドメインを越えて再利用しやすい観点 |

---

## 4. L1: 個別機能レベル

## 4.1 定義

L1 は、特定の画面、機能、項目、業務オブジェクトに強く依存した観点である。

## 4.2 例

```text
取引先コードの重複入力制御
取引先登録後の一覧反映
仕訳入力画面の借方金額必須チェック
管理者ユーザーによる部門マスタ削除
```

## 4.3 使いどころ

L1 は、元テストケースに近い粒度で記録したい場合に使う。

Step 02 の観点候補は L1 に近くてよい。

## 4.4 注意

Step 03 の最終的な観点カタログでは、L1 のまま残すかどうかを検討する。

次のような場合は、L1 のまま残してもよい。

* 特定業務固有の意味が強い
* 無理に一般化すると意味が失われる
* 根拠TCが少なく、横展開できるか判断できない
* ユーザーによるレビューが必要

---

## 5. L2: 機能横断レベル

## 5.1 定義

L2 は、特定機能に閉じず、同じ種類の機能・画面・業務オブジェクトに横展開できる観点である。

## 5.2 例

```text
マスタコードの重複入力制御
マスタ登録後の一覧反映
承認済みデータに対する変更操作の抑止
検索条件に一致するデータ抽出
出力条件に応じた帳票内容の反映確認
```

## 5.3 使いどころ

L2 は、実務上かなり使いやすい抽象度である。

特に、以下の場合に適している。

* 同じシステム内の複数機能へ横展開したい
* 業務ドメインの意味を残したい
* L3 にすると一般化しすぎる
* 観点カタログとして再利用したいが、業務文脈も残したい

## 5.4 注意

L2 は、無理に L3 へ上げる必要はない。

たとえば、会計システムの以下の観点は、L2 のまま残したほうが実用的な場合がある。

```text
締め済み会計期間に対する更新制御
連結対象会社に対する入力可否制御
帳票出力条件に応じた表示科目の反映確認
```

これらを一般化しすぎると、レビュー時に使いにくくなることがある。

---

## 6. L3: 汎用テスト設計レベル

## 6.1 定義

L3 は、特定機能や業務ドメインを越えて再利用しやすい観点である。

## 6.2 例

```text
一意制約項目の重複入力制御
必須項目未入力時の登録抑止
状態に応じた操作可否制御
権限不足ユーザーによる保護対象操作の抑止
登録・更新後の関連表示への反映確認
境界値における判定結果の確認
```

## 6.3 使いどころ

L3 は、観点カタログを横断的なテスト設計資産として使いたい場合に適している。

特に、以下の場合に有効である。

* 複数プロダクトで再利用したい
* テスト設計レビューの観点集として使いたい
* 新規機能のテスト設計時に参照したい
* 類似観点を統合したい

## 6.4 注意

L3 へ上げすぎると、観点が曖昧になる。

悪い例:

```text
入力チェック
業務ルール確認
データ確認
画面確認
```

これらは L3 ではなく、単に粗いカテゴリである。

---

## 7. 抽象化の判断手順

観点候補を抽象観点へ変換するときは、以下の順で判断する。

```text
1. 元の観点候補を確認する
2. 根拠TCを確認する
3. 確認している条件を特定する
4. 期待される制御・振る舞いを特定する
5. 個別名をどこまで一般化できるか判断する
6. 類似観点と統合できるか判断する
7. L1 / L2 / L3 を付与する
8. 抽象化理由を記録する
```

## 8. 抽象化の型

## 8.1 項目名を性質へ変換する

個別の項目名を、その項目が持つ性質へ変換する。

| L1      | L2/L3           |
| ------- | --------------- |
| 取引先コード  | マスタコード / 一意制約項目 |
| ユーザーID  | 識別子 / 一意制約項目    |
| 開始日     | 日付項目 / 期間判定項目   |
| 金額      | 数値項目 / 計算対象項目   |
| 承認ステータス | 業務状態 / 状態制御項目   |

例:

```text
取引先コードの重複入力制御
↓
一意制約項目の重複入力制御
```

---

## 8.2 画面名を表示先・反映先へ変換する

個別画面名を、表示先や反映先の性質へ変換する。

| L1      | L2/L3   |
| ------- | ------- |
| 取引先一覧   | 関連一覧画面  |
| 取引先詳細   | 関連詳細画面  |
| 検索結果一覧  | 検索結果    |
| 帳票プレビュー | 出力プレビュー |
| Excel出力 | 出力物     |

例:

```text
取引先登録後の一覧表示
↓
登録後の関連一覧画面への反映確認
```

---

## 8.3 状態名を状態制御へ変換する

個別状態を、操作可否や状態遷移の観点へ変換する。

| L1   | L2/L3           |
| ---- | --------------- |
| 締め済み | 確定済み状態          |
| 承認済み | 確定済み状態 / 承認済み状態 |
| 申請中  | 処理中状態           |
| 無効   | 非アクティブ状態        |
| 削除済み | 論理削除状態          |

例:

```text
締め済み期間の伝票を更新できない
↓
確定済み状態のデータに対する変更操作の抑止
```

---

## 8.4 操作名を操作種別へ変換する

個別操作を、操作種別として一般化する。

| L1 | L2/L3     |
| -- | --------- |
| 登録 | 作成操作      |
| 更新 | 変更操作      |
| 削除 | 削除操作      |
| 承認 | 状態変更操作    |
| 出力 | 出力操作      |
| 取込 | 外部データ取込操作 |

例:

```text
管理者以外の削除不可
↓
権限不足ユーザーによる保護対象操作の抑止
```

---

## 8.5 個別エラーを制約違反へ変換する

個別エラーを、入力制約や業務制約の観点へ変換する。

| L1      | L2/L3         |
| ------- | ------------- |
| 必須エラー   | 必須項目未入力時の登録抑止 |
| 重複エラー   | 一意制約項目の重複入力制御 |
| 桁数エラー   | 入力値の桁数制約      |
| 日付逆転エラー | 期間整合性の制御      |
| 権限エラー   | 権限不足時の操作抑止    |

例:

```text
開始日が終了日より後の場合にエラー
↓
期間項目の前後関係不整合時の登録抑止
```

## 9. 統合判断基準

## 9.1 統合してよい場合

以下の条件を満たす場合、複数の観点候補を1つの抽象観点へ統合してよい。

* 確認している条件が同じ、または非常に近い
* 期待される制御・振る舞いが同じ
* 個別対象だけが異なる
* 抽象化後も意味が失われない
* 根拠TCをすべて保持できる

例:

```text
取引先コードの重複入力制御
ユーザーIDの重複入力制御
部門コードの重複入力制御
↓
一意制約項目の重複入力制御
```

---

## 9.2 統合しないほうがよい場合

以下の場合は、無理に統合しない。

* 条件が異なる
* 期待動作が異なる
* 確認対象が異なる
* 業務的な意味が異なる
* 統合するとレビューしにくくなる
* 抽象化後に「何を確認するのか」が曖昧になる

例:

```text
必須項目未入力時のエラー表示
重複コード入力時のエラー表示
禁止文字入力時のエラー表示
```

これらはすべて「エラー表示」として統合できるように見える。
しかし、実際には確認している制約が異なるため、別観点として残したほうがよい場合が多い。

---

## 9.3 統合を保留する場合

以下の場合は、統合を保留して review 要とする。

* 根拠TCが少ない
* 推定度が中または低
* 観点候補の意味が曖昧
* 業務上の意味差がありそうだが判断できない
* 同じ言葉だが文脈が異なる
* 異なる言葉だが同じ意味か判断できない

保留した場合は、`03_cataloging_issues.md` に記録する。

## 10. 抽象化例

## 10.1 重複入力制御

| 元観点候補           | 抽象観点          | レベル |
| --------------- | ------------- | --- |
| 取引先コードの重複入力制御   | 一意制約項目の重複入力制御 | L3  |
| ユーザーIDの重複登録チェック | 一意制約項目の重複入力制御 | L3  |
| 部門コードの重複エラー     | 一意制約項目の重複入力制御 | L3  |

---

## 10.2 必須入力制御

| 元観点候補        | 抽象観点          | レベル |
| ------------ | ------------- | --- |
| 取引先名未入力時のエラー | 必須項目未入力時の登録抑止 | L3  |
| 金額未入力時の登録不可  | 必須項目未入力時の登録抑止 | L3  |
| 開始日未入力時の保存不可 | 必須項目未入力時の保存抑止 | L3  |

注意:

「登録抑止」と「保存抑止」は、対象操作の表現に合わせて使い分けてよい。
ただし、同じ意味なら統一する。

---

## 10.3 期間判定

| 元観点候補             | 抽象観点               | レベル |
| ----------------- | ------------------ | --- |
| 開始日が終了日より後の場合のエラー | 期間項目の前後関係不整合時の登録抑止 | L3  |
| 適用開始日と適用終了日の境界値   | 境界日付における期間判定       | L3  |
| 会計年度外の日付入力エラー     | 業務対象期間外の日付入力制御     | L2  |

---

## 10.4 状態制御

| 元観点候補        | 抽象観点                  | レベル |
| ------------ | --------------------- | --- |
| 締め済み期間の更新制御  | 確定済み状態のデータに対する変更操作の抑止 | L3  |
| 承認済みデータの削除不可 | 確定済み状態のデータに対する変更操作の抑止 | L3  |
| 申請中データの再申請不可 | 処理中状態における重複操作の抑止      | L2  |

---

## 10.5 権限制御

| 元観点候補            | 抽象観点                   | レベル |
| ---------------- | ---------------------- | --- |
| 一般ユーザーの削除不可      | 権限不足ユーザーによる保護対象操作の抑止   | L3  |
| 参照権限なしユーザーの画面非表示 | 権限不足ユーザーに対する保護対象情報の非表示 | L3  |
| 管理者のみ承認可能        | 権限に応じた状態変更操作の可否制御      | L3  |

---

## 10.6 反映確認

| 元観点候補    | 抽象観点             | レベル |
| -------- | ---------------- | --- |
| 登録後の一覧反映 | 登録後の関連一覧画面への反映確認 | L2  |
| 更新後の詳細反映 | 更新後の関連詳細画面への反映確認 | L2  |
| 登録後の帳票反映 | 登録・更新後の出力物への反映確認 | L2  |

注意:

一覧、詳細、帳票は確認対象が異なる。
無理に「データ反映確認」と統合しない。

## 11. 抽象化しすぎの例

以下は、抽象化しすぎである。

| 元観点候補         | 抽象化しすぎた観点 | 問題            |
| ------------- | --------- | ------------- |
| 取引先コードの重複入力制御 | 入力チェック    | 条件と期待動作が消えている |
| 締め済み期間の更新制御   | 業務ルール確認   | 何のルールか不明      |
| 登録後の一覧反映      | 画面確認      | 確認対象が不明       |
| 管理者以外の削除不可    | 権限確認      | 操作と期待動作が不明    |
| 帳票の金額反映       | 帳票確認      | 確認内容が不明       |

## 12. 個別すぎる例

以下は、抽象観点としては個別すぎる可能性がある。

| 個別すぎる観点               | 改善例                           |
| --------------------- | ----------------------------- |
| 取引先コード「C001」の重複入力制御   | 取引先コードの重複入力制御 / 一意制約項目の重複入力制御 |
| 2024/04/01 の開始日境界チェック | 境界日付における期間判定                  |
| 山田ユーザーで削除できないこと       | 権限不足ユーザーによる削除操作の抑止            |
| 第1四半期帳票のA列金額反映        | 出力条件に応じた帳票項目の反映確認             |

ただし、具体値そのものがテスト上重要な場合は、根拠や説明に残してよい。

## 13. 業務ドメイン固有表現の扱い

## 13.1 消しすぎない

業務ドメイン固有の言葉には、テスト観点として重要な意味がある場合がある。

例:

```text
締め済み期間
承認済みデータ
連結対象会社
換算レート
会計年度
表示科目
勘定科目
```

これらをすべて一般化すると、観点の実用性が下がることがある。

---

## 13.2 L2 に留める判断

以下の場合は、L3 まで上げず L2 に留める。

* 業務ルールの意味が重要
* 一般化すると確認条件が曖昧になる
* レビュー時に業務用語が必要
* 同じ用語が複数テストケースで繰り返し出ている
* ドメイン固有のリスクを含んでいる

例:

```text
締め済み会計期間に対する更新制御
```

これは、

```text
確定済み状態のデータに対する変更操作の抑止
```

へ一般化できる。
ただし、会計期間の締めという業務意味を残す必要がある場合は、L2 として残す。

---

## 14. レビュー要にすべきケース

以下の場合は、抽象観点をレビュー要とする。

* L2 と L3 のどちらにすべきか迷う
* 業務用語を消してよいか判断できない
* 複数の観点候補を統合したが、意味差が残っている
* 根拠TCが1件のみである
* 元観点候補の推定度が中または低
* カテゴリ分類に迷う
* 抽象観点名が長すぎる
* 抽象観点名が短すぎて意味が薄い
* 根拠TCの期待結果が曖昧

---

## 15. 抽象観点名の命名ルール

## 15.1 基本形

抽象観点名は、以下のいずれかの形を基本とする。

```text
〜時の〜制御
〜時の〜抑止
〜に対する〜確認
〜に応じた〜可否制御
〜における〜判定
〜後の〜反映確認
```

例:

```text
必須項目未入力時の登録抑止
一意制約項目の重複入力制御
権限不足ユーザーによる保護対象操作の抑止
状態に応じた操作可否制御
境界日付における期間判定
登録後の関連画面への反映確認
```

---

## 15.2 避ける表現

以下のような表現は避ける。

```text
〜チェック
〜確認
〜テスト
正常系
異常系
データ
画面
処理
```

これらは短すぎて、確認条件や期待動作が不明になりやすい。

ただし、元観点候補としては使ってよい。
抽象観点名では、より具体的にする。

---

## 15.3 長くなりすぎる場合

観点名が長くなりすぎる場合は、観点名をやや短くし、説明欄で補う。

例:

観点名:

```text
期間項目の前後関係不整合時の登録抑止
```

説明:

```text
開始日が終了日より後であるなど、期間を構成する日付項目の前後関係が不正な場合に、登録・保存が抑止されることを確認する。
```

---

## 16. 抽象化理由の書き方

抽象化理由には、以下を含める。

* 元観点候補
* 一般化した対象
* 統合した理由
* 抽象化レベルの理由
* 根拠TC

例:

```text
取引先コード、ユーザーID、部門コードはいずれも一意であるべき識別子として扱われ、重複値を入力した場合の登録抑止を確認しているため、一意制約項目の重複入力制御として統合した。複数機能に横展開できるため L3 とした。
```

悪い例:

```text
似ているため統合した。
```

理由が薄い。ほんのり危険な香りがする。

---

## 17. 抽象化後の品質チェック

抽象観点ごとに、以下を確認する。

```markdown id="n5fu9a"
| 確認項目 | 判定 |
|---|---|
| 元観点候補から説明できるか |  |
| 根拠TCが1件以上あるか |  |
| 条件または期待動作が残っているか |  |
| 抽象化しすぎていないか |  |
| 個別機能に依存しすぎていないか |  |
| カテゴリ名そのものになっていないか |  |
| 類似観点と重複していないか |  |
| 意味の異なる観点を無理に統合していないか |  |
| 抽象化理由を説明できるか |  |
| レビュー要否が適切か |  |
```

---

## 18. 禁止事項

抽象化時に、以下を行ってはならない。

* Step 02 にない観点候補を起点にする
* 根拠TCがない抽象観点を作る
* 一般論で観点を追加する
* 抽象化しすぎて条件や期待動作を消す
* カテゴリ名を抽象観点名にする
* 意味の異なる観点を無理に統合する
* 業務上重要な用語を理由なく消す
* 推定度が低い観点候補を確定扱いする
* 抽象化理由を書かずに統合する
