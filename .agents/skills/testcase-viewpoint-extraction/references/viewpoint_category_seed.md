# viewpoint_category_seed.md

## 1. 目的

このファイルは、抽象観点を分類するための初期カテゴリ案を定義する。

主な目的は以下である。

- 観点カテゴリの表記ゆれを抑える
- 抽象観点カタログを整理しやすくする
- Step 03 で観点IDを付与しやすくする
- レビュー時に観点の偏りを確認しやすくする

このファイルは、主に Step 03 で参照する。

---

## 2. 基本方針

## 2.1 カテゴリは観点そのものではない

カテゴリは、抽象観点を整理するための分類である。

カテゴリ名をそのまま抽象観点名にしてはならない。

悪い例:

```text
カテゴリ: 入力制約
抽象観点: 入力制約
````

良い例:

```text
カテゴリ: 入力制約
抽象観点: 必須項目未入力時の登録抑止
```

---

## 2.2 無理に分類しない

どのカテゴリに分類すべきか判断できない場合は、無理に分類しない。

その場合は、以下のいずれかを使用する。

```text
その他
分類保留
```

また、必要に応じて `03_cataloging_issues.md` に記録する。

---

## 2.3 カテゴリは追加してよい

入力テストケース群に固有の観点があり、既存カテゴリでは表現しにくい場合は、新しいカテゴリを追加してよい。

ただし、追加する場合は以下を満たすこと。

* 1件以上の抽象観点が存在する
* 既存カテゴリでは分類しにくい理由がある
* カテゴリ名が広すぎない
* カテゴリ名が特定機能に依存しすぎない

---

## 3. 観点ID接頭辞

観点IDは、原則として以下の形式にする。

```text
VP-{CATEGORY}-{連番}
```

カテゴリごとの接頭辞は以下を初期案とする。

| カテゴリ      | ID接頭辞      |
| --------- | ---------- |
| 入力制約      | VP-INPUT   |
| 業務ルール     | VP-BIZ     |
| 状態制御      | VP-STATE   |
| 権限        | VP-AUTH    |
| データ整合性    | VP-DATA    |
| 表示        | VP-VIEW    |
| 検索・抽出     | VP-SEARCH  |
| 登録・更新・削除  | VP-CRUD    |
| 帳票・出力     | VP-OUTPUT  |
| 外部連携      | VP-IF      |
| エラー・警告    | VP-ERROR   |
| 排他・同時実行   | VP-CONC    |
| 監査ログ      | VP-AUDIT   |
| 性能・応答     | VP-PERF    |
| 回帰確認      | VP-REG     |
| バッチ・非同期処理 | VP-BATCH   |
| 環境・設定     | VP-CONFIG  |
| 分類保留      | VP-UNCLEAR |
| その他       | VP-OTHER   |

## 4. カテゴリ一覧

## 4.1 入力制約

### 定義

入力値が、システムで許可される形式、範囲、条件を満たしているかを確認する観点。

### 典型的な観点

```text
必須項目未入力時の登録抑止
一意制約項目の重複入力制御
入力値の桁数制約
入力値の型制約
禁止文字入力時のエラー表示
数値項目の範囲外入力制御
日付項目の形式不正時の登録抑止
```

### 含めるもの

* 必須チェック
* 桁数チェック
* 型チェック
* 形式チェック
* 範囲チェック
* 重複チェック
* 禁止文字チェック
* NULL / 空欄の扱い
* 最大値・最小値
* 境界値

### 注意

入力制約は、業務ルールと重なることがある。

例:

```text
会計年度外の日付入力制御
```

これは単なる日付範囲チェックではなく、業務ルールとして扱うほうがよい場合もある。

---

## 4.2 業務ルール

### 定義

対象システムの業務仕様、業務条件、業務上の制約に基づく確認観点。

### 典型的な観点

```text
業務対象期間外の日付入力制御
承認状態に応じた操作可否制御
連結対象会社に対する入力可否制御
取引区分に応じた入力項目の制御
会計年度に応じた処理対象データの判定
```

### 含めるもの

* 業務期間
* 承認・締め
* 会計年度
* 対象会社
* 取引区分
* 適用条件
* 業務上の例外
* 仕様上の分岐条件

### 注意

カテゴリが広くなりやすい。
「業務ルール確認」のような抽象観点名にしてはいけない。

---

## 4.3 状態制御

### 定義

データや業務オブジェクトの状態に応じて、操作可否、表示、遷移、処理結果が変わることを確認する観点。

### 典型的な観点

```text
確定済み状態のデータに対する変更操作の抑止
処理中状態における重複操作の抑止
無効状態データの選択不可制御
削除済みデータの表示制御
状態遷移後の操作可否制御
```

### 含めるもの

* 未処理
* 処理中
* 承認済み
* 差戻し
* 締め済み
* 確定済み
* 無効
* 削除済み
* 状態遷移
* 状態別操作可否

### 注意

業務ルールや権限と重なることがある。
状態そのものが主な分岐条件である場合は、状態制御に分類する。

---

## 4.4 権限

### 定義

ユーザー、ロール、所属、権限設定に応じて、画面表示、操作可否、データ参照範囲が制御されることを確認する観点。

### 典型的な観点

```text
権限不足ユーザーによる保護対象操作の抑止
権限不足ユーザーに対する保護対象情報の非表示
ロールに応じた利用可能機能の制御
所属組織に応じた参照可能データ範囲の制御
管理者権限に限定された設定変更操作の制御
```

### 含めるもの

* ロール
* 権限
* 管理者
* 一般ユーザー
* 参照権限
* 更新権限
* 承認権限
* 所属組織
* データアクセス範囲

### 注意

単に「権限確認」としない。
どの権限条件で、何が許可・抑止されるのかを観点名または説明に残す。

---

## 4.5 データ整合性

### 定義

登録、更新、削除、計算、集計などの結果として、関連データ間の整合性が保たれることを確認する観点。

### 典型的な観点

```text
登録・更新後の関連画面への反映確認
親子データ間の整合性確認
明細合計とヘッダ合計の一致確認
関連マスタ変更後の参照データ反映確認
削除後の関連データ参照制御
```

### 含めるもの

* 登録後反映
* 更新後反映
* 削除後反映
* 親子関係
* ヘッダ・明細
* マスタ参照
* 集計値
* 関連テーブル
* 画面とDBの整合
* 出力値との整合

### 注意

表示カテゴリと迷う場合がある。
確認対象が「表示そのもの」ではなく、データの反映・一致・整合であればデータ整合性に分類する。

---

## 4.6 表示

### 定義

画面上の表示内容、表示状態、表示形式、表示制御を確認する観点。

### 典型的な観点

```text
条件に応じた項目表示・非表示制御
入力値に応じた表示内容の切替確認
一覧画面における表示順の確認
数値・日付項目の表示形式確認
メッセージ表示タイミングの確認
```

### 含めるもの

* 表示・非表示
* 活性・非活性
* 表示順
* 表示形式
* ラベル
* メッセージ
* 初期表示
* 画面遷移後表示
* 表示切替

### 注意

「登録後の一覧反映」は、表示ではなくデータ整合性に分類したほうがよい場合がある。
表示形式や表示制御が主眼なら表示に分類する。

---

## 4.7 検索・抽出

### 定義

検索条件、抽出条件、絞り込み条件に応じて、対象データが正しく取得されることを確認する観点。

### 典型的な観点

```text
検索条件に一致するデータ抽出
複数検索条件の組み合わせによる絞り込み
未指定条件を含む検索結果の確認
部分一致検索による対象データ抽出
期間条件に応じた対象データ抽出
```

### 含めるもの

* 完全一致
* 部分一致
* 前方一致
* 後方一致
* 複数条件
* 未指定条件
* 期間指定
* 範囲指定
* ソート
* ページング
* フィルタ

### 注意

検索結果の表示形式は「表示」に分類する場合がある。
検索条件に対するデータ抽出が主眼なら「検索・抽出」に分類する。

---

## 4.8 登録・更新・削除

### 定義

作成、変更、削除といったデータ変更操作そのものの成功・失敗・反映を確認する観点。

### 典型的な観点

```text
正常条件における新規登録の完了確認
既存データ更新時の変更内容保存確認
削除操作後の対象データ削除確認
論理削除後の参照可否制御
登録・更新操作のキャンセル時の変更破棄確認
```

### 含めるもの

* 登録
* 更新
* 削除
* 保存
* キャンセル
* 論理削除
* 物理削除
* 再登録
* 複製
* 変更破棄

### 注意

登録・更新・削除は広いカテゴリなので、可能であれば入力制約、状態制御、権限、データ整合性など、より具体的なカテゴリを優先する。

---

## 4.9 帳票・出力

### 定義

帳票、CSV、Excel、PDF、印刷、ファイル出力など、出力物の内容や形式を確認する観点。

### 典型的な観点

```text
出力条件に応じた帳票内容の反映確認
登録・更新後の出力物への反映確認
出力ファイルの項目順・形式確認
集計条件に応じた帳票金額の一致確認
出力対象外データの除外確認
```

### 含めるもの

* 帳票
* CSV
* Excel
* PDF
* 印刷
* ファイル出力
* プレビュー
* 出力項目
* 出力順
* 出力形式
* 出力条件
* 集計値

### 注意

帳票上の金額一致は、データ整合性にも分類できる。
出力物での確認が主眼なら、帳票・出力に分類する。

---

## 4.10 外部連携

### 定義

外部システム、外部ファイル、API、バッチ、データ取込・送信など、システム外部との連携に関する観点。

### 典型的な観点

```text
外部ファイル取込時の正常登録確認
取込データ不正時のエラー制御
外部システム送信後のステータス反映確認
連携失敗時のリトライ・エラー通知確認
連携対象外データの除外確認
```

### 含めるもの

* API連携
* ファイル取込
* ファイル送信
* データ連携
* 外部システム
* インポート
* エクスポート
* 連携エラー
* リトライ
* 連携ログ

### 注意

CSVやExcelの出力だけなら「帳票・出力」に分類する。
外部システムとのデータ授受が主眼なら「外部連携」に分類する。

---

## 4.11 エラー・警告

### 定義

エラー、警告、確認メッセージ、バリデーションメッセージの表示内容や制御を確認する観点。

### 典型的な観点

```text
制約違反時のエラーメッセージ表示確認
警告表示後の続行・中断制御
確認ダイアログにおける選択結果の反映確認
複数エラー発生時の表示優先順位確認
エラー発生時の入力値保持確認
```

### 含めるもの

* エラーメッセージ
* 警告メッセージ
* 確認ダイアログ
* バリデーションメッセージ
* メッセージ文言
* 表示タイミング
* 表示位置
* メッセージ優先順位
* エラー後の画面状態

### 注意

エラー表示は、多くの場合、入力制約や業務ルールの副次確認である。
メッセージ内容や表示制御自体が主眼である場合に、このカテゴリを使う。

---

## 4.12 排他・同時実行

### 定義

複数ユーザー、複数処理、複数セッションが同時に対象データへアクセスする場合の制御を確認する観点。

### 典型的な観点

```text
同一データ同時更新時の排他制御
編集中データに対する他ユーザー更新の抑止
後勝ち更新によるデータ不整合の防止
処理中データに対する重複実行の抑止
ロック解除後の再操作確認
```

### 含めるもの

* 排他制御
* 同時更新
* ロック
* 楽観ロック
* 悲観ロック
* 二重送信
* 二重実行
* 複数セッション
* 同時ログイン
* 競合

### 注意

同時実行条件が元テストケースに明示されていない場合、このカテゴリを推測で追加してはならない。

---

## 4.13 監査ログ

### 定義

操作履歴、変更履歴、監査ログ、アクセスログなどが、操作や処理に応じて記録されることを確認する観点。

### 典型的な観点

```text
登録操作後の監査ログ出力確認
更新前後の変更履歴記録確認
削除操作時の操作履歴記録確認
権限エラー発生時のアクセスログ記録確認
ログ出力項目の内容確認
```

### 含めるもの

* 操作ログ
* 監査ログ
* 変更履歴
* アクセスログ
* 実行履歴
* 登録者
* 更新者
* 更新日時
* 処理結果
* エラー履歴

### 注意

ログ確認が元テストケースに明示されていない場合、このカテゴリを推測で追加してはならない。

---

## 4.14 性能・応答

### 定義

処理時間、応答速度、大量データ処理、画面表示速度など、性能面の確認観点。

### 典型的な観点

```text
大量データ検索時の応答時間確認
大量明細登録時の処理完了時間確認
帳票出力時の処理時間確認
画面初期表示の応答時間確認
同時実行時の応答劣化確認
```

### 含めるもの

* 応答時間
* 処理時間
* 大量データ
* 負荷
* スループット
* タイムアウト
* 画面表示速度
* 帳票出力時間

### 注意

性能観点は、元テストケースに明示されている場合のみ扱う。
一般論として追加してはならない。

---

## 4.15 回帰確認

### 定義

既存機能や過去不具合が、変更後も期待通り動作することを確認する観点。

### 典型的な観点

```text
既存登録機能の変更後動作維持確認
修正済み不具合の再発防止確認
関連機能への変更影響確認
既存帳票出力結果の維持確認
既存入力制約の維持確認
```

### 含めるもの

* 回帰
* デグレード
* 既存機能維持
* 修正確認
* 再発防止
* 影響確認
* 過去不具合

### 注意

「回帰」という既存分類があるだけでは不十分な場合がある。
実際に何を維持確認しているのか、観点名に残す。

---

## 4.16 バッチ・非同期処理

### 定義

バッチ処理、非同期処理、スケジュール実行、バックグラウンド処理の実行結果や状態を確認する観点。

### 典型的な観点

```text
バッチ処理正常終了時の処理結果反映確認
バッチ異常終了時のエラー通知確認
非同期処理完了後のステータス反映確認
スケジュール実行対象データの抽出確認
再実行時の重複処理抑止
```

### 含めるもの

* バッチ
* 非同期
* ジョブ
* スケジュール実行
* バックグラウンド処理
* 処理ステータス
* 再実行
* 異常終了
* リトライ

### 注意

外部システム連携と重なる場合がある。
処理形態が主眼ならバッチ・非同期処理、外部授受が主眼なら外部連携に分類する。

---

## 4.17 環境・設定

### 定義

環境設定、マスタ設定、パラメータ、システム設定、ユーザー設定に応じた動作を確認する観点。

### 典型的な観点

```text
設定値に応じた入力項目の表示制御
システムパラメータ変更後の処理分岐確認
ユーザー設定に応じた初期表示確認
マスタ設定に基づく選択肢表示確認
環境差異による出力先切替確認
```

### 含めるもの

* システム設定
* ユーザー設定
* 環境設定
* パラメータ
* マスタ設定
* 初期値
* デフォルト値
* 設定切替
* 環境差異

### 注意

マスタ設定に基づく業務ルールは、業務ルールカテゴリのほうが適切な場合もある。

---

## 4.18 分類保留

### 定義

既存カテゴリに分類すべきか判断できない観点を一時的に置くカテゴリ。

### 典型的な使い方

```text
分類に迷う観点
業務意味が不明な観点
根拠TCが曖昧な観点
複数カテゴリにまたがる観点
```

### 注意

分類保留は、最終状態として望ましいわけではない。
`03_cataloging_issues.md` または `04_final_issue_log.md` に理由を記録する。

---

## 4.19 その他

### 定義

上記カテゴリに該当しないが、根拠TCがあり、抽象観点として扱う必要があるもの。

### 注意

「その他」が増えすぎる場合は、カテゴリ設計を見直す。
その他カテゴリに入れる場合も、抽象観点名は具体的に書く。

---

## 5. カテゴリ選択の優先ルール

カテゴリに迷った場合は、以下の順で判断する。

## 5.1 何が主な確認対象かを見る

同じテストケースでも、複数カテゴリに見える場合がある。

例:

```text
権限なしユーザーで削除ボタンが非表示になる
```

候補カテゴリ:

* 権限
* 表示
* 登録・更新・削除

この場合、主な確認対象が「権限による制御」なら、カテゴリは `権限` とする。

---

## 5.2 条件を優先するか、結果を優先するか

カテゴリ選択では、基本的に「条件」を優先する。

例:

```text
締め済み期間のデータを更新するとエラーになる
```

候補カテゴリ:

* 状態制御
* エラー・警告
* 登録・更新・削除

この場合、主な条件が「締め済み状態」であるため、カテゴリは `状態制御` とする。

ただし、エラーメッセージ文言や表示位置が主な確認対象なら `エラー・警告` とする。

---

## 5.3 出力物が主な確認対象なら帳票・出力

例:

```text
登録した金額が帳票に反映される
```

候補カテゴリ:

* データ整合性
* 帳票・出力

この場合、出力物での確認が主眼なら `帳票・出力` とする。
データ間の一致確認が主眼なら `データ整合性` とする。

---

## 5.4 操作種別よりも制約種別を優先する

例:

```text
必須項目未入力で登録できない
```

候補カテゴリ:

* 登録・更新・削除
* 入力制約
* エラー・警告

この場合、主な確認対象は「必須入力制約」なので、カテゴリは `入力制約` とする。

---

## 6. カテゴリと観点名の関係

抽象観点名は、カテゴリより具体的にする。

| カテゴリ   | 悪い抽象観点名 | 良い抽象観点名               |
| ------ | ------- | --------------------- |
| 入力制約   | 入力チェック  | 必須項目未入力時の登録抑止         |
| 権限     | 権限確認    | 権限不足ユーザーによる保護対象操作の抑止  |
| 状態制御   | 状態確認    | 確定済み状態のデータに対する変更操作の抑止 |
| データ整合性 | データ確認   | 登録・更新後の関連画面への反映確認     |
| 帳票・出力  | 帳票確認    | 出力条件に応じた帳票内容の反映確認     |
| 検索・抽出  | 検索確認    | 検索条件に一致するデータ抽出        |

---

## 7. カテゴリ追加の記録

新しいカテゴリを追加した場合は、`03_cataloging_issues.md` または `04_final_summary.md` に記録する。

記録例:

```markdown
| Issue ID | 種別 | 対象観点/TC | 内容 | 影響 | 推奨対応 |
|---|---|---|---|---|---|
| ISS-CAT-001 | カテゴリ追加 | VP-XXXX-001 | 既存カテゴリに該当しないため「移行データ」を追加した | 観点分類が拡張された | レビュー時にカテゴリ妥当性を確認する |
```

---

## 8. 禁止事項

カテゴリ付与時に、以下を行ってはならない。

* カテゴリ名をそのまま抽象観点名にする
* 迷ったカテゴリを根拠なく断定する
* 元テストケースに存在しない観点をカテゴリから逆算して追加する
* すべてを「業務ルール」や「その他」に寄せる
* 分類保留の理由を記録しない
* カテゴリを増やしすぎて整理不能にする
