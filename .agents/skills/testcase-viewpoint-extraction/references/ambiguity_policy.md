# ambiguity_policy.md

## 1. 目的

このファイルは、テストケースからテスト意図・観点候補・抽象観点を抽出する際に、曖昧な情報をどのように扱うかを定義する。

主な目的は以下である。

- 曖昧な記述をAIエージェントが勝手に補完しないようにする
- 推定と確定情報を区別する
- 情報不足を issue として記録する
- 後続レビューで確認すべき点を明確にする
- 成果物に推測由来の情報が混入することを防ぐ

このファイルは、主に Step 01、Step 02、Step 04 で参照する。

---

## 2. 基本方針

## 2.1 曖昧さは失敗ではなく、成果物として残す

曖昧なテストケースは、無理にきれいに補完しない。

曖昧さは、既存テストケースの改善余地を示す重要な情報である。

そのため、以下のように扱う。

```text
曖昧な情報を見つける
↓
推測で補完しない
↓
推定度を下げる
↓
レビュー要否を「要」にする
↓
issue として記録する
````

---

## 2.2 入力にない情報を確定事項にしない

入力資料に存在しない情報を、仕様・業務ルール・設計意図として確定的に書いてはならない。

悪い例:

```text
期待結果に「エラー」とあるため、必須項目チェックである。
```

良い例:

```text
期待結果に「エラー」とあるため、何らかの入力制約違反を確認している可能性がある。
ただし、必須項目チェックかどうかは入力資料からは判断できない。
```

---

## 2.3 推定は明示する

合理的に推定できる場合でも、推定であることを明示する。

推定度は以下を使用する。

| 推定度 | 意味                      |
| --- | ----------------------- |
| 高   | テストケースの記述から明確に読み取れる     |
| 中   | 妥当に推定できるが、明示されていない要素がある |
| 低   | 可能性はあるが、根拠が弱い           |

推定度が `低` の内容は、確定成果物として扱わず、issue またはレビュー対象にする。

---

## 2.4 推定を重ねない

推定を根拠にして、さらに別の推定を重ねてはならない。

禁止例:

```text
期待結果に「登録できない」とある
→ 入力チェックだと推定する
→ 必須項目チェックだと推定する
→ 必須チェック観点として抽象化する
```

この場合、最初の「入力チェックだと推定する」までは許容できる場合がある。
しかし、そこから「必須項目チェック」と断定するのは推定の積み上げである。

## 3. 曖昧さの種類

## 3.1 テストケースIDが不明

### 状態

テストケースIDが存在しない、またはどのIDがどのテストケースに対応するか分からない。

### 扱い

* Step 01 で仮IDを付与する
* 仮IDを付与したことを issue に記録する
* 元IDとして扱わない

### 仮ID形式

```text
TC-TEMP-001
TC-TEMP-002
TC-TEMP-003
```

### issue 記録例

```markdown
| Issue ID | 種別 | 対象TC | 内容 | 影響 | 推奨対応 |
|---|---|---|---|---|---|
| ISS-IN-001 | TC IDなし | TC-TEMP-001 | 元資料にテストケースIDが存在しないため仮IDを付与した | 後続のトレーサビリティで元資料との対応確認が必要 | 元資料に正式なTC IDを付与する |
```

---

## 3.2 テストケース名が曖昧

### 状態

テストケース名が抽象的すぎる。

例:

```text
登録確認
表示確認
エラー確認
正常系
異常系
パターン1
```

### 扱い

* 操作/入力と期待結果を優先して意図を判断する
* テストケース名だけで意図を断定しない
* 必要に応じて推定度を下げる
* issue に記録する

### 注意

テストケース名が曖昧でも、操作/入力と期待結果が明確であれば、推定度は高にできる場合がある。

---

## 3.3 対象機能が不明

### 状態

どの機能、画面、業務、モジュールのテストケースか分からない。

### 扱い

* 入力資料の見出し、シート名、周辺情報から読み取れる場合は、根拠を明示して記録する
* 判断できない場合は空欄または `不明` とする
* 観点抽象化時は、機能横断的に扱いすぎない
* issue に記録する

### 禁止事項

対象機能を一般常識で補ってはならない。

---

## 3.4 前提条件が不足している

### 状態

テスト実行に必要な状態や条件が記載されていない。

例:

```text
登録する
```

しかし、以下が不明。

* ログインユーザー
* 権限
* 既存データの有無
* 対象データの状態
* 対象期間
* マスタ設定
* 事前登録データ
* 環境条件

### 扱い

* 操作/入力と期待結果から読み取れる範囲で意図を抽出する
* 前提条件が意図に影響する場合は推定度を下げる
* issue に記録する

### 例

```text
操作: 既存コードを入力して登録
期待結果: 重複エラー
```

この場合、既存コードが存在することは操作から読み取れるため、重複入力制御の意図は比較的明確である。

一方で、

```text
操作: データを登録
期待結果: エラー
```

この場合、何の条件でエラーになるか不明なため、推定度は低にする。

---

## 3.5 操作/入力が不足している

### 状態

何を操作したのか、どの値を入力したのかが不明。

例:

```text
登録ボタンを押す
結果を確認する
```

入力値や条件が不明。

### 扱い

* 期待結果から読み取れる範囲で意図を推定する
* 入力条件が観点抽出に必要な場合は、観点候補を確定しない
* 推定度を中または低にする
* issue に記録する

### 禁止事項

入力値を推測で補ってはならない。

悪い例:

```text
おそらく必須項目を空欄にした。
```

良い例:

```text
エラーが期待されているが、どの入力値に起因するエラーかは不明。
```

---

## 3.6 期待結果が不足している

### 状態

期待結果が記載されていない、または確認すべき結果が不明。

例:

```text
データを登録する。
```

期待結果なし。

### 扱い

* 主意図を確定しない
* 観点候補も確定しない、または推定度を低にする
* issue に記録する
* 必要に応じてレビュー対象にする

### 注意

期待結果がない場合、テストケースとしての確認目的が判断しにくい。
この場合、無理に正常系・異常系へ分類しない。

---

## 3.7 期待結果が曖昧

### 状態

期待結果が抽象的すぎる。

例:

```text
正しく表示される
正しく登録される
エラーになる
正常に処理される
問題ないこと
仕様通りであること
```

### 扱い

* 具体的に何を確認するのかが分かる範囲で記録する
* 不明な部分は補完しない
* 推定度を中または低にする
* レビュー要否を `要` にする
* issue に記録する

### 例

入力:

```text
操作/入力: データを登録する
期待結果: 正しく表示される
```

出力例:

```text
主意図: 登録したデータが表示されることを確認する。
推定度: 中
レビュー理由: 「正しく表示される」の具体的な表示項目・値・状態が不明。
```

---

## 3.8 「エラーになる」だけが記載されている

### 状態

期待結果に「エラーになる」とだけ記載されている。

不明な要素:

* 何のエラーか
* エラー文言
* エラー表示場所
* 登録が抑止されるのか
* 入力値が保持されるのか
* 後続処理が止まるのか

### 扱い

* 「エラー発生の確認」としては扱える
* エラー原因を断定しない
* 入力や条件から原因が明確な場合のみ観点候補にする
* issue に記録する

### 例

```text
操作: 必須項目を空欄にして登録
期待結果: エラーになる
```

この場合:

```text
観点候補: 必須項目未入力時のエラー表示
推定度: 高または中
```

ただし、

```text
操作: 登録する
期待結果: エラーになる
```

この場合:

```text
観点候補: エラー発生確認
推定度: 低
```

---

## 3.9 「正しく」「正常に」などの曖昧語

### 状態

以下のような語が含まれる。

```text
正しく
正常に
適切に
問題なく
仕様通り
想定通り
反映される
処理される
```

### 扱い

* 周辺記述から具体内容が読み取れる場合のみ具体化する
* 読み取れない場合は曖昧語をそのまま残す
* issue に記録する

### 悪い例

```text
正しく表示される
→ 登録した名称、コード、住所、電話番号が一覧に表示されること
```

元資料に項目名がなければ、補完しすぎである。

### 良い例

```text
正しく表示される
→ 登録したデータが表示されることを確認していると推定する。
ただし、具体的な表示項目は不明。
```

---

## 3.10 複数意図が混在している

### 状態

1つのテストケースに複数の確認内容が含まれている。

例:

```text
登録後、一覧に表示され、詳細画面にも反映され、帳票にも出力される。
```

### 扱い

* 主意図と副次意図に分けて記録する
* 分けきれない場合は issue にする
* テストケース分割の候補として記録する
* 勝手に複数テストケースへ分割しない

### 例

```text
主意図: 登録後に一覧へ反映されることを確認する。
副次意図: 詳細画面への反映、帳票出力への反映を確認する。
レビュー要否: 要
レビュー理由: 1テストケースに複数の確認対象が含まれる。
```

---

## 3.11 操作と期待結果が対応していない

### 状態

操作内容と期待結果の対応が不明、または矛盾している。

例:

```text
操作: 削除する
期待結果: 登録完了メッセージが表示される
```

### 扱い

* 意図を確定しない
* 推定度を低にする
* issue に記録する
* 必要に応じて対象外または要レビューにする

### 禁止事項

矛盾を勝手に修正してはならない。

悪い例:

```text
期待結果は削除完了メッセージの誤記だと判断する。
```

良い例:

```text
操作は削除だが、期待結果は登録完了メッセージであり、記述に矛盾がある。
```

---

## 3.12 既存分類と内容が一致しない

### 状態

元資料の既存分類と、テストケース内容が一致しない可能性がある。

例:

```text
既存分類: 正常系
操作: 必須項目を空欄にして登録
期待結果: エラーが表示される
```

### 扱い

* テストケース内容を優先する
* 既存分類は参考情報として扱う
* issue に記録する
* 勝手に既存分類を書き換えない

### 例

```text
既存分類は「正常系」だが、内容は必須項目未入力時のエラー確認であり、分類が一致しない可能性がある。
```

---

## 3.13 業務用語の意味が不明

### 状態

テストケースに業務用語が含まれるが、その意味が入力資料だけでは判断できない。

例:

```text
締め済み
確定済み
連結対象
IFRS調整
組替
内部取引
```

### 扱い

* 用語を勝手に一般化しすぎない
* 元の業務用語を残す
* 意味が不明な場合は issue に記録する
* 抽象化時はレビュー要にする

### 注意

業務用語は、テスト観点として重要な意味を持つ場合がある。
分からないからといって、すべて一般的な言葉へ置き換えない。

---

## 3.14 対象データの状態が不明

### 状態

対象データが新規、既存、承認済み、締め済み、削除済みなど、どの状態なのか不明。

### 扱い

* 状態に依存する観点を確定しない
* 状態が期待結果に関係する場合は推定度を下げる
* issue に記録する

### 例

```text
操作: 更新する
期待結果: 更新できない
```

この場合、なぜ更新できないのか不明。
状態制御、権限、入力制約など複数の可能性があるため、推定度は低にする。

---

## 3.15 ユーザー・権限条件が不明

### 状態

操作ユーザーや権限条件が期待結果に影響しそうだが、記載がない。

例:

```text
操作: 削除する
期待結果: 削除できない
```

### 扱い

* 権限による制御と断定しない
* 他の条件による制御の可能性も残す
* issue に記録する

### 悪い例

```text
一般ユーザーなので削除できないことを確認している。
```

### 良い例

```text
削除できないことを確認しているが、権限不足によるものか、状態制御によるものかは不明。
```

---

## 3.16 具体値の意味が不明

### 状態

入力値は記載されているが、その値がなぜ選ばれたのか不明。

例:

```text
金額: 999999999
日付: 2024/04/01
コード: A0001
```

### 扱い

* 具体値は根拠として残す
* 境界値や代表値と断定しない
* 意味が推定できる場合でも推定として明示する
* issue に記録する

### 例

```text
999999999 は最大桁の確認である可能性があるが、最大値としての根拠は入力資料上では確認できない。
```

---

## 3.17 期待結果に「DB確認」「ログ確認」などがあるが詳細不明

### 状態

確認対象は記載されているが、具体的な確認内容が不明。

例:

```text
DBを確認する
ログを確認する
出力ファイルを確認する
```

### 扱い

* 確認対象は記録する
* 確認内容は補完しない
* 観点候補は具体化しすぎない
* issue に記録する

### 例

```text
観点候補: DB反映確認
推定度: 中
レビュー理由: 確認対象テーブル・項目・値が不明。
```

---

## 4. 推定度の判定基準

## 4.1 推定度: 高

以下を満たす場合、推定度を `高` とする。

* 操作/入力と期待結果が明確に対応している
* テストケース名と内容が一致している
* 確認対象が具体的に書かれている
* 入力条件が明確である
* 期待される動作が明確である

### 例

```text
操作: 既存コードを入力して登録する
期待結果: 重複エラーが表示され、登録されない
```

意図:

```text
既存コードを入力した場合に、登録が抑止されることを確認する。
```

推定度:

```text
高
```

---

## 4.2 推定度: 中

以下の場合、推定度を `中` とする。

* 意図は読み取れるが、確認項目がやや曖昧
* 期待結果に曖昧語が含まれる
* 前提条件や入力値の一部が不足している
* テストケース名から意図は読めるが、期待結果の詳細が不足している
* 既存分類はあるが、内容がやや薄い

### 例

```text
操作: データを登録する
期待結果: 一覧に正しく表示される
```

意図:

```text
登録したデータが一覧に表示されることを確認する。
```

推定度:

```text
中
```

理由:

```text
一覧表示の確認意図は読み取れるが、「正しく」の具体的な項目・値は不明。
```

---

## 4.3 推定度: 低

以下の場合、推定度を `低` とする。

* 操作/入力が不足している
* 期待結果がない
* 期待結果が曖昧すぎる
* エラー原因が不明
* 主意図が複数考えられる
* 前提条件が不明で意図が判断できない
* 操作と期待結果が対応していない
* 観点候補を根拠づけられない

### 例

```text
操作: 登録する
期待結果: エラーになる
```

意図:

```text
登録操作時にエラーが発生することを確認している可能性がある。
```

推定度:

```text
低
```

理由:

```text
どの条件で、何のエラーが発生するか不明。
```

---

## 5. レビュー要否の判定

以下の場合は、レビュー要否を `要` とする。

* 推定度が低い
* 推定度が中で、業務判断を含む
* 曖昧語が含まれる
* 期待結果が不足している
* 操作/入力が不足している
* 前提条件が不足している
* 主意図が複数考えられる
* 観点候補の粒度に迷う
* 抽象化レベルに迷う
* カテゴリ分類に迷う
* 元資料に矛盾がある
* 既存分類と内容が一致しない可能性がある
* 業務用語の意味が不明

レビュー理由は必ず記録する。

---

## 6. issue 記録ルール

## 6.1 issue として記録する条件

以下に該当する場合は issue として記録する。

* 入力資料の不足
* 入力資料の曖昧さ
* 入力資料の矛盾
* 意図抽出の不確実性
* 観点候補の根拠不足
* 抽象化の判断保留
* カテゴリ分類の判断保留
* トレーサビリティの断絶
* ユーザーレビューが必要な事項

---

## 6.2 issue の書き方

issue には以下を含める。

| 項目       | 内容                |
| -------- | ----------------- |
| Issue ID | issue の識別子        |
| 種別       | issue の分類         |
| 対象TC/観点  | 対象となるTC IDまたは観点ID |
| 内容       | 何が問題または不明か        |
| 影響       | 抽出結果にどう影響するか      |
| 推奨対応     | 何を確認・修正すべきか       |

---

## 6.3 issue の種別例

```text
TC IDなし
対象機能不明
前提条件不足
操作/入力不足
期待結果不足
期待結果曖昧
複数意図混在
操作結果不整合
既存分類不一致
業務用語不明
推定度低
観点根拠不足
抽象化判断保留
カテゴリ分類保留
トレーサビリティ未対応
根拠TCなし
```

---

## 7. 代表的な悪い補完例

## 7.1 エラー原因の補完

悪い例:

```text
期待結果が「エラーになる」なので、必須チェックである。
```

問題:

```text
エラー原因が入力資料に書かれていない。
```

修正例:

```text
エラー発生を確認しているが、エラー原因は入力資料から判断できない。
```

---

## 7.2 権限条件の補完

悪い例:

```text
削除できないため、一般ユーザー権限の確認である。
```

問題:

```text
削除不可の理由は、権限以外にも状態制御や業務ルールの可能性がある。
```

修正例:

```text
削除できないことを確認しているが、権限条件によるものかは不明。
```

---

## 7.3 境界値の補完

悪い例:

```text
999999999 を入力しているため、最大値境界チェックである。
```

問題:

```text
最大値である根拠が入力資料にない。
```

修正例:

```text
大きな数値を入力しているが、最大値または境界値として選定されたかは不明。
```

---

## 7.4 業務ルールの補完

悪い例:

```text
締め済みデータなので、会計上の確定データ保護を確認している。
```

問題:

```text
確定データ保護という目的が入力資料に明示されていない。
```

修正例:

```text
締め済みデータに対する操作制御を確認している。
会計上の確定データ保護が背景にある可能性はあるが、入力資料からは確定できない。
```

## 8. 曖昧さを残す表現例

曖昧な内容は、以下のように表現する。

```text
〜である可能性がある。
ただし、入力資料上では明示されていない。
〜と推定されるが、根拠は限定的である。
〜かどうかは判断できない。
〜の具体的内容は不明。
レビュー時に確認が必要。
```

避ける表現:

```text
〜である。
〜のためである。
当然〜である。
通常〜なので〜である。
一般的に〜だから〜である。
```

## 9. 最終成果物での扱い

Step 04 では、曖昧な内容を以下のように扱う。

| 状態     | 扱い                           |
| ------ | ---------------------------- |
| 推定度 高  | 最終成果物に含めてよい                  |
| 推定度 中  | 最終成果物に含めてよいが、必要に応じてレビュー対象にする |
| 推定度 低  | 確定扱いせず、issue または未確定として扱う     |
| 根拠TCなし | 最終成果物から除外する                  |
| 対応不明   | 未対応として記録する                   |
| 矛盾あり   | issue として記録する                |

---

## 10. 禁止事項

曖昧な情報を扱う際、以下を行ってはならない。

* 入力にない前提条件を補う
* 入力にない期待結果を補う
* エラー原因を断定する
* 権限制御と状態制御を根拠なく断定する
* 業務ルールの背景目的を断定する
* 具体値を境界値と断定する
* 「正しく」「正常に」の内容を勝手に具体化する
* 操作と期待結果の矛盾を勝手に修正する
* 推定度を付けずに断定する
* 推定を重ねる
* issue を記録せずに曖昧さを隠す
